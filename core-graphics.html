Core Graphics & Core Image
The CPU drawing layer beneath Core Animation: contexts, paths, gradients — plus CIFilter / CIKernel image processing.

    Core Graphics & Core Image — Overview

    Below Core Animation's compositing of layers sits the layer that actually draws pixels: Core Graphics (a.k.a. Quartz 2D), a C-based 2D drawing engine for paths, gradients, text, and images on the CPU. Alongside it, Core Image is a GPU-accelerated image processing pipeline of filters. Together they cover everything between "arrange rectangles" (Core Animation) and "write a shader" (Metal).

    Where each fits

    Metal / shaders   → per-pixel, on the GPU, maximum control      (your Shaders course)
Core Image        → chain image FILTERS (blur, color, distort)  on the GPU
Core Graphics     → DRAW vector content (paths, text, gradients) on the CPU
Core Animation    → COMPOSITE & animate layers                  (your CA course)

    Reach down into Core Graphics when a layer-and-image approach can't express the shape; reach into Core Image when you need to process an existing image rather than draw one.

    1. 01-drawing-model — the graphics context, the painter's model, and UIGraphicsImageRenderer.
    2. 02-paths-and-curves — CGPath / UIBezierPath, lines, arcs, and Bézier curves.
    3. 03-gradients-shadows-blends — gradients, shadows, and blend modes.
    4. 04-images-and-bitmaps — bitmap contexts, drawing & resizing images, scale and color spaces.
    5. 05-core-image-filters — CIFilter, the filter graph, and writing a CIKernel.
    6. 06-performance — when to draw, cache, or rasterize, and how to stay off the main thread.

    The mental model

    Core Graphics is a painter: you give a context a sequence of drawing commands and it lays paint down in order, each stroke covering what's beneath. Core Image is a darkroom: you describe a graph of filters and it processes an image through them on the GPU.

    How it connects

    The shapes you draw here back CAShapeLayer (CAShapeLayer). The Bézier math is the same as Parametric Curves. Core Image's per-pixel kernels are conceptually the fragment shaders from your Shaders roadmap — just expressed through a different API.

    The Drawing Model

    Core Graphics draws into a graphics context (CGContext) — an opaque destination for drawing commands. The context can be the screen (inside draw(_:)), an off-screen bitmap, a PDF, or a print page. The same drawing code targets any of them.

    The painter's model

    Core Graphics is stateful and sequential. You set state (fill color, line width, transform), then issue a paint command, and pixels are laid down immediately, on top of whatever is already there:

    override func draw(_ rect: CGRect) {
    guard let ctx = UIGraphicsGetCurrentContext() else { return }
    ctx.setFillColor(UIColor.systemBlue.cgColor)
    ctx.fill(CGRect(x: 20, y: 20, width: 100, height: 60))
    ctx.setStrokeColor(UIColor.label.cgColor)
    ctx.setLineWidth(3)
    ctx.stroke(CGRect(x: 40, y: 40, width: 100, height: 60))  // drawn ON TOP
}

    Order is everything: later commands paint over earlier ones.

    Graphics state is a stack

    The context holds a graphics state (colors, line attributes, the current transform, clipping). Save it, change it, draw, and restore — like push/pop:

    ctx.saveGState()
ctx.translateBy(x: 50, y: 50)
ctx.rotate(by: .pi / 6)
// draw rotated content here
ctx.restoreGState()   // transform/colors return to what they were

    This is the same affine-transform machinery as rotation matrices — CGAffineTransform is a 2D matrix.

    The modern way: UIGraphicsImageRenderer

    You rarely call UIGraphicsBeginImageContext anymore. UIGraphicsImageRenderer manages the context, scale, and color space correctly and returns a UIImage:

    let renderer = UIGraphicsImageRenderer(size: CGSize(width: 200, height: 200))
let image = renderer.image { ctx in
    let cg = ctx.cgContext
    cg.setFillColor(UIColor.systemPink.cgColor)
    cg.fillEllipse(in: CGRect(x: 0, y: 0, width: 200, height: 200))
}

    It picks the right scale for the screen automatically, so your drawing is crisp on Retina without manual @2x/@3x math.

    draw(_:) vs rendering an image

    Override draw(_:) when content changes with the view's bounds and you want UIKit to manage invalidation (setNeedsDisplay()). Render to a UIImage once when the content is static and you'd rather hand a finished bitmap to an image view or a CALayer.contents — that avoids re-drawing on every layout. The trade-off is covered in 06-performance.

    Flipped coordinates

    UIKit's draw(_:) gives you a context with the origin at the top-left, y-down. Raw CGContext (and Core Image) use bottom-left, y-up. When you mix them, you flip with ctx.translateBy(x:0,y:height); ctx.scaleBy(x:1,y:-1). This catches everyone once.

    Paths & Curves

    A path is a sequence of lines and curves that you can stroke (outline), fill (interior), or use as a clip. CGPath is the low-level type; UIBezierPath is the friendly UIKit wrapper. Paths back CAShapeLayer, animated strokes, custom buttons, charts, and any non-rectangular shape.

    Building a path

    let path = UIBezierPath()
path.move(to: CGPoint(x: 20, y: 100))          // pen down
path.addLine(to: CGPoint(x: 120, y: 20))       // straight segment
path.addQuadCurve(to: CGPoint(x: 220, y: 100), // one control point
                  controlPoint: CGPoint(x: 170, y: 0))
path.addCurve(to: CGPoint(x: 320, y: 100),     // cubic: two control points
              controlPoint1: CGPoint(x: 250, y: 180),
              controlPoint2: CGPoint(x: 290, y: 180))
path.close()                                    // line back to start
    • move(to:) starts a new subpath without drawing.
    • addLine, addArc, addQuadCurve (1 control point), addCurve (cubic, 2 control points).
    • close() connects back to the subpath's start.

    The cubic Bézier is the workhorse

    A cubic Bézier blends four points — two endpoints and two "handles" that pull the curve toward them — exactly the formula from Parametric Curves:

    B(t) = (1−t)³P0 + 3(1−t)²t·P1 + 3(1−t)t²·P2 + t³·P3

    This same cubic is what UIBezierPath, CAShapeLayer paths, font glyphs, and CSS/CAMediaTimingFunction timing curves are all made of. Learn to feel how the handles steer the curve and you can draw or animate anything.

    Fill rules

    When a path self-intersects, the fill rule decides what's "inside":

    • Non-zero winding (default) — counts the signed crossings; overlaps stay filled.
    • Even-odd (usesEvenOddFillRule = true) — toggles inside/outside on each crossing, creating holes. This is how you punch a hole (a "spotlight" overlay): one big rect + an inner shape, even-odd filled.

    Stroking

    lineWidth, lineCapStyle (.butt/.round/.square), lineJoinStyle (.miter/.round/.bevel), and setLineDash control the outline. Dashes + animating CAShapeLayer.strokeEnd from 0→1 produce the classic "draw-on" signature/loading animation.

    Path → CAShapeLayer (animatable)

    let shape = CAShapeLayer()
shape.path = path.cgPath
shape.strokeColor = UIColor.label.cgColor
shape.fillColor = UIColor.clear.cgColor
shape.lineWidth = 3
// animate the stroke drawing itself
let draw = CABasicAnimation(keyPath: "strokeEnd")
draw.fromValue = 0; draw.toValue = 1; draw.duration = 1.2
shape.add(draw, forKey: "draw")

    This bridges Core Graphics shapes into Core Animation; see CAShapeLayer.

    Gradients, Shadows & Blend Modes

    These three give flat shapes depth and richness. They're cheap to describe and, used well, are the difference between a "drawn rectangle" and a polished surface.

    Gradients

    Two kinds: axial (linear, along a line) and radial (out from a center). In Core Graphics you draw a CGGradient into a (usually clipped) region:

    let colors = [UIColor.systemBlue.cgColor, UIColor.systemPurple.cgColor] as CFArray
let space = CGColorSpaceCreateDeviceRGB()
let gradient = CGGradient(colorsSpace: space, colors: colors, locations: [0, 1])!

ctx.saveGState()
ctx.addPath(path.cgPath); ctx.clip()        // confine gradient to a shape
ctx.drawLinearGradient(gradient,
    start: CGPoint(x: 0, y: 0),
    end: CGPoint(x: 0, y: bounds.height),
    options: [])
ctx.restoreGState()

    For a view-wide gradient that animates, prefer CAGradientLayer (CAGradientLayer) — it's GPU-composited. Use Core Graphics gradients when the gradient must be clipped to a complex drawn path or baked into an image.

    Shadows

    A shadow is offset + blur + color. Set it on the context before drawing the shape it should apply to:

    ctx.setShadow(offset: CGSize(width: 0, height: 4),
              blur: 12,
              color: UIColor.black.withAlphaComponent(0.25).cgColor)
ctx.fill(cardRect)

    For view layers, layer.shadowPath is the performance-critical version — giving the shadow an explicit path avoids the expensive off-screen pass UIKit otherwise does to derive the shape. Always set shadowPath for moving/animating shadows.

    Blend modes

    A blend mode decides how new paint combines with what's underneath — the same modes you know from Photoshop and from shader compositing:

    ctx.setBlendMode(.multiply)   // darkens: great for tints over photos
ctx.setBlendMode(.screen)     // lightens: glows, light leaks
ctx.setBlendMode(.overlay)    // contrast
ctx.setBlendMode(.destinationOut)  // ERASE: punch a hole / mask

    .destinationOut is especially useful: draw an opaque shape with it and you cut that shape out of everything beneath — a one-line way to build masks and "scratch-off" effects.

    Putting it together: a card surface

    A convincing card is often: a rounded-rect path → clip → axial gradient fill → a subtle inner highlight stroke at the top → a soft shadowPath on the layer. Each piece is a few lines, and together they read as a real, lit surface — the same "unseen details compound" idea from interaction design.

    Images & Bitmaps

    A UIImage is a high-level, immutable picture; a CGImage is the raw bitmap underneath; a bitmap context is a canvas you can draw into and read back. Knowing the difference is the key to resizing correctly, avoiding memory blow-ups, and getting color right.

    Scale: points vs pixels

    A UIImage carries a scale (1, 2, or 3). Its size is in points; the underlying pixel dimensions are size × scale. The number one image bug is ignoring scale — drawing at image.size on a 3× screen and getting a blurry or mis-sized result. UIGraphicsImageRenderer handles scale for you:

    func resized(_ image: UIImage, to size: CGSize) -> UIImage {
    let r = UIGraphicsImageRenderer(size: size)   // picks screen scale
    return r.image { _ in image.draw(in: CGRect(origin: .zero, size: size)) }
}

    Downsampling large images (the memory trap)

    Loading a 12-megapixel photo into a 120pt thumbnail wastes ~48 MB of RAM per image — a top cause of memory-pressure crashes in feeds. Downsample at decode time with ImageIO so the full bitmap is never created:

    import ImageIO
func downsample(_ url: URL, to pointSize: CGSize, scale: CGFloat) -> UIImage? {
    let src = CGImageSourceCreateWithURL(url as CFURL, nil)!
    let maxDim = max(pointSize.width, pointSize.height) * scale
    let opts: [CFString: Any] = [
        kCGImageSourceCreateThumbnailFromImageAlways: true,
        kCGImageSourceShouldCacheImmediately: true,
        kCGImageSourceCreateThumbnailWithTransform: true,
        kCGImageSourceThumbnailMaxPixelSize: maxDim
    ]
    guard let cg = CGImageSourceCreateThumbnailAtIndex(src, 0, opts as CFDictionary) else { return nil }
    return UIImage(cgImage: cg, scale: scale, orientation: .up)
}

    This is the single highest-impact image optimization in list-heavy apps.

    Reading and writing pixels

    For per-pixel work (histograms, picking a color, generating textures), draw into a known-format bitmap context and read the buffer:

    var pixels = [UInt8](repeating: 0, count: w * h * 4)
let ctx = CGContext(data: &pixels, width: w, height: h,
    bitsPerComponent: 8, bytesPerRow: w * 4,
    space: CGColorSpaceCreateDeviceRGB(),
    bitmapInfo: CGImageAlphaInfo.premultipliedLast.rawValue)!
ctx.draw(cgImage, in: CGRect(x: 0, y: 0, width: w, height: h))
// pixels now holds RGBA bytes you can inspect or mutate

    For heavy per-pixel math, this is the slow CPU path — push it to Core Image or Metal instead (Shaders).

    Color spaces, briefly

    sRGB is the safe default for UI. Display P3 covers a wider gamut on modern displays (richer reds/greens). Mismatched color spaces cause subtle color shifts; when generating images for display, use CGColorSpace(name: CGColorSpace.sRGB) unless you specifically want P3.

    Core Image Filters

    Core Image (CI) is a non-destructive, GPU-accelerated image-processing pipeline. You describe a graph of filters; CI fuses them into an optimized program and runs it on the GPU only when you ask for output. It's the bridge between Core Graphics drawing and full Metal shaders.

    The three core types

    CIImage    – a recipe for an image (lazy; not yet rendered)
CIFilter   – a node that transforms a CIImage into another CIImage
CIContext  – the (expensive) object that actually renders a CIImage to pixels

    A CIImage is a promise, not a bitmap. Chaining filters just builds a graph. Nothing executes until a CIContext renders it — so a 10-filter chain costs one optimized GPU pass, not ten.

    Chaining filters

    import CoreImage.CIFilterBuiltins

let input = CIImage(image: uiImage)!
let blur = CIFilter.gaussianBlur()
blur.inputImage = input
blur.radius = 8

let vibrance = CIFilter.vibrance()
vibrance.inputImage = blur.outputImage
vibrance.amount = 0.6

// render once
let context = CIContext()                      // create ONCE, reuse — it's heavy
let out = vibrance.outputImage!
let cg = context.createCGImage(out, from: input.extent)!
let result = UIImage(cgImage: cg)

    The big performance rule: create one CIContext and reuse it. Allocating a context per render is the classic Core Image performance mistake.

    Useful built-ins

    Blur (gaussianBlur, boxBlur), color (colorControls, vibrance, exposureAdjust), stylize (bloom, sharpenLuminance), distortion (bumpDistortion, glassDistortion), generators (qrCodeGenerator, linearGradient), and colorMatrix for arbitrary channel mixing. There are ~200; CIFilter.filterNames(inCategories:) lists them.

    Real-time rendering

    For live effects (camera, scrubbing a slider), don't go through UIImage. Render the CIImage straight into a Metal-backed view with a Metal CIContext so the pixels never round-trip through the CPU:

    let context = CIContext(mtlDevice: MTLCreateSystemDefaultDevice()!)
// render into a MTKView's drawable each frame

    This is how camera filters stay at 60fps.

    Writing your own: CIKernel

    When no built-in fits, write a kernel in the Metal Shading Language — a per-pixel function, exactly like the fragment shaders in your Fragment Shaders notes:

    // MyKernels.ci.metal — a simple tint kernel
#include <CoreImage/CoreImage.h>
extern "C" float4 tint(coreimage::sample_t s, float3 color) {
    return float4(s.rgb * color, s.a);
}

    You load it as a CIColorKernel/CIKernel and apply it like any filter. This is the same per-pixel thinking as your shader course, wrapped in CI's filter graph and memory management.

    Drawing Performance

    Custom drawing is powerful and easy to make slow. The whole game is: draw as little as possible, as rarely as possible, off the main thread when you can, and let the GPU composite the rest.

    Know what's expensive

    • draw(_:) runs on the main thread and produces a bitmap the size of the view × screen scale. Big views redrawn often = jank.
    • Off-screen passes are the silent killer: cornerRadius + masksToBounds, shadowPath-less shadows, shouldRasterize, and masks force the GPU to render into a separate buffer and composite it back. Instruments' Core Animation tool flags these in yellow/red.

    The decision ladder

    1. Can a CALayer property do it?      → use it (corner radius, border, bg color) — GPU, free
2. Can an image/asset do it?          → ship a bitmap, set layer.contents
3. Does it change with bounds/state?  → override draw(_:), keep it minimal
4. Is it per-pixel & heavy?           → Core Image or Metal (off the CPU entirely)

    Most "I need custom drawing" turns out to be step 1 or 2.

    Cheap shadows

    // SLOW: UIKit derives the shape via an off-screen pass every frame
layer.shadowOpacity = 0.2
// FAST: tell it the shape explicitly
layer.shadowPath = UIBezierPath(roundedRect: bounds, cornerRadius: 12).cgPath

    Always set shadowPath for any shadow that moves or animates — it can be the difference between 60 and 20 fps in a scrolling list.

    Rasterize sparingly

    layer.shouldRasterize = true caches a layer as a bitmap — great for a complex static layer reused while scrolling, terrible for anything that changes (it re-rasterizes every frame). Always pair with layer.rasterizationScale = UIScreen.main.scale or it's blurry on Retina.

    Draw off the main thread

    UIGraphicsImageRenderer works on a background queue. Render expensive content there, then hop back to assign it:

    DispatchQueue.global(qos: .userInitiated).async {
    let img = renderer.image { ctx in /* expensive drawing */ }
    DispatchQueue.main.async { imageView.image = img }
}

    Combined with the ImageIO downsampling from 04-images-and-bitmaps, this keeps scrolling smooth even with heavy thumbnails.

    Measure, don't guess

    Use Instruments' Core Animation template (color off-screen-rendered layers, color blended layers) and Time Profiler to find the real cost. This pairs directly with Instruments profiling and the WWDC performance sessions (Performance Survival Guide).
