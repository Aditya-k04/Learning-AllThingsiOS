Concurrency in Practice
The async/await model, structured concurrency and task groups, actors & data isolation, @MainActor discipline.

    Concurrency in Practice — Overview

    Swift Concurrency (async/await, tasks, actors) replaced the old world of DispatchQueue callbacks and locks with a model the compiler can check for data races. Your Swift concurrency basics notes cover the syntax; this course is about using it correctly in a real app — the patterns, the pitfalls, and the discipline that keeps the main thread smooth.

    The three pillars

    async/await           – write asynchronous code that reads top-to-bottom
structured concurrency – child tasks tied to a scope: cancellation & errors propagate
actors                 – reference types that serialize access to their state (no data races)

    Layered on top: @MainActor for UI-thread safety, and Sendable — the compiler's check that a value is safe to cross concurrency boundaries.

    The mental shift

    Old model: you pick a queue and hope you don't touch shared state from two threads. New model: you mark what must run where (@MainActor, actors), and the compiler refuses to let you create a data race.

    This is the same move as elsewhere in modern Apple dev — declare intent, let the system enforce correctness.

    What you'll learn

    1. 01-async-await-model — what async/await actually do (suspension, not blocking) and Tasks.
    2. 02-structured-concurrency — async let, task groups, and cancellation propagation.

    How it connects

    This underpins networking (every request is async), SwiftData (context actor isolation), and SwiftUI itself (.task, @MainActor views). Strict concurrency checking — on by default in the Swift 6 language mode — is the through-line of modern Apple code.

    The Async/Await Model

    async/await lets you write asynchronous code that reads top-to-bottom, like synchronous code, while never blocking a thread. The key is understanding what await actually does — it's a suspension point, not a pause.

    Suspension, not blocking

    When you await, the function may suspend: it gives up its thread, lets other work run, and resumes later — possibly on a different thread — when the awaited result is ready. No thread sits idle waiting.

    func loadProfile() async throws -> Profile {
    let user = try await api.fetchUser()      // suspends here; thread freed
    let posts = try await api.fetchPosts()    // resumes, suspends again
    return Profile(user: user, posts: posts)
}

    Contrast with old blocking code that ties up a thread, or callback code that fragments logic across closures. Here the control flow is linear and errors use ordinary throw/try.

    Tasks: the unit of async work

    async functions can only be called from an async context. A Task is the bridge from sync (like a button handler) into async:

    Button("Load") {
    Task {                              // starts a new top-level async context
        do { profile = try await loadProfile() }
        catch { self.error = error }
    }
}

    A Task runs concurrently with the code that created it. It captures the current actor context (a Task {} inside a @MainActor view runs its body on the main actor unless you say otherwise).

    await ≠ parallel

    Two sequential awaits run one after another. To run work concurrently you must say so with async let or a task group (02-structured-concurrency). A common beginner mistake is expecting two awaits to overlap — they don't.

    // sequential: total = a + b
let a = await slowA(); let b = await slowB()
// concurrent: total = max(a, b)
async let x = slowA(); async let y = slowB()
let (a2, b2) = await (x, y)

    Where async runs

    By default, async functions don't promise a thread — they run on the cooperative thread pool, sized to your CPU. UI-touching code must hop to the main actor (04-mainactor-discipline). CPU-heavy work should not run on the main actor; let it run on the pool and only return results to the main actor.

    Structured Concurrency

    Structured concurrency means child tasks live within a scope: they can't outlive it, errors propagate out of it, and cancelling the parent cancels the children. This structure is what makes concurrent Swift code safe and leak-free, compared to fire-and-forget threads.

    async let: a few concurrent children

    async let starts work immediately and binds a promise; you await it where you need the value. All async lets in a scope run concurrently:

    func loadDashboard() async throws -> Dashboard {
    async let user   = api.fetchUser()
    async let feed   = api.fetchFeed()
    async let badges = api.fetchBadges()
    return try await Dashboard(user: user, feed: feed, badges: badges)  // awaits all
}

    Total time is the slowest request, not the sum. If the scope exits (or throws) before awaiting, the unawaited children are automatically cancelled — no leaks.

    Task groups: a dynamic number of children

    When the count is dynamic (one task per item), use a task group:

    func thumbnails(for ids: [Asset.ID]) async throws -> [Asset.ID: UIImage] {
    try await withThrowingTaskGroup(of: (Asset.ID, UIImage).self) { group in
        for id in ids { group.addTask { (id, try await renderThumbnail(id)) } }
        var result: [Asset.ID: UIImage] = [:]
        for try await (id, image) in group { result[id] = image }   // collect as they finish
        return result
    }
}

    The group won't return until every child finishes (or one throws and the rest are cancelled). To bound concurrency (don't launch 1,000 at once), add tasks up to a limit, then add a new one each time one completes.

    Cancellation is cooperative

    Cancellation doesn't forcibly stop a task — it sets a flag. Well-behaved code checks it and stops early:

    try Task.checkCancellation()                 // throws CancellationError if cancelled
if Task.isCancelled { return partialResult } // or bail gracefully

    URLSession and Task.sleep already check cancellation, so awaiting them propagates cancellation automatically. This is why cancelling a search Task on each keystroke (networking) just works — and why a long CPU loop should call checkCancellation() periodically or it ignores cancellation.

    Why "structured" matters

    Because children are bound to a scope, you can reason locally: when loadDashboard() returns, nothing it started is still running. No detached threads quietly continuing, no callback that fires after the screen is gone. (For genuinely independent background work, Task.detached opts out of the structure — use it sparingly.)

    Actors & Data Isolation

    An actor is a reference type that protects its mutable state by allowing only one task inside at a time. It's Swift's structural answer to data races: instead of remembering to take a lock, the type system guarantees serialized access.

    The problem actors solve

    Two tasks mutating the same class from different threads is a data race — undefined behavior, crashes, corruption. The old fix was manual locks (NSLock, serial queues), easy to get wrong. An actor makes the compiler enforce it:

    actor ImageCache {
    private var store: [URL: UIImage] = [:]
    func image(for url: URL) -> UIImage? { store[url] }   // synchronous INSIDE the actor
    func insert(_ image: UIImage, for url: URL) { store[url] = image }
}

    store can only be touched through the actor, and only one task runs the actor's code at a time. No locks, no races.

    Crossing in is async

    From outside, every access is await — because you might have to wait your turn:

    let cache = ImageCache()
await cache.insert(img, for: url)        // await: may suspend until the actor is free
let cached = await cache.image(for: url)

    Inside the actor, its own methods call each other synchronously (you're already isolated).

    Reentrancy: the subtle part

    Actors are reentrant — when an actor method awaits, it suspends and the actor can run other tasks in the meantime. So state you read before an await may have changed after it:

    func loadOnce(_ url: URL) async -> UIImage {
    if let c = store[url] { return c }
    let img = await download(url)        // SUSPENDS — another task may also be downloading url
    store[url] = img                     // last writer wins; you may have downloaded twice
    return img
}

    Guard against duplicate work by storing the in-flight Task (a [URL: Task<UIImage,Error>]), so concurrent callers await the same download — the coalescing pattern from token refresh. "Check your assumptions after every await" is the actor rule.

    Sendable: what's safe to cross boundaries

    A value sent into or out of an actor/task must be Sendable — provably safe to share. Value types of Sendable parts are automatically Sendable; final classes with immutable state can be marked @unchecked Sendable (you promise it's safe). The compiler uses Sendable to prevent you from smuggling a mutable reference across a concurrency boundary — the core of Swift 6's data-race safety.

    The MainActor is an actor

    @MainActor is a global actor representing the main thread. It's the same mechanism: code annotated @MainActor is serialized onto the main thread. That's the next lesson.

    @MainActor Discipline

    All UIKit and SwiftUI updates must happen on the main thread. @MainActor is how Swift Concurrency expresses and enforces that. The discipline is: keep UI on the main actor, push heavy work off it, and let results flow back — without ever freezing the main thread.

    What @MainActor means

    @MainActor annotates a type, method, or property as running on the main thread. The compiler then checks it — calling main-actor code from a background context requires an await hop, and accidentally touching UI off-main becomes a compile error rather than a runtime glitch.

    @MainActor
final class FeedViewModel: ObservableObject {
    @Published var items: [Item] = []         // safe: only mutated on main

    func refresh() async {
        let fetched = await loader.load()      // load() runs off-main
        items = fetched                        // back on main automatically
    }
}

    SwiftUI Views are implicitly @MainActor, as are most of UIKit. So a Task {} started inside a view runs its body on the main actor by default.

    The trap: heavy work on the main actor

    Because Task {} in a view inherits the main actor, CPU-heavy work inside it blocks the UI:

    // BAD: parsing/encoding runs on the main actor → dropped frames
Task { let report = generateHugeReport() ; self.report = report }

    Move the heavy part off-main, then return to main only to publish:

    Task {
    let report = await Task.detached(priority: .userInitiated) {
        generateHugeReport()                    // runs off the main actor
    }.value
    self.report = report                        // back on main to update UI
}

    Or put the heavy function on a background actor / make it nonisolated. The principle: the main actor is for touching UI, not for computing.

    nonisolated for pure work

    If a method on a @MainActor type does no UI work, mark it nonisolated so it can run anywhere without forcing a hop:

    @MainActor final class Formatter {
    nonisolated func format(_ x: Double) -> String { String(format: "%.1f", x) }
}

    Awaiting back to main

    You rarely call DispatchQueue.main.async anymore. Returning from an await into main-actor code resumes on the main actor automatically. When you do need an explicit hop from a non-isolated context:

    await MainActor.run { label.text = "done" }

    The payoff

    Done right, @MainActor means "UI off the main thread" bugs become compile errors, and "the app froze" bugs become obvious (you'll see heavy code on the main actor). It's the enforced version of the rule you already followed by habit.

    Pitfalls & Patterns

    A field guide to the concurrency mistakes everyone makes once, and the patterns that prevent them.

    Pitfall: expecting sequential awaits to be parallel

    let a = await load(1)      // these run ONE AFTER ANOTHER
let b = await load(2)

    Fix: async let or a task group when the work is independent (02-structured-concurrency).

    Pitfall: forgetting cancellation

    A long task keeps running after the user leaves the screen, wasting battery and possibly updating dead UI. Fix: store the Task and cancel it, or use SwiftUI's .task modifier, which cancels automatically when the view disappears:

    .task { await viewModel.subscribeToUpdates() }   // cancelled on disappear, for free

    And honor cancellation in your loops with try Task.checkCancellation().

    Pitfall: actor reentrancy assumptions

    State read before an await can change after it (03-actors-and-isolation). Re-check invariants after every suspension; coalesce duplicate in-flight work by caching the Task.

    Pitfall: blocking the main actor

    Heavy parsing/encoding/image work inside a view's Task freezes scrolling. Push it off-main (04-mainactor-discipline).

    Pitfall: Task.detached everywhere

    Detached tasks escape structured concurrency — they don't inherit cancellation, priority, or actor context, and they're easy to leak. Use them only for genuinely independent, fire-and-forget work; prefer structured async let/task groups otherwise.

    Pattern: bridging callbacks with continuations

    Wrapping an old delegate/closure API into async uses a continuation — call resume exactly once (zero = a leaked forever-suspended task; twice = a crash):

    func currentLocation() async throws -> CLLocation {
    try await withCheckedThrowingContinuation { cont in
        locationManager.requestLocation { result in
            switch result {
            case .success(let loc): cont.resume(returning: loc)
            case .failure(let err): cont.resume(throwing: err)
            }
        }
    }
}

    withCheckedContinuation traps if you misuse resume — keep it during development.

    Pattern: AsyncSequence for streams

    For a stream of values over time (locations, socket messages, notifications) use AsyncStream / AsyncSequence and consume with for await:

    for await event in eventStream {
    handle(event)        // loop ends when the stream finishes or the task is cancelled
}

    This replaces delegate callbacks and Combine pipelines for many cases, and integrates with cancellation cleanly.

    Pattern: make models Sendable by being value types

    The easiest way to satisfy strict concurrency checking is to pass structs of Sendable parts across boundaries. Value semantics (value semantics) and concurrency safety reinforce each other — another reason to prefer structs.

    🆕 WWDC26 update

    Swift 6.3 adds a third continuation type, Continuation<Success, Failure> — use it instead of CheckedContinuation/UnsafeContinuation when bridging callbacks. Double-resume becomes a compile-time error and a missing resume traps, with no fast-path overhead:

    let value = try await withContinuation { c in
    legacyAPI { result in c.resume(returning: result) }
}

    Also useful: withTaskCancellationShield { … } to protect critical cleanup/SOS work from cancellation. See What's New in Swift (WWDC26).
