    Demystify SwiftUI (WWDC21 - 10022)

    Session URL: https://developer.apple.com/videos/play/wwdc2021/10022/ YouTube mirror: https://www.youtube.com/watch?v=XwdVz0Ef1vU Speakers: Matt Ricketson, Luca Bernardi, Raj Ramamurthy (Apple SwiftUI team)

    Summary

    This session pulls back the curtain on SwiftUI's internal decision-making. It answers the question: "What does SwiftUI see when it looks at your code?" The answer is three things -- the three pillars that drive every aspect of the framework:

    1. Identity -- How SwiftUI recognizes elements as the same or distinct across multiple updates.
    2. Lifetime -- How SwiftUI tracks the existence of views and data over time.
    3. Dependencies -- How SwiftUI knows what to update and when to update it.

    Understanding these pillars gives you a mental model for predicting SwiftUI's behavior, diagnosing performance issues, and avoiding subtle bugs.

    Cross-ref: 01-swiftui-philosophy covers the declarative paradigm; this session explains the machinery underneath it.

    Pillar 1: Identity

    Identity is the most foundational concept. It is how SwiftUI distinguishes whether two views are the same view in different states or two entirely different views. This distinction determines whether SwiftUI animates a transition (different views) or smoothly updates properties in place (same view, new values).

    SwiftUI uses two forms of identity:

    1.1 Explicit Identity

    Explicit identity uses a custom or data-driven identifier that you provide. There are two mechanisms:

    Pointer Identity (UIKit)

    In UIKit, each UIView instance is a reference type with a unique pointer in memory. Two variables referencing the same view share identity because they point to the same object. SwiftUI views are value types (structs) that are created and destroyed frequently, so pointer identity is not applicable.

    Data-Driven Identity

    SwiftUI provides several ways to explicitly assign identity:

    • ForEach with Identifiable -- The id property of each element identifies the corresponding view. SwiftUI uses this to correctly animate insertions, deletions, and reordering.
    ForEach(rescueAnimals) { animal in
    AnimalCard(animal: animal)
}
// Each AnimalCard is identified by animal.id
    • ForEach with a key path -- When your type is not Identifiable, you supply a key path to a Hashable property.
    ForEach(rescueAnimals, id: \.tagID) { animal in
    AnimalCard(animal: animal)
}
    • The .id() modifier -- Assigns an explicit identity to any view. Changing the value passed to .id() tells SwiftUI to destroy the old view and create a new one (resetting all state). This is powerful but should be used intentionally.
    ScrollView {
    content
}
.id(selectedCategory) // Resets scroll position when category changes

    Cross-ref: 07-lists-and-foreach covers ForEach and Identifiable in depth.

    1.2 Structural Identity

    Structural identity is the default and most common form. SwiftUI identifies views by their type and position in the view hierarchy. You do not write any identifiers -- SwiftUI infers them from the structure of your code.

    How It Works

    The body property of a View is implicitly wrapped in a @ViewBuilder. The compiler translates your declarative code into a static, generic type hierarchy:

    var body: some View {
    VStack {
        Text("Header")
        if isLoggedIn {
            ProfileView()
        } else {
            LoginView()
        }
        Text("Footer")
    }
}

    Behind the scenes, the if/else becomes a _ConditionalContent<ProfileView, LoginView>. The "true" branch (ProfileView) and the "false" branch (LoginView) get different identities. When isLoggedIn flips, SwiftUI sees a transition between two distinct views and will animate them in/out accordingly (e.g., fade transition).

    The _ConditionalContent Pattern

    Code construct SwiftUI sees Identity implication
    if condition { A } else { B } _ConditionalContent<A, B> A and B are different views -- transitioning between them
    if condition { A } _ConditionalContent<A, EmptyView> A and EmptyView are different views
    switch with N cases Nested _ConditionalContent Each case is a distinct view

    Best Practice: Prefer Inert Modifiers Over Branches

    When both branches refer to the same view with different configuration, avoid breaking identity:

    // BAD: breaks structural identity -- two different views
if isHighlighted {
    Text(name).foregroundColor(.yellow)
} else {
    Text(name).foregroundColor(.primary)
}

// GOOD: single view, consistent identity -- smooth property animation
Text(name)
    .foregroundColor(isHighlighted ? .yellow : .primary)

    In the bad example, toggling isHighlighted causes SwiftUI to destroy one Text and create another (losing state, triggering a transition). In the good example, the same Text persists and its foreground color animates smoothly.

    Cross-ref: 02-views-and-modifiers covers how each modifier wraps the view in a new type, and 12-animation-basics explains how same-identity property changes produce smooth animations.

    1.3 AnyView: The Pitfalls

    AnyView is a type-erasing wrapper. It hides the concrete view type from SwiftUI's generic signature.

    // Anti-pattern: using AnyView to return different types
func makeView() -> AnyView {
    if showProfile {
        return AnyView(ProfileView())
    } else {
        return AnyView(LoginView())
    }
}

    Why AnyView is problematic:

    1. Destroys structural identity -- SwiftUI cannot see through AnyView to the conditional structure. It sees the same type (AnyView) in every branch, so it has no information to diff efficiently.
    2. Hampers performance -- Without type information, SwiftUI cannot statically analyze the view hierarchy. It falls back to slower diffing strategies.
    3. Hides useful compile-time information -- Generics are SwiftUI's primary optimization tool. AnyView discards them.

    The fix: use @ViewBuilder

    Instead of returning AnyView, annotate your function or computed property with @ViewBuilder:

    @ViewBuilder
func makeView() -> some View {
    if showProfile {
        ProfileView()
    } else {
        LoginView()
    }
}

    Now SwiftUI sees a _ConditionalContent<ProfileView, LoginView> and can properly track identity for each branch.

    Alternative: use generics

    When a view accepts a child view as a parameter, use a generic constraint rather than AnyView:

    // BAD
struct Container: View {
    let content: AnyView
    // ...
}

// GOOD
struct Container<Content: View>: View {
    let content: Content
    // ...
}

    Cross-ref: 23-view-composition covers generic view containers and @ViewBuilder parameters.

    Pillar 2: Lifetime

    Identity drives lifetime. A view's lifetime begins when it first appears in the hierarchy (with a given identity) and ends when it is removed or its identity changes.

    2.1 Views Are Values, Not Objects

    SwiftUI views are value types (structs). They are created and destroyed cheaply and frequently. The view struct itself is not the view's long-lived representation -- it is a transient description. SwiftUI maintains its own internal representation behind the scenes, and the struct's body is called only when SwiftUI needs an updated description.

    A view's value (the struct instance) is not its identity. The identity persists across many different values.

    2.2 State Persistence Is Tied to Identity

    When SwiftUI encounters @State or @StateObject in a view, it allocates persistent storage that is associated with that view's identity, not with the struct instance:

    Identity created  -->  @State/@StateObject initialized with initial values
      |
      v
View body called (many times)  -->  State persisted across calls
      |
      v
Identity destroyed  -->  State deallocated

    This is why:

    • If you use .id(someValue) and someValue changes, all @State and @StateObject in that view reset to their initial values -- because SwiftUI sees a new identity.
    • If a view moves to a different branch of a conditional (e.g., from if to else), its state is not carried over -- the two branches are different identities.
    // @State resets when selectedTab changes because .id() creates a new identity
DetailView(item: selectedItem)
    .id(selectedTab)

    Cross-ref: 05-state-and-binding explains @State ownership, and 15-observable-and-environment covers @StateObject and @ObservedObject lifecycle.

    2.3 Lifetime Best Practices

    1. Preserve identity whenever possible -- Prefer modifying a view's properties over switching between entirely different views. This gives smoother transitions and preserves state.
    2. Be intentional with .id() -- Only use it when you specifically want to reset state and create a fresh view instance.
    3. Understand conditional identity -- An if/else creates two separate lifetimes. If you want a single view to persist across both states, restructure the code to avoid the branch.

    Pillar 3: Dependencies

    Dependencies are how SwiftUI knows which views to update when data changes. SwiftUI builds a dependency graph connecting views to the data they read.

    3.1 The Dependency Graph

    Each view is a node in a graph. The edges connect views to the data sources they depend on (@State, @Binding, @ObservedObject, @EnvironmentObject, @Environment, etc.). When a dependency changes:

    1. SwiftUI identifies all views that depend on the changed data.
    2. It invalidates those views (marks them as needing a new body).
    3. It calls each invalidated view's body property.
    4. It compares the new body with the old one and applies the minimal set of changes to the rendered output.

    This is not a tree but a graph -- multiple views can depend on the same piece of state, and a single view can depend on multiple pieces of state.

    @State count ──────> CounterLabel.body
            \
             └─────> CounterControls.body

@Environment(\.colorScheme) ──> CounterLabel.body
                               └─> HeaderView.body

    3.2 Efficient Updates

    Because views are value types, SwiftUI can efficiently compare old and new body values to determine what actually changed. Only the true diff is applied to the rendering tree.

    Key insight: invalidation does not always mean a visible change. SwiftUI may re-evaluate a body and discover nothing has actually changed, in which case no rendering update occurs.

    3.3 Minimizing Unnecessary Updates

    The dependency graph is why view decomposition matters for performance:

    // POOR: entire body re-evaluates when count OR name changes
struct MonolithView: View {
    @State private var count = 0
    @State private var name = ""

    var body: some View {
        VStack {
            Text("Count: \(count)")
            Text("Name: \(name)")
            Button("Increment") { count += 1 }
        }
    }
}

    If you extract Text("Count: \(count)") and the button into a separate CounterView with its own @State, then changing name does not invalidate CounterView at all.

    Cross-ref: 16-data-flow dives deep into data flow patterns, source of truth, and how to structure state ownership. 25-real-world-architecture covers view decomposition strategies.

    Code Examples Discussed in the Session

    The Rescue-Animal App

    The session uses a pet-adoption app as a running example. Key scenarios:

    1. Adopting animals with ForEach -- Each animal has a unique id, and ForEach uses it to track which rows to insert, remove, or reorder.

    2. Conditional profile views -- An if/else branch shows either a logged-in profile or a sign-in prompt. The session demonstrates how SwiftUI translates this to _ConditionalContent and how the two branches get distinct identities.

    3. AnyView refactoring -- A helper function that returns AnyView in each branch is refactored to use @ViewBuilder, restoring structural identity and improving performance.

    4. .id() for scroll reset -- A ScrollView is given an .id() tied to the selected category so that scrolling resets when the user switches categories.

    Key Diagram: The Three Pillars

                     ┌──────────┐
                 │ Identity │
                 └────┬─────┘
                      │ determines
                      v
                 ┌──────────┐
                 │ Lifetime │
                 └────┬─────┘
                      │ scopes
                      v
              ┌──────────────┐
              │ Dependencies │

Identity   =  "Is this the same view or a different one?"
Lifetime   =  "How long does this view (and its state) exist?"
Dependencies = "What data does this view read, and when should it update?"

    The pillars are hierarchical: identity defines lifetime, and lifetime scopes the dependency tracking.

    Key Takeaways

    1. SwiftUI has two kinds of identity: explicit and structural. Explicit identity uses data-driven identifiers (.id(), ForEach + Identifiable). Structural identity uses view type and position in the hierarchy.

    2. Avoid AnyView. It erases type information that SwiftUI needs for efficient diffing and structural identity. Use @ViewBuilder or generics instead.

    3. Conditional branches create separate identities. An if/else in your view body means each branch is a different view. Use inert modifiers (ternary expressions) when you want a single view to persist across states.

    4. @State and @StateObject are tied to identity, not to the struct. State persists as long as the identity exists. When identity changes (e.g., the value passed to .id() changes), state resets.

    5. The dependency graph drives performance. SwiftUI only re-evaluates views whose dependencies have changed. Smaller, focused views mean more granular invalidation and better performance.

    6. Preserve identity for smooth transitions. Keeping a view's identity stable lets SwiftUI animate property changes smoothly instead of performing insert/remove transitions.

    7. some View (opaque return type) preserves type information. The compiler knows the concrete type; it is just hidden from the caller. This is fundamentally different from AnyView, which truly erases the type at runtime.

    Design with SwiftUI — WWDC23 Session 10115

    "When you use SwiftUI, you're not just creating a prototype. You're making an app."

    Overview

    This session is presented by Will and Philip, designers on the Apple Maps team at Apple. Rather than a traditional engineering talk, this is a design-focused session where Apple's own designers share how SwiftUI changed their design workflow. They demonstrate how working directly in SwiftUI influenced the design of the Maps app in watchOS 10, providing real-world examples of how SwiftUI can serve as both a design tool and a production-ready framework.

    The core message: SwiftUI is not just for developers — it is a uniquely powerful design tool that lets designers iterate faster, test on real devices, and ship production-quality work.

    1. SwiftUI as a Design Tool (1:40)

    Why SwiftUI for designers?

    Apple argues that SwiftUI occupies a unique position in the design-tool landscape:

    "SwiftUI strikes the balance between these two types of tools. It lowers the floor for simple tasks and raises the ceiling for creating new and novel things."

    Traditional design tools fall into two categories: - Low-floor tools — easy to start but limited in capability (drag-and-drop editors) - High-ceiling tools — powerful but steep learning curves (code-first frameworks)

    SwiftUI bridges both: many people with no programming background find they can learn SwiftUI and be productive in a short period of time.

    Key capabilities for designers

    • Declarative syntax — you describe what you want, not how to build it. This builds on the intuition designers already have about how interfaces are assembled.
    • Visual canvas in Xcode — see and interact with designs in real-time without compiling to a device.
    • Modifiers — change appearance just like any design tool: drop shadows, borders, fonts, layout properties (aspect ratios, padding).
    • Built-in components — no need to recreate standard UI elements like color pickers, navigation transitions, or system controls.
    • Cross-platform — one codebase targets iPhone, iPad, Apple Watch, Mac, and Apple Vision Pro.
    • Framework integration — native access to MapKit, WeatherKit, RealityKit, and more.
    • Full device access — haptic feedback, audio, cameras, gestures, sensors.
    • App Store ready — designs built in SwiftUI are not throwaway prototypes; they can ship directly.

    Layout fundamentals for designers

    SwiftUI provides a composable layout system that designers should understand:

    • Stacks: VStack (vertical), HStack (horizontal), ZStack (depth/overlay)
    • Modifiers for layout: .padding(), .aspectRatio(), .frame(), spacing parameters
    • Styling modifiers: .shadow(), .border(), .font(), .foregroundStyle()
    • Composition: views are composed from smaller, reusable pieces — this mirrors how designers think about component hierarchies

    See: 04-layout-fundamentals, 02-views-and-modifiers, 23-view-composition

    2. Getting the Details Right (5:56)

    The prototype reveals what static mocks cannot

    One of the session's strongest arguments: SwiftUI surfaces important design details that remain invisible in static design tools. When you build and run on the actual device, you confront questions that Figma or Sketch never ask:

    • How should an image look while it is loading?
    • How does a button appear when pressed?
    • What happens at the boundary between two scroll regions?
    • What does a transition feel like when interrupted mid-animation?

    Maps watchOS 10 — Zoom Speed Discovery

    The team built the Maps watchOS 10 experience entirely in SwiftUI, running on the watch itself. One of the first things they discovered:

    • The Digital Crown zoom speed was too fast — the map jumped from street level to continent level almost instantly.
    • This was invisible in static mocks. Only on-device prototyping revealed it.
    • They iterated directly in SwiftUI, adjusting sensitivity values until the zoom felt natural.

    Walking Radius Feature

    Another detail that emerged from prototyping:

    • A feature showing the walkable area around the current location with estimated walking time.
    • Required a ticker animation that revealed detail as the user scrolled with the Digital Crown.
    • They built a mini design tool in SwiftUI to visualize the walking radius overlaid on the dynamic map surface, letting them iterate on sizing and animation timing.

    Dynamic interfaces demand dynamic tools

    Modern products are rarely static — they respond to context, data, orientation, accessibility settings, and more. SwiftUI forces you to confront these realities early.

    See: 12-animation-basics, 06-controls, 18-scrollview

    3. Designing for Interaction (9:11)

    First-class interaction support

    SwiftUI provides native, performant support for:

    • Gestures — tap, long press, drag, rotation, magnification
    • Transitions — built-in and custom view transitions
    • Animations — fully interruptible, physics-based spring animations
    • Haptics — integrated haptic feedback via WKInterfaceDevice / UIFeedbackGenerator

    Compass Interaction Example

    The team demonstrated a compass feature for the Maps watchOS app:

    • Activated by tapping the location button.
    • Displays the cardinal direction based on the watch's physical orientation, reading from internal sensors.
    • The entire prototype was completed in a few hours.
    • This would have been impossible to prototype in a static design tool — sensor data and real-time orientation are inherently dynamic.

    Split-Screen Search Interaction

    A sophisticated scroll interaction for the Maps search experience:

    • Map display on top, search results list below.
    • The Digital Crown drives a custom scroll interaction:
    • Slow movements — results scroll smoothly.
    • Fast movements — the list snaps with a spring animation for a "snappy" feel.
    • Spring animations were tuned for different scroll velocities.
    • Haptic feedback fires when crossing the threshold between slow-scroll and snap modes.
    • This level of interaction tuning is only possible when designing in a real runtime environment.

    Why this matters for design

    "SwiftUI provides a great out-of-the-box experience for many interactions, and when you want something custom, it's a highly capable tool allowing you to fine-tune things until they feel just right."

    See: 14-gestures, 12-animation-basics, 13-animation-advanced

    4. Testing Your Ideas (11:50)

    Design in realistic scenarios

    One thing that is easy to overlook: testing designs in real-world conditions. SwiftUI makes this natural because you are running real code on real devices:

    • Sunlight and glare — does the UI remain readable outdoors? Only on-device testing reveals contrast issues under bright light.
    • Real data — plug in actual data sets instead of idealized placeholder content.
    • Edge cases — what happens when a route has no elevation change? When a list has 200 items? When the user has accessibility settings enabled?

    Elevation Graph Example

    The Maps team built an elevation graph showing the altitude profile of walking/biking routes:

    • Initially designed with San Francisco test data (lots of hills) — looked great.
    • When tested with New York City data (relatively flat), the graph was nearly a flat line.
    • Required adjusting the Y-axis scale to be dynamic — adapting to the elevation range of each specific route.
    • This issue only surfaced because they tested with diverse, real-world data sets.

    Parameterized Design Tools

    A powerful technique: build one-off, parameterized design tools in SwiftUI to explore a design space:

    • Create sliders or controls that let you scrub through design variations.
    • Example: adjusting corner radii, color values, animation curves, spacing, or timing — all in real-time on the device.
    • These are throwaway explorations, not production code, but SwiftUI makes them trivially fast to build.

    See: 05-state-and-binding, 11-colors-and-materials, 22-geometry-and-preferences

    5. Presenting Your Work (15:23)

    On-device demos over slide decks

    "When people can hold a demo of your work in their hands, designs explain themselves. This helps build consensus and can save everyone the hassle of more meetings."

    Key insights on presenting design work:

    • Build demos into a single SwiftUI app — combine multiple design concepts, screens, or variations into one demo app for design reviews.
    • On-device demos de-risk projects — stakeholders can evaluate rather than being told about a design. This prevents misunderstandings and builds confidence.
    • More effective than slide decks — static screenshots lose nuance around animation timing, haptic feedback, scroll physics, and real-world data presentation.
    • Demos prevent confusion — a 30-second hands-on experience communicates more than a 30-minute presentation.

    The unique advantage

    "When you use SwiftUI, you're not just creating a prototype. You're making an app."

    Unlike prototyping tools that produce throwaway artifacts, SwiftUI prototypes can evolve directly into production code. The design-to-development handoff gap effectively disappears.

    See: 23-view-composition, 25-real-world-architecture

    Design Principles Discussed

    1. Design with real materials

    Do not design in a vacuum. Use real data, real devices, real lighting conditions, and real interaction paradigms. SwiftUI enables this because your design IS the running software.

    2. Surface hidden decisions early

    Static mocks hide critical decisions (loading states, pressed states, error states, animation timing). Building in SwiftUI forces you to confront these immediately.

    3. Feel over appearance

    How something feels — scroll momentum, haptic feedback, animation spring constants — is as important as how it looks. This can only be evaluated in a real runtime.

    4. Iterate in context

    Test on the actual device, in the actual environment. A watchOS design must be evaluated on a wrist, under sunlight, during a walk — not on a 27-inch monitor.

    5. Let demos speak for themselves

    An interactive demo is worth a thousand slides. Build your design reviews around hands-on experiences.

    6. Common things easy, complex things possible

    SwiftUI's philosophy aligns with good design tooling: trivial tasks should require trivial effort, but the ceiling should be high enough for novel, custom work.

    Visual Hierarchy and Spacing Approach

    The session emphasizes that SwiftUI's built-in spacing and layout system encourages designers to think about:

    • Semantic spacing — using .padding() and stack spacing rather than pixel-perfect absolute positioning
    • Adaptive layout — designs that respond to dynamic type sizes, screen sizes, and accessibility settings
    • System typography — leveraging Apple's type styles (.title, .headline, .body, .caption) for automatic visual hierarchy
    • Material and depth — using ZStack, .background(), and material effects to create layered visual hierarchies
    • Composition over complexity — building complex interfaces from small, focused, reusable views

    See: 04-layout-fundamentals, 17-grids-and-adaptive-layout, 11-colors-and-materials, 03-text-and-images

    Key Takeaways

    1. SwiftUI is a legitimate design tool — Apple's own design team uses it as their primary design medium, not just a developer framework.
    2. Prototyping in SwiftUI reveals hidden complexity — loading states, interaction edge cases, animation interruptions, and data diversity issues surface immediately.
    3. No programming background required — Apple explicitly states that non-programmers can become productive in SwiftUI quickly thanks to its declarative model.
    4. Design and production converge — SwiftUI prototypes are not disposable; they can ship to the App Store.
    5. On-device testing is non-negotiable — sunlight readability, haptic feel, sensor interaction, and scroll physics can only be evaluated on real hardware.
    6. Build demo apps for design reviews — replace slide decks with interactive, on-device demonstrations.
    7. Create custom design tools — build throwaway parameterized explorations in SwiftUI to rapidly iterate through a design space.
    8. The framework encourages good design — by requiring you to define behavior for all states (loading, pressed, error, empty), SwiftUI makes designs more complete.

    .scrollTransition

    Exposes the content undergoing a scroll transition and a phase describing its current position relative to the visible scroll area.

.scrollTargetBehavior(.paging)

    3. View Transitions

    Custom transitions conform to the Transition protocol and implement a body(content:phase:) method. The TransitionPhase enum tells you where the view is in its lifecycle.

    TransitionPhase

    Case Meaning
    .willAppear View is about to appear (insertion)
    .isIdentity View is fully visible (at rest)
    .didDisappear View has been removed (removal)

    Custom Twirl Transition

    struct Twirl: Transition {
    func body(content: Content, phase: TransitionPhase) -> some View {
        content
            .scaleEffect(phase.isIdentity ? 1 : 0.5)
            .opacity(phase.isIdentity ? 1 : 0)
            .blur(radius: phase.isIdentity ? 0 : 10)
            .rotationEffect(
                .degrees(
                    phase == .willAppear ? 360 :
                    phase == .didDisappear ? -360 : .zero
                )
            )
            .brightness(phase == .willAppear ? 1 : 0)
    }
}

// Usage
Text("Hello")
    .transition(Twirl())

    5. Metal Shaders in SwiftUI

    Shaders are small programs that run on the GPU for every pixel. They enable effects that would be impractical on the CPU. SwiftUI itself uses Metal internally for effects like Mesh Gradients. Since iOS 17, developers can write their own.

    Metal shaders are written in Metal Shading Language (MSL), not Swift. They live in .metal files in your Xcode project.

    The <span class="xref-missing" title="note not in this site">stitchable</span> Attribute

    Every Metal function used with SwiftUI must be marked <span class="xref-missing" title="note not in this site">stitchable</span>. This attribute does two things:

    1. Makes the function visible to the Metal Framework API so SwiftUI can discover it
    2. Enables runtime stitching -- Metal can compose multiple shader functions together efficiently

    When you build your project, Xcode automatically scans all .metal files, finds every function marked <span class="xref-missing" title="note not in this site">stitchable</span>, and generates corresponding methods on ShaderLibrary via dynamic member lookup.

    ShaderLibrary

    You instantiate shaders through ShaderLibrary using the function name:

    let shader = ShaderLibrary.Ripple(
    .float2(origin),
    .float(elapsedTime),
    .float(amplitude),
    .float(frequency),
    .float(decay),
    .float(speed)
)

    Parameter wrapper types: - .float() -- single Float value - .float2() -- CGPoint or two-component vector - .color() -- SwiftUI Color - .image() -- Image - .data() -- raw Data buffer

    Three Effect Types + Custom Fills

    SwiftUI provides four ways to apply shaders:

    1. .colorEffect() -- Per-pixel color transformation

    Cannot sample neighboring pixels. Receives position and current color.

    <span class="xref-missing" title="note not in this site">stitchable</span>
half4 myColorShader(float2 position, half4 color, float time) {
    // Transform and return modified color
    return color;
}

    2. .distortionEffect() -- Per-pixel position displacement

    Returns a new float2 position indicating where to sample the original pixel.

    maxSampleOffset

    For distortionEffect and layerEffect, you must declare maxSampleOffset -- the maximum distance (in points) a pixel can be displaced. This tells SwiftUI how much extra area to render beyond the view bounds.

    Ripple Effect -- Full Example

    This is the flagship example from the session, combining a Metal shader with SwiftUI animation.

    Metal Shader (Ripple.metal)

    #include <SwiftUI/SwiftUI.h>
using namespace metal;

<span class="xref-missing" title="note not in this site">stitchable</span>
half4 Ripple(
    float2 position,
    SwiftUI::Layer layer,
    float2 origin,
    float time,
    float amplitude,
    float frequency,
    float decay,
    float speed
) {
    // Distance from tap origin to current pixel
    float distance = length(position - origin);

    // Delay based on how far this pixel is from origin
    float delay = distance / speed;

    // Adjust time by delay, clamp to 0
    time -= delay;
    time = max(0.0, time);

    // Sine wave with exponential decay
    float rippleAmount = amplitude * sin(frequency * time) * exp(-decay * time);

    // Displace along the direction from origin
    float2 n = normalize(position - origin);
    float2 newPosition = position + rippleAmount * n;

    // Sample the layer at the displaced position
    half4 color = layer.sample(newPosition);

    // Add brightness boost proportional to ripple intensity
    color.rgb += 0.3 * (rippleAmount / amplitude) * color.a;

    return color;
}

    SwiftUI Integration (RippleModifier.swift)

    struct RippleModifier: ViewModifier {
    var origin: CGPoint
    var elapsedTime: TimeInterval
    var duration: TimeInterval
    var amplitude: Double = 12
    var frequency: Double = 15
    var decay: Double = 8

Learn how SwiftUI refreshes the rendering of a view, determines what to animate, interpolates values over time, and propagates context for the current transaction.

    Presenter: Kyle, SwiftUI team at Apple

    Summary

    This session is the foundational talk in a three-part WWDC23 animation series. It breaks down SwiftUI's animation system into four core concepts:

    1. How SwiftUI refreshes views -- the anatomy of a view update and the attribute graph
    2. Animatable -- what data gets animated
    3. Animation -- how data interpolates over time
    4. Transaction -- how animation context propagates through the view hierarchy

    The session uses a "pet avatar voting" demo app throughout. In iOS 17 and aligned releases, the default animation was changed from ease-in-out to a smooth spring, which is a significant shift in defaults.

    1. Anatomy of an Update

    SwiftUI maintains an attribute graph where each node (called an attribute) maps to a fine-grained piece of the UI. Views declare dependencies on state, and when state changes, dependent attributes are invalidated.

    The update cycle

    1. An event arrives (e.g., a tap gesture fires)
    2. An update transaction is opened
    3. If any state dependencies changed, the view is invalidated
    4. SwiftUI calls body to produce a new value
    5. Downstream attributes are refreshed
    6. The rendering is updated

    When you wrap a state change in withAnimation, the animation is set on the transaction before the body is called. This means the animation flows through the attribute graph along with the new values.

    Model value vs. presentation value

    Animatable attributes conceptually hold two values:

    • Model value -- the target/final value (set immediately)
    • Presentation value -- the currently displayed value (interpolated over time by the animation)

    When no animation is active, model and presentation are identical.

    Core Animation parallel: This is analogous to how CALayer has a model layer and a presentation layer. See 08-catransaction and 07-implicit-animations.

    2. Animatable

    The Animatable Protocol

    SwiftUI determines what to animate through the Animatable protocol. Any view or modifier conforming to Animatable must define a read-write vector of the data it wants to animate:

    protocol Animatable {
    associatedtype AnimatableData: VectorArithmetic
    var animatableData: AnimatableData { get set }
}

    The key requirement is that AnimatableData conforms to VectorArithmetic.

    VectorArithmetic

    VectorArithmetic matches the mathematical definition of a vector -- it supports vector addition and scalar multiplication. This is what enables SwiftUI to interpolate between any two values generically:

    Type Dimensions
    CGFloat / Double 1D
    CGPoint / CGSize 2D (via AnimatablePair)
    CGRect 4D (via nested AnimatablePair)

    By dealing in vectors, SwiftUI animates all of these types with a single generic implementation.

    Animatable Attributes

    Built-in modifiers like scaleEffect, opacity, offset, rotationEffect, and frame are animatable attributes. When the value of an animatable attribute changes, it checks if an animation is set on the current transaction. If so, it makes a copy of the animation and uses it to drive interpolation from the old value to the new value.

    Example: scaleEffect

    When an avatar scales from 1.0 to 1.5, the animatable data is a 4D vector (width scale, height scale, anchor x, anchor y). SwiftUI interpolates across all dimensions simultaneously.

    Custom Animatable Modifiers

    You can make any ViewModifier or Shape animatable by conforming to Animatable and defining animatableData:

    struct Shake: ViewModifier, Animatable {
    var shakeCount: CGFloat = 0

    var animatableData: CGFloat {
        get { shakeCount }
        set { shakeCount = newValue }
    }

    func body(content: Content) -> some View {
        content
            .offset(x: sin(shakeCount * .pi * 2) * 5)
    }
}

    Cross-reference: See 12-animation-basics for .animation() modifier usage and 13-animation-advanced for PhaseAnimator and KeyframeAnimator.

    3. Animation

    An Animation defines how the animated data changes over time. SwiftUI categorizes animations into three buckets:

    Three Categories of Animation

    1. Timing Curve Animations

    Defined by a Bezier curve that maps elapsed time to progress. The animation has a fixed duration and the curve controls pacing.

    .animation(.easeInOut(duration: 0.5), value: selected)
.animation(.easeIn, value: selected)
.animation(.easeOut, value: selected)
.animation(.linear(duration: 0.3), value: selected)

    A timing curve animation always runs for exactly its duration. The Bezier control points determine acceleration and deceleration.

    Core Animation parallel: These map directly to CAMediaTimingFunction. See 14-animation-timing.

    2. Spring Animations

    Model a mass on a spring. Unlike timing curves, springs have no fixed duration -- they converge asymptotically. In iOS 17, the default animation became a smooth spring (non-bouncy).

    withAnimation(.spring) {
    selected.toggle()
}

// Presets
withAnimation(.smooth) { ... }    // no bounce
withAnimation(.snappy) { ... }    // quick settle
withAnimation(.bouncy) { ... }    // visible overshoot

// Custom parameters
withAnimation(.spring(duration: 0.5, bounce: 0.3)) { ... }

    Spring animations have a critical advantage: they preserve velocity when retargeted. If a spring animation is interrupted by a new animation to a different value, the spring smoothly redirects, maintaining the current velocity. This avoids the jarring "hitch" that timing curves can produce when interrupted.

    Cross-reference: See 11-caspringanimation for the Core Animation equivalent (CASpringAnimation). See related session 10158-animate-with-springs for the deep dive into spring physics.

    3. Higher-Order Animations

    These modify a base animation:

    // Repeat forever
.animation(.easeInOut.repeatForever(), value: pulsing)

// Repeat a fixed number of times
.animation(.spring.repeatCount(3), value: bouncing)

// Delay before starting
.animation(.bouncy.delay(0.2), value: selected)

// Adjust speed
.animation(.smooth.speed(2.0), value: selected)

    Higher-order animations wrap an underlying animation and transform its behavior.

    The CustomAnimation Protocol (New in iOS 17)

    The CustomAnimation protocol gives you access to the same low-level generic entry points used to implement all built-in SwiftUI animations. It has three requirements:

    protocol CustomAnimation: Hashable {
    func animate<V: VectorArithmetic>(
        value: V,
        time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> V?

    func shouldMerge<V: VectorArithmetic>(
        previous: Animation,
        value: V,
        time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> Bool

    func velocity<V: VectorArithmetic>(
        value: V,
        time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> V?
}

    animate(value:time:context:)

    The core method. Called every frame. Returns the current interpolated value, or nil when the animation is finished.

    • value -- the target delta vector to animate toward
    • time -- elapsed time since the animation began
    • context -- mutable context for storing animation state across frames

    shouldMerge(previous:value:time:context:)

    Called when a new animation starts while a previous one is still running on the same property. Returns a Bool:

    • false (default) -- both animations run simultaneously and their results combine additively. This is what timing curve animations do.
    • true -- the new animation absorbs the previous one, incorporating its state. This is what spring animations do, allowing them to preserve velocity and retarget smoothly.

    velocity(value:time:context:)

    Returns the current velocity of the animation. Used when a running animation is merged into a new one so the new animation can preserve continuity.

    Example: Custom Linear Animation

    struct MyLinearAnimation: CustomAnimation {
    var duration: TimeInterval

    func animate<V: VectorArithmetic>(
        value: V,
        time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> V? {
        if time < duration {
            // Scale the target value by progress (time / duration)
            value.scaled(by: time / duration)
        } else {
            nil  // animation complete
        }
    }

    func velocity<V: VectorArithmetic>(
        value: V,
        time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> V? {
        // Constant velocity = value / duration
        value.scaled(by: 1.0 / duration)
    }
}

// Usage
extension Animation {
    static func myLinear(duration: TimeInterval) -> Animation {
        Animation(MyLinearAnimation(duration: duration))
    }
}

    How Additive Animation Works

    SwiftUI animations work with delta vectors (not absolute values). When multiple animations run on the same property, each one contributes a delta, and the system sums them to produce the final presentation value. This is why shouldMerge returning false works cleanly -- the two animations simply add their deltas together.

    4. Transaction

    A Transaction is the vehicle that carries animation context through a view update. When withAnimation is called, it sets an animation on the root transaction. As SwiftUI walks the attribute graph to refresh stale attributes, the transaction propagates with it.

    How Transactions Flow

    1. withAnimation(.bouncy) { selected.toggle() } sets the animation on the transaction
    2. SwiftUI calls body to produce new attribute values
    3. The transaction dictionary propagates across the attribute graph
    4. When it reaches an animatable attribute, the attribute checks for an animation
    5. If found, it copies the animation and uses it to drive interpolation
    6. Once all stale attributes are refreshed, the transaction is discarded

    Core Animation parallel: This is analogous to CATransaction in Core Animation, which batches property changes and controls their animation parameters. See 08-catransaction.

    withAnimation

    The most common way to set an animation on a transaction. Wraps a state mutation:

    // Default (smooth spring in iOS 17+)
withAnimation {
    selected = true
}

// Explicit animation
withAnimation(.bouncy) {
    selected = true
}

    The .animation() View Modifier

    Scopes an animation to a specific value change and a specific subtree of the view hierarchy. The animation is injected into the transaction as it passes through that modifier.

    Avatar(pet)
    .scaleEffect(selected ? 1.5 : 1.0)
    .animation(.bouncy, value: selected)    // bouncy for scale
    .shadow(radius: selected ? 12 : 4)
    .animation(.smooth, value: selected)    // smooth for shadow

    In this example, the transaction picks up the .bouncy spring when it reaches scaleEffect, and then the .smooth spring when it reaches shadow. This lets you apply different animations to different properties within the same view.

    Scoping Animations

    The .animation() modifier creates a scope boundary. Animations set above the modifier apply to attributes above it; the modifier replaces the transaction's animation for attributes below it.

    VStack {
    Avatar(pet)
        .scaleEffect(selected ? 1.5 : 1.0)
        .animation(.bouncy, value: selected)
        // The bouncy animation is scoped to scaleEffect
        // Everything below gets the transaction's original animation
}

    Transaction Modifier

    For more advanced control, use .transaction to directly mutate the transaction:

    Avatar(pet)
    .transaction { transaction in
        transaction.animation = .snappy
    }

    TransactionKey (New in iOS 17)

    You can carry custom data in a transaction using TransactionKey, similar to EnvironmentKey:

    enum AnimationVariant {
    case primary, secondary, none
}

struct AnimationVariantKey: TransactionKey {
    static let defaultValue: AnimationVariant = .primary
}

extension Transaction {
    var animationVariant: AnimationVariant {
        get { self[AnimationVariantKey.self] }
        set { self[AnimationVariantKey.self] = newValue }
    }
}

    This enables views deep in the hierarchy to make decisions based on custom transaction context, without using environment values.

    +---> Animatable attribute (e.g., scaleEffect)
    |       |
    |       v
    |     Has animation in transaction? YES
    |       |
    |       v
    |     Copy animation, create interpolation:
    |       old value --[Animation]--> new value
    |       model value = new value (immediately)
    |       presentation value = interpolated (per frame)
    |
    +---> Non-animatable attribute
    |       |
    |       v
    |     Update immediately, no interpolation
    |
    v
Transaction discarded after all attributes refreshed
    |
    v
Render loop calls into animatable attributes each frame
until all animations converge (return nil)

    Key architectural insight: the animation does not live on the view -- it lives in the transaction, which is ephemeral. The animatable attribute captures a copy of the animation at the moment the transaction reaches it. This is why .animation() modifiers can inject different animations at different points in the view tree.

    Demo App: Pet Avatar Voting

    The session uses a pet avatar voting app to demonstrate concepts:

    1. Basic tap to select -- tapping an avatar sets selected = true, which triggers a scale effect
    2. Adding animation -- wrapping in withAnimation makes the scale change animate with the default spring
    3. Customizing with .bouncy -- withAnimation(.bouncy) adds visible overshoot
    4. Scoped animations -- using .animation(.bouncy, value:) on scaleEffect and .animation(.smooth, value:) on shadow so they animate differently

            .scaleEffect(selected ? 1.5 : 1.0)

            }
    }
}

    Key Takeaways

    1. SwiftUI animations are declarative -- you change state, and the framework handles interpolation. You never write frame-by-frame logic.

    2. Three pillars: Animatable, Animation, Transaction -- understanding these three primitives gives you complete control over SwiftUI animation.

    3. Smooth spring is the new default -- in iOS 17+, bare withAnimation { } uses a smooth spring instead of ease-in-out. Springs are preferred because they handle interruption gracefully.

    4. Springs preserve velocity on retarget -- when interrupted, spring animations smoothly redirect. Timing curves cannot do this, which can cause visual hitches.

    5. Animations propagate via transactions -- the animation is not attached to the view. It flows through the attribute graph inside a transaction, and animatable attributes capture copies of it.

    6. .animation() modifier creates scoping boundaries -- you can apply different animations to different properties by inserting .animation() modifiers between them.

    7. CustomAnimation protocol unlocks full control -- the same generic infrastructure backing built-in animations is available to you via CustomAnimation.

    8. Additive animation via delta vectors -- SwiftUI uses delta vectors so multiple concurrent animations on the same property combine naturally.

    9. VectorArithmetic is the key abstraction -- any type conforming to VectorArithmetic can be animated. This includes scalars, points, sizes, rects, and custom types.

    10. TransactionKey enables custom animation context -- carry arbitrary metadata through the transaction, similar to environment keys.

    Wind Your Way Through Advanced Animations in SwiftUI

    WWDC23 Session 10157 -- Presented by Tim (SwiftUI team)

    Summary

    This session explores the advanced animation APIs introduced in SwiftUI for iOS 17: PhaseAnimator for multi-step state-driven animations, KeyframeAnimator for timeline-based per-property control, and KeyframeTimeline for manual keyframe evaluation outside of view modifiers. It builds on the foundational concepts covered in session 10156 and demonstrates how to construct complex, choreographed motion sequences that go far beyond simple state-to-state transitions.

    The session emphasizes a core principle of SwiftUI animation: animations are interruptible, physics-based, and integrated throughout the framework. The new APIs extend this philosophy to multi-step and timeline-driven use cases.

    Key Concepts at a Glance

    API Use Case Granularity
    PhaseAnimator Multi-step sequences with discrete states All properties animate together per phase
    KeyframeAnimator Timeline-based choreography Each property animated independently via tracks
    KeyframeTimeline Manual keyframe evaluation (e.g., charts, scrubbing) Programmatic access to values at any time t
    CustomAnimation Build your own animation curves Low-level control over interpolation

    PhaseAnimator

    PhaseAnimator cycles through a sequence of discrete phases, animating all view modifications between each step automatically. You define the phases and, optionally, the animation curve to use when transitioning to each phase.

    How It Works

    1. Define a sequence of phases (a simple array or a CaseIterable enum).
    2. Provide a content closure that receives the current phase and applies modifiers.
    3. Optionally provide an animation closure that returns the animation to use when transitioning to that phase.
    4. SwiftUI advances through the phases in order, looping back to the start.

    Continuous (Ambient) Animation

    Without a trigger, the animator loops continuously while the view is visible. This is ideal for ambient effects like pulsing indicators.

    // Simple two-phase pulse: opacity toggles between 1.0 and 0.5
OverdueReminderView()
    .phaseAnimator([false, true]) { content, phase in
        content
            .foregroundStyle(phase ? .red : .primary)
            .opacity(phase ? 1.0 : 0.5)
    } animation: { _ in
        .easeInOut(duration: 1.0)
    }

    Triggered (Event-Driven) Animation

    Supply a trigger value. When it changes, the animator runs through all phases once and settles back at the first phase.

    @State private var reactionCount = 0

ReactionView()
    .phaseAnimator(
        Phase.allCases,
        trigger: reactionCount
    ) { content, phase in
        content
            .scaleEffect(phase.scale)
            .offset(y: phase.verticalOffset)
    } animation: { phase in
        switch phase {
        case .initial: .smooth
        case .move:    .easeInOut(duration: 0.3)
        case .scale:   .spring(duration: 0.3, bounce: 0.7)
        }
    }

Button("React") { reactionCount += 1 }

    Custom Phase Enum Pattern

    The recommended pattern is to define an enum conforming to CaseIterable with computed properties for each animatable value:

    enum Phase: CaseIterable {
    case initial, move, scale

    var verticalOffset: Double {
        switch self {
        case .initial:  0
        case .move:    -64
        case .scale:   -64
        }
    }

    var scale: Double {
        switch self {
        case .initial: 1.0
        case .move:    1.0
        case .scale:   1.8
        }
    }
}

    This pattern keeps animation values co-located with their phase, making complex multi-property sequences easy to reason about. See 13-animation-advanced for more PhaseAnimator examples and exercises.

    When to Use PhaseAnimator

    • Repeating ambient animations (pulsing, glowing, breathing effects).
    • Event-driven multi-step animations that need graceful interruption.
    • Sequences where all properties should animate together per step.
    • Quick prototyping -- fewer moving parts than keyframes.

    KeyframeAnimator

    KeyframeAnimator provides timeline-level control by letting you define keyframe tracks for individual properties, each with precise timing. Unlike phases (where all properties animate together between discrete states), keyframes allow each property to follow its own independent timeline.

    How It Works

    1. Define a struct holding all animatable properties with default values.
    2. Attach the .keyframeAnimator modifier with initialValue set to an instance of that struct.
    3. In the content closure, apply the current interpolated values to the view.
    4. In the keyframes closure, declare one KeyframeTrack per property.

    AnimationValues Struct

    struct AnimationValues {
    var scale = 1.0
    var verticalStretch = 1.0
    var verticalTranslation = 0.0
    var angle = Angle.zero
}

    Full Example: Emoji Reaction Animation

    The session demonstrates a rich reaction animation that coordinates scale, vertical translation, stretch, and rotation independently:

    @State private var trigger = 0

EmojiView()
    .keyframeAnimator(
        initialValue: AnimationValues(),
        trigger: trigger
    ) { content, value in
        content
            .rotationEffect(value.angle)
            .scaleEffect(value.scale)
            .scaleEffect(y: value.verticalStretch)
            .offset(y: value.verticalTranslation)
    } keyframes: { _ in
        KeyframeTrack(\.scale) {
            LinearKeyframe(1.0, duration: 0.36)
            SpringKeyframe(1.5, duration: 0.8, spring: .bouncy)
            SpringKeyframe(1.0, spring: .bouncy)
        }

        KeyframeTrack(\.verticalStretch) {
            CubicKeyframe(0.8, duration: 0.1)
            CubicKeyframe(1.2, duration: 0.15)
            CubicKeyframe(1.0, duration: 0.15)
            CubicKeyframe(0.9, duration: 0.12)
            CubicKeyframe(1.0, duration: 0.1)
        }

        KeyframeTrack(\.verticalTranslation) {
            LinearKeyframe(0.0, duration: 0.1)
            SpringKeyframe(20.0, duration: 0.15, spring: .bouncy)
            SpringKeyframe(-60.0, duration: 1.0, spring: .bouncy)
            SpringKeyframe(0.0, spring: .bouncy)
        }

        KeyframeTrack(\.angle) {
            CubicKeyframe(.zero, duration: 0.58)
            CubicKeyframe(.degrees(16), duration: 0.125)
            CubicKeyframe(.degrees(-16), duration: 0.125)
            CubicKeyframe(.degrees(16), duration: 0.125)
            CubicKeyframe(.zero, duration: 0.125)
        }
    }
    .onTapGesture { trigger += 1 }

    Keyframe Types

    Each keyframe type controls how interpolation happens between values:

    Type Interpolation Best For
    LinearKeyframe Constant-speed linear interpolation in vector space Uniform motion, delays (hold a value)
    SpringKeyframe Spring-physics interpolation Natural settling, bouncy transitions
    CubicKeyframe Cubic Bezier curve (Catmull-Rom spline equivalent) Smooth ease-in/ease-out curves
    MoveKeyframe Immediate jump, no interpolation Instant snaps, resetting to a value

    SwiftUI preserves velocity between keyframes for natural-feeling motion, even when mixing keyframe types within a single track.

    When to Use KeyframeAnimator

    • Complex choreographed animations where different properties need different timelines.
    • Pre-planned sequences with precise timing requirements.
    • Reaction effects, character animations, or onboarding sequences.
    • Situations where you want full control over the motion profile.

    Limitations

    • Not interruptible mid-animation like spring-based state animations. Keyframe animations are pre-planned sequences, not interactive.
    • Performance-sensitive: updates happen on every frame. Avoid expensive operations in the content closure.
    • Not suitable for state-driven interactive changes (use springs or PhaseAnimator instead).

    KeyframeTimeline

    KeyframeTimeline lets you evaluate keyframes programmatically outside of view modifiers. You initialize it with an initial value and keyframe tracks (same API as keyframeAnimator), then query it for values at any point in time.

    let timeline = KeyframeTimeline(initialValue: CGPoint.zero) {
    KeyframeTrack(\.x) {
        LinearKeyframe(100, duration: 0.5)
        SpringKeyframe(0, duration: 0.5, spring: .bouncy)
    }
    KeyframeTrack(\.y) {
        CubicKeyframe(-50, duration: 0.3)
        CubicKeyframe(0, duration: 0.7)
    }
}

// Query properties
let totalDuration: TimeInterval = timeline.duration
let pointAtHalf: CGPoint = timeline.value(time: 0.5)

    Use Cases

    • Swift Charts integration: drive chart data points along a keyframe-defined path.
    • Scroll-driven scrubbing: map a scroll offset to a time value and evaluate the timeline.
    • GeometryReader coordination: position views based on keyframe-evaluated coordinates.
    • Map camera animation: the session demonstrates mapCameraKeyframeAnimator for coordinated map camera movements (center, heading, distance).

    CustomAnimation Protocol

    The CustomAnimation protocol (introduced in iOS 17, covered in depth in session 10156) gives you access to the same low-level entry points that SwiftUI uses internally for all built-in animations. It lets you build entirely custom animation curves.

    Protocol Requirements

    Method Required? Purpose
    animate(value:time:context:) Yes Return the interpolated value at a given time, or nil when finished
    shouldMerge(previous:value:time:context:) No (default: false) Whether to merge with a previous animation of the same type
    velocity(value:time:context:) No Report current velocity for smooth handoffs

    How animate Works

    The method receives a delta vector (the difference between old and new values), the elapsed time since the animation began, and a mutable context for storing state. It returns: - A scaled version of the delta vector representing the current animated offset, or - nil to signal the animation is complete.

    Because it operates on VectorArithmetic, implementations are generic and work with animatable data of any dimension.

    Example: Custom Linear Animation

    struct MyLinearAnimation: CustomAnimation {
    var duration: TimeInterval

    func animate<V: VectorArithmetic>(
        value: V, time: TimeInterval,
        context: inout AnimationContext<V>
    ) -> V? {
        if time <= duration {
            value.scaled(by: time / duration)
        } else {
            nil // animation complete
        }
    }

    func velocity<V: VectorArithmetic>(
        value: V, time: TimeInterval,
        context: AnimationContext<V>
    ) -> V? {
        value.scaled(by: 1.0 / duration)
    }
}

// Register as an Animation static member
extension Animation {
    static func myLinear(duration: TimeInterval) -> Animation {
        Animation(MyLinearAnimation(duration: duration))
    }
}

// Usage
withAnimation(.myLinear(duration: 0.5)) {
    offset = 100
}

    shouldMerge Behavior

    • Timing-curve animations (linear, easeIn, etc.) return false -- concurrent animations are combined additively by the system.
    • Spring animations return true -- the new animation incorporates the state of the previous one, preserving velocity and retargeting to the new value. This produces the natural "redirect" feel when a user interrupts an in-progress animation.

    See 13-animation-advanced for a full CustomAnimation example building an elastic bounce curve.

    Choosing Between PhaseAnimator and KeyframeAnimator

    Consideration PhaseAnimator KeyframeAnimator
    Complexity Low -- define states, SwiftUI handles transitions Higher -- explicit timeline per property
    Property independence All properties animate together per phase Each property has its own track and timeline
    Interruption Graceful -- can restart from any phase Not interruptible mid-sequence
    Timing control Per-phase animation curve Per-keyframe timing and interpolation type
    Repeating Built-in (no trigger = continuous loop) Must re-trigger manually
    Best for State-driven multi-step, ambient effects Choreographed sequences with precise timing

    Performance Considerations

    • Keyframe updates run every frame. Keep the content closure lightweight -- avoid layout recalculations, network calls, or complex view tree changes.
    • Use .drawingGroup() for complex view hierarchies animated via keyframes to flatten into a single Metal render pass.
    • PhaseAnimator is cheaper for simple multi-step sequences since it only triggers view updates at phase boundaries, not every frame.
    • Profile with Instruments using the Core Animation template to catch dropped frames.

    Key Takeaways

    1. PhaseAnimator is the simplest path to multi-step animations -- define phases, optionally set per-phase animation curves, and SwiftUI handles the rest.
    2. KeyframeAnimator unlocks full choreographic control by letting each property follow its own independent timeline with mixed interpolation types.
    3. KeyframeTimeline decouples keyframe evaluation from views, enabling integration with Charts, MapKit, scroll-driven effects, and any other programmatic use case.
    4. CustomAnimation opens the door to entirely bespoke animation curves using the same infrastructure SwiftUI uses internally.
    5. SwiftUI preserves velocity between keyframes automatically, producing natural-feeling motion without manual bookkeeping.
    6. Keyframe animations are pre-planned, not interactive -- use springs or PhaseAnimator when animations need to respond to ongoing user input.
    7. The four keyframe types (Linear, Spring, Cubic, Move) give you granular control over the interpolation profile within each track.

    Animate with Springs

    Session 10158 -- WWDC23 Speaker: Jacob Xiao (Apple, SwiftUI team) Duration: ~23 minutes Chapters: Why springs | How springs work | How to use springs

    Discover how to bring life to your app with animation by taking advantage of springs. This session explains why springs are the ideal animation primitive, the physics behind how they work, and how to use the new Spring APIs across SwiftUI, UIKit, and Core Animation.

    Summary

    Apple made springs the default animation in SwiftUI starting with iOS 17. This session argues that springs are superior to ease curves for UI animation because they maintain velocity continuity, handle interruptions gracefully, and produce physically natural motion. It introduces a new duration/bounce parameterization that replaces the harder-to-reason-about mass/stiffness/damping model, and adds a first-class Spring type that can be used for both driving animations and performing custom spring math.

    1. Why Springs

    The Problem with Ease Curves

    Traditional ease-in/ease-out (Bezier) animations are pre-specified curves. They have two critical shortcomings:

    1. No initial velocity support -- When a user releases a gesture, the animation has a specific velocity at the moment of release. An ease curve always starts from zero velocity, so there is a velocity discontinuity (a visible jerk) when the gesture ends and the animation begins.

    2. Interruption problems -- If the user interrupts an in-flight ease animation (e.g., tapping again mid-animation), the curve restarts from a fixed shape. The position may be continuous, but the velocity jumps, causing a stutter.

    Why Springs Solve This

    Springs are the only type of animation that maintains continuity of both position and velocity in all cases:

    • Static start -- When there is no initial velocity, a spring starts gently and accelerates, just like an ease-out.
    • Gesture handoff -- A spring can incorporate any initial velocity, so the animation picks up right where the gesture left off. No jerk.
    • Interruption -- When a spring animation is interrupted, the system preserves the current velocity and feeds it into the new spring. The result is seamless.

    A spring animation does not mean a bouncy animation. Non-bouncy springs (bounce = 0) are used throughout iOS and are the default.

    Multi-Property Animations

    With ease curves, every animated property has the same fixed duration, so they all land at the same instant -- which can feel mechanical. With springs, different properties (e.g., position, scale, opacity) naturally settle at different rates, producing a more organic, staggered finish.

    2. How Springs Work

    Physical Model

    A spring is modeled as a mass on the end of a spring attached to a fixed point (the target value). Three physical quantities govern its behavior:

    Parameter Description
    Mass Weight of the object on the spring
    Stiffness How rigid the spring is (force per unit extension)
    Damping Friction that slows the motion over time

    These parameters are hard to reason about in terms of the resulting animation "feel." That is why Apple introduced a new parameterization.

    Duration and Bounce (New Parameterization)

    Instead of mass/stiffness/damping, springs can now be described with two intuitive parameters:

    Parameter Range Description
    Duration > 0 The perceptual duration -- defines the pace of the spring
    Bounce -1.0 to 1.0 How much the spring oscillates. 0 = smooth, positive = bouncy, negative = flat

    Perceptual duration is deliberately designed to be predictable: changing the bounce does not change the perceived duration, and vice versa. This is in contrast to the settling duration, which is the actual time until the spring's displacement is negligible and varies unpredictably with parameters.

    Spring Categories

    Bounce Value Name Behavior
    > 0 Bouncy Overshoots the target and oscillates (damped cosine wave x exponential decay)
    = 0 Smooth Approaches target gradually with a long tail (linear x exponential decay)
    < 0 Flat Two-exponential blend; flat approach, good for scroll-deceleration feel

    The Math

    Bouncy springs (bounce > 0)

    The motion is a damped cosine wave: a cosine curve multiplied by an exponential decay envelope.

    position(t) = A * cos(omega_d * t + phi) * e^(-zeta * omega_n * t)
    • The cosine provides oscillation.
    • The exponential decay pulls the oscillation toward zero.
    • Different initial velocities shift and scale the cosine wave so the spring can match any starting velocity.

    Smooth springs (bounce = 0) -- critically damped

    position(t) = (A + B*t) * e^(-omega_n * t)

    A linear ramp multiplied by exponential decay. No oscillation, smooth approach.

    Flat springs (bounce < 0) -- overdamped

    position(t) = A * e^(-alpha * t) + B * e^(-beta * t)

    Two exponential decays added together. Produces a flat, decelerating approach similar to scroll deceleration.

    Conversion Formulas

    Given duration and bounce, the equivalent physical parameters (with mass fixed at 1) are:

    mass = 1

stiffness = (2 * pi / duration)^2

damping:
  if bounce >= 0:  4 * pi * (1 - bounce) / duration
  if bounce <  0:  4 * pi / (duration + 4 * pi * bounce)

    Note: damping here is the damping coefficient, not the damping ratio. The damping ratio zeta = damping / (2 * sqrt(stiffness * mass)).

    3. How to Use Springs

    Spring Presets

    SwiftUI ships three presets that cover the most common use cases:

    Preset Bounce Character
    .smooth 0 No bounce, gradual deceleration (most versatile)
    .snappy ~0.15 Slight bounce, feels brisk and responsive
    .bouncy ~0.30 Noticeable bounce, playful and energetic

    Each preset can be tuned:

    // Default smooth spring
withAnimation(.smooth) { /* ... */ }

// Snappy with custom duration
withAnimation(.snappy(duration: 0.4)) { /* ... */ }

// Bouncy with extra bounce
withAnimation(.bouncy(extraBounce: 0.1)) { /* ... */ }

    Custom Springs

    For full control, specify duration and bounce directly:

    // SwiftUI
withAnimation(.spring(duration: 0.6, bounce: 0.2)) {
    // state changes
}

// UIKit (iOS 17+)
UIView.animate(duration: 0.6, bounce: 0.2) {
    // property changes
}

// Core Animation (iOS 17+)
let anim = CASpringAnimation(perceptualDuration: 0.6, bounce: 0.2)

    The Spring Model Type

    A new struct, Spring, lets you create a reusable spring model, inspect its physical properties, and evaluate its motion at any point in time:

    // Create from duration/bounce
let mySpring = Spring(duration: 0.5, bounce: 0.2)

// Inspect physical parameters
mySpring.mass       // 1.0
mySpring.stiffness  // derived from duration
mySpring.damping    // derived from duration + bounce

// Create from physical parameters
let physicsSpring = Spring(mass: 1, stiffness: 100, damping: 10)

// Use as an animation
withAnimation(.spring(physicsSpring)) {
    // state changes
}

    Evaluating Spring Values Programmatically

    You can call methods on Spring to get the position and velocity at any time, which is useful for building custom animations or driving non-animatable properties:

    let mySpring = Spring(duration: 0.4, bounce: 0.2)

// Get position at a given time
let position = mySpring.value(target: 1, time: time)

// Get velocity at a given time
let velocity = mySpring.velocity(target: 1, time: time)

    Custom CustomAnimation Conformance

    You can build a fully custom animation backed by a spring:

    func animate<V: VectorArithmetic>(
    value: V, time: Double, context: inout AnimationContext<V>
) -> V? {
    spring.value(
        target: value,
        initialVelocity: context.initialVelocity,
        time: effectiveTime(time: time, context: context)
    )
}

    Completion Handlers

    SwiftUI's new animation completion handler (iOS 17+) uses the perceptual duration rather than the settling duration. This means the completion fires at a predictable time regardless of bounce.

    Spring Preview (Xcode)

    Xcode provides a spring animation preview in the canvas that lets you visualize the spring curve as you adjust duration and bounce parameters, giving immediate feedback on how the animation will feel.

    Practical Guidance

    Choosing Duration

    • Start with the default (~0.5s).
    • Shorten for small, quick UI transitions (buttons, toggles).
    • Lengthen for large, sweeping transitions (sheet presentations, hero animations).

    Choosing Bounce

    Bounce Feel Good for
    0 Smooth, subtle Most UI, professional apps
    ~0.15 Brisk, snappy Responsive controls, tab switches
    ~0.30 Playful, energetic Games, social apps, delightful moments
    > 0.40 Very bouncy Use with caution -- can feel excessive
    < 0 Flat, decelerating Scroll-like behavior

    Best Practices

    1. Use springs as default -- withAnimation { } already uses a spring in iOS 17+.
    2. Start with presets -- .smooth, .snappy, .bouncy cover most needs.
    3. Tune duration first, then adjust bounce.
    4. Be consistent -- pick a spring character and use it across the entire app.
    5. Avoid extreme bounce -- values above 0.4 often feel unnatural in production UI.
    6. Leverage velocity continuity -- connect gesture velocities to spring animations for fluid handoff.

    Key Takeaways

    1. Springs maintain velocity continuity, making them fundamentally superior to ease curves for interactive animations.
    2. Spring != bouncy -- non-bouncy springs (bounce = 0) are the most common and versatile.
    3. Duration/bounce replaces mass/stiffness/damping as the recommended way to parameterize springs, because the parameters are perceptually predictable and independent.
    4. Three presets (.smooth, .snappy, .bouncy) make it easy to get started.
    5. Perceptual duration is stable -- it does not shift when you change bounce, unlike settling duration.
    6. Spring model type enables programmatic spring evaluation (.value(), .velocity()) for custom animation logic.
    7. Springs are now the default in SwiftUI -- just call withAnimation.
    8. Unified API -- the same duration/bounce parameterization works across SwiftUI, UIKit (UIView.animate), and Core Animation (CASpringAnimation).

    Demystify SwiftUI Performance (WWDC23 - 10160)

    Speaker: Matt Ricketson, SwiftUI Team Duration: ~21 minutes Video: Apple Developer | YouTube

    Prerequisites

    This is an advanced session. Before watching, you should understand:

    • SwiftUI identity (implicit vs. explicit) -- see 02-views-and-modifiers
    • View lifetime vs. view identity -- see 01-swiftui-philosophy
    • The predecessor session: Demystify SwiftUI (WWDC21, session 10022)

    Summary

    This session teaches how to build a mental model for SwiftUI performance and write faster, more efficient code. It covers the common causes behind hangs (delays in responding to user interaction) and hitches (perceivable animation glitches like scroll pauses or skipped frames), both of which often originate from slow SwiftUI view updates.

    The talk is structured around three pillars:

    1. Dependencies -- understanding what causes views to re-evaluate
    2. Faster view updates -- reducing the cost of each re-evaluation
    3. Identity in List and Table -- keeping collection-based views performant

    1. The Performance Feedback Loop

    Before diving into SwiftUI specifics, the session establishes a general performance debugging workflow:

    Symptom --> Measure --> Diagnose Root Cause --> Optimize --> Re-Measure
    Step Description
    Symptom Slow navigation, broken animations, spinning wait cursor
    Measure Verify the symptom exists with profiling tools
    Root Cause Understand how things should work, then find the deviation
    Optimize Apply a targeted fix
    Re-Measure Confirm the fix actually worked

    The rest of the session focuses on building the mental model needed for the Root Cause step.

    2. Dependencies

    See 16-data-flow and 05-state-and-binding for foundational data-flow concepts.

    How the Dependency Graph Works

    Every SwiftUI view has dependencies -- values that, when changed, cause the view's body to be re-evaluated. Dependencies come from two sources:

    1. Parent view values -- properties passed down from ancestor views (e.g., a dog: Dog property)
    2. Dynamic properties -- property wrappers like @State, @Binding, @Environment, @ObservedObject, @StateObject, and the new @Observable macro

    SwiftUI maintains an internal dependency graph. Each node is a view, and each edge represents a dependency relationship. When a dependency changes, SwiftUI walks the graph and re-evaluates only the views that depend on the changed value. Each arrow in the graph represents a view's body -- connecting inputs to outputs.

    Debugging with _printChanges()

    The session introduces Self._printChanges() as the primary debugging tool for understanding why a view's body is being called:

    struct ScalableDogImage: View {
    @State private var scaleToFill = false
    var dog: Dog

    var body: some View {
        let _ = Self._printChanges()
        dog.image
            .resizable()
            .aspectRatio(contentMode: scaleToFill ? .fill : .fit)
            .frame(maxHeight: scaleToFill ? 500 : nil)
            .padding(.vertical, 16)
            .onTapGesture {
                withAnimation { scaleToFill.toggle() }
            }
    }
}

    Important caveats about _printChanges(): - The underscore prefix means it is an unsupported, debugging-only API - Never ship it to the App Store -- remove before release - It prints to the console which dependency triggered the body evaluation - Output format: ViewName: _propertyName changed. or ViewName: @self changed. (the view value itself changed)

    Reducing Dependencies (Key Optimization)

    The core insight: pass only the data a view actually reads, not an entire model object.

    Before -- unnecessary dependency on full Dog object:

    struct ScalableDogImage: View {
    @State private var scaleToFill = false
    var dog: Dog  // Entire Dog passed, but only .image is used

    var body: some View {
        dog.image
            .resizable()
            .aspectRatio(contentMode: scaleToFill ? .fill : .fit)
    }
}

    In this pattern, when any property of Dog changes (e.g., dog.name), this view's body is re-evaluated even though it only reads dog.image. The @self changed message from _printChanges() reveals this -- the view value itself changed because the dog property (the entire struct) was different.

    After -- scoped dependency:

    struct ScalableDogImage: View {
    @State private var scaleToFill = false
    var image: Image  // Only what the view actually needs

    var body: some View {
        image
            .resizable()
            .aspectRatio(contentMode: scaleToFill ? .fill : .fit)
    }
}

    Now the view only re-evaluates when the image itself changes.

    @Observable vs ObservedObject -- Performance Differences

    See 15-observable-and-environment for a deep-dive on the Observation framework.

    This is one of the most impactful performance differences introduced in iOS 17:

    Aspect ObservableObject + @Published @Observable (iOS 17+)
    Granularity Whole-object -- any @Published property change invalidates every view observing the object Per-property -- only views that read the changed property are invalidated
    Dependency tracking Manual (you must split models or extract views) Automatic (SwiftUI tracks which properties each body accesses at runtime)
    View wrapper @ObservedObject / @StateObject @State (for owned) / plain property / @Environment
    Over-invalidation risk High -- common source of performance bugs Low -- scoped by design

    Example of the problem with ObservableObject:

    // Old pattern -- ObservableObject
class DogModel: ObservableObject {
    @Published var name: String
    @Published var age: Int
    @Published var image: Image
    @Published var favoriteActivities: [String]
}

struct DogNameView: View {
    @ObservedObject var model: DogModel
    var body: some View {
        Text(model.name)
        // Re-evaluates when age, image, OR activities change too!
    }
}

    Same scenario with @Observable:

    // New pattern -- @Observable
@Observable class DogModel {
    var name: String
    var age: Int
    var image: Image
    var favoriteActivities: [String]
}

struct DogNameView: View {
    var model: DogModel
    var body: some View {
        Text(model.name)
        // Only re-evaluates when model.name changes
        // SwiftUI automatically tracks that body only reads .name
    }
}

    The @Observable macro eliminates an entire class of performance bugs by making dependency scoping automatic. For existing apps, migrating from ObservableObject to @Observable can yield significant performance improvements with relatively little code change.

    Dependency Reduction Strategies

    1. Extract child views -- break large views into smaller ones so each only depends on what it reads
    2. Pass specific values instead of entire model objects
    3. Use @Observable for automatic per-property tracking (iOS 17+)
    4. Use @ObservedObject carefully on older targets -- split models or use view extraction to limit invalidation scope

    3. Faster View Updates

    See 23-view-composition for patterns on structuring views effectively.

    Once a dependency triggers a view update, the update itself needs to be fast. The session identifies three categories of slow updates:

    3a. Expensive Dynamic Property Instantiation

    Dynamic properties like @StateObject and @State may run initialization code that blocks the main thread:

    Problem -- synchronous fetch in initializer:

    @Observable class FetchModel {
    var dogs: [Dog]

    init() {
        fetchDogs()  // Blocks the main thread!
    }

    func fetchDogs() {
        // Network call or heavy computation
    }
}

    When SwiftUI instantiates this model (e.g., via @State private var model = FetchModel()), the init() blocks until fetchDogs() completes.

    Solution -- async fetch with .task modifier:

    See 21-async-patterns for .task and structured concurrency in SwiftUI.

    @Observable class FetchModel {
    var dogs: [Dog] = []

    init() {}  // Cheap, instant

    func fetchDogs() async {
        // Network call runs off main thread
    }
}

struct DogRootView: View {
    @State private var model = FetchModel()

    var body: some View {
        DogList(model.dogs)
            .task {
                await model.fetchDogs()  // Runs asynchronously
            }
    }
}

    The .task modifier ties the async work to the view's lifecycle and automatically cancels it if the view disappears.

    3b. Expensive Work in body

    The body property runs on the main thread and must return quickly. Common offenders:

    Offender Impact Fix
    Expensive string interpolation Repeated String(format:) or NumberFormatter calls Cache formatted strings in the model
    Inline data filtering array.filter { ... } called every body evaluation Pre-compute and cache the filtered result
    Bundle lookups Bundle.main.url(forResource:) on every render Cache the result once
    Heap allocations Creating class instances inside body Move allocation outside body
    Heavy computation Sorting, mapping, complex math Move to model layer or use .task

    Rule of thumb: The body should only compose views -- all data transformation belongs in the model layer.

    3c. Slow Identification

    Views that are expensive to identify (e.g., using complex Hashable conformances as IDs) slow down every update cycle. Keep identifiers cheap to create and compare -- prefer simple types like Int, String, or UUID.

    4. Identity in List and Table

    See 07-lists-and-foreach for fundamentals of List and ForEach.

    How List Gathers IDs

    Critical insight: List and Table gather all row identifiers eagerly -- even for rows not currently visible. This is required for:

    • Scroll bar sizing
    • Programmatic scrolling
    • Selection management
    • Consistency guarantees

    The ForEach generic signature reveals how identity works:

    ForEach<Data, ID, Content>
// Data: the collection
// ID: the identifier key path
// Content: the view builder for each element

    Row count equation:

    Total Rows = Number of Elements x Number of Views Per Element

    The "views per element" count must be constant (determinable at compile time). If it is not constant, SwiftUI must evaluate the body of every element to discover identifiers, defeating lazy loading.

    Anti-Patterns That Kill List Performance

    Anti-Pattern 1 -- Conditional views inside ForEach (variable view count):

    // BAD: View count per element is 0 or 1 (not constant)
List {
    ForEach(dogs) { dog in
        if dog.likesFetch {
            DogCell(dog)
        }
    }
}

    SwiftUI cannot know how many views each element produces without evaluating every body. This forces eager evaluation of all rows.

    Anti-Pattern 2 -- AnyView type erasure:

    // BAD: AnyView hides the view structure from SwiftUI
List {
    ForEach(dogs) { dog in
        AnyView(DogCell(dog))
    }
}

    AnyView erases type information, preventing SwiftUI from reasoning about the view count statically.

    Correct Patterns

    Filter in the data layer, not the view layer:

    // GOOD: Pre-filter the data
List {
    ForEach(tennisBallDogs) { dog in
        DogCell(dog)  // Always exactly 1 view per element
    }
}

    See 20-search-and-filtering for data filtering patterns in SwiftUI.

    Cache the filtered collection in your model:

    @Observable class DogModel {
    var dogs: [Dog] = []

    var tennisBallDogs: [Dog] {
        dogs.filter { $0.likesFetch }
    }
}

    Sectioned lists with nested ForEach (acceptable):

    // GOOD: Outer ForEach over sections, inner ForEach over items
struct DogsByToy: View {
    var model: DogModel

    var body: some View {
        List {
            ForEach(model.dogToys) { toy in
                Section(toy.name) {
                    ForEach(model.dogs(toy: toy)) { dog in
                        DogCell(dog)
                    }
                }
            }
        }
    }
}

    This is fine because each inner ForEach produces a constant number of views per element (1 DogCell per dog).

    Table Improvements in iOS 17 / macOS Sonoma

    iOS 17 introduced a streamlined Table API that enforces constant row counts:

    Before (iOS 16 -- verbose, allows anti-patterns):

    struct DogTable: View {
    var dogs: [Dog]
    var body: some View {
        Table(of: Dog.self) {
            // Column definitions...
        } rows: {
            ForEach(dogs) { dog in
                TableRow(dog)
            }
        }
    }
}

    After (iOS 17 -- streamlined, constant row count enforced):

    struct DogTable: View {
    var dogs: [Dog]
    var body: some View {
        Table(of: Dog.self) {
            // Column definitions...
        } rows: {
            ForEach(dogs)  // Implicit TableRow -- constant row count guaranteed
        }
    }
}

    Breaking change note: In iOS 16, row IDs were based on the ForEach element value. In iOS 17, row IDs are based on the table's data type. If you need the old behavior for back-deployment:

    ForEach(dogs, id: \.self) { dog in
    TableRow(dog.bestFriend)
}

    List/Table Identity Rules

    Rule Why
    Keep identifiers cheap to create IDs are gathered eagerly for all rows
    Keep view count per element constant Variable counts force eager body evaluation
    Avoid AnyView in list content Type erasure prevents static analysis of view counts
    Avoid conditionals that produce 0 or 1 views Non-constant view count
    Filter data, not views Move filtering to the model layer
    Use explicit stacks if you need multiple views per row Wraps them in a single container
    Flatten nested ForEach (except sections) Reduces identity resolution overhead

    5. Performance Tools and Profiling Tips

    Tools Mentioned

    Tool Purpose
    Self._printChanges() Console output showing which dependency triggered a body evaluation (debug only)
    Instruments -- Time Profiler Identify slow function calls during hangs and hitches
    Instruments -- SwiftUI View Body instrument Shows body evaluation count and average duration per view type
    Instruments -- Animation Hitches Detect and diagnose animation hitches

    Profiling Best Practices

    1. Always profile on a real device, not the Simulator
    2. Use Cmd+I (Product > Profile) to build in profiling mode
    3. In the SwiftUI View Body instrument, focus on Count and Avg Duration columns
    4. Look for views with unexpectedly high counts (over-invalidation) or high durations (expensive bodies)
    5. Use _printChanges() in debug builds to trace exactly why a view is updating
    6. After applying a fix, always re-measure to confirm improvement

    The _printChanges() Output Guide

    Output Meaning
    ViewName: _propertyName changed. A specific dynamic property triggered the update
    ViewName: @self changed. The view's value itself changed (parent passed new data)
    ViewName: _someState changed, @self changed. Multiple dependencies changed simultaneously

    6. Key Takeaways

    1. Build a mental model early. Understand the dependency graph, body evaluation, and identity system before performance problems arise.

    2. Reduce dependencies. Pass only the specific data a view reads. Extract sub-views. Use @Observable for automatic per-property tracking.

    3. Keep body cheap. No network calls, no heavy computation, no expensive allocations. Move all data transformation to the model layer. Use .task for async work.

    4. Use @Observable over ObservableObject. The per-property tracking eliminates whole-object invalidation, one of the most common performance pitfalls.

    5. Maintain constant view counts in List/Table. Never use conditionals or AnyView inside ForEach. Filter your data, not your views.

    6. Cache computed data. Pre-compute filtered/sorted/formatted data in your model rather than recalculating in every body evaluation.

    7. Use cheap identifiers. List and Table gather all IDs eagerly. Expensive Hashable conformances directly impact load and scroll performance.

    8. Leverage iOS 17 / macOS Sonoma improvements. Streamlined Table API, Observation framework, and internal SwiftUI optimizations can yield dramatic improvements for bigger lists and tables with minimal code changes.

    9. Profile on real devices. Use Instruments (Time Profiler, SwiftUI View Body, Animation Hitches) and _printChanges() to find and verify fixes.

    Ultimate Application Performance Survival Guide

    Performance optimization can seem like a daunting task -- with many metrics to track and tools to use. This session is a survival guide that helps you understand tooling, metrics, and paradigms to optimize your app's performance and ensure a great user experience.

    Speaker: Shefali Saboo, Performance Tools Engineer at Apple Duration: ~25 minutes Platforms: iOS, macOS, watchOS, tvOS

    Overview

    This session serves as a comprehensive map of Apple's performance optimization ecosystem. Rather than deep-diving into any single metric, it provides the framework for understanding what to measure, where to measure it, and which tools to use at each stage of your app's lifecycle. It covers eight key performance metrics and walks through the tooling available at every phase of development.

    The Performance Lifecycle: Three Phases

    Apple structures performance work around the app development lifecycle, with different tools suited to each phase:

    Phase 1: Development (Xcode + Instruments)

    • Use Instruments to profile your app in controlled environments
    • Use Xcode Gauges (CPU, Memory, Energy, Disk, Network) in the Debug Navigator for real-time monitoring
    • Write XCTest performance tests to catch regressions in CI
    • This is your tightest feedback loop -- profile early and often

    Phase 2: Beta / TestFlight (MetricKit + Xcode Organizer)

    • Deploy to real users on real devices via TestFlight
    • MetricKit collects on-device telemetry and delivers payloads to your app
    • New in iOS 15: diagnostics (including hang reports) are delivered immediately after an issue occurs, not on the previous 24-hour cadence
    • Triage crashes and performance regressions using Xcode Organizer

    Phase 3: Production (Xcode Organizer + App Store Connect API)

    • Xcode Organizer aggregates anonymized performance data from consenting users
    • App Store Connect API lets you query performance metrics programmatically for dashboards and alerts
    • Compare metrics across app versions to detect regressions
    • Monitor trends over time

    The Core Loop: Measure -> Profile -> Optimize

      ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ MEASURE  │ --> │ PROFILE  │ --> │ OPTIMIZE │
  └──────────┘     └──────────┘     └──────────┘
       ^                                  │
       └──────────────────────────────────┘
    1. Measure -- Collect data using Xcode Organizer, MetricKit, or XCTest to identify what is slow
    2. Profile -- Use Instruments to drill into why it is slow
    3. Optimize -- Fix the issue, ship, and measure again to verify improvement

    The Eight Performance Metrics

    1. Battery Usage

    What it is: Excessive CPU, networking, or location usage drains the user's battery and degrades the overall device experience.

    Key subsystems to monitor: - CPU -- Intensive processing, especially on the main thread - Networking -- Frequent or large network requests; failure to batch - Location -- Using continuous GPS when significant-change monitoring would suffice

    How to diagnose: | Tool | When | What it shows | |------|------|---------------| | Energy Gauge (Xcode Debug Navigator) | Development | Real-time energy impact (Low / High / Very High) | | Time Profiler (Instruments) | Development | CPU hotspots by function/call stack | | Location Energy Model (Instruments) | Development | Energy impact of Core Location usage | | Xcode Organizer | Production | Aggregated battery usage across user base | | MetricKit (MXCPUMetric, MXNetworkTransferMetric, MXLocationActivityMetric) | Beta / Production | On-device telemetry payloads |

    How to fix: - Reduce unnecessary background CPU work - Batch network requests; use URLSession background transfers - Use the lowest-accuracy location mode that meets your needs - Defer non-critical work using BGTaskScheduler

    2. Launch Time

    What it is: The time from the user tapping your app icon to the first frame being rendered on screen. Slow launch times frustrate users and can lead to system termination (watchdog kills).

    Thresholds: - Target: < 400ms for a responsive feel - The system watchdog will terminate your app if launch takes too long (~20 seconds, varies by platform)

    How to diagnose: | Tool | When | What it shows | |------|------|---------------| | App Launch template (Instruments) | Development | Full launch timeline with thread activity | | XCTApplicationLaunchMetric (XCTest) | Development / CI | Automated launch time measurement | | Xcode Organizer | Production | Aggregated launch times by percentile | | MetricKit (MXAppLaunchMetric) | Beta / Production | Launch duration telemetry |

    How to fix: - Defer non-essential work out of application(_:didFinishLaunchingWithOptions:) - Reduce dylib load counts; use fewer dynamic frameworks - Minimize static initializer work (+load, __attribute__((constructor))) - Lazy-load view controllers and data - Avoid synchronous network requests at launch

    3. Hang Rate

    What it is: A hang is a period of unresponsiveness where the main thread is blocked for a perceptible amount of time. The app appears frozen -- buttons don't respond, scrolling stops, the UI goes blank.

    Definition: The main thread is unresponsive for >= 250ms.

    Severity thresholds: | Duration | Classification | Impact | |----------|---------------|--------| | < 100ms | Imperceptible | Users won't notice | | 100-250ms | Borderline | May feel sluggish subconsciously | | 250-500ms | Micro hang | Noticeable; tools begin reporting | | > 500ms | Hang | Clearly broken feel | | > 1s | Severe hang | Users may force-quit |

    How to diagnose: | Tool | When | What it shows | |------|------|---------------| | Thread State Trace (Instruments) | Development | Main thread state over time | | System Call Trace (Instruments) | Development | Blocking system calls on main thread | | Hang Detection (Xcode Organizer) | Production | Hang rate and diagnostic reports | | MetricKit (MXAppResponsivenessMetric) | Beta / Production | Hang diagnostics with call stacks | | XCTest responsiveness tests | Development / CI | Automated hang detection |

    New in iOS 15: Hang diagnostics from MetricKit are now delivered immediately after the issue occurs, enabling rapid root-cause analysis.

    How to fix: - Move heavy work off the main thread (use GCD, async/await, OperationQueue) - Avoid synchronous I/O, networking, or database calls on the main thread - Break large main-thread tasks into smaller chunks - Use os_signpost to mark critical sections for profiling - See #Related Sessions -- session 10258 goes deep on this topic

    4. Scrolling & Hitches

    What it is: A hitch is a user-perceivable animation glitch -- a frame that appears on screen later than expected, causing visible stutter during scrolling or animation.

    The Render Loop: The iOS render pipeline operates in a continuous loop with these phases:

    App Process                         Render Server              Display
┌────────┐  ┌────────┐            ┌─────────────┐  ┌─────────┐  ┌─────────┐
│ Event  │->│ Commit │ ---------> │Render Prepare│->│ Render  │->│ Display │
│ Phase  │  │ Phase  │            │              │  │ Execute │  │         │
└────────┘  └────────┘            └─────────────┘  └─────────┘  └─────────┘
                                          ▲
                                     VSYNC deadline

    Each phase must complete before the next VSYNC. When a phase misses its deadline, the frame is delayed and a hitch occurs.

    Types of hitches: - Commit hitch -- The app takes too long in the event/commit phase (e.g., expensive layout, slow cellForRow) - Render hitch -- The render server takes too long (e.g., offscreen rendering, complex layer hierarchies)

    Hitch Time Ratio: The primary metric is hitch time ratio = total hitch time / scroll duration, measured in ms/s (milliseconds of hitching per second of scrolling).

    Hitch Time Ratio Quality Action
    < 5 ms/s Good Mostly unnoticeable
    5-10 ms/s Warning User notices interruptions; investigate
    > 10 ms/s Critical Severely impacts UX; fix immediately

    How to diagnose: | Tool | When | What it shows | |------|------|---------------| | Animation Hitches (Instruments) | Development | Per-frame hitch analysis | | XCTOSSignpostMetric.scrollDecelerationMetric (XCTest) | Development / CI | Automated scroll performance tests | | mxSignpostAnimationIntervalBegin (MetricKit) | Beta / Production | Custom animation hitch telemetry | | Xcode Organizer (Scroll Hitch Rate) | Production | Aggregated hitch rates |

    New in iOS 15: Use MXSignpost to tag custom animations for hitch telemetry in production:

    func startAnimating() {
    mxSignpostAnimationIntervalBegin(
        log: MXMetricManager.makeLogHandle(category: "animation_telemetry"),
        name: "custom_animation")
}

func animationDidComplete() {
    mxSignpost(OSSignpostType.end,
        log: MXMetricManager.makeLogHandle(category: "animation_telemetry"),
        name: "custom_animation")
}

    XCTest scroll performance test example:

    func testScrollingAnimationPerformance() throws {
    app.launch()
    app.staticTexts["Meal Planner"].tap()
    let foodCollection = app.collectionViews.firstMatch

    let measureOptions = XCTMeasureOptions()
    measureOptions.invocationOptions = [.manuallyStop]

    measure(metrics: [XCTOSSignpostMetric.scrollDecelerationMetric],
            options: measureOptions) {
        foodCollection.swipeUp(velocity: .fast)
        stopMeasuring()
        foodCollection.swipeDown(velocity: .fast)
    }
}

    How to fix: - Use cell prefetching (UICollectionViewDataSourcePrefetching) - Avoid expensive layout calculations during scroll - Reduce offscreen rendering (shadows, masks, rounded corners without cornerRadius + masksToBounds) - Use CALayer.shouldRasterize carefully - Flatten layer hierarchies where possible

    5. Memory Footprint

    What it is: Memory is a shared, finite resource. If your app exceeds the system's memory limit, it will be terminated immediately (jetsam / OOM kill). Even if not terminated, high memory pressure causes the system to evict other apps and degrade overall device performance.

    Key metrics: - Peak Memory -- The maximum memory your app uses during a session - Memory at Suspension -- Memory held when your app moves to background (high values increase chance of background termination)

    How to diagnose: | Tool | When | What it shows | |------|------|---------------| | Memory Gauge (Xcode Debug Navigator) | Development | Real-time memory usage | | Allocations (Instruments) | Development | Every heap allocation with backtrace | | Leaks (Instruments) | Development | Retain cycles and leaked objects | | VM Tracker (Instruments) | Development | Virtual memory regions and dirty pages | | Xcode Organizer (Memory section) | Production | Peak memory and memory at suspension | | MetricKit (MXMemoryMetric) | Beta / Production | Memory telemetry with MXSignpost intervals |

    MXSignpost for memory-critical sections:

    func saveAppAssets() {
    mxSignpost(OSSignpostType.begin,
        log: MXMetricManager.makeLogHandle(category: "memory_telemetry"),
        name: "custom_memory")

    // save app metadata

    mxSignpost(OSSignpostType.end,
        log: MXMetricManager.makeLogHandle(category: "memory_telemetry"),
        name: "custom_memory")
}

    How to fix: - Use autoreleasepool in tight loops - Downscale images to the display size (UIGraphicsImageRenderer or ImageIO) - Respond to didReceiveMemoryWarning and purge caches - Use NSCache instead of Dictionary for caching (auto-evicts under pressure) - Audit for retain cycles (especially in closures and delegates) - Reduce memory at suspension by releasing resources in applicationDidEnterBackground - See related session: Detect and Diagnose Memory Issues (WWDC21-10180)

    6. Disk Writes

    What it is: Excessive disk writes wear out the user's NAND flash storage, degrading long-term device health. The system monitors writes and generates diagnostic reports for apps that exceed thresholds.

    Threshold: Disk Write Diagnostic Reports are generated when an app writes > 1 GB in a 24-hour period.

    7. Terminations

    What it is: Unexpected app terminations degrade user trust. The system terminates apps for various reasons: excessive memory, watchdog timeouts (slow launch/resume), and background task violations.

    Types of terminations: - Jetsam (memory) -- App exceeded memory limit - Watchdog -- App took too long to launch, resume, or respond to a system event - CPU resource limit -- App used excessive CPU in the background - Crash -- Unhandled exception or signal

    Understand and Eliminate Hangs from Your App

    WWDC21 Session 10258 -- Presented by Anubhav, OS Performance Team at Apple

    Summary

    This session teaches how to identify, diagnose, and eliminate hangs -- periods of unresponsiveness where an app appears stuck, lags, or spins. It covers the fundamental architecture of the main thread and the run loop, explains why hangs happen, demonstrates diagnostic tools like Time Profiler and System Trace, introduces MetricKit for field diagnostics, and walks through common anti-patterns and their fixes using a demo app called Desserted (a recipe app for drinks and desserts).

    What Is a Hang?

    A hang is when an app becomes unresponsive to user input. The user taps, swipes, or otherwise interacts, but nothing happens -- the UI freezes, animations stall, and the app feels broken.

    Formal Definition

    A hang is reported when the main thread is busy doing work or waiting on another thread or system resource, causing a delay in updating the view content by at least 250 milliseconds.

    Why 250ms?

    • Research shows users perceive UI updates within 100ms as "instant"
    • A delay of 250ms or more is definitively noticeable and frustrating
    • Events are buffered during a hang and cannot be handled, compounding the problem
    • If the hang is long enough, the system watchdog will terminate the app entirely

    Hang Severity Categories

    Category Duration User Impact
    Micro hang < 400ms Brief stutter, noticeable but tolerable
    Hang 250ms -- several seconds Clearly unresponsive, user may retry input
    Severe hang Multiple seconds App feels broken, user may force-quit
    Watchdog kill ~20s (foreground) System terminates the app

    The Main Thread and the Run Loop

    The Main Run Loop

    The main run loop is a loop that the application's main thread enters to run event handlers in response to incoming events -- primarily user interactions. Each "turn" of the run loop follows this cycle:

    User Input --> Run Loop Receives Event --> Process Event --> Update UI --> Wait for Next Event

    All of this happens on the main thread and within one turn of the run loop. If processing the event takes too long, there is a delay between user input and any UI update.

    Event Handling Phase vs. Commit Phase

    Each turn of the run loop has two critical phases:

    1. Event Handling Phase

    • The app receives an event (tap, scroll, etc.)
    • Your code runs in response (e.g., touchesBegan, action handlers, gesture recognizer callbacks)
    • Decisions are made about how to update the UI
    • This is where most of your app logic runs

    2. Commit Phase (Commit Transaction)

    • After event handling completes, the system records that views need layout or display
    • During the commit transaction, the system calls:
    • layoutSubviews -- for views that need layout
    • drawRect: / draw(_:) -- for views that need display
    • The resulting layer tree is sent to the render server (a separate process) which composites the individual UI layers and renders the next frame

    A hang in either phase blocks the entire run loop turn. Work in the commit phase (expensive layoutSubviews or draw calls) is just as capable of causing hangs as work in the event handling phase.

    Render Server

    The render server is a separate process responsible for: - Compositing individual UI layers - Rendering the next frame to the display - Operating independently of the app process

    Since the render server is a separate process, once the commit transaction is sent, the app's main thread is free again. The bottleneck is everything that happens before that handoff.

    Causes of Hangs

    The high-level cause for all hangs is too much work being done on, or on behalf of, the main thread.

    Direct Causes

    1. Expensive computation on the main thread -- CPU-intensive work (image processing, data parsing, complex calculations) running synchronously
    2. Synchronous I/O on the main thread -- File reads/writes, database queries, network requests that block until complete
    3. Synchronous IPC / framework calls -- Querying system frameworks that internally perform expensive operations (e.g., Contacts framework)
    4. Lock contention -- Main thread waiting to acquire a lock held by another thread (@synchronized, os_unfair_lock, pthread_mutex, dispatch_sync)
    5. Blocking on another thread -- Using dispatch_sync to a queue where work is still executing, or waiting on a semaphore

    Indirect Causes

    1. Work dispatched to the main queue from background threads -- Even if you dispatch async to the main queue, that block must execute on the main thread. A flood of small blocks on the main queue accumulates
    2. Priority inversion -- Main thread waits on a lower-priority thread that gets preempted by the scheduler. Semaphores do not propagate priority, which can lengthen a hang due to preemption
    3. Expensive commit phase work -- Heavy layoutSubviews or drawRect: implementations that run during the commit transaction

    Demo App: Desserted

    The session uses a recipe app called Desserted to demonstrate real-world hang scenarios and fixes.

    Hang #1: Contact Querying on Every Tap

    Problem: Desserted has a social feature icon that only appears if a contact has been added as a friend. The app queries all contacts on every tap into a detail view, causing the main thread to block while the Contacts framework performs expensive operations.

    Fix: Use NotificationCenter observers -- register for the CNContactStoreDidChange notification (or ABDatabaseChangedExternally) so the app reacts to changes lazily. Cache the result and only re-query when the notification fires, dispatching the query to a background queue.

    Hang #2: CPU-Intensive Image Processing

    Problem: To add rounded corners to images, Desserted creates a bitmap-based graphics context, converts the image to a bitmap, applies a UIBezierPath clip, then converts back to a UIImage. This is CPU-intensive, uses excessive memory, and blocks the main thread.

    Fix: Use CoreAnimation instead of CPU-based image manipulation. Setting layer.cornerRadius and layer.masksToBounds = true on a CALayer leverages the GPU and is essentially free:

    // BAD: CPU-based rounding (causes hangs)
func roundedImage(_ image: UIImage) -> UIImage {
    UIGraphicsBeginImageContextWithOptions(size, false, 0)
    let path = UIBezierPath(roundedRect: rect, cornerRadius: radius)
    path.addClip()
    image.draw(in: rect)
    let result = UIGraphicsGetImageFromCurrentImageContext()
    UIGraphicsEndImageContext()
    return result!
}

// GOOD: GPU-based rounding via CoreAnimation (instant)
imageView.layer.cornerRadius = 12
imageView.layer.masksToBounds = true

    Hang #3: Missing Caches for Formatted Assets

    Problem: Ingredient image tiles are generated from scratch every time they are needed -- an expensive operation repeated on each view appearance.

    Fix: Use NSCache to store formatted assets. The first generation is expensive, but subsequent accesses are a quick memory read:

    let imageCache = NSCache<NSString, UIImage>()

func formattedImage(for key: String) -> UIImage {
    if let cached = imageCache.object(forKey: key as NSString) {
        return cached
    }
    let image = generateExpensiveImage(for: key)  // expensive
    imageCache.setObject(image, forKey: key as NSString)
    return image
}

    Important: Have an accurate cache invalidation mechanism. Balance between stale data and constant regeneration.

    Common Hang Patterns and Fixes

    Pattern 1: Synchronous Networking

    // BAD: Blocks main thread until network response arrives
let data = try! Data(contentsOf: url)

// GOOD: Use URLSession's async API
URLSession.shared.dataTask(with: url) { data, response, error in
    guard let data = data else { return }
    DispatchQueue.main.async {
        self.updateUI(with: data)
    }
}.resume()

    Pattern 2: Synchronous File I/O

    // BAD: Reading large files synchronously on main thread
let contents = try! String(contentsOfFile: path)

// GOOD: Dispatch to background queue
DispatchQueue.global(qos: .userInitiated).async {
    let contents = try? String(contentsOfFile: path)
    DispatchQueue.main.async {
        self.display(contents)
    }
}

    Pattern 3: dispatch_sync to a Busy Queue

    // BAD: Main thread blocks waiting for the queue
serialQueue.sync {
    expensiveWork()
}

// GOOD: Use async and a completion handler
serialQueue.async {
    let result = expensiveWork()
    DispatchQueue.main.async {
        self.handleResult(result)
    }
}

    Pattern 4: Lock Contention

    Locks that can cause main thread hangs: - @synchronized / objc_sync_enter - os_unfair_lock - pthread_mutex_lock - dispatch_sync (acts as a lock) - DispatchSemaphore.wait() -- does not propagate priority, so a lower-priority thread holding the resource can be preempted, extending the hang

    Fix: Minimize critical sections, prefer lock-free designs, or use dispatch_async instead of dispatch_sync.

    Pattern 5: Expensive Commit Phase Work

    // BAD: Heavy layout computation
override func layoutSubviews() {
    super.layoutSubviews()
    // Complex constraint recalculations, dynamic sizing
    performExpensiveLayoutCalculation()
}

// GOOD: Pre-compute layout values off main thread, cache results

    Strategies for Eliminating Hangs

    1. Use Caches (NSCache)

    • Cache frequently used assets, computed values, formatted images
    • Replace expensive repeated computations with quick memory reads
    • Ensure proper cache invalidation

    2. Observe Notifications (NotificationCenter)

    • Replace polling / on-demand expensive queries with reactive notification observers
    • Let the system notify you of changes instead of checking on every interaction
    • Dispatch observer callbacks to background queues

    3. Find Asynchronous API Alternatives

    • Many Apple frameworks offer both sync and async variants
    • Always prefer the async version (e.g., URLSession.dataTask over Data(contentsOf:))
    • With Swift concurrency: prefer async/await patterns

    4. Use Grand Central Dispatch (GCD)

    • DispatchQueue.global().async -- move work to a background thread
    • Pre-warming computation -- dispatch work early on a background queue, then sync to retrieve results only when needed:
    let prefetchQueue = DispatchQueue(label: "com.app.prefetch")

// Start work early
prefetchQueue.async {
    self.precomputeExpensiveData()
}

// Later, when result is needed (blocks briefly if not done)
prefetchQueue.sync {
    // Result is already computed or finishes quickly
    return self.cachedResult
}
    • Always dispatch back to the main queue for UI updates:
    DispatchQueue.global(qos: .userInitiated).async {
    let result = self.computeResult()
    DispatchQueue.main.async {
        self.label.text = result
    }
}

    5. Leverage the GPU via CoreAnimation

    • Use CALayer properties (cornerRadius, shadowPath, etc.) instead of CPU-based drawing
    • Let the render server and GPU handle visual effects

    Tools for Finding and Diagnosing Hangs

    Development Phase

    Time Profiler (Instruments)

    • Records call stacks at regular intervals
    • Shows where CPU time is being spent on the main thread
    • Use to identify hot spots and expensive functions
    • Look for tall stacks on the main thread during hang intervals

    System Trace (Instruments)

    • Low-level view of thread scheduling, context switches, and system calls
    • Shows when the main thread is blocked vs. running
    • Reveals lock contention, IPC waits, and priority inversions
    • More detailed than Time Profiler for understanding why the main thread is blocked

    Thread Performance Checker (Xcode)

    • Runtime tool that warns about performance anti-patterns
    • Detects work on the wrong thread (e.g., UI updates on background threads)

    Post-Ship / Field Diagnostics

    MetricKit -- MXHangDiagnostic

    • Available since iOS 14
    • Collects call trees for hangs experienced by real users in the field
    • Aggregates call stacks taken during a hang into a tree format similar to what the Time Profiler presents
    • Enables you to prioritize fixes based on which hangs customers are most frequently hitting
    • Delivered as part of the 24-hour diagnostic payload via MXMetricManager
    // Registering for MetricKit diagnostics
class AppDelegate: UIResponder, UIApplicationDelegate, MXMetricManagerSubscriber {
    func application(_ application: UIApplication,
                     didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {
        MXMetricManager.shared.add(self)
        return true
    }

    func didReceive(_ payloads: [MXDiagnosticPayload]) {
        for payload in payloads {
            if let hangDiagnostics = payload.hangDiagnostics {
                for hang in hangDiagnostics {
                    // hang.callStackTree -- aggregated call stacks

                    analyzeHang(hang)
                }
            }
        }
    }
}

    Xcode Organizer -- Hang Reports

    • Shows hang rate per app version as a chart
    • Useful for triaging regressions between releases
    • Helps baseline and quantify the performance of your app over time
    • Available under the Metrics tab in Xcode Organizer

    Xcode Organizer -- Hang Rate Metric

    • Percentage of app usage time spent in hangs
    • Compare across app versions to track improvements or regressions
    • Filter by device type and OS version

    Key Takeaways

    1. A hang is a main thread block of 250ms or more -- users notice, events queue up, and the experience degrades
    2. The main run loop has two phases where hangs can occur: event handling and the commit transaction
    3. The #1 cause is too much work on the main thread -- either directly or by waiting on other threads/resources
    4. Use the right tools at the right time: - During development: Time Profiler and System Trace in Instruments - After shipping: MetricKit (MXHangDiagnostic) and Xcode Organizer hang reports
    5. Four elimination strategies: caching (NSCache), notification observers, async API alternatives, and GCD (dispatch_async)
    6. Leverage the GPU for visual operations -- use CALayer properties instead of CPU-based image processing
    7. Beware of priority inversion -- semaphores do not propagate priority and can extend hangs
    8. Even async dispatch to the main queue runs on the main thread -- be mindful of how much work returns to the main queue
    9. Baseline your app's hang rate in Xcode Organizer so you can measure the impact of optimizations
    10. Use MetricKit to prioritize -- fix the hangs that the most users are hitting first

    Principles of Great Design (WWDC26 — 250)

    A ~16-minute design-philosophy talk. Not an API session — it's the why that should sit above every stack decision you make. Eight principles, in order, each building toward the last.

    Core thesis: Design is not aesthetics or behavior — it's creating with intention. Every feature spends users' time, attention, and trust. So deciding what not to build matters as much as what you do build.

    Session Timeline

    Timestamp Principle

    The eight are ordered deliberately: the first seven are the work; Delight is what falls out when you do the other seven well.

    The Eight Principles

    1. Purpose

    Design experiences that genuinely serve people; respect and adapt to their lives. - Before writing code, validate the feature has real purpose and delivers genuine value. - Clarity and thoughtful consideration over feature count. - Engineering tie-in: the cheapest, fastest, most bug-free feature is the one you correctly decide not to build. When a feature doesn't serve a real purpose, the highest-leverage engineering decision is to recognize that and not build it. The skill is in the judgment — distinguishing "this has no genuine purpose" (cut it) from "this is hard but valuable" (build it well).

    2. Agency

    Put people in control; let them explore at their own pace. Don't force predetermined paths. - Build forgiveness in: easy undo, confirmation for destructive actions, confidence to try without fear. - Engineering tie-in: undo/redo, non-destructive defaults, reversible/interruptible interactions (see interruptible interaction model — letting users grab and reverse a transition is agency expressed in motion).

    3. Responsibility

    Respect privacy as a fundamental human right. - Request only the permissions and data you actually need. - Anticipate harms — especially with AI features — and safeguard users from negative outcomes. - Remove features when risk outweighs value.

    4. Familiarity

    Let people apply existing knowledge to new designs. - Use metaphors that are neither too literal nor too abstract. - Follow established conventions for common actions. - Behavioral consistency: things that look similar should work similarly.

    5. Flexibility

    Adapt to different contexts, devices, and abilities. - Design for each platform's unique strengths (not lowest-common-denominator). - Consider the full range of audience capabilities (accessibility is flexibility). - Offer personalization when no single solution fits everyone.

    6. Simplicity

    Simplicity ≠ minimalism. It's about removing friction. - Concise language; strong visual hierarchy; clear affordances for discoverability. - Every element must earn its place. - Context can make an interface simpler by supplying clarity instead of more controls.

    7. Craft

    Uncompromising attention to detail builds trust. - Invest in beautiful typography, responsive animations, solid performance, and iterative refinement. - Great design has longevity — it evolves with platforms and user needs. - Engineering tie-in: this is where the whole stack-decisions and physics-of-motion discipline pays off. "Responsive animations" = picking the right rung so motion is 60/120fps and gesture-driven, not janky. See smooth 60/120fps.

    8. Delight

    The natural result of executing all the other principles well — not superficial embellishment ("confetti" is called out explicitly as the wrong idea of delight). - Identify the emotion users should feel. - Reinforce that emotion through every design decision. - Emerges from combining all principles with genuine care.

    Key Takeaways

    1. Intention over decoration — design is deciding what to build and what not to.
    2. Every feature has a cost in the user's time, attention, and trust — spend it deliberately.
    3. Delight is earned, not sprinkled on — it's the emergent payoff of purpose → craft, not animations bolted on at the end.
    4. Agency and Responsibility are first-class, not afterthoughts — undo, privacy, and AI-harm anticipation are design, not legal checkboxes.
    5. Craft is where engineering meets design — typography, motion, and performance are design decisions you implement.

    WWDC18 Session 803 -- Designing Fluid Interfaces

    "When you align the interface to the way we think and move, something kind of magical happens. It really stops feeling like a computer, and starts feeling like a seamless extension of us."

    Overview

    This session is widely regarded as one of the best WWDC talks ever delivered. Presented by three members of Apple's Human Interface team, it reveals the design philosophy and physics behind iPhone X's gesture-driven interface. Unlike a typical design talk, it goes deep into the math and engineering behind making interfaces feel alive -- covering spring physics, rubber banding, momentum projection, and interruptible animations.

    The central thesis: an interface is fluid when it behaves according to the way people think, not the way machines think. Fluid interfaces are responsive, interruptible, and redirectable.

    Speakers

    Speaker Section Focus
    Chan Karunamuni Design Principles Core philosophy, the eight properties of fluid interfaces
    Nathan de Vries Motion Design Spring physics, animation curves, momentum projection
    Marcos Alonso Gesture Design Tap, swipe, one-to-one tracking, multi-gesture handling

    Part 1: Interface Design Principles (Chan Karunamuni)

    Chan lays out eight properties that make an interface feel fluid:

    1. Responsiveness (Minimized Latency)

    • People are extremely sensitive to latency; even small delays break the illusion of direct manipulation.
    • Every interaction -- taps, swipes, presses -- must respond instantly.
    • Look for delays everywhere in the interaction chain.

    2. Constant Redirection and Interruption

    • Users should be able to redirect mid-gesture without waiting for an animation to finish.
    • Multi-axis gestures enable parallel thought and action.
    • Example: iPhone X's home gesture combines horizontal app switching with vertical home navigation simultaneously.
    • See: Animation Timing and UIPropertyAnimator for interruptible animation APIs.

    3. Spatial Consistency

    • Objects should enter and exit along symmetric, predictable paths that match real-world persistence.
    • Reinforces gesture expectations and aids discovery.
    • Example: iOS navigation -- slide right to push, slide right to pop. The object "lives" to the right.

    4. Direction Hinting

    • Intermediate transition states should grow toward the final state.
    • Leverages the human ability to predict motion trajectories.
    • Example: Control Center modules grow upward toward your finger as you press.

    5. Lightweight Interactions with Amplified Output

    • Short, effortless inputs should produce magnified results.
    • Preserve gesture energy/momentum through velocity analysis.
    • Example: Scrolling momentum -- a quick flick sends content flying.

    6. Soft Boundaries (Rubber Banding)

    • Gracefully indicate interface edges without harsh walls.
    • Content should never just "stop" -- it should communicate the boundary softly.
    • Maintains constant feedback with the user.
    • See: #Rubber Banding Formula below.

    7. Smooth Frame Motion

    • Prevent visual strobing through appropriate velocity and frame rate.
    • Higher frame rates enable faster on-screen movement.
    • Techniques: motion blur, motion stretching.
    • Example: iPhone X app launch animation stretches the icon to reduce strobing.

    8. Behavior Over Animation

    • Design dynamic systems that respond to users, not prescripted animation sequences.
    • Objects should have variable mass reflecting conceptual weight.
    • Example: Photos feel lighter than apps in interactions.

    Part 2: Motion Design (Nathan de Vries)

    Seamless Motion

    • Reference physical world behaviors (e.g., car braking deceleration curves).
    • Maintain unbroken motion trajectories even when behaviors change.
    • No perceptible stopping points -- content should always appear ready to move.

    Why Springs?

    Springs are the foundation of fluid motion because they model elasticity -- the natural tendency of materials to return to a resting state. Unlike duration-based animations, spring animations:

    • Are interruptible at any point
    • Preserve velocity when redirected
    • Produce natural-feeling deceleration
    • Never feel "stuck" waiting for a timeline to end

    See: CASpringAnimation for the Core Animation implementation.

    Spring Physics

    A spring system can be modeled as a ball attached to a spring. The ball's behavior is governed by three physical properties:

    Property Effect
    Mass Inertia of the object; heavier = slower to start/stop
    Stiffness Force applied toward the target; higher = snappier
    Damping Friction that slows the ball; controls overshoot

    The Problem with Raw Physics Parameters

    Mass, stiffness, and damping are hard for designers to reason about. The session introduces two designer-friendly parameters instead:

    Designer Parameter Physics Equivalent Description
    Damping (ratio) Damping ratio (zeta) Controls overshoot. 1.0 = critically damped (no overshoot). 0.0 = undamped (oscillates forever).
    Response Frequency response Controls speed to target. Replaces "duration" -- because springs don't have a fixed duration.

    Avoid the word "duration" when thinking about springs. Springs settle, they don't end.

    Spring Parameter Conversion Formulas

    UISpringTimingParameters requires mass, stiffness, damping, and initialVelocity. To convert from the designer-friendly dampingRatio and response:

    extension UISpringTimingParameters {
    /// Attempt to use designer-friendly damping/response instead of raw physics.
    ///
    /// - Parameters:
    ///   - damping: Damping ratio. 1.0 = no overshoot; < 1.0 = bouncy.
    ///   - response: How quickly the spring settles (seconds). ~0.3-0.5 typical.
    ///   - initialVelocity: Starting velocity (optional).
    convenience init(damping: CGFloat, response: CGFloat, initialVelocity: CGVector = .zero) {
        let stiffness = pow(2 * .pi / response, 2)
        let damp = 4 * .pi * damping / response
        self.init(mass: 1, stiffness: stiffness, damping: damp, initialVelocity: initialVelocity)
    }
}

    The underlying equations (mass normalized to 1):

    stiffness = (2 * pi / response)^2
damping_coefficient = 4 * pi * damping_ratio / response

    See also: SwiftUI Animation Basics -- SwiftUI's .spring(response:dampingFraction:blendDuration:) directly exposes these designer-friendly parameters, inspired by this very talk.

    Character Development -- Tuning Springs

    The session outlines a methodology for tuning spring behavior:

    1. Start at 100% damping (no overshoot) -- smooth, graceful, seamless motion that doesn't distract.
    2. Add bounciness when gestures carry momentum -- if a gesture has momentum and there's no overshoot, it feels broken or unsatisfying. Reward momentum with a little bounce.
    3. Use overshoot as a utility hint -- e.g., the Flashlight button on iPhone X uses pressure feedback with subtle bounce to communicate activation thresholds.
    4. Combine motion with sound and haptics -- motion, sound, and haptics should form a cohesive tactile experience.
    5. Maintain consistency -- all spring behaviors in an app should feel like a "family of behaviors."

    Momentum Projection Formula

    One of the most important formulas in the talk. Given a gesture's velocity, project where it would naturally come to rest using deceleration:

    /// Projects a position based on velocity and deceleration.
/// Uses UIScrollView's deceleration rate for familiarity with scrolling physics.
func project(value: CGFloat, velocity: CGFloat, decelerationRate: CGFloat = UIScrollView.DecelerationRate.normal.rawValue) -> CGFloat {
    return value + (velocity / 1000) * decelerationRate / (1 - decelerationRate)
}

    Or in simplified form:

    projected_position = current_position + velocity^2 / (2 * deceleration_rate)

    Why this matters: Use the projected endpoint (not just the current position) to decide what the gesture "intended." This is how iOS decides whether a swipe should dismiss a view, which corner a PiP window should snap to, etc.

    This formula works for: - Positions (linear) - Scales (pinch gestures) - Rotations - Any combination thereof

    Part 3: Gesture Design (Marcos Alonso)

    Tap Gesture

    • Highlight immediately on touch-down (instant feedback).
    • Confirm only on touch-up (not touch-down).
    • Create an extra margin around the button to prevent accidental cancellation.
    • Allow cancellation by dragging outside; reconfirm by dragging back in.
    • Use UIControl subclass (not UIButton) when customizing tap interactions.

    Swipe Gesture

    • Detect after a 10-point hysteresis distance to differentiate from taps.
    • Maintain one-to-one touch-content tracking during the gesture.
    • Track position history and velocity, not just the final position.
    • Direction determination follows the hysteresis threshold.

    One-to-One Tracking

    • The core iOS interaction principle: content follows the finger directly.
    • This breaks naturally at boundaries (rubber banding), communicating scroll limits.
    • Applies across all Apple platforms (including Apple TV's focus engine).

    Continuous Feedback

    • The UI should respond during the gesture, not just at completion.
    • Use gesture recognizer methods that provide full gesture data (position, velocity, pressure, touch size).
    • Avoid delayed-feedback gesture recognizers.

    See: SwiftUI Gestures for declarative gesture handling in SwiftUI.

    Multiple Gesture Handling

    • Detect all possible gestures in parallel from the start of an interaction.
    • Cancel non-matching gestures once intent becomes clear.
    • Avoid delays when possible (double-tap is an exception -- necessitates ~500ms delay).

    Rubber Banding Formula

    Rubber banding is the soft resistance you feel when scrolling past the edge of content. The formula used by UIScrollView:

    /// Attempt to apply rubber banding to a value that has exceeded its bounds.
///
/// - Parameters:
///   - offset: How far past the boundary the value has gone.
///   - dimension: The size of the scrollable area (width or height).
///   - coefficient: Resistance factor. UIScrollView uses 0.55.
/// - Returns: The rubber-banded offset (always less than the raw offset).
func rubberBand(offset: CGFloat, dimension: CGFloat, coefficient: CGFloat = 0.55) -> CGFloat {
    return (coefficient * abs(offset) * dimension) / (dimension + coefficient * abs(offset))
}

    The math:

    f(x, d, c) = (x * d * c) / (d + c * x)

    Where: - x = distance past the boundary (offset) - d = dimension of the content area (width or height) - c = coefficient (UIScrollView uses 0.55)

    This is a logarithmic decay curve: the further you drag past the edge, the more resistance you feel. It asymptotically approaches the dimension value but never reaches it.

    Simpler alternative (less accurate but useful for quick prototyping):

    // Power-based rubber banding
let rubberBanded = pow(offset, 0.7)

    Use an exponent between 0 and 1. Larger exponent = less resistance; smaller = more resistance.

    See: Math for Shaders -- the same logarithmic/asymptotic curves appear in shader math for clamping and easing.

    Interruptible Animations with UIPropertyAnimator

    The session advocates for UIPropertyAnimator as the primary animation tool because it supports:

    • Interruption: Pause, reverse, or redirect animations mid-flight.
    • Scrubbing: Map gesture progress to animation fraction (fractionComplete).
    • Spring timing: Use UISpringTimingParameters for physics-based motion.
    • Velocity preservation: When interrupted, the animator preserves the object's current velocity, preventing jarring stops.
    let animator = UIPropertyAnimator(
    duration: 0,  // Ignored for spring timing
    timingParameters: UISpringTimingParameters(damping: 0.8, response: 0.4)
)
animator.isInterruptible = true
animator.addAnimations {
    view.center = targetPoint
}
animator.startAnimation()

    See: Animation Timing and Advanced SwiftUI Animation for more on interruptible animation patterns.

    The session covers strategies for discoverability in gesture-heavy UIs:

    Strategy Description Example
    Visual cues Clip content to indicate scrollable areas; use paging indicators Partially visible card edges
    Elevation Place interactive elements on visually separate planes Floating action buttons
    Behavioral teaching Reinforce discrete animations with gesture parallels Safari tab close animation mirrors swipe-to-delete
    Explanations Sparse, upfront guidance for repeated gestures First-launch coach marks

    Key Takeaways

    1. Latency is the enemy of fluidity. Every millisecond counts. Optimize the entire touch-to-pixel pipeline.
    2. Springs, not durations. Replace UIView.animate(withDuration:) with spring-based animations using UIPropertyAnimator + UISpringTimingParameters. Springs are interruptible, velocity-preserving, and feel natural.
    3. Project momentum, don't just read position. Use the projection formula to determine user intent, not just where their finger currently is.
    4. Rubber band at boundaries instead of hard-stopping. Soft boundaries communicate limits without breaking the interaction flow.
    5. Design behaviors, not animations. A fluid interface is a dynamic system that responds to real-time input, not a sequence of canned transitions.
    6. Reward momentum with bounce. If a gesture carries velocity, let the UI overshoot slightly. Critically-damped springs for precise gestures; underdamped for energetic ones.
    7. Keep your finger on the content. One-to-one tracking is the foundation. The user should feel like they're directly manipulating the object.
    8. Fun is an emergent property. When responsiveness, interruption, motion, and gesture design all align, the interface becomes genuinely enjoyable to use.
