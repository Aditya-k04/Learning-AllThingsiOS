View controllers, the responder chain, and UIScrollView / UICollectionView internals — the substrate under SwiftUI & Core Animation.

    UIKit Internals — Overview

    SwiftUI is the future, but it is a thin declarative skin over machinery that is still pure UIKit and Core Animation underneath. Every ScrollView, every navigation push, every gesture you write in SwiftUI ultimately becomes a UIScrollView, a UINavigationController transition, or a UIGestureRecognizer. When an interaction misbehaves — a scroll that fights your gesture, a cell that recycles wrong, a hit-test that misses — the answer is almost always one layer down, in UIKit.

    This course is the substrate. It explains how UIKit actually works so that the fluid interactions you build on top behave predictably.

    The mental model

    UIKit is built from three intertwined systems:

    The view hierarchy decides what is drawn and where. The responder chain decides who handles events. The run loop decides when any of it happens.

    Master those three and almost every UIKit mystery dissolves.

    What you'll learn

    1. 01-view-controller-lifecycle — the exact order of viewDidLoad → viewWillAppear → layout → viewDidAppear, and why putting work in the wrong one causes bugs.
    2. 02-responder-chain — how touches and actions travel from a view up through controllers, and how target/action and first responder really resolve.
    3. 03-uiscrollview-mechanics — contentOffset, contentInset, paging, deceleration, and the delegate callbacks that drive scroll-linked effects.
    4. 04-compositional-layout — UICollectionViewCompositionalLayout: items, groups, sections, and fractional sizing.
    5. 05-diffable-data-sources — snapshot-based updates that animate correctly and end an entire class of crashes.
    6. 06-uikit-swiftui-interop — UIViewRepresentable, UIHostingController, and how to drop down to UIKit from SwiftUI when you must.

    View Controller Lifecycle

    A UIViewController is a coordinator between a chunk of screen (its view) and the rest of your app. Its lifecycle methods fire in a precise order, and the single most common UIKit bug is doing the right work in the wrong method.

    The order of events

    init(coder:) / init(nibName:bundle:)   // object created, view NOT loaded yet
loadView()                              // creates self.view (rarely overridden)
viewDidLoad()                           // view exists; runs ONCE
viewWillAppear(_:)                      // about to become visible; runs EVERY time
viewWillLayoutSubviews()                // before layout pass
viewDidLayoutSubviews()                 // after layout — frames are now valid
viewDidAppear(_:)                       // fully on screen; animations safe here
viewWillDisappear(_:) / viewDidDisappear(_:)

    The two distinctions that matter most:

    • viewDidLoad runs once; viewWillAppear runs every time the view appears. Set up subviews and one-time configuration in viewDidLoad. Refresh data that may have changed (e.g. after returning from a pushed screen) in viewWillAppear.
    • Frames are not valid until viewDidLayoutSubviews. In viewDidLoad, view.bounds is often still the size from the nib/storyboard, not the real device size. Never compute positions from bounds in viewDidLoad.

    Why "don't read frames in viewDidLoad" matters

    override func viewDidLoad() {
    super.viewDidLoad()
    // WRONG: bounds may be 0,0,0,0 or a placeholder size
    gradient.frame = view.bounds
}

override func viewDidLayoutSubviews() {
    super.viewDidLayoutSubviews()
    // RIGHT: bounds now reflects the actual laid-out size
    gradient.frame = view.bounds
}

    viewDidLayoutSubviews can fire many times (rotation, keyboard, trait changes), so keep it cheap and idempotent — assign frames, don't allocate.

    Appearance is not loading

    A controller's view can be loaded (in memory) without being visible. isViewLoaded tells you the first; viewIfLoaded lets you touch the view without forcing it to load. Accessing self.view triggers loadView immediately — a subtle cause of "why did my view load so early?"

    Containment

    Custom container controllers (carousels, custom tab bars) must keep the parent/child relationship correct or child lifecycle methods won't fire:

    addChild(child)
view.addSubview(child.view)
child.view.frame = containerFrame
child.didMove(toParent: self)
// removal mirrors this: willMove(toParent: nil); removeFromSuperview; removeFromParent()

    Skipping these calls is why a child's viewWillAppear or rotation callbacks silently never run.

    The Responder Chain

    Every interactive object in UIKit — views, view controllers, the app itself — is a UIResponder. Events (touches, presses, motion, remote-control) travel along a chain of responders until something handles them. Understanding the chain explains hit-testing, becomeFirstResponder, target/action with nil, and the keyboard.

    Two phases: find, then bubble

    Phase 1 — hit-testing (top down). When a touch lands, UIKit walks the view hierarchy from the window downward to find the deepest view that contains the point and is eligible:

    hitTest(_:with:) → recursively calls pointInside(_:with:) on subviews

    A view is skipped if isHidden, alpha < 0.01, or isUserInteractionEnabled == false. This is why a button "doesn't work" when an invisible overlay sits on top with interaction enabled, or why a subview drawn outside its parent's bounds can't be tapped (clipsToBounds doesn't matter — hit-testing does).

    Phase 2 — the responder chain (bottom up). Once the hit view is found, if it doesn't handle the event it passes it to its next responder:

    view → superview → … → view controller → window → UIApplication → AppDelegate

    The chain in practice

    // A view's next responder is usually its superview;
// the top view's next responder is its view controller.
override var next: UIResponder? { ... }  // UIKit wires this for you

    target: nil actions ride this chain. This is the trick behind UIMenuController, UIKeyCommand, and sendAction:

    // Sends up the chain until SOMEONE implements save(_:)
UIApplication.shared.sendAction(#selector(MyResponder.save(_:)), to: nil, from: self, for: nil)

    Any responder that implements save(_:) handles it. That's how the system "find" command or copy/paste reaches whatever is appropriate without hard wiring.

    First responder

    The first responder is the object that receives keyboard and text input. Calling becomeFirstResponder() on a text field is what shows the keyboard; resignFirstResponder() dismisses it. canBecomeFirstResponder gates it.

    textField.becomeFirstResponder()   // show keyboard, focus this field
view.endEditing(true)              // resign whoever is first responder

    Custom views can become first responder too — that's how you build a view that accepts key commands or shows an input accessory.

    Why this matters for fluid UI

    Hit-testing is where gesture conflicts begin. If a scroll view eats a touch your custom control wanted, you override hitTest or point(inside:) to expand/shrink the touch target, or use UIGestureRecognizerDelegate (next-level control, see Interaction Math for the geometry).

    UIScrollView Mechanics

    UIScrollView is the most important view in UIKit. UITableView, UICollectionView, UITextView, and SwiftUI's ScrollView are all built on it. Scroll-linked animations — parallax, sticky headers, the stretchy hero image, the wavy carousel — are all just functions of contentOffset.

    The core geometry

    contentSize    – the size of the scrollable content (bigger than the bounds)
contentOffset  – how far scrolled: the top-left of the visible window
contentInset   – padding around the content (safe areas, pull-to-refresh)
bounds.size    – the visible viewport

    A scroll view doesn't move its content. It moves its own bounds.origin — which is contentOffset. Subviews are drawn relative to bounds, so changing the origin slides everything. This is the same bounds-origin trick from Layer Geometry.

    contentOffset drives effects

    The key realization: a scroll effect is f(contentOffset), computed every frame the user scrolls — not an animation. This is category-1 from Who Drives Time.

    func scrollViewDidScroll(_ sv: UIScrollView) {
    let y = sv.contentOffset.y
    // Stretchy header: when pulled past the top (negative offset), grow the image.
    if y < 0 {
        let scale = 1 + (-y / header.bounds.height)
        header.transform = CGAffineTransform(scaleX: scale, y: scale)
    }
    // Parallax: move a background slower than the scroll.
    background.transform = CGAffineTransform(translationX: 0, y: y * 0.5)
}

    scrollViewDidScroll fires on every offset change — keep it cheap (no allocations, no layout).

    The delegate timeline of a swipe

    scrollViewWillBeginDragging
scrollViewDidScroll            // repeatedly, while finger moves
scrollViewWillEndDragging(_:withVelocity:targetContentOffset:)   // finger lifts
scrollViewDidEndDragging(_:willDecelerate:)
scrollViewDidScroll            // repeatedly, during deceleration
scrollViewDidEndDecelerating

    scrollViewWillEndDragging is the powerful one: you receive the velocity and a mutable targetContentOffset pointer. Rewrite that pointer to implement custom paging or snap-to-item:

    func scrollViewWillEndDragging(_ sv: UIScrollView, withVelocity v: CGPoint,
                               targetContentOffset t: UnsafeMutablePointer<CGPoint>) {
    let page = (t.pointee.x / itemWidth).rounded()
    t.pointee.x = page * itemWidth   // snap to the nearest card
}

    Paging, insets, and the safe area

    isPagingEnabled snaps in viewport-sized steps. For custom card widths, leave it off and snap via targetContentOffset above. contentInsetAdjustmentBehavior controls whether the system adds insets for the nav bar / safe area — a frequent source of "my content starts 100pt too low."

    Compositional Layout

    UICollectionViewCompositionalLayout (iOS 13+) replaced the pain of custom UICollectionViewLayout subclasses for 90% of cases. It is built from a strict, composable hierarchy — once you internalize it, almost any grid, list, carousel, or magazine layout becomes a few lines.

    The hierarchy: Item → Group → Section

    An item is one cell. A group arranges items along one axis. A section stacks groups. A layout holds sections.

    // Item: half the group's width, full height
let item = NSCollectionLayoutItem(layoutSize: .init(
    widthDimension: .fractionalWidth(0.5),
    heightDimension: .fractionalHeight(1.0)))
item.contentInsets = .init(top: 4, leading: 4, bottom: 4, trailing: 4)

// Group: full width, fixed height, lays items horizontally
let group = NSCollectionLayoutGroup.horizontal(layoutSize: .init(
    widthDimension: .fractionalWidth(1.0),
    heightDimension: .absolute(180)),
    subitems: [item])     // → two items per row

let section = NSCollectionLayoutSection(group: group)
let layout = UICollectionViewCompositionalLayout(section: section)

    Fractional sizing is the superpower

    Dimensions come in three flavors:

    • .fractionalWidth(0.5) / .fractionalHeight(1.0) — relative to the container (group or section).
    • .absolute(180) — fixed points.
    • .estimated(120) — a starting guess that self-sizes to Auto Layout content (great for text).

    Because items size fractionally to their group, the same layout reflows correctly across screen sizes and orientations with no math on your part.

    Per-section layouts

    The real power: each section can have a completely different layout via the section provider, so a single collection view can mix a horizontal carousel, a grid, and a list:

    let layout = UICollectionViewCompositionalLayout { sectionIndex, env in
    switch sectionIndex {
    case 0: return Self.carouselSection()   // orthogonal scrolling
    default: return Self.gridSection()
    }
}

    Orthogonal scrolling (carousels for free)

    Set the section's orthogonalScrollingBehavior and the section scrolls horizontally inside a vertical collection view — the App Store's shelves, with no nested scroll views:

    section.orthogonalScrollingBehavior = .groupPagingCentered

    Supplementary items

    Headers, footers, and badges are NSCollectionLayoutBoundarySupplementaryItem (pinned headers via pinToVisibleBounds = true) and NSCollectionLayoutSupplementaryItem (badges anchored to an item corner).

    Try it

    Adjust items-per-group, group count, spacing, and axis below to see how fractional sizing reflows the section.

    Diffable Data Sources

    The classic UICollectionViewDataSource / UITableViewDataSource API made you responsible for keeping your model and the view in sync via performBatchUpdates. Get the counts wrong and you crash: "invalid number of items." Diffable data sources (iOS 13+) flip the model: you describe the desired state as a snapshot, and UIKit computes the diff and animates it.

    Snapshots describe state, not changes

    enum Section { case main }

let dataSource = UICollectionViewDiffableDataSource<Section, Item.ID>(
    collectionView: cv) { cv, indexPath, id in
        let cell = cv.dequeueConfiguredReusableCell(using: reg, for: indexPath, item: id)
        return cell
    }

func apply(_ items: [Item], animate: Bool = true) {
    var snapshot = NSDiffableDataSourceSnapshot<Section, Item.ID>()
    snapshot.appendSections([.main])
    snapshot.appendItems(items.map(\.id))
    dataSource.apply(snapshot, animatingDifferences: animate)
}

    You never call insertItems / deleteItems. You hand over the new truth; UIKit figures out what moved, inserted, or deleted and animates each correctly.

    Identifiers must be Hashable and stable

    The diff is computed on identifiers, not objects. Use a stable ID (not array index), and make sure equality is meaningful:

    Store item identifiers in the snapshot, keep the full objects in your store, and look them up in the cell provider. This avoids re-rendering everything when one field changes.

    If two different items hash equal, the diff misbehaves (cells vanish or duplicate) — the single most common diffable bug.

    Cell registration replaces dequeue-by-string

    let reg = UICollectionView.CellRegistration<UICollectionViewListCell, Item.ID> { cell, _, id in
    var config = cell.defaultContentConfiguration()
    config.text = store[id]?.title
    cell.contentConfiguration = config
}

    No more reuse-identifier strings, no force casts. UIContentConfiguration is the modern, value-type way to fill a cell — the same configuration objects work for table and collection list cells.

    Reloading vs reconfiguring

    reconfigureItems (iOS 15+) updates a visible cell in place without replacing it — cheaper and flicker-free for content changes where identity is unchanged. Use it instead of reloadItems when only the data inside a cell changed.

    Why it matters

    Diffable data sources eliminate an entire class of crashes and give you correct, free animations — which is exactly the "describe the state, let the system interpolate" philosophy you see in SwiftUI and in the dependency graph.

    UIKit ↔ SwiftUI Interop

    You rarely rewrite an app wholesale. Real apps are hybrids: a SwiftUI screen hosting a MKMapView, or a UIKit app embedding a new SwiftUI feature. Three bridges connect the two worlds, and knowing which to reach for keeps the boundary clean.

    SwiftUI hosting UIKit: UIViewRepresentable

    Wrap any UIView so SwiftUI can place it:

    struct BlurView: UIViewRepresentable {
    func makeUIView(context: Context) -> UIVisualEffectView {
        UIVisualEffectView(effect: UIBlurEffect(style: .systemMaterial))
    }
    func updateUIView(_ view: UIVisualEffectView, context: Context) {
        // called whenever SwiftUI state this view depends on changes
    }
}
    • makeUIView runs once to create the view.
    • updateUIView runs every time relevant SwiftUI state changes — push new values into UIKit here.
    • For UIViewController, use UIViewControllerRepresentable (same shape: makeUIViewController / updateUIViewController).

    Talking back: the Coordinator

    UIKit communicates via delegates and target/action, which are reference-based. The Coordinator is the bridge object that adopts those delegates and forwards into SwiftUI:

    func makeCoordinator() -> Coordinator { Coordinator(self) }

class Coordinator: NSObject, UIScrollViewDelegate {
    let parent: MyRepresentable
    init(_ parent: MyRepresentable) { self.parent = parent }
    func scrollViewDidScroll(_ sv: UIScrollView) {
        parent.offset = sv.contentOffset.y   // push UIKit event → SwiftUI binding
    }
}

    UIKit hosting SwiftUI: UIHostingController

    The reverse bridge. UIHostingController is a UIViewController whose body is a SwiftUI view — drop it into a navigation stack, a container, or as a child:

    let host = UIHostingController(rootView: ProfileView(user: user))
navigationController?.pushViewController(host, animated: true)

    For embedding SwiftUI inside a UIKit cell or view, add host.view as a subview and wire up containment (see 01-view-controller-lifecycle). On iOS 16+, UIHostingConfiguration lets a collection/table cell be SwiftUI directly — the cleanest hybrid path.

    The sizing gotcha

    The classic friction is sizing: a UIHostingController doesn't always report its SwiftUI content's intrinsic size to UIKit Auto Layout. Set host.sizingOptions = .intrinsicContentSize (iOS 16+) or pin constraints explicitly. Conversely, a UIViewRepresentable should implement sizeThatFits (iOS 16+) so SwiftUI's layout system can measure it.

    When to drop down

    Reach for UIKit from SwiftUI when you need: precise scroll/gesture control SwiftUI doesn't expose, a mature UIKit-only view (MKMapView, PHPicker, AVPlayerViewController), or text editing power (UITextView/TextKit). Everything else stays SwiftUI.
