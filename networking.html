    Networking — Overview

    Almost every app talks to a server. Apple's URLSession is a capable, modern HTTP stack — with async/await it's genuinely pleasant, and you rarely need a third-party library like Alamofire for typical REST work. This course builds a clean, testable networking layer from the primitives up.

    What you'll learn

    1. 01-urlsession — the session, requests, configurations, and the response model.
    2. 02-async-await-requests — modern request code, cancellation, and a typed request layer.
    3. 03-codable — modelling JSON with Codable, keys, dates, and nested/odd shapes.
    4. 04-errors-and-retries — a real error taxonomy, status-code handling, retries and backoff.
    5. 05-caching-auth-and-testing — URLCache, auth tokens, and testing the layer with URLProtocol.

    The mental model

    A request is a value (URLRequest), a session is the engine (URLSession), and a response is data + metadata ((Data, URLResponse)). Your job is to turn a typed request into a typed model, handling failure honestly at every step.

    The shape of a good networking layer

    Endpoint (typed)  →  URLRequest  →  URLSession  →  (Data, HTTPURLResponse)
        ↓                                                    ↓
   decode/encode  ←——————————  validate status  ←———————————┘
        ↓
   typed Model  /  typed Error

    Keep the transport (sending bytes) separate from modelling (Codable) and policy (retries, auth). That separation is what makes the layer testable.

    How it connects

    Networking is almost entirely async, so it leans on the async/await model and structured cancellation. The decoded models often flow into SwiftData for offline use, and the whole layer is a prime target for the dependency injection in App Architecture so you can swap in a fake for tests.

    URLSession

    URLSession is the engine. It manages a pool of connections, applies a configuration (caching, timeouts, headers), and hands you data via tasks. Understanding its pieces means you almost never reach for a third-party HTTP library.

    Sessions and configurations

    // .shared is fine for simple cases; a custom session lets you set policy.
let config = URLSessionConfiguration.default
config.timeoutIntervalForRequest = 15
config.httpAdditionalHeaders = ["Accept": "application/json"]
config.requestCachePolicy = .useProtocolCachePolicy
config.waitsForConnectivity = true            // queue until network is available
let session = URLSession(configuration: config)

    Three configuration flavors: .default (disk-cached, credential storage), .ephemeral (nothing persisted — "private" sessions), and .background (uploads/downloads that continue after the app is suspended, delivered to your app on relaunch).

    Building a request

    A URLRequest is a value you configure:

    var req = URLRequest(url: URL(string: "https://api.example.com/users/42")!)
req.httpMethod = "POST"
req.setValue("application/json", forHTTPHeaderField: "Content-Type")
req.httpBody = try JSONEncoder().encode(payload)

    Build URLs safely with URLComponents so query items are percent-encoded correctly:

    var comps = URLComponents(string: "https://api.example.com/search")!
comps.queryItems = [URLQueryItem(name: "q", value: "swift ui"),
                    URLQueryItem(name: "page", value: "2")]
let url = comps.url!     // …/search?q=swift%20ui&page=2

    The response is data + metadata

    A request returns (Data, URLResponse). The URLResponse is really an HTTPURLResponse carrying the status code and headers — and a non-2xx status is not an error as far as URLSession is concerned. A 404 or 500 still "succeeds" at the transport level; you must check the code yourself (covered in 04-errors-and-retries):

    let (data, response) = try await session.data(for: req)
guard let http = response as? HTTPURLResponse else { throw NetError.nonHTTP }
guard (200..<300).contains(http.statusCode) else { throw NetError.status(http.statusCode) }

    Task types

    dataTask (in-memory Data, the common case), uploadTask (large bodies/files, progress), and downloadTask (streams to a file on disk — use for large payloads to avoid loading them into RAM). With async/await you mostly use session.data(for:), session.upload(for:from:), and session.download(for:).

    Async/Await Requests

    URLSession's async methods turn callback spaghetti into linear, readable code that supports cancellation for free. This lesson builds a small, typed request layer you'd actually ship.

    The basic call

    func fetchUser(id: Int) async throws -> User {
    let url = URL(string: "https://api.example.com/users/\(id)")!
    let (data, response) = try await URLSession.shared.data(from: url)
    guard let http = response as? HTTPURLResponse, (200..<300).contains(http.statusCode)
    else { throw NetError.badStatus }
    return try JSONDecoder().decode(User.self, from: data)
}

    No completion handlers, no nesting, errors propagate with throw. Call it from a Task or another async function.

    A typed endpoint layer

    Don't scatter URL strings across the app. Describe endpoints as values and have one function execute them — this is the seam you'll later mock for tests and inject per dependency injection:

    struct Endpoint<Response: Decodable> {
    var path: String
    var method = "GET"
    var query: [URLQueryItem] = []
    var body: Data? = nil
}

protocol APIClient { func send<R>(_ e: Endpoint<R>) async throws -> R }

struct LiveAPIClient: APIClient {
    let base = URL(string: "https://api.example.com")!
    let session = URLSession.shared
    func send<R>(_ e: Endpoint<R>) async throws -> R {
        var comps = URLComponents(url: base.appending(path: e.path), resolvingAgainstBaseURL: false)!
        comps.queryItems = e.query.isEmpty ? nil : e.query
        var req = URLRequest(url: comps.url!)
        req.httpMethod = e.method; req.httpBody = e.body
        let (data, resp) = try await session.data(for: req)
        try validate(resp)                       // status-code check from lesson 04
        return try JSONDecoder.api.decode(R.self, from: data)
    }
}

// Usage reads like the domain, not like HTTP:
let user: User = try await api.send(Endpoint(path: "/users/42"))

    Cancellation is structured

    An async request is tied to its Task. Cancel the task (or let it cancel when its scope ends) and the underlying request is cancelled — no manual URLSessionTask bookkeeping:

    let task = Task { try await api.send(searchEndpoint(query)) }
// later, e.g. when the user types again:
task.cancel()        // throws CancellationError / URLError.cancelled into the awaiter

    This is why search-as-you-type is trivial with async/await: cancel the in-flight task on each keystroke. It builds directly on structured concurrency.

    Parallel requests

    Need several resources at once? async let runs them concurrently and awaits together — total time is the slowest request, not the sum:

    async let profile = api.send(profileEndpoint)
async let posts   = api.send(postsEndpoint)
let screen = try await ProfileScreen(profile: profile, posts: posts)

    Codable

    Codable (= Encodable & Decodable) is Swift's built-in serialization. For JSON APIs it turns bytes into typed models with almost no boilerplate — and gives you precise escape hatches when the JSON is messy.

    The happy path

    struct User: Codable, Identifiable {
    let id: Int
    let name: String
    let email: String
}
let user = try JSONDecoder().decode(User.self, from: data)

    If property names match the JSON keys and types line up, that's the whole story.

    Key mismatches: keyDecodingStrategy

    APIs love snake_case. Rather than annotate every property, convert globally:

    let decoder = JSONDecoder()
decoder.keyDecodingStrategy = .convertFromSnakeCase   // created_at → createdAt

    For one-off renames, CodingKeys:

    struct Post: Codable {
    let id: Int
    let bodyText: String
    enum CodingKeys: String, CodingKey {
        case id
        case bodyText = "content"     // JSON "content" → bodyText
    }
}

    Dates and other strategies

    decoder.dateDecodingStrategy = .iso8601                 // "2026-06-09T12:00:00Z"
// or a custom formatter for non-standard dates:
decoder.dateDecodingStrategy = .formatted(myFormatter)

    Centralize these in one configured decoder (JSONDecoder.api) and reuse it everywhere so every model decodes consistently.

    Messy JSON: custom init(from:)

    When the shape doesn't map cleanly — a field that's sometimes a string and sometimes a number, a flattened nested object, defaults for missing keys — implement the decoder yourself:

    struct Product: Decodable {
    let id: Int
    let price: Double
    let inStock: Bool

    init(from decoder: Decoder) throws {
        let c = try decoder.container(keyedBy: CodingKeys.self)
        id = try c.decode(Int.self, forKey: .id)
        // price may arrive as "12.99" (string) or 12.99 (number)
        if let d = try? c.decode(Double.self, forKey: .price) { price = d }
        else { price = Double(try c.decode(String.self, forKey: .price)) ?? 0 }
        // default when key is absent
        inStock = (try? c.decode(Bool.self, forKey: .inStock)) ?? true
    }
    enum CodingKeys: String, CodingKey { case id, price, inStock }
}

    Decoupling API shape from your domain

    A useful pattern: decode a …DTO that mirrors the JSON exactly, then map it to a clean domain model. Your app code never depends on the API's quirks, and when the API changes you fix one mapping function. This DTO→domain boundary pairs with clean data flow.

    Errors & Retries

    Networking fails constantly — flaky connections, expired tokens, server hiccups, rate limits. Production-grade code treats failure as a first-class path, with a clear error taxonomy and a retry policy that doesn't make things worse.

    A real error taxonomy

    Collapse the many failure modes into a domain error your UI can reason about:

    enum NetError: Error {
    case offline                 // URLError.notConnectedToInternet
    case timedOut
    case cancelled               // user navigated away — usually silent
    case badStatus(Int, Data?)   // non-2xx; keep the body for server messages
    case decoding(Error)         // shape mismatch — a bug, not a transient failure
    case unauthorized            // 401 — trigger token refresh
    case rateLimited(retryAfter: TimeInterval?)  // 429
    case server                  // 5xx — retryable
}

    Map raw failures into it:

    func classify(_ error: Error) -> NetError {
    if let u = error as? URLError {
        switch u.code {
        case .notConnectedToInternet, .dataNotAllowed: return .offline
        case .timedOut: return .timedOut
        case .cancelled: return .cancelled
        default: return .server
        }
    }
    return .server
}

    Status codes carry meaning

    func validate(_ resp: URLResponse, _ data: Data) throws {
    guard let http = resp as? HTTPURLResponse else { throw NetError.server }
    switch http.statusCode {
    case 200..<300: return
    case 401:       throw NetError.unauthorized
    case 429:       throw NetError.rateLimited(retryAfter:
                        http.value(forHTTPHeaderField: "Retry-After").flatMap(TimeInterval.init))
    case 400..<500: throw NetError.badStatus(http.statusCode, data)
    default:        throw NetError.server
    }
}

    Retry only what's retryable — with backoff

    Retrying a 400 (your bad request) is pointless; retrying a 503 or a timeout is sensible. Use exponential backoff with jitter so a recovering server isn't hammered by every client at once:

    func withRetry<T>(max: Int = 3, _ op: @escaping () async throws -> T) async throws -> T {
    var attempt = 0
    while true {
        do { return try await op() }
        catch let e as NetError {
            attempt += 1
            let retryable = { if case .server = e { return true }
                              if case .timedOut = e { return true }
                              if case .rateLimited = e { return true }; return false }()
            guard retryable, attempt < max else { throw e }
            let base = pow(2.0, Double(attempt))            // 2,4,8 s
            let jitter = Double.random(in: 0...0.5)
            try await Task.sleep(for: .seconds(base + jitter))
        }
    }
}

    For 429, honor the server's Retry-After instead of your own backoff.

    Surface failure honestly

    In the UI, distinguish offline (show a retry affordance), cancelled (show nothing — expected), decoding (a bug — log it, don't silently swallow), and server (transient — retry then apologize). Mapping each NetError case to a specific UI state is what makes an app feel robust rather than mysteriously broken.

    Caching, Auth & Testing

    Three pieces that turn a working request into a production layer: caching (speed + offline), authentication (tokens done right), and testing (proving it works without hitting a real server).

    Caching with URLCache

    URLSession has an HTTP cache out of the box. If the server sends proper Cache-Control/ETag headers, responses are cached and revalidated automatically:

    URLCache.shared = URLCache(memoryCapacity: 16_000_000, diskCapacity: 200_000_000)
config.requestCachePolicy = .useProtocolCachePolicy   // respect server headers (default)

    For app-level caching (offline browsing, instant reloads), persist decoded models into SwiftData and treat the network as a refresh — the "read from store, revalidate in background" pattern.

    Authentication tokens

    The common scheme: a short-lived access token sent on every request, and a long-lived refresh token used to get a new access token when it expires (401). The hard part is doing the refresh once even when many requests fail at the same moment:

    actor TokenProvider {
    private var token: String?
    private var refreshTask: Task<String, Error>?

    func valid() async throws -> String {
        if let t = token { return t }
        // coalesce concurrent refreshes into one in-flight task
        if let task = refreshTask { return try await task.value }
        let task = Task { try await refresh() }
        refreshTask = task
        defer { refreshTask = nil }
        let t = try await task.value; token = t; return t
    }
}

    Using an actor here is exactly the data-isolation tool from actors — it serializes access so two requests can't kick off two refreshes. Never store tokens in UserDefaults; use the Keychain (shipping covers secure storage alongside it).

    Testing without a server: URLProtocol

    The cleanest way to test networking is to intercept requests with a custom URLProtocol and return canned responses — fast, deterministic, no network:

    final class MockURLProtocol: URLProtocol {
    static var handler: ((URLRequest) throws -> (HTTPURLResponse, Data))?
    override class func canInit(with: URLRequest) -> Bool { true }
    override class func canonicalRequest(for r: URLRequest) -> URLRequest { r }
    override func startLoading() {
        guard let h = Self.handler else { return }
        do {
            let (resp, data) = try h(request)
            client?.urlProtocol(self, didReceive: resp, cacheStoragePolicy: .notAllowed)
            client?.urlProtocol(self, didLoad: data)
            client?.urlProtocolDidFinishLoading(self)
        } catch { client?.urlProtocol(self, didFailWithError: error) }
    }
    override func stopLoading() {}
}

// In the test:
let config = URLSessionConfiguration.ephemeral
config.protocolClasses = [MockURLProtocol.self]
let session = URLSession(configuration: config)
MockURLProtocol.handler = { _ in (HTTPURLResponse(/*200*/), userJSON) }

    Alternatively, hide the network behind the APIClient protocol from 02-async-await-requests and inject a fake implementation — usually simpler for unit tests. Both approaches are the testing seam covered in unit testing patterns.
