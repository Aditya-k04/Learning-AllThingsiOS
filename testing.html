Testing & Instruments
Swift Testing & XCTest, unit/snapshot/UI tests, and profiling hangs and performance with Instruments.

    Testing & Instruments — Overview

    Tests let you change code without fear; Instruments lets you find out why something is slow instead of guessing. Together they're what turn a working prototype into something you can maintain and ship with confidence.

    What you'll learn

    1. 01-xctest-and-swift-testing — the two test frameworks: classic XCTest and the modern Swift Testing.
    2. 02-unit-testing-patterns — what to test, the arrange-act-assert shape, and testing async code.
    3. 03-snapshot-and-ui-tests — verifying rendered UI and end-to-end flows.
    4. 04-instruments — the profiler: Time Profiler, Allocations, Core Animation, and more.
    5. 05-performance-and-hangs — diagnosing hangs, hitches, and memory growth.

    The testing pyramid

            ╱ UI tests ╲          few — slow, brittle, full end-to-end flows
      ╱ integration ╲         some — modules working together
    ╱   unit tests    ╲       many — fast, isolated, the foundation

    Write mostly unit tests (fast, pure logic), some integration tests, and a few UI tests for critical flows. Inverting this — lots of slow UI tests, few unit tests — gives slow, flaky suites that no one trusts.

    What makes code testable

    Tests reward the architecture from the previous course: logic in ViewModels/reducers (not views), dependencies injected so you can substitute fakes (DI), and side effects pushed to the edges. If something is hard to test, that's usually a design smell, not a testing problem.

    How it connects

    The seams you test were built in networking (mockable client) and unidirectional flow (pure reducers). The performance half pairs with the WWDC sessions on eliminating hangs and the performance survival guide.

    XCTest & Swift Testing

    iOS has two test frameworks now. XCTest is the long-standing one (still used for UI and performance tests). Swift Testing (introduced 2024) is the modern, expressive successor for unit tests. Know both — you'll see XCTest in every existing codebase and Swift Testing in new ones.

    Swift Testing (the modern way)

    Macro-based, expressive, and parallel by default:

    import Testing
@testable import MyApp

@Test func parsesISODate() throws {
    let date = try DateParser.parse("2026-06-09T12:00:00Z")
    #expect(date.timeIntervalSince1970 == 1_780_315_200)
}

@Test("rejects malformed input")
func rejectsGarbage() {
    #expect(throws: ParseError.self) { try DateParser.parse("nonsense") }
}
    • @Test marks a function; no test prefix needed, and tests can have readable names.
    • #expect(...) is one assertion macro for everything; on failure it shows the actual sub-values (#expect(a == b) prints both a and b).
    • #require(...) is like #expect but stops the test if it fails (use to safely unwrap before continuing).

    Parameterized tests

    One test, many inputs — no copy-paste:

    @Test(arguments: [("", false), ("a@b.co", true), ("nope", false)])
func validatesEmail(_ input: String, _ expected: Bool) {
    #expect(EmailValidator.isValid(input) == expected)
}

    Each argument runs as its own case and reports independently.

    Suites and setup

    Group tests in a struct/final class annotated @Suite. A fresh instance is created per test, so stored properties are your isolated setup — no shared mutable state between tests:

    @Suite struct CartTests {
    let cart = Cart()                      // fresh for every @Test
    @Test func startsEmpty() { #expect(cart.items.isEmpty) }
}

    XCTest (still essential)

    XCTest is required for UI tests (XCUIApplication) and performance tests (measure {}), and it's what you'll meet in existing projects:

    import XCTest
final class CartXCTests: XCTestCase {
    func testAddItem() {
        let cart = Cart()
        cart.add(.sample)
        XCTAssertEqual(cart.items.count, 1)
    }
    func testParsePerformance() {
        measure { _ = try? DateParser.parse(big) }   // records baseline timing
    }
}

    The two coexist in the same test bundle, so you can adopt Swift Testing for new unit tests while keeping XCTest for UI/performance.

    Unit Testing Patterns

    Knowing the framework is easy; knowing what and how to test is the skill. Good unit tests are fast, isolated, deterministic, and read like a specification of behavior.

    Arrange–Act–Assert

    Every test has the same three-beat shape:

    @Test func applyingDiscountReducesTotal() {
    // Arrange
    let cart = Cart(items: [.init(price: 100), .init(price: 50)])
    // Act
    cart.apply(.percentage(10))
    // Assert
    #expect(cart.total == 135)
}

    Keep each test focused on one behavior. A test that asserts ten things tells you little when it fails.

    Test behavior, not implementation

    Assert what the code does, not how. Tests coupled to internal details break every refactor and discourage improvement. Test through the public interface: given inputs/actions, expect outputs/state. The pure reducers from unidirectional flow are ideal here — they're just (state, action) -> state.

    Inject fakes for dependencies

    This is where DI pays off. Replace the real network/clock/store with controllable test doubles:

    struct StubAPI: APIClient {
    var result: Result<User, Error>
    func user(_ id: Int) async throws -> User { try result.get() }
}

@Test func showsErrorStateOnFailure() async {
    let vm = ProfileViewModel(api: StubAPI(result: .failure(NetError.offline)))
    await vm.load(id: 1)
    #expect(vm.state.isFailed)
}

    No network, deterministic, runs in milliseconds — and you can test the failure path, which is hard to trigger against a real server.

    Testing async code

    Swift Testing supports async tests directly — just await:

    @Test func loadsConcurrently() async throws {
    let dash = try await loader.loadDashboard()
    #expect(dash.feed.isEmpty == false)
}

    For code that emits over time (an AsyncStream), collect values with for await and assert the sequence. Inject a controlled clock instead of real Task.sleep so time-based logic (debounce, retry backoff) runs instantly and deterministically.

    What's worth testing

    • High-value: business logic, reducers/ViewModels, parsing/decoding, edge cases (empty, nil, huge, malformed), error paths, money/date math.
    • Low-value: trivial getters, framework behavior (don't test that URLSession works), exact pixel layout (that's snapshot territory).

    Aim for confidence, not a coverage number. 100% coverage of trivial code is worse than 60% covering the logic that actually breaks.

    Determinism is non-negotiable

    A flaky test is worse than no test — it trains people to ignore failures. Eliminate the usual sources: real network (inject fakes), real time (inject clocks), shared state (fresh instances per test), and ordering dependence (tests run in parallel; never rely on order).

    Snapshot & UI Tests

    Unit tests verify logic; these two verify the rendered result. Snapshot tests catch unintended visual changes; UI tests drive the real app like a user. Both are valuable and both are slower and more brittle than unit tests — use them deliberately.

    Snapshot testing

    A snapshot test renders a view to an image once (the "reference"), then fails future runs if the rendering differs. It's the fastest way to catch accidental visual regressions across many states.

    // using a snapshot library (e.g. swift-snapshot-testing)
func testProfileCard() {
    let view = ProfileCard(user: .sample)
    assertSnapshot(of: view, as: .image(layout: .device(.iPhone15)))
}
    • The first run records the reference image; commit it. Later runs diff against it.
    • Test multiple states cheaply: light/dark, Dynamic Type sizes, empty/loaded/error, RTL. Each is one line and guards a whole appearance.
    • Brittleness is the cost: legitimate design changes require re-recording references. Keep snapshots for stable components, not rapidly-iterating screens.

    SwiftUI previews are the informal cousin — #Preview with multiple trait variations gives you the same coverage visually during development.

    UI (end-to-end) tests

    XCUITest launches the actual app and interacts through the accessibility tree — the closest thing to a real user:

    final class CheckoutUITests: XCTestCase {
    func testCheckoutFlow() {
        let app = XCUIApplication()
        app.launchArguments = ["-uitesting"]      // app uses stub data in this mode
        app.launch()
        app.buttons["Add to Cart"].tap()
        app.tabBars.buttons["Cart"].tap()
        app.buttons["Checkout"].tap()
        XCTAssertTrue(app.staticTexts["Order Confirmed"].waitForExistence(timeout: 5))
    }
}

    Keys to non-flaky UI tests:

    • Find by accessibility identifier, not by visible label or position: button.accessibilityIdentifier = "checkout". Labels change and break tests; identifiers are stable.
    • waitForExistence, never sleep. Async UI appears when it appears; fixed sleeps are both slow and flaky.
    • Stub the backend. Launch the app in a test mode that injects deterministic data (via a launch argument), so a flaky network never fails a UI test.
    • Keep them few. Cover the handful of critical flows (sign-in, purchase), not every screen — UI tests are minutes, unit tests are milliseconds.

    The accessibility dividend

    UI tests navigate the same accessibility tree VoiceOver uses. Writing testable UI (good identifiers and labels) makes your app more accessible at the same time — the work pays off twice, and connects to accessibility.

    Instruments

    Instruments is Xcode's profiler — a suite of tools that record what your app actually does at runtime: where CPU time goes, what's allocating memory, why the UI hitches. The golden rule of performance work: measure first. Optimizing without profiling is guessing, and you'll usually fix the wrong thing.

    How to profile

    Cmd-I (Product → Profile) builds in Release configuration (optimizations on — critical; Debug numbers lie) and launches Instruments. Pick a template, record while you reproduce the problem, then read the trace.

    The instruments you'll use most

    • Time Profiler — samples the call stack ~1000×/sec to show where CPU time is spent. The "heaviest stack trace" tells you the hot path. Look for your code high in the tree, and for unexpected main-thread work.
    • Allocations — every allocation and what's still alive. Use it to find memory growth and large transient allocations (e.g. full-size images you forgot to downsample, images & bitmaps).
    • Leaks — flags memory that's unreachable but not freed, usually a retain cycle (ARC). Pair with the Memory Graph debugger to see the cycle.
    • Core Animation / Animation Hitches — surfaces dropped frames, off-screen rendering, and blended layers; the tool for jank in scrolling and animation (drawing performance).
    • SwiftUI — shows view body re-evaluations: which views recompute, how often, and why. The profiler for "my SwiftUI is slow."

    Reading a Time Profiler trace

    1. Filter to the main thread (most UI jank lives there).
    2. Turn on "Hide System Libraries" and "Invert Call Tree" to surface your hottest leaf functions.
    3. The function with the highest self-weight is where the time goes. Often it's something surprising — JSON decoding on the main thread, a layout pass triggered in a loop, an unindexed fetch (persistence performance).

    The Memory Graph debugger

    Not strictly Instruments, but essential: pause the app and click the memory-graph button to see the live object graph. Purple ! badges flag leaks; trace the arrows to find which strong reference forms a cycle, then break it with weak/unowned.

    A workflow

    1. Reproduce the problem with a profiler attached (Release build).
2. Find the single biggest contributor (don't micro-optimize noise).
3. Fix it. Re-profile to confirm the number moved.
4. Repeat until it's fast enough — then stop.

    Performance work obeys Amdahl's law: fix the dominant cost first; shaving 5% off something that's 2% of the time is wasted effort.

    Performance & Hangs

    Users don't read your code — they feel its performance. Three distinct problems show up as "the app feels bad": hangs (the UI freezes), hitches (a frame is dropped mid-animation), and memory growth (it gets slow then gets killed). Each has a different cause and a different fix.

    Hangs: the main thread is blocked

    A hang is the main thread doing synchronous work so nothing can respond. Anything over ~250ms is perceptible; over a few seconds the watchdog may kill the app. Causes and fixes:

    • Synchronous I/O on main (disk reads, a big JSON decode, an unindexed Core Data fetch) → move it off the main actor (@MainActor discipline).
    • Heavy layout/draw on main → simplify, cache, or rasterize (drawing performance).
    • A lock or await waiting on something slow while holding up the UI.

    Apple's Eliminate Hangs session is the definitive treatment; the Time Profiler shows you exactly which main-thread call is blocking.

    Hitches: a missed frame

    The display refreshes every ~16.7ms (or ~8.3ms at 120Hz ProMotion). If a frame's work overruns that budget, the frame is dropped — a hitch — visible as a stutter in a scroll or animation. Causes:

    • Commit phase too long: too many layers, off-screen passes (un-pathed shadows, cornerRadius + masking), overdraw.
    • Main-thread work during the animation (even brief) steals from the frame budget.

    Use the Animation Hitches instrument; it attributes each hitch to the commit or render phase. The fixes are the layer-level ones from how rendering works: fewer off-screen passes, shadowPath, avoid blending, do nothing heavy on main during animation.

    Memory: growth and pressure

    Persistent growth that never comes back  → a LEAK (retain cycle) → Leaks/Memory Graph
A big spike on a screen                  → large transient allocs → downsample images, page data
Steady climb under scrolling             → caches without limits  → bound your caches

    iOS doesn't swap; when memory runs low the system kills your app. Honor didReceiveMemoryWarning / the memory-pressure source by dropping caches. The biggest single offender in feed apps is full-resolution images — always downsample (images & bitmaps).

    Launch time

    A slow launch is the first impression. Keep the main-thread work before first frame minimal: defer non-critical setup, avoid synchronous network/disk in init/didFinishLaunching, and watch static initializers. Measure with the App Launch instrument; the target is well under a second to first meaningful frame.

    The discipline

    Set a budget (60/120fps, <1s launch, a memory ceiling), measure against it with Instruments, fix the dominant cost, and re-measure. Performance is a feature you maintain, not a one-time pass — and it's invisible when done right, which is exactly the point.
