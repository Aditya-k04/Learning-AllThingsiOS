Persistence & SwiftData
Where data lives: SwiftData, the Core Data engine underneath, the file system, and migrations.

    Persistence & SwiftData — Overview

    Every app needs to remember things — between launches, offline, across devices. iOS gives you a ladder of options from a single value to a full object graph. The skill is matching the tool to the data, and knowing the Core Data engine that still sits under the modern API.

    The ladder of options

    @AppStorage / UserDefaults   – small key-values: flags, prefs, last-selected tab
Files (Codable → disk)       – documents, caches, exported data, App Group sharing
Keychain                     – secrets: tokens, passwords (encrypted)
SwiftData / Core Data        – a queryable object GRAPH with relationships & migrations
CloudKit (+ SwiftData)       – the above, synced across the user's devices

    Don't reach for a database to store a boolean, and don't store a thousand related records in a JSON file. Pick the lightest tool that fits the shape and size of the data.

    What you'll learn

    1. 01-where-data-lives — UserDefaults, the sandbox directories, and what belongs where.
    2. 02-swiftdata — the modern @Model API: models, queries, the context.
    3. 03-core-data-engine — what SwiftData is built on, and when you still meet Core Data.
    4. 04-files-and-codable — saving Codable to disk and sharing via App Groups.
    5. 05-migrations-and-performance — evolving your schema safely and keeping it fast.

    The mental model

    Persistence has two halves: a store (where bytes live) and a context (your in-memory working set that you mutate and then save). SwiftData and Core Data both follow this; understanding it explains saving, undo, and threading.

    How it connects

    Decoded network models (Codable) often land in SwiftData for offline use. Widgets read a shared store via an App Group (widget interactivity). And the @Observable models SwiftData produces plug straight into the Observation framework.

    Where Data Lives

    Before reaching for a database, know the lightweight stores and the app sandbox. Putting data in the wrong directory is a real bug — it can get purged, fail to back up, or get your app rejected.

    UserDefaults / @AppStorage

    For small, simple values — preferences, flags, the last-selected tab. Backed by a property list, loaded into memory at launch.

    UserDefaults.standard.set(true, forKey: "hasOnboarded")

// SwiftUI binds to it directly:
@AppStorage("hasOnboarded") private var hasOnboarded = false

    @AppStorage is a property wrapper that reads/writes UserDefaults and invalidates the view when it changes — the data-flow integration from state storage. Rule: small values only. It's not for images, large arrays, or models.

    The app sandbox directories

    Your app can only read/write inside its sandbox. The directories differ in backup and purge behavior — this is the part people get wrong:

    Documents/        – user-created, user-visible data. Backed up to iCloud/iTunes.
Library/Application Support/  – app-generated data you want to keep. Backed up. (DBs go here.)
Library/Caches/   – re-creatable data. NOT backed up; system may PURGE under pressure.
tmp/              – scratch. Can vanish any time.

    let support = try FileManager.default.url(for: .applicationSupportDirectory,
    in: .userDomainMask, appropriateFor: nil, create: true)

    Common mistakes: putting a cache in Documents (bloats the user's iCloud backup → possible rejection), or putting the database in Caches (the OS deletes it and the app "loses everything").

    Excluding from backup

    For large re-downloadable files you keep in Application Support, mark them so they don't inflate backups:

    var url = fileURL
var values = URLResourceValues(); values.isExcludedFromBackup = true
try url.setResourceValues(values)

    Keychain for secrets

    Tokens, passwords, keys — never UserDefaults (it's plain plist). The Keychain is encrypted and survives reinstalls (optionally). The API is C-ish; most apps wrap it or use a tiny helper:

    let query: [String: Any] = [
    kSecClass as String: kSecClassGenericPassword,
    kSecAttrAccount as String: "authToken",
    kSecValueData as String: token.data(using: .utf8)!]
SecItemAdd(query as CFDictionary, nil)

    App Group: sharing with extensions

    Widgets, share extensions, and your app are separate processes. To share data, put it in an App Group container (enabled in capabilities) — a directory and UserDefaults suite both targets can reach:

    let shared = UserDefaults(suiteName: "group.com.you.app")!
let dir = FileManager.default.containerURL(forSecurityApplicationGroupIdentifier: "group.com.you.app")!

    This is how a widget timeline provider reads what the app wrote.

    SwiftData

    SwiftData (iOS 17+) is Apple's modern persistence framework: you annotate plain Swift classes with @Model, and they become a queryable, relational, observable store — with deep SwiftUI integration. It's the recommended default for structured data in new apps.

    Define a model

    import SwiftData

@Model
final class Task {
    var title: String
    var isDone: Bool
    var createdAt: Date
    @Relationship(deleteRule: .cascade) var subtasks: [Subtask] = []

    init(title: String, isDone: Bool = false) {
        self.title = title; self.isDone = isDone; self.createdAt = .now
    }
}

    @Model rewrites the class to track changes and persist properties. Relationships are just typed properties; deleteRule: .cascade means deleting a task deletes its subtasks.

    The container and context

    ModelContainer  – owns the store file (the database on disk)
ModelContext    – your in-memory working set: insert/delete/edit, then save

    In SwiftUI you wire the container once and get a context from the environment:

    @main struct MyApp: App {
    var body: some Scene {
        WindowGroup { ContentView() }
            .modelContainer(for: Task.self)     // creates the store
    }
}

struct ContentView: View {
    @Environment(\.modelContext) private var context
    @Query(sort: \Task.createdAt, order: .reverse) private var tasks: [Task]   // live results

    var body: some View {
        List(tasks) { task in Text(task.title) }
        Button("Add") { context.insert(Task(title: "New")) }   // auto-saves
    }
}

    @Query is the magic

    @Query runs a fetch and keeps the view in sync: insert, delete, or edit a model anywhere and every @Query showing it updates automatically. It's the persistence equivalent of the dependency graph — you describe what you want, the framework keeps it current. Filter and sort declaratively:

    @Query(filter: #Predicate<Task> { !$0.isDone }, sort: \.createdAt) var open: [Task]

    Saving, deleting, undo

    context.insert(_:) and context.delete(_:) stage changes; SwiftData autosaves by default, or call try context.save() explicitly. Enable undo with container.mainContext.undoManager for free edit/undo.

    Threading

    The main context is main-actor bound. For heavy imports, use a background ModelContext (via ModelContainer), do the work off-main, and save — the same main-actor discipline from @MainActor discipline. Pass model IDs (PersistentIdentifier) across actors, not the model objects.

    The Core Data Engine

    SwiftData is a modern Swift skin over Core Data, the framework that has powered iOS persistence for over a decade. You'll still meet Core Data in existing codebases, in advanced scenarios SwiftData doesn't yet cover, and in the concepts that explain SwiftData's behavior.

    The Core Data stack

    NSManagedObjectModel    – the schema (entities, attributes, relationships)
NSPersistentStoreCoordinator – connects the model to the actual store file (SQLite)
NSManagedObjectContext  – the in-memory scratchpad of managed objects
NSPersistentContainer   – bundles all of the above together (the modern entry point)

    SwiftData's ModelContainer/ModelContext map directly onto NSPersistentContainer/NSManagedObjectContext. Same architecture, friendlier API.

    Managed objects are faulted

    Core Data is lazy. A fetched object can be a fault — a placeholder that hasn't loaded its data yet; touching a property fires the fault and loads from SQLite. This makes fetching large graphs cheap but means I/O can happen at surprising times. SwiftData inherits this laziness, which is why you keep model access on the right actor.

    Contexts and threading (the classic footgun)

    A managed object belongs to the context that created it and must not be used on another thread. Cross threads by passing the NSManagedObjectID (SwiftData: PersistentIdentifier) and re-fetching in the target context.

    The standard pattern is a main-queue context for UI and background contexts for imports, merging changes between them. SwiftData automates much of this, but the rule is the same and explains its actor constraints from 02-swiftdata.

    When you still need Core Data directly

    • Fine-grained fetch control: NSFetchedResultsController (efficient table/collection backing), batch updates/deletes (NSBatchDeleteRequest) that skip loading objects into memory.
    • Complex migrations beyond SwiftData's current support.
    • Existing apps with a large Core Data model you're not ready to port.

    SwiftData and Core Data can even share the same store, so migration can be incremental.

    CloudKit sync

    Both can sync to the user's private CloudKit database with little code — NSPersistentCloudKitContainer for Core Data, or .modelContainer(..., cloudKitDatabase: .automatic)-style configuration for SwiftData. The store mirrors to iCloud and merges across the user's devices, with constraints (every attribute must be optional or have a default, no unique constraints) that shape your model design.

    Files & Codable Storage

    Sometimes a database is overkill. For documents, exported data, caches, and sharing between your app and its extensions, writing Codable values straight to disk is simpler, transparent, and fast.

    Codable → disk in three lines

    func save<T: Encodable>(_ value: T, to url: URL) throws {
    let data = try JSONEncoder().encode(value)
    try data.write(to: url, options: [.atomic])    // atomic = no half-written files
}
func load<T: Decodable>(_ type: T.Type, from url: URL) throws -> T {
    try JSONDecoder().decode(T.self, from: Data(contentsOf: url))
}

    .atomic writes to a temp file and renames — so a crash mid-write never corrupts the existing file. For large or performance-sensitive data, prefer PropertyListEncoder (binary plist) or a compact binary format over JSON.

    Choosing the directory

    Use the right sandbox location from 01-where-data-lives: user documents in Documents, app-managed data in Application Support, throwaway/recreatable data in Caches. Create the directory before writing:

    let dir = try FileManager.default.url(for: .applicationSupportDirectory,
    in: .userDomainMask, appropriateFor: nil, create: true)
let fileURL = dir.appendingPathComponent("draft.json")

    Sharing via App Groups

    To hand data to a widget or share extension, write into the App Group container both processes can reach:

    let groupID = "group.com.you.app"
let dir = FileManager.default
    .containerURL(forSecurityApplicationGroupIdentifier: groupID)!
try save(model, to: dir.appendingPathComponent("widget-feed.json"))
// the widget's TimelineProvider then loads it synchronously

    This is the concrete mechanism behind the timeline provider reading your app's data.

    Large binaries: write files, store paths

    Don't stuff images or video into JSON/SwiftData as Data — it bloats the store and memory. Write the binary to a file and persist only its filename/relative path in your model. SwiftData's @Attribute(.externalStorage) automates exactly this: large Data properties are written to separate files and referenced, keeping the database lean.

    Document-based apps

    If the file is the document (a drawing app, an editor), use FileDocument (SwiftUI) or UIDocument (UIKit). They handle reading/writing, conflict resolution, autosave, and iCloud Drive integration — don't hand-roll that for true document apps.

    When to use files vs SwiftData

    Files: a handful of blobs, exported/imported documents, simple caches, cross-process sharing. SwiftData: many records, relationships, querying/sorting/filtering, live UI binding. If you find yourself writing search/sort logic over a folder of JSON files, that's the signal to graduate to 02-swiftdata.

    Migrations & Performance

    Two things separate a toy store from a shippable one: it can evolve without losing user data when you change the schema, and it stays fast as data grows. Both need a little upfront thought.

    Why migrations matter

    The moment you ship, users have data on disk in your current schema. Add a property, rename an entity, or change a relationship, and the new app's schema no longer matches the old store. Without a migration plan, the store fails to open — and the app crashes on launch or wipes data.

    SwiftData versioned schemas

    SwiftData expresses each schema version as a type, and a migration plan describes how to move between them:

    enum SchemaV1: VersionedSchema {
    static var versionIdentifier = Schema.Version(1, 0, 0)
    static var models: [any PersistentModel.Type] { [TaskV1.self] }
}
enum SchemaV2: VersionedSchema {
    static var versionIdentifier = Schema.Version(2, 0, 0)
    static var models: [any PersistentModel.Type] { [Task.self] }  // added `priority`
}

enum MigrationPlan: SchemaMigrationPlan {
    static var schemas: [any VersionedSchema.Type] { [SchemaV1.self, SchemaV2.self] }
    static var stages: [MigrationStage] {
        [.lightweight(fromVersion: SchemaV1.self, toVersion: SchemaV2.self)]
    }
}
    • Lightweight migrations are automatic: adding/removing optional properties, adding entities. No code needed beyond declaring the stage.
    • Custom migrations run a closure to transform data (split a name field, compute a new value) when the change isn't mechanical.

    The same lightweight-vs-heavyweight distinction exists in Core Data; SwiftData just makes the common case declarative. Always test a migration by installing the old version, adding data, then upgrading.

    Performance: fetch less, fetch lazily

    • Predicate at the store, not in Swift. Filter with #Predicate so SQLite does the work and returns few rows — never fetch everything and .filter in memory.
    • Sort in the query, paginate with fetchLimit/fetchOffset for large lists.
    • Only load what you show. Core Data faulting means relationships load on access; avoid walking a whole graph in a list cell.
    • Batch heavy writes on a background context and save once, rather than inserting thousands of objects one save at a time.
    @Query(filter: #Predicate<Task> { $0.isDone == false },
       sort: [SortDescriptor(\.createdAt, order: .reverse)]) var open: [Task]

    Indexing

    For properties you filter or sort on frequently, add an index so lookups stay fast as the table grows (@Attribute(.unique) for uniqueness, and schema indexes for hot query paths). Without an index, a filter becomes a full-table scan — fine at 100 rows, painful at 100,000.

    Measure with Instruments

    Use the SwiftData/Core Data and Time Profiler instruments to catch slow fetches and main-thread saves — the same profiling workflow as Instruments. A janky scroll over a list is often a too-broad fetch or main-thread save, not a UI problem.
