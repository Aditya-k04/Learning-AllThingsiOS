    App Intents, Accessibility & Shipping — Overview

    A finished app isn't just features — it integrates with the system, works for everyone, and actually makes it to the App Store. This course covers three things every shippable app needs and most tutorials skip: exposing actions to the system, accessibility, and the build-and-release pipeline.

    What you'll learn

    1. 01-app-intents-and-siri — App Intents: one definition that powers Siri, Shortcuts, Spotlight, and widgets.
    2. 02-accessibility — VoiceOver, Dynamic Type, contrast, and Reduce Motion done properly.
    3. 03-swift-package-manager — dependencies and your own packages with SPM.
    4. 04-code-signing — certificates, identifiers, profiles — demystified.
    5. 05-xcode-cloud-and-app-store — CI/CD and the path through App Store Connect to release.

    The mental model

    Features make the app do things. Shipping makes those things reachable — by the system (Intents), by every user (accessibility), and by the world (signing + the store). Skipping this layer is how good apps never get used.

    Why it's worth your time

    App Intents turn your features into system capabilities (a huge reach multiplier). Accessibility is both the right thing and required — and it makes UI tests easier (UI tests). And signing/CI is the gauntlet every app must pass; understanding it turns a recurring source of dread into a routine. This is the difference between "it runs on my simulator" and "it's in people's hands."

    How it connects

    App Intents are the same mechanism behind interactive widgets. SPM is the tool behind modularization. And accessibility identifiers are what make UI tests robust.

    App Intents & Siri

    App Intents is the framework that exposes your app's actions to the system. One Swift definition becomes a Siri command, a Shortcuts action, a Spotlight result, and an interactive widget button — all at once. It's one of the highest-leverage things you can add, because it multiplies where your features can be invoked.

    An intent is a verb

    import AppIntents

struct AddTaskIntent: AppIntent {
    static var title: LocalizedStringResource = "Add Task"
    static var description = IntentDescription("Adds a task to your list.")

    @Parameter(title: "Title") var taskTitle: String

    func perform() async throws -> some IntentResult & ProvidesDialog {
        TaskStore.shared.add(taskTitle)
        return .result(dialog: "Added “\(taskTitle)”.")
    }
}

    perform() is the action; @Parameters are the inputs the system can prompt for. Because the intent is a serializable value, the system can run it from anywhere — even when your app isn't open (the same property that makes interactive widgets work).

    Entities and queries

    To act on your data ("call my plumber", "open the Acme project"), model it as an AppEntity with a query so the system can resolve names to objects:

    struct ProjectEntity: AppEntity, Identifiable {
    let id: UUID
    var name: String
    static var typeDisplayRepresentation: TypeDisplayRepresentation = "Project"
    var displayRepresentation: DisplayRepresentation { .init(title: "\(name)") }
    static var defaultQuery = ProjectQuery()
}

struct ProjectQuery: EntityQuery {
    func entities(for ids: [UUID]) async throws -> [ProjectEntity] { /* look up */ }
    func suggestedEntities() async throws -> [ProjectEntity] { /* recents */ }
}

    Now an intent can take a ProjectEntity parameter and Siri/Shortcuts will let the user pick or speak one.

    Shortcuts and Siri phrases

    Expose intents to Siri and the Shortcuts app with AppShortcutsProvider, including spoken phrases:

    struct MyShortcuts: AppShortcutsProvider {
    static var appShortcuts: [AppShortcut] {
        AppShortcut(intent: AddTaskIntent(),
            phrases: ["Add a task in \(.applicationName)",
                      "New \(.applicationName) task"],
            shortTitle: "Add Task", systemImageName: "plus.circle")
    }
}

    These appear automatically in Spotlight and Shortcuts, and Siri understands the phrases — no separate Siri code.

    Why this is worth it

    App Intents is a small amount of code for an outsized payoff: your features become OS-level capabilities the user can automate, voice-trigger, and pin. It's also future-facing — it's how features surface in newer system experiences. Define your key actions as intents and you get reach you'd otherwise never build.

    Accessibility

    Accessibility (a11y) makes your app usable by people with vision, motor, hearing, or cognitive differences. It's the right thing to do, it's expected on Apple platforms, and — a nice side effect — accessible UI is more testable and often better designed for everyone. SwiftUI gives you a lot for free; the rest is small, deliberate additions.

    VoiceOver: the screen reader

    VoiceOver reads the UI aloud and lets users navigate by swiping. Standard SwiftUI controls are already accessible; your job is to make custom views and decorative compositions read sensibly:

    Image("heart").onTapGesture { like() }
    .accessibilityLabel("Like")                    // what it is
    .accessibilityValue(isLiked ? "On" : "Off")    // its current state
    .accessibilityHint("Double tap to toggle")     // what happens
    .accessibilityAddTraits(.isButton)             // how it behaves

// Combine a decorative cluster into one element:
HStack { Image(systemName: "star.fill"); Text("4.8"); Text("(2.3k)") }
    .accessibilityElement(children: .combine)      // reads "4.8, 2.3k" as one

    Mark purely decorative images .accessibilityHidden(true) so VoiceOver skips noise.

    Dynamic Type

    Users can scale text system-wide, up to huge accessibility sizes. Use semantic font styles (.body, .headline) which scale automatically — never hard-coded point sizes:

    Text("Title").font(.headline)            // scales with the user's setting
// custom font that still scales:
Text("Hi").font(.custom("Inter", size: 17, relativeTo: .body))

    Then make sure layouts survive large type: prefer flexible layouts, allow wrapping, and test at the largest accessibility size. Switch HStacks to VStacks at huge sizes with @Environment(\.dynamicTypeSize).

    Contrast and color

    Don't rely on color alone to convey meaning (color-blind users) — pair it with text or an icon. Meet contrast ratios, and respect Increase Contrast and Reduce Transparency. Use semantic colors (.primary, .secondary, asset colors with dark variants) so the system adapts them.

    Reduce Motion

    Some users get motion sickness from large animations. Honor their setting and swap big movement for a cross-fade:

    @Environment(\.accessibilityReduceMotion) private var reduceMotion
withAnimation(reduceMotion ? .none : .spring) { expand.toggle() }

    (This is exactly the prefers-reduced-motion courtesy your own learning site implements.)

    VoiceOver-friendly = test-friendly

    Accessibility identifiers and labels are what XCUITest uses to find elements (UI tests). Adding them makes the app both accessible and robustly testable — the same work, two payoffs.

    Audit it

    Use the Accessibility Inspector (in Xcode) to audit a running app — it flags missing labels, small touch targets, and contrast issues. Better: turn on VoiceOver on a device and try to use your app with the screen off. It's humbling and instantly shows what's broken.

    Swift Package Manager

    Swift Package Manager (SPM) is the built-in tool for adding third-party dependencies and for splitting your own code into reusable modules. It replaced CocoaPods/Carthage for most projects and is integrated directly into Xcode.

    Adding a dependency

    In Xcode: File → Add Package Dependencies, paste a repo URL, choose a version rule. Or in a Package.swift:

    dependencies: [
    .package(url: "https://github.com/pointfreeco/swift-composable-architecture",
             from: "1.0.0")
],

    Version rules matter

    SPM uses semantic versioning. Choose how much you trust upstream:

    from: "1.2.0"            → >=1.2.0 and <2.0.0  (accept minor/patch — common default)
.upToNextMinor(from:)    → >=1.2.0 and <1.3.0  (conservative)
.exact("1.2.3")          → pin precisely       (reproducible, but you miss fixes)
branch: "main"           → bleeding edge       (avoid for releases)

    The resolved versions are recorded in Package.resolved — commit it so everyone (and CI) builds the exact same dependency versions.

    Authoring your own package

    A package is a folder with a Package.swift manifest describing products (what it exposes), targets (compilation units), and their dependencies — the structure behind modularization:

    // swift-tools-version: 5.10
import PackageDescription
let package = Package(
    name: "DesignSystem",
    platforms: [.iOS(.v17)],
    products: [.library(name: "DesignSystem", targets: ["DesignSystem"])],
    targets: [
        .target(name: "DesignSystem", resources: [.process("Resources")]),
        .testTarget(name: "DesignSystemTests", dependencies: ["DesignSystem"])
    ])

    Reference local packages by path (.package(path: "../DesignSystem")) to build a multi-module app from sibling folders.

    Resources and access control

    Bundle assets, JSON, or .metal files with .process(...)/.copy(...) and load them via Bundle.module. Remember that types are internal by default — anything a consumer of your package must use needs to be public (or open). Forgetting public is the most common "why can't I import this?" confusion.

    Practical tips

    • Keep dependency count low; each one is code you don't control, plus build time and a supply-chain surface.
    • Prefer well-maintained packages with real release tags over branch: main.
    • If Xcode's package cache misbehaves, File → Packages → Reset Package Caches fixes most resolution gremlins.

    Code Signing

    Code signing is the part of iOS development everyone dreads, usually because it's treated as magic. It isn't — it's a small set of pieces that together prove who built an app and what it's allowed to do. Understand the pieces and the dread goes away.

    The four pieces

    Certificate          – proves WHO you are (a public/private key pair issued by Apple).
                       Development cert (run on your devices) vs Distribution cert (ship).
App ID / Bundle ID   – uniquely identifies the app (com.you.app) and its capabilities.
Entitlements         – what the app may DO (push, App Groups, iCloud, HealthKit…).
Provisioning Profile – ties cert + App ID + entitlements + (for dev) allowed devices
                       together into one file Xcode embeds in the build.

    A signed build = your binary + a provisioning profile that says "this certificate is allowed to run this bundle ID with these entitlements (on these devices)."

    Automatic signing (use this)

    For most apps, Xcode's Automatically manage signing is the right choice: check the box, pick your Team, and Xcode creates and renews certificates and profiles for you.

    Target → Signing & Capabilities → ✔ Automatically manage signing → select Team

    Add a capability (Push, App Groups, Sign in with Apple) in that same tab and Xcode updates the App ID's entitlements and regenerates the profile automatically. This is the 90% path.

    When you manage it manually

    CI servers, enterprise distribution, and teams with strict control sometimes manage signing by hand: create the App ID and capabilities in the developer portal, generate a distribution certificate, create a profile, and download both. Tools like fastlane match store a shared set of certificates/profiles in a git repo so every machine and CI runner signs identically — the usual fix for "works on my Mac, fails on the build server."

    Distribution flavors

    • Development — run on registered devices during dev.
    • App Store — upload to App Store Connect for TestFlight/App Store.
    • Ad Hoc — distribute to a fixed list of registered devices outside the store.
    • Enterprise — in-house distribution (special program).

    The certificate type and profile must match the distribution method, which is why "the archive won't upload" is usually a profile/cert mismatch.

    Common errors, decoded

    • "No profiles for 'com.you.app' were found" → the App ID/profile doesn't exist or doesn't match the bundle ID; let automatic signing regenerate it.
    • "Provisioning profile doesn't include signing certificate" → your cert and the profile are out of sync; refresh.
    • "Entitlement not in profile" → you added a capability but the profile is stale; toggle automatic signing or regenerate.

    Almost every signing error is "the four pieces don't agree." Find which one is stale and refresh it.

    Xcode Cloud & the App Store

    The last mile: getting a signed build tested by real users and onto the App Store, ideally automatically. This lesson covers continuous integration with Xcode Cloud, TestFlight, and the App Store Connect release flow.

    Continuous integration

    CI runs your tests on every push and produces builds without anyone manually archiving. Xcode Cloud is Apple's built-in option (no servers to manage, integrated with App Store Connect):

    App Store Connect → Xcode Cloud → create a workflow:
  Start condition:  push to main / PR opened
  Actions:          Build → Test (your XCTest/Swift Testing suites)
  Post-action:      Archive → deliver to TestFlight

    Alternatives — GitHub Actions, fastlane, Bitrise — do the same with more control and (often) lower cost at scale; many teams use fastlane lanes (fastlane beta) invoked from GitHub Actions. Whatever the tool, the pipeline is the same: build → test → sign → upload.

    The build → release pipeline

    1. Archive          a Release build (signed with a Distribution profile)
2. Upload           to App Store Connect (Xcode Organizer, `xcodebuild`, or fastlane `pilot`)
3. Processing       Apple processes the build (a few minutes to an hour)
4. TestFlight       distribute to internal testers instantly; external testers after a brief review
5. Submit           attach the build to an App Store version, fill metadata, submit for review
6. Review → Release  approval, then release (manually, automatically, or phased)

    TestFlight

    TestFlight is how you get real-world testing before release. Internal testers (up to 100 team members) get builds immediately; external testers (up to 10,000, via public link) get them after a lightweight beta review. Builds expire after 90 days. Use it to catch device-specific bugs, gather feedback, and validate analytics before going live.

    App Store submission essentials

    A submission needs: an app icon and screenshots for required device sizes, a description and keywords, a privacy policy URL, and the Privacy Nutrition Label (declare what data you collect and why) plus the privacy manifest (PrivacyInfo.xcprivacy) for the app and SDKs. Incomplete or inaccurate privacy info is a common rejection.

    Surviving review

    Most rejections are avoidable: provide a demo account if login is required, don't ship placeholder/crashing content, request permissions with clear purpose strings (the NS…UsageDescription keys), and make sure every feature works without the reviewer guessing. Read the rejection carefully — it cites a specific guideline — fix that, and reply in Resolution Center.

    Phased release

    For App Store releases you can roll out gradually (phased release) over 7 days, so a bad build reaches a small fraction first. Pair it with crash monitoring (Xcode Organizer's crash reports / MetricKit) so you catch regressions before they hit everyone — the production analog of the profiling in performance & hangs.

    The whole journey

    From here you've gone end to end: write a feature, test it (testing), make it reachable (App Intents) and usable by everyone (accessibility), sign it (code signing), and ship it. That's a complete iOS engineer's loop.
