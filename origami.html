    Origami Studio

    Origami Studio is Meta's free prototyping tool for designing real, physics-based interactions — the kind that ship in Facebook, Instagram, WhatsApp, and Quest. Where Figma prototyping is about linking screens, Origami is about behavior: how something moves, responds to a finger, and feels. It's the natural companion to your interaction-math and fluid-interface work — the same ideas (springs, gesture-as-function, velocity) made visual and hands-on.

    Origami is Mac-only, completely free with no subscriptions or feature limits, and pairs with Origami Live, a free iOS app that runs your prototype on a real device so you can feel the interaction in your hand.

    The three areas of the interface

    Origami's window has three panels, and almost everything you do lives in one of them:

    Layers (left)    – the visual hierarchy: groups, text, images, shapes, video.
                   Like the layers panel in Figma/Sketch.
Viewer (center)  – the live preview. It updates instantly as you build — no "play" button.
Patch Editor     – the node graph where the LOGIC lives. This is the heart of Origami.
(bottom/right)

    You design appearance in Layers and behavior in the Patch Editor. The Viewer is always live, which is what makes Origami feel immediate.

    Patches: visual programming

    A patch is a node with inputs (left) and outputs (right). You wire an output of one patch into the input of another to build logic — no text code. A simple interaction reads like a sentence drawn left-to-right:

    [ Interaction ]──tap──▶[ Switch ]──on/off──▶[ Pop Animation ]──progress──▶ Layer.Scale

    That graph means: "when the layer is tapped, flip a switch, and spring the layer's scale to a new value." The whole tool is composing these little data-flow graphs — exactly the model-the-behavior-as-a-function mindset from modeling visuals as functions.

    The core patches (learn these first)

    The library has hundreds of patches, but ~15–20 cover almost everything:

    • Interaction — detects tap, press, drag, scroll on a layer; outputs position, and crucially velocity.
    • Switch — flip-flops between on/off (a tap toggles a state).
    • Pop Animation — a spring: animates a value with bounciness and speed. Origami's signature, physical feel.
    • Classic Animation — duration + easing curve (when you want a fixed timing, not a spring).
    • Transition — remaps a 0→1 progress value into a range (e.g. progress → scale 1.0…1.2). This is lerp/remap from interpolation & easing.
    • Pulse, Option Switch / Option Picker, Delay, Hit Area, Conditional, math patches (Add, Multiply, Progress), and Scroll.

    Master those and you can build most interactions; the rest you look up as needed.

    Pop animation = a spring

    The Pop patch is a critically-tunable spring with two dials — Bounciness and Speed — which map directly to damping and stiffness. Drag a card and let go, and a Pop animation carries it home with the finger's velocity. That's the exact gesture→physics handoff from velocity & rubber-banding and the spring you already studied — Origami just exposes it as two sliders.

    The demo above is that spring: drag the ball and release, or set critical damping, and watch how bounciness/speed translate into feel. Tuning a Pop patch in Origami is the same intuition.

    Driving values from a gesture (the key idea)

    Origami's power is that a value can be a continuous function of a drag, not a canned animation. The Interaction patch outputs the drag position; you feed it through a Transition (remap) into a layer property, and the layer tracks the finger 1:1:

    [ Interaction ]──drag.Y──▶[ Transition: 0…300 → 1.0…0.5 ]──▶ Card.Scale
                          (as you drag down 300pt, scale goes 1.0 → 0.5)

    This is the interface-as-a-function-of-the-gesture principle — category 1 from who drives time — made tangible. Build it here, feel it on Origami Live, then reproduce it in SwiftUI.

    Components and reuse

    A Component is a self-contained, reusable patch+layer module with its own inputs and outputs — like a function or a SwiftUI view. Build a toggle once, expose its "isOn" input and "changed" output, and drop it in everywhere. Components keep big prototypes manageable, the same way modularization does for code.

    Figma import & handoff

    You don't have to design in Origami. Import a Figma (or Sketch) file and your layers come in with their hierarchy intact, ready to wire up interactions. The common workflow: design in Figma → import to Origami → prototype the interaction → test on Origami Live → spec it for engineering. Origami is the "make it move" step between static design and real code.

    Why it's worth your time

    Origami trains the exact instinct your whole learning stack is about: thinking of an interaction as driven values and physics, not as screens and canned transitions. It's a fast, no-code sandbox to explore a feel before committing to SwiftUI/UIKit — and because its model (interaction → remap → spring) mirrors what you've learned, the translation to code is almost mechanical.

    A first project (do this)

    1. Import or draw a card layer.
    2. Add an Interaction patch on it; wire drag position into the card's Position so it follows your finger.
    3. On release, route through a Pop Animation back to the origin — feel the spring.
    4. Add a Transition so the card's scale shrinks as it's dragged away.
    5. Open it in Origami Live on your phone and tune the Pop's bounciness/speed until it feels right.
    6. Now rebuild the same thing in SwiftUI (designing the feel) and compare.

    Ten of those reps and the gap between "an interaction you admire" and "an interaction you can build" closes fast.
