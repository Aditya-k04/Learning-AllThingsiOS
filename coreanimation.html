    Core Animation Curriculum Roadmap

    Start here — the plain idea

    Core Animation is the layer of code that actually puts pixels on the screen, no matter which framework you write in. Think of UIKit and SwiftUI as two different remote controls — but they both end up talking to the same TV. That TV is Core Animation.

    This roadmap is just the map of the territory. The big idea to hold onto: when you write UIView.animate or a SwiftUI .animation(), you're not doing the animating yourself — you're filling out a request form that gets handed down to Core Animation, which hands it to a separate render process, which hands it to the GPU. This course walks down that chain so the "magic" stops being magic.

    You don't need to memorize the lesson list. Just know there's a single road: pixels first (how rendering works), then the box that holds a picture (CALayer), then making that box move (animations).

    This curriculum takes you from zero to mastery of Core Animation — the rendering engine that powers every pixel on iOS, macOS, tvOS, and watchOS. Whether you see a UIKit view animate across the screen or a SwiftUI transition dissolve, Core Animation is doing the heavy lifting underneath.

    Why Core Animation?

    UIKit's UIView.animate(withDuration:) and SwiftUI's .animation() are convenience wrappers. They ultimately create CAAnimation objects on CALayer trees. Understanding Core Animation gives you:

    • Precise control over every animation parameter
    • The ability to build effects that high-level APIs cannot express
    • Deep performance insight — you'll know why frames drop
    • The vocabulary to read and write render-level code in any Apple framework

    How Rendering Works

    Start here — the plain idea

    Here's the thing that trips everyone up: your app doesn't actually draw to the screen. It never touches a single pixel. Instead, your app writes a to-do list — "this box is here, that one is half-transparent, this color is blue" — and slides that list under the door to a completely separate program. That other program (the render server) does the real drawing on the GPU.

    Think of it like ordering at a restaurant. You don't go into the kitchen and cook; you write down what you want and hand the ticket to the kitchen. The big payoff: the moment you hand off the ticket, you're free. Your app can move on to the next thing while the kitchen is still plating the last order. That hand-off is exactly why animations stay smooth even when your code is briefly busy.

    Almost everything in this lesson is a consequence of that one split: your app describes, someone else draws. The phases, the three layer trees, the "commit" — they're all just the machinery of writing a clean ticket and passing it along once per frame.

    Before writing a single animation, you need to understand what happens between your code changing a property and photons hitting the user's retina. This lesson maps the entire iOS rendering pipeline.

    The Rendering Pipeline

    Every frame on iOS passes through this pipeline:

    Your App Process          Render Server (backboardd)       Hardware
┌──────────────┐         ┌──────────────────────┐        ┌─────────┐
│ Layout       │         │                      │        │         │
│ Display      │ ──────► │ Decode + Render      │ ────►  │  GPU    │ ──► Display
│ Prepare      │  (IPC)  │ (Metal commands)     │        │         │
│ Commit       │         │                      │        │         │
└──────────────┘         └──────────────────────┘        └─────────┘

    Your app does NOT render pixels. It builds a description (the layer tree) and hands it off. The render server — a separate system process — does the actual GPU work. This is why Core Animation is fast: your app can proceed to the next frame while the render server is still compositing the previous one.

    The Four Phases in Your App

    Each frame, Core Animation runs these phases inside your app process:

    1. Layout

    • layoutSubviews() / layoutSublayers() are called
    • AutoLayout constraints are solved
    • Frames and bounds are calculated

    2. Display

    • draw(_:) / draw(in:) are called if needsDisplay is set
    • Core Graphics drawing commands execute (into a backing store bitmap)
    • This is the most expensive phase if you do custom drawing

    3. Prepare

    • Image decoding (compressed PNG/JPEG → raw bitmap)
    • Image format conversion if needed (e.g., non-GPU-friendly formats)

    4. Commit

    • The layer tree snapshot is serialized
    • Sent to the render server via IPC (Mach ports)
    • Everything is batched — nothing renders mid-run-loop
    // All three of these property changes are batched into
// a single commit at the end of the current run loop iteration.
layer.position = CGPoint(x: 100, y: 200)
layer.opacity = 0.5
layer.backgroundColor = UIColor.blue.cgColor
// The render server receives ONE commit with all three changes.

    The Three Layer Trees

    Core Animation maintains three parallel copies of every layer:

    Model Layer

    • The layer object your code touches: myLayer.position = ...
    • Reflects the target state — where things should end up
    • Accessed via layer itself

    Presentation Layer

    • Reflects the current on-screen state during an animation
    • Updates ~60-120 times per second
    • Accessed via layer.presentation()

    Render Layer

    • Private, lives in the render server process
    • Actually composited by the GPU
    • You cannot access this directly
    // During an animation from x=0 to x=300:
print(layer.position.x)                // 300 (model layer — final value)
print(layer.presentation()?.position.x) // 147.3 (presentation — current visual position)
// The render layer is inaccessible but closely matches the presentation layer.

    This three-tree architecture is why animations appear smooth even when your main thread is briefly busy. The render server drives the presentation/render trees independently.

    Frame Timing

    Display Refresh Rate Frame Budget
    Standard 60 Hz 16.67 ms
    ProMotion 120 Hz 8.33 ms
    ProMotion (adaptive) 24-120 Hz Varies

    If your commit phase takes longer than the frame budget, you drop a frame. Dropped frames are visible — the user perceives jank.

    CADisplayLink

    CADisplayLink is a timer synchronized to the display refresh rate:

    class RenderLoop {
    private var displayLink: CADisplayLink?

    func start() {
        displayLink = CADisplayLink(target: self, selector: #selector(tick))
        displayLink?.add(to: .main, forMode: .common)
    }

    @objc func tick(_ link: CADisplayLink) {
        let frameDuration = link.targetTimestamp - link.timestamp
        // frameDuration ≈ 0.01667 at 60Hz, ≈ 0.00833 at 120Hz

        let actualDuration = link.duration
        // Use this to drive manual animations
    }

    func stop() {
        displayLink?.invalidate()
        displayLink = nil
    }
}

    See 20-concurrency-basics.md — CADisplayLink runs on the main run loop; blocking the main thread blocks rendering.

    Off-Screen Rendering

    Normally, the GPU composites layers directly to the frame buffer (on-screen). Certain effects force the GPU to first render to a separate texture, then composite that texture — this is off-screen rendering.

    Triggers for off-screen rendering: - cornerRadius combined with masksToBounds = true (in some cases) - mask (layer masking) - shadowPath NOT set when shadows are used - shouldRasterize = true - Complex opacity with sublayers (group opacity)

    Why it's expensive: 1. GPU must create a temporary texture (memory allocation) 2. GPU must switch render targets (context switch) 3. The temporary texture must be composited back 4. This happens every frame unless rasterized

    // EXPENSIVE: shadow without path — GPU must compute shape every frame
layer.shadowColor = UIColor.black.cgColor
layer.shadowOpacity = 0.5
layer.shadowRadius = 10

// FAST: provide shadowPath so GPU knows the shape
layer.shadowPath = UIBezierPath(roundedRect: layer.bounds,
                                 cornerRadius: layer.cornerRadius).cgPath

    The Render Server: backboardd

    On iOS, the render server is a process called backboardd. On macOS, it's WindowServer. Key facts:

    • It runs in a separate process with elevated GPU priority
    • Your app sends layer tree snapshots to it via IPC
    • It executes Metal render commands on the GPU
    • It drives the presentation layer updates
    • If your app crashes, the last frame remains on screen (because backboardd still has the layer tree)

    This architecture means animation interpolation happens outside your app. Once you commit an animation, your app can go to sleep and the animation continues smoothly.

    Identifying Frame Drops

    Instruments: Core Animation Template

    1. Open Instruments → Choose "Core Animation" template
    2. Look for red bars in the "Commits" track — these are slow commits
    3. "Offscreen Rendering" track shows off-screen render passes
    4. Target: all commits under 8ms (for ProMotion) or 16ms (for 60Hz)

    Quick Debug Options (in Simulator)

    • Debug → Color Blended Layers (green = opaque, red = blended)
    • Debug → Color Off-Screen Rendered (yellow highlight)
    • Debug → Color Misaligned Images (magenta = not pixel-aligned)
    // Programmatic frame drop detection
let link = CADisplayLink(target: self, selector: #selector(checkFrame))
link.add(to: .main, forMode: .common)

@objc func checkFrame(_ link: CADisplayLink) {
    let elapsed = link.timestamp - previousTimestamp
    if elapsed > 0.02 { // more than 20ms = dropped frame at 60Hz
        print("Frame drop detected: \(elapsed * 1000)ms")
    }
    previousTimestamp = link.timestamp
}

    The Mental Model

    This separation is what makes Core Animation powerful: - You describe what should happen - The system handles when (frame timing) and how (GPU commands) - Your main thread is free to process events, layout, and logic

    CALayer Fundamentals

    Start here — the plain idea

    A CALayer is just a lightweight box that holds a picture and some properties; the view is the heavy thing, the layer is what actually gets drawn. Every UIView you've ever used is secretly carrying one of these boxes around — when you set the view's background color, the view quietly turns around and sets it on its layer.

    So why care about the box directly? Because the view comes with a lot of baggage it doesn't always need: touch handling, accessibility, the responder chain, Auto Layout. If all you want is something pretty on screen — a colored shape, a shadow, an image — a bare layer is the cheaper tool. It's like the difference between hiring a full employee versus printing a poster. The poster can't answer the phone, but if you only need it to look good on the wall, that's fine.

    One thing to tuck away early: a layer is a class, a reference type. Two variables pointing at the same layer point at the same box — change it through one and the other sees it too. That bites people, so notice it now.

    CALayer is the fundamental unit of visual content in Core Animation. Every UIView is backed by a CALayer. But layers can also exist independently — and understanding them directly unlocks the full power of the rendering engine.

    What Is CALayer?

    A CALayer is an object that manages image-based content and lets you perform animations on that content. It stores:

    • A bitmap (the backing store) that the GPU composites
    • Geometry (position, size, transform)
    • Visual properties (color, border, shadow, opacity)
    • A tree structure (sublayers, superlayer)

    Layers are not views. They do not handle: - Touch events or gesture recognizers - Accessibility - The responder chain - Auto Layout directly

    See 10-structs-vs-classes.md — CALayer is a class (reference type), not a struct. Multiple variables can point to the same layer.

    Every UIView Has a Layer

    let view = UIView(frame: CGRect(x: 0, y: 0, width: 200, height: 200))
view.backgroundColor = .systemBlue

// The layer is always there:
print(type(of: view.layer))  // CALayer
print(view.layer.bounds)     // (0, 0, 200, 200)

// UIView properties are often pass-throughs to the layer:
view.alpha = 0.5
print(view.layer.opacity)   // 0.5 — same thing

    When you set view.backgroundColor, UIKit sets view.layer.backgroundColor. When you set view.frame, UIKit sets the layer's bounds and position. The view is a thin wrapper.

    CALayer Without UIView

    Layers can exist and render without any UIView:

    class LayerDemoViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()

        // Create a standalone layer
        let redSquare = CALayer()
        redSquare.frame = CGRect(x: 50, y: 100, width: 150, height: 150)
        redSquare.backgroundColor = UIColor.systemRed.cgColor
        redSquare.cornerRadius = 12

        // Add it directly to the view's layer
        view.layer.addSublayer(redSquare)
    }
}

    This is lighter weight than creating a UIView — no responder chain overhead, no Auto Layout involvement. Use standalone layers when you need pure visual elements that don't require interaction.

    Key Properties Overview

    let layer = CALayer()

// Geometry
layer.frame = CGRect(x: 20, y: 20, width: 100, height: 100)
layer.bounds = CGRect(origin: .zero, size: CGSize(width: 100, height: 100))

// Visual

// Corner & Border

    Important: Color properties take CGColor, not UIColor. This is because CALayer is shared across UIKit and AppKit, and CGColor is the cross-platform type.

    Setting Layer Contents

    The contents property is typed as Any? — a legacy of its Objective-C origins. In practice, you assign a CGImage:

    let imageLayer = CALayer()
imageLayer.frame = CGRect(x: 20, y: 200, width: 200, height: 200)
imageLayer.contentsScale = UIScreen.main.scale
imageLayer.contentsGravity = .resizeAspect

if let image = UIImage(named: "landscape") {
    imageLayer.contents = image.cgImage
}

view.layer.addSublayer(imageLayer)

    Layer Delegation

    CALayer has a delegate protocol for custom drawing and layout:

    class LayerDrawingController: UIViewController, CALayerDelegate {
    private let customLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        customLayer.frame = CGRect(x: 20, y: 100, width: 200, height: 200)
        customLayer.delegate = self
        customLayer.contentsScale = UIScreen.main.scale
        view.layer.addSublayer(customLayer)

        // Trigger drawing
        customLayer.setNeedsDisplay()
    }

    // Called when the layer needs to draw its contents
    func draw(_ layer: CALayer, in ctx: CGContext) {
        // Draw a red circle
        ctx.setFillColor(UIColor.systemRed.cgColor)
        ctx.fillEllipse(in: layer.bounds)

        // Draw a white cross
        ctx.setStrokeColor(UIColor.white.cgColor)
        ctx.setLineWidth(4)

        let mid = CGPoint(x: layer.bounds.midX, y: layer.bounds.midY)
        ctx.move(to: CGPoint(x: mid.x - 30, y: mid.y))
        ctx.addLine(to: CGPoint(x: mid.x + 30, y: mid.y))
        ctx.move(to: CGPoint(x: mid.x, y: mid.y - 30))
        ctx.addLine(to: CGPoint(x: mid.x, y: mid.y + 30))
        ctx.strokePath()
    }

    // Called when the layer needs to layout its sublayers
    func layoutSublayers(of layer: CALayer) {
        // Adjust sublayer frames here if needed
    }
}

    The Display Cycle

    Layers use a display cycle similar to views:

    // Mark that contents need redrawing (does NOT draw immediately)
layer.setNeedsDisplay()

// Force immediate redraw if needed (rarely used)
layer.displayIfNeeded()

// Check if redraw is pending
if layer.needsDisplay() {
    print("Layer will redraw on next commit")
}

    The setNeedsDisplay() call marks the layer as dirty. The actual draw(_:in:) call happens during the Display phase of the next render loop iteration. This batching is efficient — even if you call setNeedsDisplay() ten times, the layer draws only once.

    Subclassing CALayer

    You can subclass CALayer for reusable custom-drawing layers:

    class GradientCircleLayer: CALayer {
    override func draw(in ctx: CGContext) {
        guard let gradient = CGGradient(
            colorsSpace: CGColorSpaceCreateDeviceRGB(),
            colors: [UIColor.systemPurple.cgColor,
                     UIColor.systemPink.cgColor] as CFArray,
            locations: [0.0, 1.0]
        ) else { return }

        let center = CGPoint(x: bounds.midX, y: bounds.midY)
        let radius = min(bounds.width, bounds.height) / 2

        ctx.addEllipse(in: bounds)
        ctx.clip()
        ctx.drawRadialGradient(gradient,
                                startCenter: center, startRadius: 0,
                                endCenter: center, endRadius: radius,
                                options: [])
    }
}

// Usage:
let circle = GradientCircleLayer()
circle.frame = CGRect(x: 50, y: 400, width: 120, height: 120)
circle.contentsScale = UIScreen.main.scale
circle.setNeedsDisplay()
view.layer.addSublayer(circle)

    Thread Safety

    Layer property access must happen on the main thread in most cases:

    // CORRECT: main thread
DispatchQueue.main.async {
    layer.position = CGPoint(x: 100, y: 200)
    layer.opacity = 0.5
}

// DANGEROUS: background thread
DispatchQueue.global().async {
    layer.position = CGPoint(x: 100, y: 200)  // May crash or corrupt state
}

    The exception: you can read presentation() layer values from any thread during animation, but even this is discouraged. Stick to main thread for all layer work.

    Layer Geometry

    Start here — the plain idea

    Here's the mental model that makes all of this click: a layer is held up by a single pin. The position is where the pin is stuck into the parent. The anchorPoint is where on the layer that pin pokes through — dead center by default, but you can move it to a corner. The bounds is just how big the layer is. Everything about where a layer sits comes from those three facts.

    And frame? Frame is not real. It's not stored anywhere — it's a convenience that the system calculates on the fly from the pin location, the pin's spot on the layer, and the size. Think of it as a derived readout, like a speedometer: useful to glance at, but it's reporting something, not controlling it. This is why frame goes haywire the moment you rotate a layer — a tilted box doesn't have a clean upright rectangle anymore.

    Once "position is where the pin is, anchorPoint is where the pin goes through the layer" lives in your head, the famous gotcha (move the anchorPoint and the layer jumps) stops being surprising — you moved the hole, not the pin, so the box swings to a new spot.

    Understanding layer geometry is understanding where things appear on screen and why. The relationship between bounds, position, anchorPoint, and frame is the single most important spatial concept in Core Animation.

    The Four Geometry Properties

    bounds

    The layer's own coordinate system — its internal size and origin.

    let layer = CALayer()
layer.bounds = CGRect(x: 0, y: 0, width: 200, height: 100)
// The layer thinks it is 200 wide and 100 tall.
// bounds.origin is usually (0,0) but can be changed for scrolling effects.

    position

    Where the layer sits in its superlayer's coordinate space. Specifically, it defines where the layer's anchorPoint is placed in the superlayer.

    layer.position = CGPoint(x: 150, y: 250)
// The layer's anchorPoint is at (150, 250) in the superlayer's coordinates.

    anchorPoint

    A normalized coordinate (0,0 to 1,1) within the layer that position refers to. Default is (0.5, 0.5) — the center.

    layer.anchorPoint = CGPoint(x: 0.5, y: 0.5)  // center (default)
layer.anchorPoint = CGPoint(x: 0, y: 0)        // top-left
layer.anchorPoint = CGPoint(x: 1, y: 1)        // bottom-right

    frame

    A derived property. It is NOT stored. It is computed from bounds, position, anchorPoint, and transform.

    // frame is calculated, not stored:
// frame.origin.x = position.x - (bounds.width * anchorPoint.x)
// frame.origin.y = position.y - (bounds.height * anchorPoint.y)
// frame.size = bounds.size (when transform is identity)

let layer = CALayer()
layer.bounds = CGRect(x: 0, y: 0, width: 200, height: 100)
layer.position = CGPoint(x: 150, y: 250)
layer.anchorPoint = CGPoint(x: 0.5, y: 0.5)

print(layer.frame)
// CGRect(x: 50, y: 200, width: 200, height: 100)
// Because: 150 - (200 * 0.5) = 50,  250 - (100 * 0.5) = 200

    Setting Frame vs. Setting Bounds + Position

    When you set frame, Core Animation internally sets bounds.size and recalculates position:

    // These two approaches are equivalent:
// Approach 1: set frame
layer.frame = CGRect(x: 50, y: 200, width: 200, height: 100)

// Approach 2: set bounds and position (with default anchorPoint 0.5, 0.5)
layer.bounds.size = CGSize(width: 200, height: 100)
layer.position = CGPoint(x: 150, y: 250)  // center of the frame

    Warning: frame is undefined when transform is not identity. If you've rotated or scaled a layer, do NOT read or write frame. Use bounds and position instead.

    Changing AnchorPoint Without Moving the Layer

    The classic gotcha: changing anchorPoint moves the layer because position stays the same but now refers to a different point.

    let layer = CALayer()
layer.frame = CGRect(x: 50, y: 200, width: 200, height: 100)
// position is (150, 250), anchorPoint is (0.5, 0.5)

layer.anchorPoint = CGPoint(x: 0, y: 0)
// position is STILL (150, 250), but now the TOP-LEFT is at (150, 250)
// The layer visually jumped right and down!
print(layer.frame) // CGRect(x: 150, y: 250, width: 200, height: 100)

    Fix: Recalculate position after changing anchorPoint:

    extension CALayer {
    func setAnchorPointWithoutMoving(_ newAnchorPoint: CGPoint) {
        let oldFrame = frame
        anchorPoint = newAnchorPoint
        // Restore the frame, which recalculates position
        let newPosition = CGPoint(
            x: oldFrame.origin.x + oldFrame.width * newAnchorPoint.x,
            y: oldFrame.origin.y + oldFrame.height * newAnchorPoint.y
        )
        position = newPosition
    }
}

// Usage:
layer.setAnchorPointWithoutMoving(CGPoint(x: 0, y: 0))
// Layer stays in place visually, but rotations now pivot from top-left.

    This is critical for rotational animations. If you want a layer to rotate around its top-left corner, set anchorPoint to (0, 0) — but use this helper to prevent the visual shift.

    See 15-transform-3d.md — anchorPoint determines the center of rotation and scaling transforms.

    anchorPointZ

    For 3D transforms, anchorPointZ specifies the anchor's position along the Z axis:

    layer.anchorPointZ = -50  // anchor point is 50 points "behind" the layer
// Useful when rotating in 3D — changes the radius of rotation

    Coordinate Conversion

    Convert points and rects between layer coordinate spaces:

    let parentLayer = CALayer()
parentLayer.frame = CGRect(x: 100, y: 100, width: 300, height: 300)

let childLayer = CALayer()
childLayer.frame = CGRect(x: 50, y: 50, width: 100, height: 100)
parentLayer.addSublayer(childLayer)

// Convert a point from child's coordinate space to parent's
let pointInChild = CGPoint(x: 10, y: 10)
let pointInParent = parentLayer.convert(pointInChild, from: childLayer)
print(pointInParent) // (60, 60) — offset by child's origin

// Convert a point from parent's coordinate space to child's
let parentPoint = CGPoint(x: 80, y: 80)
let childPoint = childLayer.convert(parentPoint, from: parentLayer)
print(childPoint) // (30, 30) — relative to child's origin

// Convert a rect
let rectInChild = CGRect(x: 0, y: 0, width: 50, height: 50)
let rectInParent = parentLayer.convert(rectInChild, from: childLayer)
print(rectInParent) // (50, 50, 50, 50)

    Hit Testing

    Layers support hit testing to determine which layer a point falls within:

    class HitTestDemoController: UIViewController {
    let blueLayer = CALayer()
    let redLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        blueLayer.frame = CGRect(x: 50, y: 100, width: 200, height: 200)
        blueLayer.backgroundColor = UIColor.systemBlue.cgColor

        redLayer.frame = CGRect(x: 50, y: 50, width: 100, height: 100)
        redLayer.backgroundColor = UIColor.systemRed.cgColor
        blueLayer.addSublayer(redLayer)

        view.layer.addSublayer(blueLayer)
    }

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        guard let touch = touches.first else { return }
        let point = touch.location(in: view)

        // hitTest returns the deepest sublayer containing the point
        if let hitLayer = view.layer.hitTest(point) {
            if hitLayer === redLayer {
                print("Tapped red layer")
            } else if hitLayer === blueLayer {
                print("Tapped blue layer (but not red)")
            }
        }

        // contains() checks a single layer
        let pointInBlue = blueLayer.convert(point, from: view.layer)
        print("Point is in blue layer: \(blueLayer.contains(pointInBlue))")
    }
}

    hitTest(_:) walks the sublayer tree back-to-front and returns the deepest layer whose bounds contain the point. Unlike UIView hit testing, it does NOT check isUserInteractionEnabled (layers don't have that concept).

    contentsCenter

    A CGRect in the unit coordinate space (0,0 to 1,1) that defines a stretchable region, similar to 9-patch/9-slice images:

    let layer = CALayer()
layer.contents = UIImage(named: "rounded-button-bg")?.cgImage
layer.contentsScale = UIScreen.main.scale

// The center 60% is stretchable; edges (20% on each side) are fixed
layer.contentsCenter = CGRect(x: 0.2, y: 0.2, width: 0.6, height: 0.6)

// Now the image stretches gracefully when the layer resizes
layer.frame = CGRect(x: 20, y: 300, width: 300, height: 60) // wide button

    zPosition

    Controls visual stacking order among sibling layers:

    let layerA = CALayer()
layerA.frame = CGRect(x: 50, y: 100, width: 150, height: 150)
layerA.backgroundColor = UIColor.systemRed.cgColor

let layerB = CALayer()
layerB.frame = CGRect(x: 100, y: 150, width: 150, height: 150)
layerB.backgroundColor = UIColor.systemBlue.cgColor

view.layer.addSublayer(layerA)  // index 0
view.layer.addSublayer(layerB)  // index 1 — draws on top

// Override visual order without changing sublayer array:
layerA.zPosition = 1  // Now red draws on top of blue

    Key distinction: zPosition changes visual ordering only. The sublayers array order is unchanged. For hit testing, zPosition is also considered.

    Layer Contents

    Start here — the plain idea

    A layer is basically an empty picture frame. This lesson is all about the different ways to slide a picture into that frame. The simplest way is to hand it a finished image (a CGImage) — like dropping a printed photo into the frame. The other way is to draw the picture yourself, stroke by stroke, with Core Graphics — like painting directly onto the canvas inside the frame.

    The properties around contents are just the framing controls. contentsGravity is "how should the photo sit in the frame — stretched, centered, cropped?" contentsRect is "only show this corner of the photo" (which is the whole trick behind sprite sheets — one image, show one tile at a time). contentsScale is the easy-to-forget one: it tells the layer how many real pixels live in each point, and forgetting it is the number one reason a layer looks blurry on a Retina screen.

    shouldRasterize is a different idea: "this picture is expensive to draw and it isn't changing, so draw it once and photocopy the result." Great for static complexity, terrible for anything that animates — because then you're re-photocopying every frame.

    A layer is a rectangle of visual content. This lesson covers every way to get content into that rectangle — from assigning images to drawing with Core Graphics to rasterization for performance.

    The contents Property

    contents is typed as Any?. This is a historical artifact from Objective-C bridging. In practice, you assign a CGImage:

    let photoLayer = CALayer()
photoLayer.frame = CGRect(x: 20, y: 100, width: 300, height: 200)

// Assign a CGImage (not UIImage!)
let uiImage = UIImage(named: "mountain")
photoLayer.contents = uiImage?.cgImage

view.layer.addSublayer(photoLayer)

    If you assign something that isn't a CGImage, the layer will display nothing (or crash in debug). The type erasure exists because CALayer predates Swift and even predates the modern Objective-C type system.

    contentsScale

    contentsScale tells Core Animation how many pixels correspond to one point in the layer. On Retina displays, this should be 2.0 or 3.0.

    let layer = CALayer()
layer.frame = CGRect(x: 20, y: 100, width: 100, height: 100)

// WITHOUT setting contentsScale — image looks blurry on Retina
layer.contents = UIImage(named: "icon")?.cgImage

// FIX: match the screen scale
layer.contentsScale = UIScreen.main.scale

// Or match the image's scale
if let image = UIImage(named: "icon") {
    layer.contents = image.cgImage
    layer.contentsScale = image.scale
}

    Rule: Always set contentsScale when assigning a CGImage to contents. Forgetting this is the #1 cause of blurry layers.

    contentsGravity

    Controls how the contents image is positioned and scaled within the layer's bounds. Analogous to UIView.contentMode:

    let layer = CALayer()
layer.frame = CGRect(x: 0, y: 0, width: 300, height: 200)
layer.contents = UIImage(named: "photo")?.cgImage
layer.contentsScale = UIScreen.main.scale

// Scaling modes (change image size)
layer.contentsGravity = .resize            // stretch to fill, ignoring aspect
layer.contentsGravity = .resizeAspect      // fit inside, preserving aspect (letterbox)
layer.contentsGravity = .resizeAspectFill  // fill layer, preserving aspect (may crop)

// Positional modes (no scaling, position at edges/corners/center)
layer.contentsGravity = .center
layer.contentsGravity = .top
layer.contentsGravity = .bottom
layer.contentsGravity = .left
layer.contentsGravity = .right
layer.contentsGravity = .topLeft
layer.contentsGravity = .topRight
layer.contentsGravity = .bottomLeft
layer.contentsGravity = .bottomRight

    Mapping to UIView:

    CALayerContentsGravity UIView.ContentMode
    .resize .scaleToFill
    .resizeAspect .scaleAspectFit
    .resizeAspectFill .scaleAspectFill
    .center .center
    .top .top

    contentsRect

    A CGRect in unit coordinates (0,0 to 1,1) defining which portion of the image to display. This is powerful for sprite sheets and cropping:

    let layer = CALayer()
layer.frame = CGRect(x: 50, y: 100, width: 100, height: 100)
layer.contents = UIImage(named: "spritesheet")?.cgImage
layer.contentsScale = UIScreen.main.scale

// Show only the top-left quarter of the image
layer.contentsRect = CGRect(x: 0, y: 0, width: 0.5, height: 0.5)

// Show only the bottom-right quarter
layer.contentsRect = CGRect(x: 0.5, y: 0.5, width: 0.5, height: 0.5)

// Show the full image (default)
layer.contentsRect = CGRect(x: 0, y: 0, width: 1, height: 1)

    Sprite Sheet Example

    class SpriteSheetLayer {
    let layer = CALayer()
    let columns: Int
    let rows: Int

    init(image: UIImage, columns: Int, rows: Int, size: CGSize) {
        self.columns = columns
        self.rows = rows

        layer.frame = CGRect(origin: .zero, size: size)
        layer.contents = image.cgImage
        layer.contentsScale = image.scale
        layer.contentsGravity = .resize

        // Start at frame 0
        showFrame(column: 0, row: 0)
    }

    func showFrame(column: Int, row: Int) {
        let w = 1.0 / CGFloat(columns)
        let h = 1.0 / CGFloat(rows)
        layer.contentsRect = CGRect(
            x: CGFloat(column) * w,
            y: CGFloat(row) * h,
            width: w,
            height: h
        )
    }
}

    contentsCenter (9-Slice Scaling)

    contentsCenter defines a stretchable inner rectangle. The four corners remain fixed, the edges stretch in one direction, and the center stretches in both.

    let buttonLayer = CALayer()
buttonLayer.frame = CGRect(x: 20, y: 300, width: 280, height: 50)
buttonLayer.contents = UIImage(named: "button-background")?.cgImage
buttonLayer.contentsScale = UIScreen.main.scale

// Preserve the outer 25% on each edge; stretch the middle 50%
buttonLayer.contentsCenter = CGRect(x: 0.25, y: 0.25, width: 0.5, height: 0.5)

    Custom Drawing with Core Graphics

    Via CALayerDelegate

    class DrawingController: UIViewController, CALayerDelegate {
    let chartLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        chartLayer.frame = CGRect(x: 20, y: 100, width: 340, height: 200)
        chartLayer.backgroundColor = UIColor.systemGray6.cgColor
        chartLayer.contentsScale = UIScreen.main.scale
        chartLayer.delegate = self
        view.layer.addSublayer(chartLayer)

        chartLayer.setNeedsDisplay()
    }

    func draw(_ layer: CALayer, in ctx: CGContext) {
        let data: [CGFloat] = [0.2, 0.5, 0.3, 0.8, 0.6, 0.9, 0.4, 0.7]
        let barWidth = layer.bounds.width / CGFloat(data.count)

        for (index, value) in data.enumerated() {
            let barHeight = value * layer.bounds.height
            let barRect = CGRect(
                x: CGFloat(index) * barWidth + 2,
                y: layer.bounds.height - barHeight,
                width: barWidth - 4,
                height: barHeight
            )
            ctx.setFillColor(UIColor.systemBlue.cgColor)
            ctx.fill(barRect)
        }
    }
}

    Via CALayer Subclass

    class PieChartLayer: CALayer {
    var slices: [(value: CGFloat, color: UIColor)] = [] {
        didSet { setNeedsDisplay() }
    }

    override func draw(in ctx: CGContext) {
        let center = CGPoint(x: bounds.midX, y: bounds.midY)
        let radius = min(bounds.width, bounds.height) / 2 - 4
        let total = slices.reduce(0) { $0 + $1.value }

        var startAngle: CGFloat = -.pi / 2  // start at top

        for slice in slices {
            let endAngle = startAngle + (slice.value / total) * 2 * .pi

            ctx.move(to: center)
            ctx.addArc(center: center, radius: radius,
                       startAngle: startAngle, endAngle: endAngle,
                       clockwise: false)
            ctx.closePath()
            ctx.setFillColor(slice.color.cgColor)
            ctx.fillPath()

            startAngle = endAngle
        }
    }
}

// Usage:
let pie = PieChartLayer()
pie.frame = CGRect(x: 50, y: 350, width: 200, height: 200)
pie.contentsScale = UIScreen.main.scale
pie.slices = [
    (30, .systemRed),
    (45, .systemBlue),
    (25, .systemGreen)
]
view.layer.addSublayer(pie)

    setNeedsDisplay vs displayIfNeeded

    // Schedule a redraw for the next render pass (lazy, efficient)
layer.setNeedsDisplay()

// Redraw immediately if the layer is marked dirty
layer.displayIfNeeded()

// Check if a redraw is pending
if layer.needsDisplay() {
    print("Will redraw soon")
}

    setNeedsDisplay() is the normal path. displayIfNeeded() is for rare cases where you need the backing store updated immediately (e.g., before snapshotting).

    Rasterization

    shouldRasterize tells Core Animation to cache the layer's rendered output as a bitmap:

    let complexLayer = CALayer()
// ... add many sublayers with shadows, gradients, etc.

complexLayer.shouldRasterize = true
complexLayer.rasterizationScale = UIScreen.main.scale  // IMPORTANT!

    When to rasterize: - Complex layer trees that don't change every frame - Layers with expensive shadows (without shadowPath) - Static decorative elements with many sublayers

    When NOT to rasterize: - Layers whose content changes frequently (forces re-rasterization) - Animated layers (the cache becomes stale every frame) - Very large layers (huge texture in GPU memory)

    Memory cost: Each rasterized layer creates a texture in GPU memory. A 200x200 point layer at 3x scale = 600x600 pixels x 4 bytes = 1.44 MB of GPU memory.

    // Check rasterization in Instruments:
// "Color Hits Green and Misses Red" debug option
// Green = cache was reused (good)
// Red = cache was rebuilt (bad — means content changed)

    See 12-animation-basics.md — SwiftUI's .drawingGroup() modifier uses rasterization under the hood.

    Layer Appearance

    Start here — the plain idea

    This lesson is the styling toolbox: color, rounded corners, borders, shadows, transparency, and masks. Think of it like CSS for a layer — each property is a knob that changes how the box looks without changing what's inside it.

    Two ideas are worth getting straight up front. First, the mask: a mask is just a stencil. Wherever the stencil is solid, the layer shows through; wherever it's see-through, the layer is hidden. The stencil's color doesn't matter at all — only its transparency does. That single rule explains rounded-corner masks, fade-out gradients, and any weird shape you want.

    Second, the recurring villain of this lesson is off-screen rendering. Some effects (a shadow with no shadowPath, group opacity, certain corner-clipping) force the GPU to draw to a scratch pad first and then copy it back — every frame. That's slow. The fix is almost always "tell the GPU the shape ahead of time" (that's what shadowPath does) so it doesn't have to figure it out from scratch. The famous two-layer shadow trick exists for the same reason: clipping and shadowing fight each other, so you split them across two layers.

    This lesson covers every visual property you can set on a CALayer to control its look — colors, corners, borders, shadows, opacity, and masks. You'll also learn the critical performance considerations and the classic two-layer shadow trick.

    backgroundColor

    Layer colors use CGColor, not UIColor. This is because CALayer is framework-agnostic (used by both UIKit and AppKit):

    let layer = CALayer()
layer.frame = CGRect(x: 20, y: 100, width: 200, height: 200)

// CORRECT: use .cgColor
layer.backgroundColor = UIColor.systemBlue.cgColor

// Also valid: create CGColor directly
layer.backgroundColor = CGColor(red: 0.2, green: 0.4, blue: 0.8, alpha: 1.0)

// WRONG: this won't compile
// layer.backgroundColor = UIColor.systemBlue  // Type mismatch

    Corner Radius

    cornerRadius rounds the layer's corners:

    let card = CALayer()
card.frame = CGRect(x: 20, y: 100, width: 300, height: 180)
card.backgroundColor = UIColor.white.cgColor
card.cornerRadius = 16

view.layer.addSublayer(card)

    By default, cornerRadius only affects the layer's background and border. Sublayers and contents are NOT clipped. To clip content to the rounded corners, set masksToBounds:

    card.masksToBounds = true  // Now sublayers and contents are clipped to corners

bordered.cornerRadius = 12

bordered.borderWidth = 2.0
bordered.borderColor = UIColor.systemGray3.cgColor

view.layer.addSublayer(bordered)

    The border is drawn inside the layer bounds. A borderWidth of 4 means 4 points of the layer's edges are consumed by the border. The border always follows cornerRadius.

    Shadows

    Shadows are one of the most used (and most mis-used) layer features:

    let shadowLayer = CALayer()
shadowLayer.frame = CGRect(x: 40, y: 100, width: 240, height: 160)
shadowLayer.backgroundColor = UIColor.white.cgColor
shadowLayer.cornerRadius = 12

// Shadow properties
shadowLayer.shadowColor = UIColor.black.cgColor  // CGColor
shadowLayer.shadowOffset = CGSize(width: 0, height: 4)  // direction
shadowLayer.shadowRadius = 8   // blur amount
shadowLayer.shadowOpacity = 0.2 // 0.0 to 1.0 (default is 0 = no shadow)

view.layer.addSublayer(shadowLayer)

    Shadow Performance

    Without a shadowPath, Core Animation must compute the shadow shape by examining the layer's contents and sublayers every frame. This triggers expensive off-screen rendering.

    // SLOW: no shadowPath — GPU computes shape from pixel data
shadowLayer.shadowColor = UIColor.black.cgColor
shadowLayer.shadowOpacity = 0.3
shadowLayer.shadowRadius = 10
// Off-screen render pass every frame!

// FAST: explicit shadowPath — GPU knows the shape, no off-screen pass
shadowLayer.shadowPath = UIBezierPath(
    roundedRect: shadowLayer.bounds,
    cornerRadius: shadowLayer.cornerRadius
).cgPath

    Always set shadowPath for static layers. For layers that change size, update shadowPath in layoutSublayers():

    class ShadowedCardLayer: CALayer {
    override func layoutSublayers() {
        super.layoutSublayers()
        shadowPath = UIBezierPath(
            roundedRect: bounds,
            cornerRadius: cornerRadius
        ).cgPath
    }
}

    Opacity

    layer.opacity = 0.5   // 50% transparent (Float, not CGFloat)
layer.isOpaque = true  // Optimization hint: tells compositor there are no transparent pixels

    isOpaque doesn't change visual appearance — it's a performance hint. When true, the compositor skips blending calculations for this layer. Set it when your layer genuinely has no transparency.

    Group opacity: When a layer with sublayers has opacity < 1.0, Core Animation may need to composite the sublayers into a temporary buffer first, then apply opacity to the whole group. This is another off-screen render trigger.

    // If you see visual artifacts with group opacity:
layer.allowsGroupOpacity = false  // Applies opacity to each sublayer independently

    Masking

    The mask property takes another CALayer whose alpha channel defines visibility:

    class MaskDemoController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()

        // The content layer (what you want to show)
        let imageLayer = CALayer()
        imageLayer.frame = CGRect(x: 40, y: 100, width: 250, height: 250)
        imageLayer.contents = UIImage(named: "landscape")?.cgImage
        imageLayer.contentsScale = UIScreen.main.scale
        imageLayer.contentsGravity = .resizeAspectFill

        // The mask layer (alpha channel controls visibility)
        let maskLayer = CAShapeLayer()
        maskLayer.frame = imageLayer.bounds
        maskLayer.path = UIBezierPath(
            roundedRect: imageLayer.bounds,
            cornerRadius: 50
        ).cgPath
        maskLayer.fillColor = UIColor.black.cgColor  // only alpha matters

        imageLayer.mask = maskLayer

        view.layer.addSublayer(imageLayer)
    }
}

    How masks work: - Where the mask layer is opaque (alpha = 1.0) → content is fully visible - Where the mask layer is transparent (alpha = 0.0) → content is hidden - The mask layer's color hue doesn't matter — only the alpha channel - The mask uses the masked layer's coordinate space

    The Two-Layer Shadow Trick

    The problem: You want a layer with both rounded corners AND a drop shadow. But masksToBounds = true (needed for rounded corners on content) clips the shadow.

    // This WON'T work — shadow is clipped:
let broken = CALayer()
broken.cornerRadius = 16
broken.masksToBounds = true    // clips content to rounded corners
broken.shadowOpacity = 0.3     // shadow is ALSO clipped! invisible!

    The solution: Use two layers — one for the shadow, one for the rounded content.

    class ShadowedRoundedCardView: UIView {
    private let shadowContainer = CALayer()
    private let contentLayer = CALayer()

    override init(frame: CGRect) {
        super.init(frame: frame)
        setup()
    }

    required init?(coder: NSCoder) {
        super.init(coder: coder)
        setup()
    }

    private func setup() {
        // Outer layer: shadow (no masksToBounds)
        shadowContainer.shadowColor = UIColor.black.cgColor
        shadowContainer.shadowOffset = CGSize(width: 0, height: 4)
        shadowContainer.shadowRadius = 12
        shadowContainer.shadowOpacity = 0.25
        shadowContainer.cornerRadius = 16
        shadowContainer.backgroundColor = UIColor.white.cgColor
        layer.addSublayer(shadowContainer)

        // Inner layer: rounded corners with clipping
        contentLayer.cornerRadius = 16
        contentLayer.masksToBounds = true
        contentLayer.backgroundColor = UIColor.white.cgColor
        shadowContainer.addSublayer(contentLayer)
    }

    override func layoutSubviews() {
        super.layoutSubviews()
        shadowContainer.frame = bounds
        contentLayer.frame = shadowContainer.bounds

        // Performance: always set shadowPath
        shadowContainer.shadowPath = UIBezierPath(
            roundedRect: bounds,
            cornerRadius: 16
        ).cgPath
    }
}

    Now the shadow renders on the outer container (not clipped), and the inner layer clips its content to rounded corners.

    See 22-cafilter-private.md — private CAFilter APIs can achieve combined effects that normally require this two-layer trick.

    Sublayer Management

    Start here — the plain idea

    Layers are organized like a family tree, or like nested folders. Every layer can hold child layers inside it, and each child knows its one parent. When you build any real interface — a card with a header, body, and avatar — you're really just arranging this tree.

    The one rule that controls who covers whom is dead simple: layers are painted in the order they sit in the sublayers list, back to front. The last one added paints last, so it lands on top — exactly like stacking sheets of paper. If you want to cheat that order without rearranging the list, zPosition is the override: bump a layer's zPosition and it floats to the front visually while staying put in the array.

    Two niceties worth knowing the purpose of: layoutSublayers() is the spot where you reposition children when the parent resizes (so your card adapts to any size), and sublayerTransform is a single transform applied to all children at once — its main job is faking 3D perspective for a whole group instead of setting it on each child.

    Layers form a tree — every layer can have sublayers, and every sublayer has exactly one superlayer. Managing this tree is how you compose complex visual hierarchies. This lesson covers adding, removing, ordering, and laying out sublayers.

    Adding Sublayers

    let parentLayer = CALayer()
parentLayer.frame = CGRect(x: 0, y: 0, width: 400, height: 400)
parentLayer.backgroundColor = UIColor.systemGray6.cgColor

let childA = CALayer()
childA.frame = CGRect(x: 20, y: 20, width: 100, height: 100)
childA.backgroundColor = UIColor.systemRed.cgColor

let childB = CALayer()
childB.frame = CGRect(x: 60, y: 60, width: 100, height: 100)
childB.backgroundColor = UIColor.systemBlue.cgColor

// Add sublayers — order matters for drawing
parentLayer.addSublayer(childA)  // drawn first (back)
parentLayer.addSublayer(childB)  // drawn second (front)

view.layer.addSublayer(parentLayer)

    Insertion Methods

    // Insert at a specific index (0 = backmost)
parentLayer.insertSublayer(childC, at: 0)

// Insert below a specific sibling
parentLayer.insertSublayer(childC, below: childA)

// Insert above a specific sibling
parentLayer.insertSublayer(childC, above: childA)

    Removing Sublayers

    // Remove a specific sublayer
childA.removeFromSuperlayer()

// Remove all sublayers
parentLayer.sublayers?.forEach { $0.removeFromSuperlayer() }

// Alternative: set sublayers to nil (also removes all)
parentLayer.sublayers = nil

    When you call removeFromSuperlayer(), the layer is removed from the tree immediately. If nothing else holds a strong reference to it, it's deallocated.

    See 18-memory-management.md — superlayer is a weak reference. The sublayer array holds strong references to children. Removing a sublayer releases the parent's strong reference.

    The Sublayer Array

    let parent = CALayer()
let a = CALayer()
let b = CALayer()
let c = CALayer()

parent.addSublayer(a)
parent.addSublayer(b)
parent.addSublayer(c)

// Access the sublayer array
if let children = parent.sublayers {
    print(children.count)  // 3
    print(children[0] === a)  // true — first added, index 0
    print(children[2] === c)  // true — last added, index 2
}

// Access the parent from a child
print(b.superlayer === parent)  // true

    Drawing Order

    Sublayers are drawn back-to-front based on their index in the sublayers array. Lower index = drawn first = behind higher index layers.

    // Drawing order: A (back) → B → C (front)
parent.addSublayer(a)  // index 0
parent.addSublayer(b)  // index 1
parent.addSublayer(c)  // index 2

    zPosition Overrides Visual Order

    zPosition changes the visual stacking without changing the sublayers array:

    a.zPosition = 10  // Now A is visually in front of B and C
// But parent.sublayers is still [a, b, c]

    // Example: bring a specific layer to front visually
func bringToFront(_ layer: CALayer) {
    guard let siblings = layer.superlayer?.sublayers else { return }
    let maxZ = siblings.map(\.zPosition).max() ?? 0
    layer.zPosition = maxZ + 1
}

// Example: send to back
func sendToBack(_ layer: CALayer) {
    guard let siblings = layer.superlayer?.sublayers else { return }
    let minZ = siblings.map(\.zPosition).min() ?? 0
    layer.zPosition = minZ - 1
}

    Important: zPosition is a CGFloat. It defaults to 0 for all layers. Two layers with the same zPosition are drawn in sublayer array order.

    Reordering Sublayers

    There is no built-in bringSubLayerToFront method like UIView has. To reorder in the sublayer array itself:

    extension CALayer {
    func bringSublayerToFront(_ sublayer: CALayer) {
        sublayer.removeFromSuperlayer()
        addSublayer(sublayer)  // appends to end = frontmost
    }

    func sendSublayerToBack(_ sublayer: CALayer) {
        sublayer.removeFromSuperlayer()
        insertSublayer(sublayer, at: 0)  // index 0 = backmost
    }

    func moveSublayer(_ sublayer: CALayer, toIndex index: Int) {
        sublayer.removeFromSuperlayer()
        insertSublayer(sublayer, at: UInt32(index))
    }
}

    Custom Layout with layoutSublayers()

    Override layoutSublayers() in a CALayer subclass to position children when the layer's bounds change:

    class HorizontalStackLayer: CALayer {
    var spacing: CGFloat = 8

    override func layoutSublayers() {
        super.layoutSublayers()

        guard let children = sublayers, !children.isEmpty else { return }

        let totalSpacing = spacing * CGFloat(children.count - 1)
        let itemWidth = (bounds.width - totalSpacing) / CGFloat(children.count)
        let itemHeight = bounds.height

        for (index, child) in children.enumerated() {
            child.frame = CGRect(
                x: CGFloat(index) * (itemWidth + spacing),
                y: 0,
                width: itemWidth,
                height: itemHeight
            )
        }
    }
}

// Usage:
let stack = HorizontalStackLayer()
stack.frame = CGRect(x: 20, y: 100, width: 340, height: 80)
stack.spacing = 10

for color in [UIColor.systemRed, .systemBlue, .systemGreen, .systemOrange] {
    let item = CALayer()
    item.backgroundColor = color.cgColor
    item.cornerRadius = 8
    stack.addSublayer(item)
}

view.layer.addSublayer(stack)
// Children are automatically laid out horizontally

    sublayerTransform

    Apply a transform to ALL sublayers. This is most commonly used to add perspective for 3D effects:

    let container = CALayer()
container.frame = CGRect(x: 0, y: 100, width: 400, height: 400)

// Add perspective to all sublayers
var perspective = CATransform3DIdentity
perspective.m34 = -1.0 / 500  // perspective foreshortening
container.sublayerTransform = perspective

// Now any sublayer with a 3D rotation will show perspective
let card = CALayer()
card.frame = CGRect(x: 100, y: 100, width: 200, height: 200)
card.backgroundColor = UIColor.systemBlue.cgColor
card.transform = CATransform3DMakeRotation(.pi / 6, 0, 1, 0)  // 30° Y rotation
container.addSublayer(card)
// The card appears to rotate in 3D with proper perspective!

    Without sublayerTransform, each sublayer would need its own m34 perspective value set on its individual transform.

    Implicit Animations

    Start here — the plain idea

    Implicit animation = change a layer property and Core Animation tweens it for you, no code. Set a standalone layer's color to red and it doesn't snap — it fades to red over a quarter second, automatically. That's the "implicit" part: you never asked for an animation, you just changed a value, and the system filled in all the in-between frames.

    What's really happening under the hood is a tiny lookup. Every time you set an animatable property, the layer quietly asks, "is there an action I should run for this change?" If it finds one (and by default it does), it builds an animation from the old value to the new value and plays it. Think of it as the layer having a standing instruction: "whenever my color changes, ease into it."

    This also explains the famous surprise: view.layer.opacity = 0 doesn't animate, but a standalone layer's does. The reason is that UIView deliberately answers "no action" to that lookup — it wants you to use UIView.animate instead. So implicit animations are alive and well; UIView just mutes them on its own layer.

    Core Animation was designed around a radical idea: changing a property should animate automatically. When you set a layer's opacity to 0.0, it doesn't snap -- it fades. This behavior is called an implicit animation, and it is the default for every standalone CALayer.

    See 02-calayer-fundamentals.md -- this lesson builds on the layer properties introduced there.

    How Implicit Animations Work

    When you change an animatable property on a CALayer, Core Animation intercepts the change and creates a CABasicAnimation behind the scenes. The layer transitions from the old value to the new value over a default duration of 0.25 seconds.

    import UIKit

class ImplicitAnimationViewController: UIViewController {
    let colorLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        colorLayer.frame = CGRect(x: 100, y: 200, width: 150, height: 150)
        colorLayer.backgroundColor = UIColor.systemBlue.cgColor
        colorLayer.cornerRadius = 12
        view.layer.addSublayer(colorLayer)
    }

    override func touchesBegan(_ touches: Set<UITouch>, with event: UIEvent?) {
        // This single line triggers an implicit animation
        colorLayer.backgroundColor = UIColor.systemRed.cgColor
    }
}

    Tap the screen and the layer smoothly transitions from blue to red over 0.25 seconds. You wrote zero animation code.

    Animatable Properties

    Not every property animates implicitly. The following properties do:

    Property Type What Animates
    position CGPoint Layer moves
    bounds CGRect Layer resizes
    opacity Float Layer fades
    backgroundColor CGColor Color transitions
    cornerRadius CGFloat Corners round/square
    transform CATransform3D Scale, rotate, translate
    borderWidth CGFloat Border grows/shrinks
    borderColor CGColor Border color transitions
    shadowRadius CGFloat Shadow blur changes
    shadowOffset CGSize Shadow moves
    shadowOpacity Float Shadow fades
    shadowColor CGColor Shadow color transitions
    shadowPath CGPath Shadow shape morphs
    contents Any (CGImage) Cross-fades image
    contentsRect CGRect Content crop animates
    sublayerTransform CATransform3D Child perspective changes

    If a property is not in this list -- such as isHidden, name, or delegate -- setting it takes effect immediately with no animation.

    Why UIView Layers Don't Implicitly Animate

    Try this and nothing animates:

    view.layer.opacity = 0.0  // snaps immediately -- no animation

    UIView disables implicit animations on its backing layer. During every property change, UIView's layer delegate returns NSNull() as the action, which tells Core Animation "do nothing." This is intentional -- UIView wants you to use UIView.animate(withDuration:) instead.

    Standalone layers (layers you create yourself with CALayer()) are not backed by a UIView and do implicitly animate.

    let standalone = CALayer()
standalone.frame = CGRect(x: 50, y: 50, width: 80, height: 80)
standalone.backgroundColor = UIColor.green.cgColor
view.layer.addSublayer(standalone)

// This WILL animate because standalone is not a UIView's backing layer
standalone.opacity = 0.0

    The CAAction Protocol

    When a property changes, Core Animation looks for an object that conforms to CAAction:

    public protocol CAAction {
    func run(forKey event: String, object anObject: Any, arguments dict: [AnyHashable: Any]?)
}

    CABasicAnimation conforms to CAAction. So when the system finds an action, it creates and runs an animation.

    Action Search Order

    Core Animation searches for actions in this exact order:

    1. Layer delegate: calls action(for:forKey:) on the delegate. If it returns NSNull(), stop -- no action. If it returns an action object, use it. If it returns nil, continue searching.
    2. Layer's actions dictionary: checks layer.actions[key]. If found, use it.
    3. Layer's style dictionary: checks recursively (rarely used).
    4. Class default: calls CALayer.defaultAction(forKey:).

    UIView intercepts at step 1. When you call UIView.animate(withDuration:), UIView's delegate returns a real CABasicAnimation instead of NSNull() -- that is how animation blocks work.

    Disabling Specific Implicit Animations

    Use the actions dictionary with NSNull() to suppress animation on specific properties:

    let layer = CALayer()

// This now snaps instantly -- no animation

// But this still animates because we only disabled "position"
layer.opacity = 0.5

    To disable all implicit animations, use CATransaction:

    CATransaction.begin()

layer.position = CGPoint(x: 200, y: 400)
layer.opacity = 0.5
CATransaction.commit()

    Getting the Default Action

    You can inspect what action Core Animation would use for a key:

    if let action = CALayer.defaultAction(forKey: "opacity") {
    print(type(of: action))  // CABasicAnimation
}

    For a UIView's backing layer, action(for:forKey:) returns NSNull() outside an animation block and a CABasicAnimation inside one.

    How UIView.animate Works Under the Hood

    When you write:

    UIView.animate(withDuration: 0.5) {
    myView.alpha = 0.0
}

    Here is what actually happens:

    1. UIKit opens a CATransaction with duration 0.5.
    2. Inside the closure, setting alpha changes layer.opacity.
    3. The layer asks its delegate (UIView) for an action.
    4. Because we are inside an animation block, UIView returns a CABasicAnimation configured with the right fromValue, toValue, and duration.
    5. The animation is added to the layer.
    6. CATransaction.commit() is called implicitly.

    This is why you must change properties inside the animation closure -- the delegate only returns real actions while the animation block is active.

    CATransaction

    Start here — the plain idea

    A CATransaction is just an envelope. You stuff a bunch of layer changes into it, seal it, and Core Animation delivers them all together in one frame. Without an envelope, changes might trickle out one at a time and the screen would tear or flicker. Think of it like saving a document: you make a batch of edits, then hit save once — everything lands at the same moment.

    Here's the part people don't realize: you're always inside an envelope, even when you don't make one. Every run loop tick, Core Animation quietly opens an implicit transaction, collects everything you changed, and commits it at the end. So when three property changes animate together "for free," that's the implicit envelope doing its job.

    You only reach for an explicit transaction (begin() … commit()) when you want to set the rules on that envelope: make the animations last longer, give them an easing curve, turn animation off entirely for setup code, or run a completion block when they finish. And the punchline: UIView.animate is literally a friendly wrapper that opens one of these envelopes for you.

    Every property change you make on a CALayer is part of a transaction. CATransaction is the mechanism Core Animation uses to batch property changes into atomic commits, ensuring that related changes are rendered together in the same frame. Without transactions, each property change would render independently, causing visual tearing and inconsistency.

    See 07-implicit-animations.md -- implicit animations rely on transactions to control their timing and behavior.

    Implicit Transactions

    You do not need to create a transaction manually. Every run loop cycle, Core Animation wraps all property changes in an implicit transaction. When the run loop finishes, the transaction commits and all changes render together.

    // These three changes happen in the same implicit transaction
// They all animate together, starting at the same time
layer.position = CGPoint(x: 200, y: 300)
layer.opacity = 0.5
layer.backgroundColor = UIColor.red.cgColor

    All three animations begin simultaneously when the current run loop cycle ends, because they are batched into one implicit transaction.

    Explicit Transactions

    Wrap property changes in CATransaction.begin() and CATransaction.commit() to create an explicit transaction with custom parameters.

    CATransaction.begin()
CATransaction.setAnimationDuration(1.0)

layer.position = CGPoint(x: 200, y: 300)
layer.opacity = 0.5
layer.backgroundColor = UIColor.red.cgColor

CATransaction.commit()

    All three properties now animate over 1 second instead of the default 0.25.

    Setting Animation Duration

    CATransaction.setAnimationDuration(_:) controls how long implicit animations last within the transaction.

    CATransaction.begin()
CATransaction.setAnimationDuration(2.0)  // 2 seconds

layer.cornerRadius = 40
layer.borderWidth = 4
layer.borderColor = UIColor.white.cgColor

CATransaction.commit()

    This only affects implicit animations. Explicit animations (those you create with CABasicAnimation) use their own duration property.

    Setting the Timing Function

    CATransaction.setAnimationTimingFunction(_:) controls the easing curve for all implicit animations in the transaction.

    CATransaction.begin()
CATransaction.setAnimationDuration(0.8)
CATransaction.setAnimationTimingFunction(
    CAMediaTimingFunction(name: .easeInEaseOut)
)

layer.position = CGPoint(x: 300, y: 500)

CATransaction.commit()

    Available built-in timing functions:

    Name Behavior
    .linear Constant speed
    .easeIn Starts slow, ends fast
    .easeOut Starts fast, ends slow
    .easeInEaseOut Slow start and end
    .default Slightly ease-in ease-out (the CA default)

    Disabling All Implicit Animations

    CATransaction.setDisableActions(true) suppresses every implicit animation within the transaction. The properties change instantly.

    CATransaction.begin()
CATransaction.setDisableActions(true)

// All of these snap immediately -- no animation
layer.position = CGPoint(x: 100, y: 100)
layer.opacity = 1.0
layer.backgroundColor = UIColor.blue.cgColor
layer.cornerRadius = 0

CATransaction.commit()

    This is essential for initial setup. When you first configure a layer's properties, you typically don't want them to animate from their default values.

    override func viewDidLoad() {
    super.viewDidLoad()

    let layer = CALayer()
    view.layer.addSublayer(layer)

    // Suppress animations during initial configuration
    CATransaction.begin()
    CATransaction.setDisableActions(true)
    layer.frame = CGRect(x: 50, y: 100, width: 200, height: 200)
    layer.backgroundColor = UIColor.systemTeal.cgColor
    layer.cornerRadius = 20
    layer.shadowRadius = 10
    layer.shadowOpacity = 0.3
    layer.shadowOffset = CGSize(width: 0, height: 4)
    CATransaction.commit()
}

    Completion Blocks

    CATransaction.setCompletionBlock { } executes a closure after all animations in the transaction finish.

    CATransaction.begin()
CATransaction.setAnimationDuration(0.5)
CATransaction.setCompletionBlock {
    print("All animations complete!")
    // Chain the next animation, update state, etc.
}

layer.position = CGPoint(x: 250, y: 400)
layer.opacity = 0.3

CATransaction.commit()

    The completion block fires once, after the longest animation in the transaction finishes.

    See 07-closures.md -- completion blocks are escaping closures.

    Nesting Transactions

    Transactions can be nested. Inner transactions inherit settings from outer transactions unless explicitly overridden.

    CATransaction.begin()
CATransaction.setAnimationDuration(2.0)  // outer: 2 seconds

layer1.opacity = 0.0  // animates over 2 seconds

    CATransaction.begin()
    CATransaction.setAnimationDuration(0.3)  // inner: 0.3 seconds

    layer2.opacity = 0.0  // animates over 0.3 seconds

    CATransaction.commit()  // inner commit

layer3.opacity = 0.0  // animates over 2 seconds (back to outer)

CATransaction.commit()  // outer commit

    Every begin() must be balanced with a commit(). Forgetting a commit() causes undefined behavior.

    The UIView.animate Connection

    UIView.animate(withDuration:animations:completion:) creates a CATransaction internally:

    // This UIKit code:
UIView.animate(withDuration: 0.5, animations: {
    myView.alpha = 0.0
    myView.center = CGPoint(x: 200, y: 300)
}, completion: { finished in
    print("Done")
})

// Is roughly equivalent to:
CATransaction.begin()
CATransaction.setAnimationDuration(0.5)
CATransaction.setCompletionBlock {
    print("Done")
}
// UIView configures its delegate to return CABasicAnimation actions
myView.layer.opacity = 0.0
myView.layer.position = CGPoint(x: 200, y: 300)
CATransaction.commit()

    The key difference is that UIView.animate also modifies the action lookup so that backing layers return real animations instead of NSNull().

    CATransaction.flush()

    flush() forces the current implicit transaction to commit immediately rather than waiting for the run loop to end. You rarely need this.

    // Force immediate commit
layer.backgroundColor = UIColor.red.cgColor
CATransaction.flush()  // commit now, don't wait for run loop

    One use case: when you need layout to be finalized before measuring. But in practice, layoutIfNeeded() is preferred for that purpose.

    Thread Safety: lock() and unlock()

    CATransaction.lock() and CATransaction.unlock() provide thread safety when modifying layer properties from background threads. However, you should almost never modify layers off the main thread.

    // Only if you must touch layers from a background thread (rare)
CATransaction.lock()
layer.contents = newImage
CATransaction.unlock()

    In modern code, dispatch to the main queue instead:

    DispatchQueue.main.async {
    layer.contents = newImage
}

    Practical Example: Multi-Property Animation With Completion

    func animateCardAppearance(_ cardLayer: CALayer) {
    // Start off-screen, transparent, and small
    CATransaction.begin()
    CATransaction.setDisableActions(true)
    cardLayer.opacity = 0
    cardLayer.transform = CATransform3DMakeScale(0.8, 0.8, 1.0)
    cardLayer.position = CGPoint(x: view.center.x, y: view.center.y + 50)
    CATransaction.commit()

    // Animate to final state
    CATransaction.begin()
    CATransaction.setAnimationDuration(0.6)
    CATransaction.setAnimationTimingFunction(
        CAMediaTimingFunction(name: .easeOut)
    )
    CATransaction.setCompletionBlock {
        print("Card is fully visible")
    }

    cardLayer.opacity = 1
    cardLayer.transform = CATransform3DIdentity
    cardLayer.position = view.center

    CATransaction.commit()
}

    This pattern -- disable actions for setup, then animate to the target state -- appears constantly in production code.

    Common Mistakes

    Forgetting to commit: Every begin() needs a matching commit(). Use a consistent pattern where you write both immediately, then fill in the middle.

    Mixing with UIView.animate: If you nest a CATransaction inside a UIView.animate block, the transaction settings may conflict. Generally, use one or the other.

    Setting duration on explicit animations: CATransaction.setAnimationDuration only affects implicit animations. If you add a CABasicAnimation manually, set its duration property directly.

    CABasicAnimation

    Start here — the plain idea

    CABasicAnimation is you taking the wheel. Implicit animations animate for you automatically; here you fill out the form yourself: "animate this property, from here, to there, over this long," then hand the form to the layer. That's the whole object — a from, a to, a duration, and a key path naming the property to move.

    The one idea that will save you hours is the model-vs-presentation split. When you add an animation, it only paints the presentation layer — the temporary on-screen copy. The real value (the model layer) never changed. So the moment the animation ends, the layer snaps back to where it "truly" was, and beginners think their code is broken. It isn't. The fix is simple: also set the property on the model layer to its final value, so when the animation lifts, there's nothing to snap back to. Think of the animation as a movie playing over a still photo — when the movie ends, you see the photo, so make sure the photo already shows the ending.

    Everything fancier — springs, keyframes, groups — is built on this same form. Learn this one well.

    CABasicAnimation is the explicit animation workhorse of Core Animation. Unlike implicit animations where you just change a property, explicit animations give you full control: you create an animation object, configure its parameters, and add it to a layer. This is the foundation that CASpringAnimation, CAKeyframeAnimation, and CAAnimationGroup all build upon.

    See 12-animation-basics.md -- SwiftUI's animation system creates Core Animation objects under the hood.

    Creating a Basic Animation

    Every CABasicAnimation targets a single animatable property identified by a string key path.

    let fadeOut = CABasicAnimation(keyPath: "opacity")
fadeOut.fromValue = 1.0
fadeOut.toValue = 0.0
fadeOut.duration = 0.5

layer.add(fadeOut, forKey: "fadeOut")

    The forKey parameter when adding is a label you choose. It lets you identify, replace, or remove the animation later. It is not the same as the keyPath.

    fromValue, toValue, and byValue

    These three properties define the animation endpoints. You can combine them in different ways:

    Set Behavior
    fromValue + toValue Animates from fromValue to toValue
    fromValue + byValue Animates from fromValue to fromValue + byValue
    toValue only Animates from current presentation value to toValue
    byValue only Animates from current value by the delta byValue
    fromValue only Animates from fromValue to current model value

    The most common combination is fromValue + toValue, but toValue-only is useful when you want to animate from wherever the layer currently is.

    // Animate from wherever the layer is to position (300, 400)
let move = CABasicAnimation(keyPath: "position")
move.toValue = CGPoint(x: 300, y: 400)
move.duration = 0.6
layer.add(move, forKey: "move")

    Key Paths Reference

    The keyPath string accesses nested properties using dot notation:

    "position" CGPoint Center position
    "position.x" CGFloat Horizontal position only
    "position.y" CGFloat Vertical position only
    "bounds" CGRect Size and origin
    "bounds.size" CGSize Size only
    "bounds.size.width" CGFloat Width only
    "backgroundColor" CGColor Fill color
    "cornerRadius" CGFloat Corner rounding
    "borderWidth" CGFloat Border thickness
    "borderColor" CGColor Border color
    "shadowRadius" CGFloat Shadow blur
    "shadowOffset" CGSize Shadow displacement
    "shadowOpacity" Float Shadow visibility
    "shadowColor" CGColor Shadow color
    "transform" CATransform3D Full 3D transform
    "transform.scale" CGFloat Uniform scale
    "transform.scale.x" CGFloat Horizontal scale
    "transform.scale.y" CGFloat Vertical scale
    "transform.rotation.z" CGFloat Z-axis rotation (radians)
    "transform.rotation.x" CGFloat X-axis rotation
    "transform.translation" CGSize Translation offset
    "transform.translation.x" CGFloat Horizontal translation

    The "Jump Back" Problem

    This is the single most important concept when learning explicit animations. Try this:

    let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = layer.position.x
anim.toValue = 300
anim.duration = 1.0
layer.add(anim, forKey: "moveRight")

    The layer slides right... then jumps back to its original position. Why?

    Model Layer vs Presentation Layer

    Core Animation maintains two layer trees:

    • Model layer (layer): the "truth" -- the values you set in code.
    • Presentation layer (layer.presentation()): what is actually on screen during animation.

    When you add an animation, it only affects the presentation layer. The model layer's position.x never changed. When the animation completes and is removed, the presentation layer snaps back to match the model layer.

    Fix 1: Set the Model Layer First (Preferred)

    let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = layer.position.x
anim.toValue = 300
anim.duration = 1.0

// Set the final value on the model layer BEFORE adding the animation
layer.position.x = 300

layer.add(anim, forKey: "moveRight")

    Now when the animation is removed, the model layer already has the correct final value. No jump.

    Fix 2: fillMode + isRemovedOnCompletion (Less Preferred)

    let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = layer.position.x
anim.toValue = 300
anim.duration = 1.0
anim.fillMode = .forwards
anim.isRemovedOnCompletion = false

layer.add(anim, forKey: "moveRight")

    This keeps the animation alive after completion, so the presentation layer stays at the final value. But the model layer is still wrong, which causes bugs: hit testing uses the model layer, and if you add another animation later, it starts from the wrong position. Use Fix 1 instead.

    Animation Delegate

    Conform to CAAnimationDelegate to receive callbacks:

    class AnimationController: NSObject, CAAnimationDelegate {
    let layer = CALayer()

    func startAnimation() {
        let anim = CABasicAnimation(keyPath: "opacity")
        anim.fromValue = 1.0
        anim.toValue = 0.0
        anim.duration = 0.5
        anim.delegate = self

        layer.opacity = 0.0  // Fix 1: set final value first
        layer.add(anim, forKey: "fadeOut")
    }

    func animationDidStart(_ anim: CAAnimation) {
        print("Animation started")
    }

    func animationDidStop(_ anim: CAAnimation, finished flag: Bool) {
        if flag {
            print("Animation completed naturally")
        } else {
            print("Animation was interrupted or removed")
        }
    }
}

    The finished flag is false if the animation was removed before completing (e.g., by calling removeAnimation(forKey:) or adding a new animation with the same key).

    Additive Animations

    By default, an animation replaces the layer's property value during its run. With isAdditive = true, the animation value is added to the model layer value instead.

    // Move 100 points to the right from wherever the layer is
let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = 0    // start: model value + 0
anim.toValue = 100    // end: model value + 100
anim.duration = 0.5
anim.isAdditive = true

layer.position.x += 100  // update model layer
layer.add(anim, forKey: "moveRight")

    Additive animations are crucial for interruptible animations. If a new animation starts while one is running, additive animations compose naturally instead of jumping.

    See 12-caanimationgroup.md -- additive animations become powerful when combined in groups.

    class ExplicitAnimationViewController: UIViewController {
    let boxLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        scale.toValue = 1.3
        scale.duration = 0.2
        scale.autoreverses = true

        // Update model layer
        boxLayer.position = target

        // Add both animations
        boxLayer.add(move, forKey: "moveToTouch")
        boxLayer.add(scale, forKey: "popScale")
    }
}

    CAKeyframeAnimation

    Start here — the plain idea

    If CABasicAnimation is "go from A to B," then CAKeyframeAnimation is "hit these stops along the way: A, then C, then F, then B." You hand it a list of values and it walks through all of them in order. Think of it like setting waypoints on a road trip instead of just a start and a destination.

    Two extra dials make it powerful. keyTimes lets you say when each stop happens (spend 80% of the time on the first leg, then rush the rest) — it's like marking each waypoint with a clock time. And instead of a list of values, you can hand it an actual path (a CGPath) and the layer will ride along it like a train on a track — perfect for arcs, orbits, and figure-eights.

    The classic uses fall right out of this: a shake is just a list of left-right wiggles; a bounce is a list of heights that get smaller; a blinking cursor is "discrete mode" — no smooth blending, just snap between on and off. Same tool, different waypoints.

    CABasicAnimation interpolates between two values. CAKeyframeAnimation interpolates through an arbitrary number of values, giving you frame-by-frame control over the animation curve. You can also animate along a CGPath, making it the go-to tool for path-based motion, shake effects, and any animation that needs more than a simple A-to-B transition.

    See 09-cabasicanimation.md -- CAKeyframeAnimation inherits from CAPropertyAnimation, the same parent as CABasicAnimation.

    Creating a Keyframe Animation

    The simplest form provides an array of values to animate through:

    let colorAnim = CAKeyframeAnimation(keyPath: "backgroundColor")
colorAnim.values = [
    UIColor.systemRed.cgColor,
    UIColor.systemOrange.cgColor,
    UIColor.systemYellow.cgColor,
    UIColor.systemGreen.cgColor,
    UIColor.systemBlue.cgColor
]
colorAnim.duration = 3.0

layer.backgroundColor = UIColor.systemBlue.cgColor  // final value
layer.add(colorAnim, forKey: "colorCycle")

    The animation passes through each color in sequence, spending equal time on each segment by default.

    Controlling Timing with keyTimes

    keyTimes is an array of NSNumber values from 0.0 to 1.0 that define when each keyframe occurs relative to the total duration.

    let bounce = CAKeyframeAnimation(keyPath: "position.y")
bounce.values = [300, 100, 250, 150, 220, 180, 200]
bounce.keyTimes = [0, 0.2, 0.4, 0.55, 0.7, 0.85, 1.0]
bounce.duration = 1.5

layer.position.y = 200  // final value
layer.add(bounce, forKey: "bounce")

    Rules for keyTimes: - Must start with 0.0 and end with 1.0 - Must have the same count as values - Values must be monotonically increasing

    Without keyTimes, the values are evenly distributed across the duration.

    Path Animations

    For position animations, you can provide a CGPath instead of a values array. The layer follows the path.

    let path = UIBezierPath()
path.move(to: CGPoint(x: 50, y: 400))
path.addCurve(
    to: CGPoint(x: 300, y: 400),
    controlPoint1: CGPoint(x: 100, y: 100),
    controlPoint2: CGPoint(x: 250, y: 600)
)

let follow = CAKeyframeAnimation(keyPath: "position")
follow.path = path.cgPath
follow.duration = 2.0

layer.position = CGPoint(x: 300, y: 400)  // final value
layer.add(follow, forKey: "followPath")

    Circular Path

    let circlePath = UIBezierPath(
    arcCenter: view.center,
    radius: 100,
    startAngle: 0,
    endAngle: .pi * 2,
    clockwise: true
)

let orbit = CAKeyframeAnimation(keyPath: "position")
orbit.path = circlePath.cgPath
orbit.duration = 3.0
orbit.repeatCount = .infinity

layer.add(orbit, forKey: "orbit")

    Rotation Along the Path

    rotationMode controls whether the animated object rotates to follow the path tangent:

    let follow = CAKeyframeAnimation(keyPath: "position")
follow.path = curvePath.cgPath
follow.duration = 2.0
follow.rotationMode = .rotateAuto  // object faces direction of travel

layer.add(follow, forKey: "followWithRotation")

    Without .rotateAuto, the layer maintains its original orientation while moving along the path. With it, the layer rotates so its right edge always faces forward.

    Use .rotateAutoReverse if the object should face backward along the path.

    Calculation Mode

    calculationMode determines how the animation interpolates between keyframe values:

    Mode Behavior
    .linear Straight-line interpolation between each pair of values (default)
    .discrete No interpolation -- jumps to each value instantly
    .paced Constant speed -- ignores keyTimes and distributes evenly by distance
    .cubic Smooth curve through values using Catmull-Rom splines
    .cubicPaced Smooth curve with constant speed 
    let smooth = CAKeyframeAnimation(keyPath: "position")
smooth.values = [
    CGPoint(x: 50, y: 300),
    CGPoint(x: 150, y: 100),
    CGPoint(x: 250, y: 300),
    CGPoint(x: 350, y: 100)
].map { NSValue(cgPoint: $0) }
smooth.calculationMode = .cubic  // smooth curve instead of sharp corners
smooth.duration = 2.0

layer.add(smooth, forKey: "smoothPath")

    .paced is especially useful when your keyframe positions are unevenly spaced but you want the animation to move at constant velocity.

    Spline Control (Cubic Mode)

    When using .cubic calculation mode, you can fine-tune the curve with Kochanek-Bartels spline parameters:

    • tensionValues: controls tightness of the curve (range -1 to 1, default 0). Positive = tighter.
    • continuityValues: controls sharpness at keyframes (range -1 to 1, default 0). Positive = sharper.
    • biasValues: controls curve direction bias (range -1 to 1, default 0).
    let anim = CAKeyframeAnimation(keyPath: "position")
anim.values = points.map { NSValue(cgPoint: $0) }
anim.calculationMode = .cubic
anim.tensionValues = [0, 0.5, 0.5, 0]  // tighter in the middle
anim.duration = 2.0

layer.add(anim, forKey: "spline")

    These arrays should have the same count as values. In practice, the defaults work well for most cases.

    Shake Animation

    A classic keyframe recipe -- shake a layer horizontally to indicate an error:

    func shakeLayer(_ layer: CALayer) {
    let shake = CAKeyframeAnimation(keyPath: "transform.translation.x")
    shake.values = [0, -12, 12, -12, 12, -6, 6, -3, 3, 0]
    shake.keyTimes = [0, 0.1, 0.2, 0.3, 0.4, 0.55, 0.7, 0.8, 0.9, 1.0]
    shake.duration = 0.5

    layer.add(shake, forKey: "shake")
}

    Using transform.translation.x instead of position.x avoids needing to update the model layer, since the translation resets to 0 at the end.

    Bounce-In Animation

    Simulate a ball dropping and bouncing:

    func bounceIn(_ layer: CALayer, to finalY: CGFloat) {
    let bounce = CAKeyframeAnimation(keyPath: "position.y")
    bounce.values = [
        -50,        // start above screen
        finalY,     // hit target
        finalY - 40, // first bounce up
        finalY,     // back down
        finalY - 15, // second bounce
        finalY,     // settle
        finalY - 5,  // tiny bounce
        finalY       // done
    ]
    bounce.keyTimes = [0, 0.3, 0.45, 0.55, 0.65, 0.75, 0.85, 1.0]
    bounce.duration = 1.0
    bounce.timingFunctions = [
        CAMediaTimingFunction(name: .easeIn),   // falling
        CAMediaTimingFunction(name: .easeOut),   // bouncing up
        CAMediaTimingFunction(name: .easeIn),    // falling
        CAMediaTimingFunction(name: .easeOut),   // bouncing up
        CAMediaTimingFunction(name: .easeIn),    // falling
        CAMediaTimingFunction(name: .easeOut),   // bouncing up
        CAMediaTimingFunction(name: .easeIn)     // settling
    ]

    layer.position.y = finalY
    layer.add(bounce, forKey: "bounceIn")
}

    Note: timingFunctions applies per-segment (between consecutive keyframes), so it has one fewer element than values.

    Per-Segment Timing Functions

    You can set a different easing curve for each segment between keyframes:

    let anim = CAKeyframeAnimation(keyPath: "opacity")
anim.values = [0, 1, 1, 0]
anim.keyTimes = [0, 0.2, 0.8, 1.0]
anim.timingFunctions = [
    CAMediaTimingFunction(name: .easeOut),  // fade in quickly
    CAMediaTimingFunction(name: .linear),    // hold
    CAMediaTimingFunction(name: .easeIn)     // fade out slowly
]
anim.duration = 3.0

layer.add(anim, forKey: "fadeInHoldFadeOut")

    See 13-animation-advanced.md for SwiftUI's KeyframeAnimator, which provides similar multi-step control with a declarative API.

    The foundation

    • CAKeyframeAnimation = animate through a list of waypoint values, not just A-to-B.
    • values = the ordered stops the property passes through.
    • keyTimes = when each stop happens (0 to 1); must start at 0, end at 1, and match the values count.
    • path = ride the layer along a CGPath like a train on a track — ideal for arcs and orbits.
    • rotationMode = .rotateAuto = turn the layer to face the direction it's traveling.
    • calculationMode = how to blend between stops: linear, discrete (snap), paced (constant speed), cubic (smooth).
    • timingFunctions = a separate easing curve per segment; one fewer than the number of values.
    • transform.translation trick = animate translation (which resets to 0) for shakes, so you skip updating the model layer.

    CASpringAnimation

    Start here — the plain idea

    A CASpringAnimation is what you reach for when you want motion that feels alive instead of mechanical. With a basic animation you describe the curve directly ("ease in, ease out"). With a spring you don't draw the curve at all — you describe a tiny physics toy (a weight on a spring) and let the simulation produce the curve for you. That's why springs feel natural: real things move like this.

    There are only four dials, and they map to physical intuition. mass is how heavy the thing is (heavy = slow and lumbering). stiffness is how strong the spring is (stiff = snappy). damping is friction (low = lots of bouncing, high = it just glides in and stops). initialVelocity is how fast it's already moving when the animation starts — which is the secret to handing a flick gesture off to a spring without a visible hiccup.

    One rule to internalize early: let the spring decide how long it runs. Set duration = settlingDuration and the system computes exactly when the bouncing has died down. Pick a number yourself and you'll either cut the bounce off mid-air or leave it sitting still wasting time.

    CASpringAnimation simulates a physical spring attached to a layer property. Instead of defining an animation curve mathematically, you define physical properties -- mass, stiffness, damping, and initial velocity -- and the simulation produces natural, organic motion. This is the same physics engine that powers UIKit's spring animations and SwiftUI's .spring() modifier.

    See 09-cabasicanimation.md -- CASpringAnimation is a subclass of CABasicAnimation, so everything from that lesson applies here.

    Creating a Spring Animation

    let spring = CASpringAnimation(keyPath: "position.y")
spring.fromValue = layer.position.y
spring.toValue = 400
spring.mass = 1.0
spring.stiffness = 100.0
spring.damping = 10.0
spring.initialVelocity = 0.0

// IMPORTANT: use settlingDuration, not a fixed duration
spring.duration = spring.settlingDuration

layer.position.y = 400  // update model layer
layer.add(spring, forKey: "springMove")

    The Four Parameters

    mass (default: 1.0)

    The weight of the object on the spring. Heavier objects take longer to accelerate and decelerate, producing slower, more lumbering motion with more momentum.

    spring.mass = 0.5   // light -- quick, responsive
spring.mass = 1.0   // normal
spring.mass = 3.0   // heavy -- slow, ponderous

    stiffness (default: 100.0)

    The force of the spring. Higher stiffness means the spring pulls harder toward the target, producing snappier motion.

    spring.stiffness = 50    // loose spring -- gentle, slow
spring.stiffness = 100   // normal
spring.stiffness = 500   // tight spring -- very snappy

    damping (default: 10.0)

    Friction that resists the spring's motion. Higher damping means less oscillation. This is not the same as SwiftUI's dampingFraction (more on that below).

    spring.damping = 5    // low friction -- lots of bouncing
spring.damping = 10   // moderate bouncing
spring.damping = 30   // minimal bouncing

    initialVelocity (default: 0.0)

    The starting speed of the animation in points per second. Positive values move toward the target, negative values move away. This is critical for gesture-driven animations where you want the animation to match the user's finger velocity.

    spring.initialVelocity = 20   // already moving toward target
spring.initialVelocity = 0    // starting from rest
spring.initialVelocity = -5   // initially moving away from target

    settlingDuration

    settlingDuration is a computed property that tells you how long the spring simulation takes to settle within a negligible threshold of the target value. Always use it as your animation duration:

    spring.duration = spring.settlingDuration

    If you set a shorter duration, the animation clips -- it ends before the spring settles, causing a visual pop. If you set a longer duration, the spring sits at its final value doing nothing for the extra time. settlingDuration gives you exactly the right length.

    Damping Ratio

    The damping ratio describes the spring's behavior category. It is not a direct property on CASpringAnimation, but you can calculate it:

    dampingRatio = damping / (2 * sqrt(mass * stiffness))
    Ratio Behavior
    < 1.0 Under-damped: bouncy, oscillates past the target before settling
    = 1.0 Critically-damped: reaches the target as fast as possible without any overshoot
    > 1.0 Over-damped: slow approach without overshoot, sluggish feel 
    // Under-damped: bouncy (ratio ~0.5)
let bouncy = CASpringAnimation(keyPath: "transform.scale")
bouncy.mass = 1.0
bouncy.stiffness = 100.0
bouncy.damping = 10.0  // ratio = 10 / (2 * sqrt(100)) = 0.5
// oscillates before settling

// Critically-damped: no overshoot (ratio = 1.0)
let critical = CASpringAnimation(keyPath: "transform.scale")
critical.mass = 1.0
critical.stiffness = 100.0
critical.damping = 20.0  // ratio = 20 / (2 * sqrt(100)) = 1.0
// fastest without bouncing

// Over-damped: sluggish (ratio ~1.5)
let sluggish = CASpringAnimation(keyPath: "transform.scale")
sluggish.mass = 1.0
sluggish.stiffness = 100.0
sluggish.damping = 30.0  // ratio = 30 / (2 * sqrt(100)) = 1.5
// slow approach, no bounce

    UIKit's dampingRatio vs CA's damping

    UIView.animate(withDuration:delay:usingSpringWithDamping:initialSpringVelocity:) takes a dampingRatio parameter (0 to 1), not the raw damping coefficient. To convert:

    // Given a desired dampingRatio, compute the CA damping value
let mass: CGFloat = 1.0
let stiffness: CGFloat = 100.0
let dampingRatio: CGFloat = 0.7

let damping = dampingRatio * 2 * sqrt(mass * stiffness)
// damping = 0.7 * 2 * 10 = 14.0

    And to go the other direction:

    let dampingRatio = damping / (2 * sqrt(mass * stiffness))

    Interactive Springs: Using Gesture Velocity

    When an animation follows a gesture (pan, swipe), pass the gesture's velocity as initialVelocity so the animation starts smoothly from the gesture's speed:

    func handlePan(_ gesture: UIPanGestureRecognizer) {
    let translation = gesture.translation(in: view)
    let velocity = gesture.velocity(in: view)

    switch gesture.state {
    case .changed:
        layer.position.y = originalY + translation.y

    case .ended:
        let targetY: CGFloat = velocity.y > 0 ? 600 : 200

        let spring = CASpringAnimation(keyPath: "position.y")
        spring.fromValue = layer.presentation()?.position.y ?? layer.position.y
        spring.toValue = targetY
        spring.mass = 1.0
        spring.stiffness = 200.0
        spring.damping = 20.0
        spring.initialVelocity = velocity.y / (targetY - layer.position.y)
        spring.duration = spring.settlingDuration

        layer.position.y = targetY
        layer.add(spring, forKey: "panSpring")

    default:
        break
    }
}

    Note that initialVelocity for CASpringAnimation is measured in units relative to the total animation distance, not in points per second. Divide the gesture velocity by the animation distance to get the correct value.

    Comparison with SwiftUI Springs

    SwiftUI's spring API uses response and dampingFraction instead of mass/stiffness/damping:

    // SwiftUI
withAnimation(.spring(response: 0.5, dampingFraction: 0.7)) { ... }

// The equivalent CASpringAnimation:
let spring = CASpringAnimation(keyPath: "position")
spring.mass = 1.0
spring.stiffness = pow(2 * .pi / 0.5, 2)  // ~157.9
spring.damping = 0.7 * 2 * sqrt(spring.mass * spring.stiffness)  // ~17.6
spring.duration = spring.settlingDuration

    The conversion formulas: - stiffness = (2 * pi / response)^2 * mass - damping = dampingFraction * 2 * sqrt(mass * stiffness)

    See 12-animation-basics.md for SwiftUI's spring presets: .bouncy, .smooth, .snappy.

    CAAnimationGroup

    Start here — the plain idea

    A single animation moves one property. But real UI moments — a card appearing, a spinner spinning — usually move several things at once: it fades and scales and slides. A CAAnimationGroup is just a bag you drop several animations into so they play as one coordinated unit, sharing a clock and a single finish line. Think of it like a conductor in front of an orchestra: each player (animation) has its own part, but they all start and stop on the conductor's beat.

    The group's duration is that master clock. A sub-animation that's shorter finishes early and waits; one that's longer gets cut off when the clock runs out. And here's the trick that makes groups feel designed rather than dumped-together: beginTime lets you delay individual animations so they cascade — fade first, then scale a beat later, then slide — instead of everything firing at once. That tiny stagger is the difference between "amateur" and "Apple."

    Two reusable ideas ride along in this lesson. speed/timeOffset let you pause and scrub an animation like a video player. And additive animations ("move by this much from wherever you are, ending at zero offset") are the secret to interruptible motion — tap again mid-flight and it redirects smoothly instead of snapping.

    Individual animations target a single property. CAAnimationGroup lets you synchronize multiple animations -- position, scale, opacity, color -- so they play together as a coordinated unit with shared timing, a single completion callback, and the ability to stagger start times. Any CAAnimation subclass can be included: CABasicAnimation, CAKeyframeAnimation, CASpringAnimation, or even nested groups.

    See 09-cabasicanimation.md, 10-cakeyframeanimation.md, and 11-caspringanimation.md -- groups combine these individual animation types.

    Creating an Animation Group

    let scale = CABasicAnimation(keyPath: "transform.scale")
scale.fromValue = 0.5
scale.toValue = 1.0

let fade = CABasicAnimation(keyPath: "opacity")
fade.fromValue = 0.0
fade.toValue = 1.0

let move = CABasicAnimation(keyPath: "position.y")
move.fromValue = layer.position.y + 50
move.toValue = layer.position.y

let group = CAAnimationGroup()
group.animations = [scale, fade, move]
group.duration = 0.6

// Set final model values
layer.transform = CATransform3DIdentity
layer.opacity = 1.0

layer.add(group, forKey: "appearAnimation")

    All three animations start simultaneously and run over 0.6 seconds.

    Group Timing vs Individual Timing

    The group's duration is the master clock. Sub-animation durations interact with it in important ways:

    • If a sub-animation has no duration set (0.0, the default), it inherits the group's duration.
    • If a sub-animation has a shorter duration than the group, it finishes early and holds its final value for the remainder.
    • If a sub-animation has a longer duration than the group, it gets clipped at the group's duration.
    let group = CAAnimationGroup()
group.duration = 2.0

let quickFade = CABasicAnimation(keyPath: "opacity")
quickFade.fromValue = 0.0
quickFade.toValue = 1.0
quickFade.duration = 0.3  // finishes at 0.3s, holds at 1.0 until 2.0s

let slowMove = CABasicAnimation(keyPath: "position.x")
slowMove.fromValue = 0
slowMove.toValue = 300
slowMove.duration = 5.0  // clipped at 2.0s -- only gets 40% through

group.animations = [quickFade, slowMove]
layer.add(group, forKey: "timing")

    Staggered Animations with beginTime

    Each sub-animation has a beginTime property. Within a group, this is relative to the group's start (not to CACurrentMediaTime()).

    let group = CAAnimationGroup()
group.duration = 1.5

let fade = CABasicAnimation(keyPath: "opacity")
fade.fromValue = 0.0
fade.toValue = 1.0
fade.duration = 0.3
fade.beginTime = 0.0  // starts immediately

let scale = CABasicAnimation(keyPath: "transform.scale")
scale.fromValue = 0.5
scale.toValue = 1.0
scale.duration = 0.5
scale.beginTime = 0.2  // starts 0.2s after group begins

let move = CABasicAnimation(keyPath: "position.y")
move.fromValue = layer.position.y + 30
move.toValue = layer.position.y
move.duration = 0.6
move.beginTime = 0.4  // starts 0.4s after group begins

group.animations = [fade, scale, move]
layer.add(group, forKey: "staggered")

    The stagger creates a cascading effect: first the layer fades in, then it scales, then it moves.

    Repeating and Autoreversing

    These properties work on the group level, affecting all sub-animations together:

    let group = CAAnimationGroup()
group.animations = [pulseScale, colorShift]
group.duration = 1.0
group.repeatCount = .infinity   // loop forever
group.autoreverses = true       // bounce back after each cycle

    autoreverses plays the group forward, then backward, then forward... Each forward+backward pair counts as one repeat.

    Sub-animations can also have their own autoreverses:

    let pulse = CABasicAnimation(keyPath: "transform.scale")
pulse.fromValue = 1.0
pulse.toValue = 1.2
pulse.duration = 0.5
pulse.autoreverses = true  // this one bounces within the group

    Speed and Time Control

    speed

    Multiplier on playback speed. Applies to the entire group.

    group.speed = 2.0   // everything plays at double speed
group.speed = 0.5   // half speed -- slow motion
group.speed = 0.0   // paused

    timeOffset

    Jump into the middle of the animation. Combined with speed = 0, this lets you scrub animations manually.

    group.speed = 0.0
group.timeOffset = 0.5  // show the state at 0.5s into the animation

    Pausing and Resuming

    func pauseAnimation(on layer: CALayer) {
    let pausedTime = layer.convertTime(CACurrentMediaTime(), from: nil)
    layer.speed = 0.0
    layer.timeOffset = pausedTime
}

func resumeAnimation(on layer: CALayer) {
    let pausedTime = layer.timeOffset
    layer.speed = 1.0
    layer.timeOffset = 0.0
    layer.beginTime = 0.0
    let timeSincePause = layer.convertTime(CACurrentMediaTime(), from: nil) - pausedTime
    layer.beginTime = timeSincePause
}

    This pause/resume pattern works on any layer, not just groups.

    See 14-animation-timing.md for a deep dive on the CAMediaTiming protocol.

    Additive Animations

    When isAdditive = true, the animation value is added to the model layer value instead of replacing it. This is fundamental for building interruptible animations.

    // Non-additive: "move to 300"
let move = CABasicAnimation(keyPath: "position.x")
move.fromValue = 100
move.toValue = 300

// Additive: "move 200 points to the right from wherever you are"
let moveRelative = CABasicAnimation(keyPath: "position.x")
moveRelative.fromValue = 200   // start offset: +200 from model
moveRelative.toValue = 0       // end offset: +0 from model (at model value)
moveRelative.isAdditive = true

    The Additive Pattern

    The key insight: animate FROM the delta TO zero, and update the model layer to the target. The animation adds a decreasing offset that smoothly disappears.

    func moveLayer(_ layer: CALayer, toX targetX: CGFloat) {
    let delta = targetX - layer.position.x

    let anim = CABasicAnimation(keyPath: "position.x")
    anim.fromValue = -delta  // start: model + (-delta) = old position
    anim.toValue = 0         // end: model + 0 = target position
    anim.isAdditive = true
    anim.duration = 0.3

    layer.position.x = targetX  // update model immediately
    layer.add(anim, forKey: "moveX")
}

    If you call moveLayer again while the first animation is running, the new animation composes with the remaining offset from the old one, creating a smooth redirection instead of a jump.

    The foundation

    • CAAnimationGroup = a bag of animations played as one coordinated unit with a shared clock and finish.
    • group duration = the master clock; shorter sub-animations finish early and hold, longer ones get clipped.
    • beginTime = delay a sub-animation relative to the group's start, so motions cascade instead of firing together.
    • repeatCount / autoreverses = loop or bounce the whole group; each forward+backward pair is one repeat.
    • speed = playback multiplier; 0 pauses, 2 is double-time, 0.5 is slow-mo.
    • timeOffset = jump to a moment in the timeline; with speed 0 it lets you scrub like a video player.
    • pause/resume pattern = freeze with speed 0 + timeOffset, then restore by rebasing beginTime.
    • additive animation = animate an offset that decays to zero so interrupted motions redirect smoothly instead of snapping.

    CATransition

    Start here — the plain idea

    Most animations move a thing from A to B — a position, an opacity, a scale. CATransition answers a different question: what do you show when the content itself gets swapped? You had photo A, now you want photo B. There's no number to interpolate between two pictures, so the trick is sleight of hand.

    Think of it as a stage magician's cape. The moment before you change the content, Core Animation snapshots the old look, drapes that snapshot over the layer, then reveals the new content underneath while the snapshot fades, pushes, or curls away. You don't animate the swap; you animate the cover-up of the swap. That's why the pattern is always "add the transition, then change the content" — the transition is the cape going up right before the switch.

    CABasicAnimation and CAKeyframeAnimation animate property changes -- a layer moves, fades, or scales. CATransition is a different paradigm: it animates content changes. When you swap an image, change text, or replace a sublayer, a transition provides the visual bridge between old content and new content. Think page curls, pushes, fades, and reveals.

    See 09-cabasicanimation.md -- CATransition inherits from CAAnimation but operates on layer content rather than individual properties.

    Creating a Transition

    let transition = CATransition()
transition.type = .push
transition.subtype = .fromRight
transition.duration = 0.4

layer.add(transition, forKey: kCATransition)

// Now change the content -- the transition animates the swap
layer.contents = newImage.cgImage

    The key step: add the transition to the layer, then immediately change the content. Core Animation captures the old content as a snapshot and animates from the snapshot to the new content.

    Transition Types

    CATransition.type defines the animation style:

    Type Effect
    .fade Cross-dissolve between old and new content
    .moveIn New content slides in over the old content
    .push New content pushes old content out
    .reveal Old content slides away to reveal new content underneath 
    // Fade (the most common and subtle)
transition.type = .fade
transition.duration = 0.3

// Move-in from bottom (like a sheet)
transition.type = .moveIn
transition.subtype = .fromBottom
transition.duration = 0.35

// Push from left (like a navigation push)
transition.type = .push
transition.subtype = .fromLeft
transition.duration = 0.3

// Reveal from top (old content slides down to show new)
transition.type = .reveal
transition.subtype = .fromTop
transition.duration = 0.4

    Subtypes

    CATransition.subtype specifies the direction for directional transitions (.moveIn, .push, .reveal):

    Subtype Direction
    .fromLeft Content enters from the left
    .fromRight Content enters from the right
    .fromTop Content enters from the top
    .fromBottom Content enters from the bottom

    .fade ignores the subtype since it has no direction.

    Private Transition Types

    Core Animation includes several private transition types that are not part of the public API. They work but are undocumented and may be rejected by App Store review if detected. Use them for personal projects, debugging, or understanding the framework.

    // 3D cube rotation
transition.type = CATransitionType(rawValue: "cube")
transition.subtype = .fromRight

// 3D flip (card flip effect)
transition.type = CATransitionType(rawValue: "flip")
transition.subtype = .fromLeft

// OpenGL flip (similar to flip but different rendering)
transition.type = CATransitionType(rawValue: "oglFlip")
transition.subtype = .fromBottom

// Page curl (like turning a book page)
transition.type = CATransitionType(rawValue: "pageCurl")
transition.subtype = .fromRight

// Page uncurl (reverse of pageCurl)
transition.type = CATransitionType(rawValue: "pageUnCurl")
transition.subtype = .fromLeft

// Ripple effect (circular ripple emanating from center)
transition.type = CATransitionType(rawValue: "rippleEffect")

// Suck effect (content gets sucked into a point)
transition.type = CATransitionType(rawValue: "suckEffect")

// Camera iris (circular open/close like a camera aperture)
transition.type = CATransitionType(rawValue: "cameraIris")
// Also: "cameraIrisHollowOpen" and "cameraIrisHollowClose"

    Partial Transitions

    startProgress and endProgress let you use only part of the transition animation:

    let transition = CATransition()
transition.type = .push
transition.subtype = .fromRight
transition.duration = 0.5

// Only play from 20% to 80% of the transition
transition.startProgress = 0.2
transition.endProgress = 0.8

    Values range from 0.0 to 1.0. This is useful for creating variations of standard transitions -- for example, starting a push already partially in view.

    Transitioning Images

    The most common use case: swapping images on a layer with a smooth cross-dissolve.

    class ImageTransitionViewController: UIViewController {
    let imageLayer = CALayer()

    let images = ["photo1", "photo2", "photo3", "photo4"]

}

    Using kCATransition

    The forKey parameter when adding a transition is conventionally kCATransition (which equals the string "transition"). Using this key ensures that adding a new transition replaces the old one rather than stacking.

    // Both are equivalent
layer.add(transition, forKey: kCATransition)
layer.add(transition, forKey: "transition")

    You can use a custom key if you want multiple transitions active simultaneously, but this is rare and can produce unpredictable results.

    Custom Transition Patterns

    For transitions that go beyond built-in types, you can use snapshot-based approaches:

    func customMorphTransition(on layer: CALayer, to newContents: CGImage) {
    // Capture current appearance as snapshot
    UIGraphicsBeginImageContextWithOptions(layer.bounds.size, false, 0)
    guard let context = UIGraphicsGetCurrentContext() else { return }
    layer.render(in: context)
    let snapshot = UIGraphicsGetImageFromCurrentImageContext()
    UIGraphicsEndImageContext()

    // Create a snapshot overlay layer
    let snapshotLayer = CALayer()
    snapshotLayer.frame = layer.bounds
    snapshotLayer.contents = snapshot?.cgImage
    layer.addSublayer(snapshotLayer)

    // Update actual content
    layer.contents = newContents

    // Animate the snapshot away
    let fade = CABasicAnimation(keyPath: "opacity")
    fade.fromValue = 1.0
    fade.toValue = 0.0
    fade.duration = 0.5

    let scale = CABasicAnimation(keyPath: "transform.scale")
    scale.fromValue = 1.0
    scale.toValue = 1.1
    scale.duration = 0.5

    let group = CAAnimationGroup()
    group.animations = [fade, scale]
    group.duration = 0.5

    CATransaction.begin()
    CATransaction.setCompletionBlock {
        snapshotLayer.removeFromSuperlayer()
    }
    snapshotLayer.opacity = 0
    snapshotLayer.add(group, forKey: "dissolve")
    CATransaction.commit()
}

    This pattern gives you complete control -- you can add blur, 3D rotation, or any effect to the snapshot while revealing the new content underneath.

    CIFilter for Custom Transitions (macOS)

    On macOS, CATransition has a filter property that accepts a CIFilter for custom transition effects:

    Animation Timing

    Start here — the plain idea

    An animation has two separate questions baked into it: where it goes (0 to 1, start to finish) and how it gets there over time. Timing is all about that second question. A timing function is like a gas pedal — same trip, but you can ease out of the driveway, floor it in the middle, or coast to a gentle stop. Linear motion feels robotic precisely because the pedal never moves.

    The deeper idea is that every layer has its own private clock, and you can mess with that clock. Speed is the playback rate (set it to 0 and time freezes; set it to 0.5 for slow-mo). timeOffset is the scrubber bar — it says "show me the frame at this moment." Freeze the clock with speed = 0 and drag timeOffset with your finger, and suddenly you're hand-scrubbing the animation, which is exactly how interactive swipe-to-dismiss works.

    fillMode is the side question: what should the layer show before the animation starts and after it ends? By default the animation vanishes on completion and the layer snaps back to its real ("model") value — which surprises everyone the first time.

    Animation timing is the difference between an animation that feels mechanical and one that feels alive. This lesson covers CAMediaTimingFunction for pacing curves and the CAMediaTiming protocol for controlling playback — start time, speed, repetition, fill modes, and the powerful technique of scrubbing animations by hand.

    CAMediaTimingFunction

    A CAMediaTimingFunction defines the pacing curve of an animation — how the interpolated value progresses over time. Internally, it is a cubic bezier curve with two control points.

    Built-in Timing Functions

    // Five built-in curves
let linear      = CAMediaTimingFunction(name: .linear)          // constant speed
let easeIn      = CAMediaTimingFunction(name: .easeIn)          // slow start, fast end
let easeOut     = CAMediaTimingFunction(name: .easeOut)          // fast start, slow end
let easeInOut   = CAMediaTimingFunction(name: .easeInEaseOut)   // slow start, slow end
let defaultCurve = CAMediaTimingFunction(name: .default)         // subtle ease-in/out

let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = 0
anim.toValue = 300
anim.duration = 0.6
anim.timingFunction = easeOut
layer.add(anim, forKey: "move")

    See 09-cabasicanimation.md for the full CABasicAnimation API.

    Custom Cubic Bezier

    When built-in curves are not enough, create your own with control points. The curve goes from (0, 0) to (1, 1). You supply the two intermediate control points:

    // CSS ease-in-out equivalent
let css = CAMediaTimingFunction(controlPoints: 0.42, 0.0, 0.58, 1.0)

// Apple's material standard curve (used in many system animations)
let material = CAMediaTimingFunction(controlPoints: 0.2, 0.0, 0.0, 1.0)

// Aggressive ease-out (spring-like feel without spring math)
let springLike = CAMediaTimingFunction(controlPoints: 0.0, 0.7, 0.3, 1.0)

let anim = CABasicAnimation(keyPath: "transform.scale")
anim.fromValue = 0.8
anim.toValue = 1.0
anim.duration = 0.35
anim.timingFunction = material
layer.add(anim, forKey: "pop")

    Visualizing the Bezier Curve

    The control point graph has time (0 to 1) on the X axis and progress (0 to 1) on the Y axis. The curve determines how far along the animation is at any given moment:

    Progress (Y)
1.0 ┤                      ●
    │                  ╱
    │              ╱
    │          ╱
    │      ╱          ← ease-out: fast start, slow finish
    │   ╱
    │ ╱
0.0 ●─────────────────────
    0.0        Time (X)   1.0

    A steeper slope means faster motion at that point. A flat slope means the animation is nearly paused.

    The CAMediaTiming Protocol

    Both CAAnimation and CALayer adopt CAMediaTiming. This protocol provides fine-grained playback control.

    beginTime

    Delays when the animation starts. Measured in the parent's time space (usually the layer's time, which maps to CACurrentMediaTime()).

    let anim = CABasicAnimation(keyPath: "opacity")
anim.fromValue = 1.0
anim.toValue = 0.0
anim.duration = 0.5
anim.beginTime = CACurrentMediaTime() + 1.0  // starts 1 second from now
anim.fillMode = .backwards  // show fromValue during the delay
layer.add(anim, forKey: "fadeOut")

    duration

    How long the animation takes, in seconds. Default is 0, which inherits the transaction duration (0.25s).

    speed

    A multiplier on the rate of time. Default is 1.0.

    anim.speed = 2.0   // animation plays in half the duration
anim.speed = 0.5   // animation takes twice as long
anim.speed = 0.0   // animation is paused (frozen in time)

    timeOffset

    Jumps the animation to a specific point in time. Unlike beginTime, this does not delay — it shifts the window of the animation cycle.

    // Jump to the halfway point of a 1-second animation
anim.duration = 1.0
anim.timeOffset = 0.5   // starts visually at the 50% mark

    repeatCount and repeatDuration

    anim.repeatCount = 3              // repeat 3 times total
anim.repeatCount = .infinity      // repeat forever

anim.repeatDuration = 5.0         // repeat for 5 seconds (regardless of single-cycle duration)

    autoreverses

    When true, the animation plays forward then backward, counting as one full cycle:

    anim.autoreverses = true
anim.repeatCount = .infinity  // bounces back and forth forever

    Fill Modes

    fillMode controls what the layer displays BEFORE the animation begins and AFTER it ends. By default, an animation is removed on completion and the layer snaps to its model value.

    Fill Mode Before Animation After Animation
    .removed (default) Model value Model value
    .forwards Model value Final animation value
    .backwards fromValue (or initial) Model value
    .both fromValue Final animation value 
    let anim = CABasicAnimation(keyPath: "position.x")
anim.fromValue = 0
anim.toValue = 300
anim.duration = 1.0
anim.beginTime = CACurrentMediaTime() + 0.5

// Without fillMode, the layer sits at its model position during the 0.5s delay,
// then jumps to 0 when the animation starts.

anim.fillMode = .backwards  // shows fromValue (0) during the delay
layer.add(anim, forKey: "slide")

    isRemovedOnCompletion

    When set to false, the animation stays attached to the layer after completion. Pair with .fillMode = .forwards (or .both) to keep the final visual state:

    anim.fillMode = .forwards
anim.isRemovedOnCompletion = false
// The layer visually stays at the toValue, but the model layer has NOT changed.
// Always update the model layer to match, or you'll hit bugs with hit testing.

    Warning: Keeping animations attached leaks memory if you never remove them. Always prefer updating the model layer to its final value.

    Pausing and Resuming Animations

    Use the layer's speed and timeOffset to pause and resume all animations on that layer:

    func pauseLayer(_ layer: CALayer) {
    let pausedTime = layer.convertTime(CACurrentMediaTime(), from: nil)
    layer.speed = 0.0
    layer.timeOffset = pausedTime
}

func resumeLayer(_ layer: CALayer) {
    let pausedTime = layer.timeOffset
    layer.speed = 1.0
    layer.timeOffset = 0.0
    layer.beginTime = 0.0
    let timeSincePause = layer.convertTime(CACurrentMediaTime(), from: nil) - pausedTime
    layer.beginTime = timeSincePause
}

    This works because setting speed = 0 freezes the layer's local time. Setting timeOffset controls which frozen moment is displayed.

    Scrubbing Animations with Gestures

    You can drive an animation manually by setting speed = 0 and updating timeOffset from a gesture:

    class ScrubViewController: UIViewController {
    let animatedLayer = CALayer()

    override func viewDidLoad() {
        super.viewDidLoad()

        animatedLayer.frame = CGRect(x: 50, y: 200, width: 80, height: 80)
        animatedLayer.backgroundColor = UIColor.systemBlue.cgColor
        view.layer.addSublayer(animatedLayer)

        // Set up animation but keep it paused
        let anim = CABasicAnimation(keyPath: "position.x")
        anim.fromValue = 50
        anim.toValue = 300
        anim.duration = 1.0
        animatedLayer.speed = 0.0   // pause the layer
        animatedLayer.add(anim, forKey: "scrub")

        let pan = UIPanGestureRecognizer(target: self, action: #selector(handlePan))
        view.addGestureRecognizer(pan)
    }

    @objc func handlePan(_ gesture: UIPanGestureRecognizer) {
        let translation = gesture.translation(in: view)
        let progress = min(max(translation.x / 250.0, 0.0), 1.0)
        animatedLayer.timeOffset = CFTimeInterval(progress)  // 0.0 to 1.0 maps to animation
    }
}

    This technique powers interactive transitions in navigation controllers and sheet dismissals.

    Hierarchical Timing

    A layer's speed affects all animations on that layer AND its sublayers. This creates a timing hierarchy:

    let container = CALayer()
container.speed = 0.5  // everything inside runs at half speed

let child = CALayer()
child.speed = 2.0      // this child runs at 2x relative to container
// Effective speed of child = 0.5 * 2.0 = 1.0 (normal speed)
container.addSublayer(child)

    This is useful for slow-motion effects: set the root layer's speed to 0.1 and every animation in the entire UI slows down.

    The foundation

    • CAMediaTimingFunction = the gas pedal — a bezier curve that decides the pacing (ease-in, ease-out, linear, or your own control points).
    • beginTime = a delay; when the animation starts on the parent's clock.
    • duration = how long one cycle lasts in seconds.
    • speed = playback rate; 1 is normal, 0.5 is slow-mo, 0 is frozen.
    • timeOffset = the scrubber; which moment of the cycle is showing right now.
    • fillMode = what the layer displays before and after the animation (.backwards shows the start value during a delay; .forwards holds the end value).
    • isRemovedOnCompletion = whether the animation detaches when done; keep it attached only with care, since the real model value hasn't actually changed.
    • hierarchical timing = a parent layer's speed multiplies its children's, so one clock can slow an entire subtree.

    3D Transforms

    Start here — the plain idea

    A transform is just a recipe for "take every point of this layer and move it here instead." A 2D transform can shift, scale, and spin flat on the glass. CATransform3D adds a third axis (Z, depth) so layers can also lean toward or away from you. The screen is still flat, of course — Core Animation is faking depth by squishing the far edge smaller, exactly like your eyes do.

    The matrix is a 4x4 grid of numbers, but you almost never touch the numbers — you call helpers like CATransform3DMakeRotation. The one number worth knowing by name is m34: it's the perspective dial. Leave it at 0 and a rotation just looks like a layer getting narrower (flat, lifeless). Set it to a tiny negative value like -1/500 and the same rotation suddenly has a near edge and a far edge — real depth. Think of m34's denominator as how far the viewer is standing from the screen; closer means more dramatic foreshortening.

    Last gotcha: order matters. Combining transforms is matrix multiplication, and "rotate then move" lands somewhere different than "move then rotate" — because rotating first tilts the very axis you're about to move along.

    CATransform3D is a 4x4 matrix that positions layers in three-dimensional space. While the screen is flat, Core Animation can simulate perspective projection, letting you build card flips, rotating carousels, and cube transitions entirely on the GPU.

    CATransform3D Basics

    A CATransform3D is a struct with 16 fields (m11 through m44) representing a 4x4 homogeneous transformation matrix. You rarely touch the fields directly — Apple provides constructor and combination functions.

    Identity

    The identity transform means "no transformation applied":

    let identity = CATransform3DIdentity
layer.transform = identity  // resets all transforms

    Translation

    Move a layer along X, Y, and/or Z axes:

    // Create a translation from scratch
let move = CATransform3DMakeTranslation(100, 0, 0)  // shift right 100pt
layer.transform = move

// Apply translation to an existing transform
let combined = CATransform3DTranslate(existingTransform, 0, 50, 0)  // also shift down 50pt

    Moving along Z pushes the layer "into" or "out of" the screen. Without perspective, this has no visible effect (orthographic projection). We'll fix that shortly.

    Scale

    let scaleUp = CATransform3DMakeScale(2.0, 2.0, 1.0)    // double width and height
let scaleDown = CATransform3DMakeScale(0.5, 0.5, 1.0)  // half size

// Apply to existing transform
let scaled = CATransform3DScale(existingTransform, 1.5, 1.5, 1.0)

    Rotation

    Rotation takes an angle in radians and an axis vector (x, y, z):

    // Rotate 45 degrees around the Z axis (flat spin)
let spin = CATransform3DMakeRotation(.pi / 4, 0, 0, 1)

// Rotate 180 degrees around the Y axis (horizontal flip)
let flip = CATransform3DMakeRotation(.pi, 0, 1, 0)

// Rotate 30 degrees around the X axis (tilt forward/back)
let tilt = CATransform3DMakeRotation(.pi / 6, 1, 0, 0)

// Apply to existing
let rotated = CATransform3DRotate(existingTransform, .pi / 2, 0, 0, 1)

    Concatenating Transforms

    Combine transforms by multiplying them. Order matters — matrix multiplication is not commutative.

    let rotate = CATransform3DMakeRotation(.pi / 4, 0, 0, 1)
let translate = CATransform3DMakeTranslation(100, 0, 0)

// Rotate THEN translate: the layer rotates in place, then moves along the rotated axis
let rt = CATransform3DConcat(rotate, translate)

// Translate THEN rotate: the layer moves right, then rotates around the new position
let tr = CATransform3DConcat(translate, rotate)

layer.transform = rt  // try both and see the difference

    The Make versions create a fresh transform. The non-Make versions (CATransform3DTranslate, CATransform3DRotate, CATransform3DScale) apply to an existing transform, effectively concatenating.

    The m34 Perspective Trick

    By default, Core Animation uses orthographic projection — no foreshortening. A layer rotated around Y just gets narrower, like squishing it, rather than showing depth.

    To enable perspective, modify m34 of the transform matrix. This value controls the vanishing point distance:

    var transform = CATransform3DIdentity
transform.m34 = -1.0 / 500.0  // perspective: smaller denominator = more dramatic

// Now apply a rotation and it will look 3D
transform = CATransform3DRotate(transform, .pi / 4, 0, 1, 0)
layer.transform = transform
    m34 Value Effect
    -1.0 / 2000.0 Subtle, far perspective
    -1.0 / 500.0 Standard, natural look
    -1.0 / 200.0 Dramatic, close perspective
    0.0 (default) Orthographic, no depth

    The negative sign is required because the Z axis points toward the viewer. Think of the denominator as the distance (in points) between the viewer and the screen.

    sublayerTransform

    Instead of setting m34 on every sublayer's transform, set it once on the parent layer's sublayerTransform. This applies perspective to all children:

    var perspective = CATransform3DIdentity
perspective.m34 = -1.0 / 500.0
containerLayer.sublayerTransform = perspective

// Now any rotation on child layers will have perspective automatically
let card = CALayer()
card.transform = CATransform3DMakeRotation(.pi / 6, 0, 1, 0)
containerLayer.addSublayer(card)

    This is the standard approach for any scene with multiple 3D-transformed layers.

    See 03-layer-geometry.md for how anchorPoint affects the pivot of transforms.

    isDoubleSided

    By default, layers are double-sided — when rotated 180 degrees, you see a mirrored version of the content. Set isDoubleSided = false to hide the back face:

    layer.isDoubleSided = false
layer.transform = CATransform3DMakeRotation(.pi, 0, 1, 0)
// Layer is invisible — you're looking at the back

    This is essential for card flip effects where front and back are separate layers.

    anchorPoint and anchorPointZ

    The anchorPoint determines the pivot point for all transforms. anchorPointZ extends this into the Z dimension:

    // Rotate around the left edge (like a door hinge)
layer.anchorPoint = CGPoint(x: 0, y: 0.5)
layer.transform = CATransform3DMakeRotation(.pi / 4, 0, 1, 0)

// anchorPointZ shifts the rotation axis along Z
layer.anchorPointZ = 50  // pivot point is 50pt "behind" the layer

    Card Flip Animation

    A classic effect: two layers representing front and back of a card.

    func setupCardFlip(in container: CALayer) {

    flipBack.toValue = 0

    flipBack.fillMode = .forwards
    flipBack.isRemovedOnCompletion = false

}

    Rotating Carousel

    Arrange N layers in a circle in 3D space:

    func buildCarousel(in container: CALayer, count: Int, radius: CGFloat) {
    var perspective = CATransform3DIdentity
    perspective.m34 = -1.0 / 800.0
    container.sublayerTransform = perspective

    let angleStep = (2.0 * CGFloat.pi) / CGFloat(count)

    for i in 0..<count {
        let panel = CALayer()
        panel.frame = CGRect(x: 0, y: 0, width: 120, height: 160)
        panel.position = CGPoint(x: container.bounds.midX, y: container.bounds.midY)
        panel.backgroundColor = UIColor(
            hue: CGFloat(i) / CGFloat(count), saturation: 0.7, brightness: 0.9, alpha: 1
        ).cgColor

        var transform = CATransform3DIdentity
        transform = CATransform3DRotate(transform, angleStep * CGFloat(i), 0, 1, 0)
        transform = CATransform3DTranslate(transform, 0, 0, radius)
        panel.transform = transform

        container.addSublayer(panel)
    }
}

    CAShapeLayer

    Start here — the plain idea

    A normal layer shows a bitmap — a grid of pixels you handed it. A CAShapeLayer is different: you give it a path (a vector outline, "go here, curve to there"), and it draws that outline live on the GPU. Because it's math, not pixels, it stays crisp at any size and costs almost nothing to redraw. Think of it as the difference between a photo of a circle and the instruction "draw a circle of radius 50."

    The magic property is strokeEnd (and its partner strokeStart). They're a fraction from 0 to 1 saying "how much of this outline should I actually draw?" Set strokeEnd to 0.5 and you see the first half of the path. Animate it from 0 to 1 and the line appears to draw itself, like an invisible pen tracing the shape. That single trick is the whole secret behind progress rings, signature replays, and the satisfying animated checkmark — you're not moving anything, just revealing more of a path over time.

    CAShapeLayer draws vector shapes defined by CGPath objects. Strokes and fills are GPU-accelerated, and key properties like strokeEnd are animatable — making it the go-to layer for progress indicators, custom drawing, and path-based animations.

    Basics: Path, Fill, and Stroke

    A CAShapeLayer renders a CGPath. You typically build paths using UIBezierPath for convenience, then assign .cgPath:

    let shapeLayer = CAShapeLayer()
shapeLayer.frame = CGRect(x: 50, y: 100, width: 200, height: 200)

let rect = UIBezierPath(roundedRect: shapeLayer.bounds, cornerRadius: 20)
shapeLayer.path = rect.cgPath

shapeLayer.fillColor = UIColor.systemBlue.cgColor
shapeLayer.strokeColor = UIColor.white.cgColor
shapeLayer.lineWidth = 3.0

view.layer.addSublayer(shapeLayer)

    Set fillColor to nil for stroke-only shapes (outlines, progress rings). Set strokeColor to nil for fill-only shapes.

    See 10-drawing-and-shapes.md for SwiftUI's Shape protocol, which compiles down to path-based rendering similar to CAShapeLayer.

    Stroke Properties

    lineCap

    Controls the shape of line endpoints:

    shapeLayer.lineCap = .butt    // flat, flush with endpoint (default)
shapeLayer.lineCap = .round   // semicircle beyond the endpoint
shapeLayer.lineCap = .square  // half-square beyond the endpoint

    lineJoin

    Controls the shape of corners where two line segments meet:

    shapeLayer.lineJoin = .miter  // sharp point (default)
shapeLayer.lineJoin = .round  // rounded corner
shapeLayer.lineJoin = .bevel  // flat cut corner

    miterLimit

    When using .miter join, very sharp angles produce extremely long spikes. miterLimit caps how far the miter extends:

    shapeLayer.miterLimit = 4.0  // default is 10; lower = more aggressive clipping

    Dashed Lines

    Create dashed strokes with lineDashPattern and lineDashPhase:

    // 6pt dash, 3pt gap, repeating
shapeLayer.lineDashPattern = [6, 3]

// More complex: 10pt dash, 5pt gap, 2pt dot, 5pt gap
shapeLayer.lineDashPattern = [10, 5, 2, 5]

// Offset the pattern (animate this for "marching ants")
shapeLayer.lineDashPhase = 0

    Marching Ants Animation

    let march = CABasicAnimation(keyPath: "lineDashPhase")
march.fromValue = 0
march.toValue = shapeLayer.lineDashPattern!.reduce(0) { $0 + $1.intValue }
march.duration = 0.5
march.repeatCount = .infinity
shapeLayer.add(march, forKey: "marchingAnts")

    strokeStart and strokeEnd

    These are the most powerful properties of CAShapeLayer. They define which portion of the path is drawn, as a fraction from 0.0 to 1.0. Both are animatable.

    shapeLayer.strokeStart = 0.0  // default: beginning of path
shapeLayer.strokeEnd = 1.0    // default: end of path

// Draw only the first half:
shapeLayer.strokeEnd = 0.5

// Draw the middle third:
shapeLayer.strokeStart = 0.33
shapeLayer.strokeEnd = 0.66

    Drawing Progress Animation

    Animate strokeEnd from 0 to 1 to reveal a path progressively:

    let path = UIBezierPath()

path.addLine(to: CGPoint(x: 100, y: 180))
path.addLine(to: CGPoint(x: 280, y: 40))

    Performance Notes

    CAShapeLayer strokes are GPU-accelerated and do not trigger off-screen rendering for simple cases. However:

    • For static shapes that do not animate, consider rasterizing with shouldRasterize = true

    CATextLayer

    Start here — the plain idea

    A CATextLayer is the bare-metal way to put text on screen. A UILabel is a whole view — it carries accessibility, Auto Layout, touch handling, the entire UIKit lifecycle. A CATextLayer skips all that and just paints letters straight into the layer tree. Think of it as the difference between hiring an employee and renting a vending machine: lighter, faster, but it won't answer questions (no VoiceOver, no constraints).

    Under the hood it works by rasterizing your text into a bitmap once, then letting the GPU composite that bitmap like any other layer. That detail explains the single most common bug: you must set contentsScale = UIScreen.main.scale, or it rasterizes at 1x and looks blurry on every Retina screen. It also explains why changing the string isn't directly animatable — there's no number between "Hello" and "Goodbye" to interpolate, so you reach for a CATransition to fade the swap instead.

    CATextLayer renders text directly in the layer tree using hardware-accelerated compositing. It supports plain strings, attributed strings, and multiple alignment and truncation modes. When you need text rendering without the overhead of UIKit views, CATextLayer is the tool.

    Basic Setup

    let textLayer = CATextLayer()
textLayer.frame = CGRect(x: 20, y: 100, width: 280, height: 60)
textLayer.string = "Hello, Core Animation"
textLayer.font = "Helvetica-Bold" as CFTypeRef
textLayer.fontSize = 24
textLayer.foregroundColor = UIColor.label.cgColor
textLayer.alignmentMode = .center

view.layer.addSublayer(textLayer)

    Critical: You must set contentsScale to match the screen or text will appear blurry on Retina displays:

    textLayer.contentsScale = UIScreen.main.scale

    This is the single most common mistake with CATextLayer. The layer rasterizes text to a bitmap, and without the correct scale, it renders at 1x resolution on a 2x or 3x display.

    Font Configuration

    The font property accepts multiple types:

    // String font name
textLayer.font = "Avenir-Medium" as CFTypeRef

// CGFont
if let cgFont = CGFont("HelveticaNeue" as CFString) {
    textLayer.font = cgFont
}

// CTFont (Core Text)
let ctFont = CTFontCreateWithName("Georgia" as CFString, 18, nil)
textLayer.font = ctFont

// fontSize is separate from the font reference
textLayer.fontSize = 20

    When using a CTFont that already encodes a size, fontSize still overrides it. Set both for clarity.

    See 03-strings.md for String fundamentals and Unicode handling in Swift.

    Alignment and Wrapping

    alignmentMode

    textLayer.alignmentMode = .left       // left-aligned (default)
textLayer.alignmentMode = .right      // right-aligned
textLayer.alignmentMode = .center     // centered
textLayer.alignmentMode = .justified  // justified (stretches words)
textLayer.alignmentMode = .natural    // follows locale (LTR or RTL)

    isWrapped

    When true, text wraps to multiple lines within the layer's bounds:

    textLayer.isWrapped = true
textLayer.string = "This is a longer string that will wrap to multiple lines within the layer bounds."

    When false (default), text stays on a single line and is clipped or truncated.

    truncationMode

    Controls where the ellipsis appears when text does not fit:

    textLayer.truncationMode = .end     // "Hello Wor..." (default)
textLayer.truncationMode = .start   // "...ld, Hello"
textLayer.truncationMode = .middle  // "Hel...llo"

    Truncation only applies when isWrapped is false or when wrapped text exceeds the layer height.

    Attributed Strings

    CATextLayer supports NSAttributedString for rich text with mixed styles:

    let attributed = NSMutableAttributedString(string: "Bold and Colored")

// Make "Bold" bold
attributed.addAttribute(.font,
    value: UIFont.boldSystemFont(ofSize: 22),
    range: NSRange(location: 0, length: 4))

    range: NSRange(location: 9, length: 7))

    When using attributed strings, the layer's font, fontSize, and foregroundColor properties serve as defaults for any ranges that do not specify those attributes.

    CATextLayer vs UILabel

    Feature CATextLayer UILabel
    Accessibility Not accessible by default Full VoiceOver support
    Auto Layout No intrinsic content size Supports constraints
    Hit testing Layer hit test only UIView responder chain
    Performance Lighter, no UIKit overhead Heavier, full view lifecycle
    Rich text NSAttributedString NSAttributedString + markdown

    Use CATextLayer when: - You are working in a pure layer tree (no UIKit views) - You need to render many text elements efficiently (e.g., a custom chart with labels) - You want to animate text properties at the Core Animation level - You are building a Metal + Core Animation hybrid rendering pipeline

    Use UILabel when: - You need accessibility (VoiceOver) - You need Auto Layout integration - You need tap handling, gestures, or responder chain - Standard UIKit view hierarchy

    Performance Considerations

    CATextLayer renders text to a bitmap once when properties change (or when setNeedsDisplay() is called). This bitmap is then composited by the GPU like any other layer content. For large amounts of non-interactive text — think ticker displays, debug overlays, HUD labels — it is significantly cheaper than UILabel because it skips the entire UIView lifecycle.

    However, every time the string changes, the text is re-rasterized on the CPU. For text that changes every frame (a live counter), this CPU cost adds up. Consider using a CADisplayLink with throttled updates or pre-rendering digits.

    CAGradientLayer

    Start here — the plain idea

    A CAGradientLayer is a layer that fills itself with a smooth color ramp — you hand it a list of colors and it blends between them. The key thing to internalize: you're not drawing a gradient or generating a bitmap of one. You're just declaring "go from blue to purple," and the GPU computes every in-between pixel for free during compositing. That's why gradients are one of the cheapest effects you can use — no rasterization, no off-screen pass.

    Two ideas unlock most of it. First, startPoint and endPoint live in unit coordinates — (0,0) is top-left, (1,1) is bottom-right — so the direction scales automatically with the layer and you never recompute it on resize. Second, locations lets you say where along that line each color sits. Push a color's stop around and you slide the band. That second idea is the whole trick behind a shimmer: animate the locations (even past the 0–1 edges) and a bright band appears to slide across, which is exactly how skeleton-loading screens shimmer.

    CAGradientLayer renders smooth color transitions directly on the GPU. It supports linear (axial), radial, and conic gradients, and its properties are all animatable — making it the foundation for shimmer effects, gradient masks, and dynamic color transitions.

    Basic Linear Gradient

    let gradient = CAGradientLayer()
gradient.frame = CGRect(x: 20, y: 100, width: 300, height: 200)

gradient.colors = [
    UIColor.systemBlue.cgColor,
    UIColor.systemPurple.cgColor
]

view.layer.addSublayer(gradient)

    By default, the gradient runs top-to-bottom. The colors array must contain CGColor values (not UIColor).

    See 11-colors-and-materials.md for SwiftUI's LinearGradient, RadialGradient, and AngularGradient views.

    Direction: startPoint and endPoint

    startPoint and endPoint define the gradient direction in unit coordinate space — (0, 0) is top-left, (1, 1) is bottom-right:

    // Top to bottom (default)
gradient.startPoint = CGPoint(x: 0.5, y: 0.0)
gradient.endPoint   = CGPoint(x: 0.5, y: 1.0)

// Left to right
gradient.startPoint = CGPoint(x: 0.0, y: 0.5)
gradient.endPoint   = CGPoint(x: 1.0, y: 0.5)

// Diagonal: top-left to bottom-right
gradient.startPoint = CGPoint(x: 0.0, y: 0.0)
gradient.endPoint   = CGPoint(x: 1.0, y: 1.0)

// Bottom-right to top-left (reversed diagonal)
gradient.startPoint = CGPoint(x: 1.0, y: 1.0)
gradient.endPoint   = CGPoint(x: 0.0, y: 0.0)

    The coordinates are relative, so they automatically scale with the layer's frame. You never need to update them when the frame changes.

    Color Stops: locations

    By default, colors are spaced evenly. The locations property overrides this with explicit stops (0.0 to 1.0):

    gradient.colors = [
    UIColor.red.cgColor,
    UIColor.yellow.cgColor,
    UIColor.green.cgColor
]

// Even distribution (default behavior, equivalent to not setting locations)
gradient.locations = [0.0, 0.5, 1.0]

// Push yellow toward the top
gradient.locations = [0.0, 0.2, 1.0]

// Push yellow toward the bottom
gradient.locations = [0.0, 0.8, 1.0]

    The number of values in locations must match the number of values in colors.

    Multi-Stop Gradients

    Use three or more colors for complex gradients:

    // Sunset gradient
gradient.colors = [
    UIColor(red: 0.98, green: 0.36, blue: 0.35, alpha: 1).cgColor,  // coral
    UIColor(red: 0.93, green: 0.50, blue: 0.55, alpha: 1).cgColor,  // rose
    UIColor(red: 0.45, green: 0.31, blue: 0.59, alpha: 1).cgColor,  // purple
    UIColor(red: 0.16, green: 0.16, blue: 0.29, alpha: 1).cgColor   // dark navy
]
gradient.locations = [0.0, 0.35, 0.65, 1.0]

    Gradient Types

    Axial (Linear) — Default

    gradient.type = .axial  // this is the default

    Radial

    Radiates outward from startPoint to endPoint:

    gradient.type = .radial
gradient.startPoint = CGPoint(x: 0.5, y: 0.5)  // center
gradient.endPoint = CGPoint(x: 1.0, y: 1.0)    // edge (determines radius)

gradient.colors = [
    UIColor.white.cgColor,
    UIColor.systemBlue.cgColor
]

    The gradient radiates from startPoint in a circle whose radius reaches endPoint. The aspect ratio of the layer can stretch the circle into an ellipse.

    Conic (Angular)

    Available in iOS 12+. The gradient sweeps around a center point:

    gradient.type = .conic
gradient.startPoint = CGPoint(x: 0.5, y: 0.5)  // center of rotation
gradient.endPoint = CGPoint(x: 0.5, y: 0.0)    // where 0.0 starts (top)

gradient.colors = [
    UIColor.red.cgColor,
    UIColor.yellow.cgColor,
    UIColor.green.cgColor,
    UIColor.cyan.cgColor,
    UIColor.blue.cgColor,
    UIColor.magenta.cgColor,
    UIColor.red.cgColor   // repeat first color for seamless loop
]

    Conic gradients are perfect for color wheels, angular progress indicators, and pie-chart-style visuals.

    Animating Gradients

    All gradient properties are implicitly animatable when changed inside a CATransaction, and explicitly animatable with CABasicAnimation:

    Animate Colors

colorAnim.toValue = [UIColor.systemOrange.cgColor, UIColor.systemRed.cgColor]
colorAnim.duration = 1.5
gradient.colors = [UIColor.systemOrange.cgColor, UIColor.systemRed.cgColor]
gradient.add(colorAnim, forKey: "colorShift")

    Animate Locations

    let locAnim = CABasicAnimation(keyPath: "locations")
locAnim.fromValue = [0.0, 0.5, 1.0]
locAnim.toValue = [0.0, 0.9, 1.0]
locAnim.duration = 0.8
gradient.locations = [0.0, 0.9, 1.0] as [NSNumber]
gradient.add(locAnim, forKey: "locationShift")

    Animate Direction

    let dirAnim = CABasicAnimation(keyPath: "startPoint")

dirAnim.toValue = CGPoint(x: 1, y: 1)

dirAnim.repeatCount = .infinity

    Shimmer / Skeleton Loading Effect

    A sliding highlight that indicates loading state:

    func addShimmer(to layer: CALayer) {
    let shimmer = CAGradientLayer()
    shimmer.frame = layer.bounds
    shimmer.colors = [
        UIColor.systemGray5.cgColor,
        UIColor.systemGray6.cgColor,
        UIColor.systemGray5.cgColor
    ]
    shimmer.locations = [0.0, 0.5, 1.0]
    shimmer.startPoint = CGPoint(x: 0.0, y: 0.5)
    shimmer.endPoint = CGPoint(x: 1.0, y: 0.5)

    layer.addSublayer(shimmer)

    // Animate locations to slide the highlight across
    let anim = CABasicAnimation(keyPath: "locations")
    anim.fromValue = [-1.0, -0.5, 0.0]
    anim.toValue = [1.0, 1.5, 2.0]
    anim.duration = 1.5
    anim.repeatCount = .infinity
    shimmer.add(anim, forKey: "shimmer")
}

    The trick is using location values outside [0, 1] — the highlight slides through the visible area.

    The mask uses the alpha channel: black = opaque, clear = transparent. The gradient creates a smooth fade.

    Performance

    Gradient layers are lightweight. The GPU interpolates colors directly during compositing — no bitmap rasterization needed. This makes them one of the cheapest visual effects available:

    • No off-screen rendering
    • No CPU rasterization
    • Animating properties is GPU-driven
    • Safe to use many gradient layers simultaneously

    The only cost concern is overdraw: stacking many semi-transparent gradient layers forces the GPU to blend each one per-pixel.

    CAReplicatorLayer

    Start here — the plain idea

    A CAReplicatorLayer is a copy machine for layers. You build one sublayer, tell it "make 12 copies," and it stamps that copy out 12 times — each one nudged a little further by the same transform. The deep part: it doesn't actually create 12 layer objects. The GPU draws your one layer's pixels and re-stamps them with different transforms, so 100 copies cost almost the same as 1. Think of it like a rubber stamp versus hand-painting 100 dots.

    Two dials do the heavy lifting. instanceTransform is applied cumulatively — copy 1 gets it once, copy 2 gets it twice — so a small rotation per copy fans your layers into a circle, and a small translation lines them into a row. instanceDelay staggers the animation across copies: copy 2 starts its bounce a beat after copy 1. Combine "rotate each copy around a center" with "delay each copy slightly" and you've built a loading spinner without ever writing a loop. That stagger is the soul of every replicator effect — ripples, EQ bars, cascades.

    CAReplicatorLayer efficiently clones its sublayers with incremental transforms, color shifts, and animation delays. You provide one sublayer, and the replicator draws N copies — each offset, rotated, or tinted from the previous. This is vastly more efficient than creating N separate layers, and the staggered animation delay produces elegant ripple and cascade effects.

    Core Properties

    let replicator = CAReplicatorLayer()
replicator.frame = CGRect(x: 0, y: 0, width: 300, height: 300)

replicator.instanceCount = 6           // total copies including the original
replicator.instanceDelay = 0.15        // seconds between each copy's animation start
replicator.instanceTransform = CATransform3DMakeTranslation(40, 0, 0)  // shift each copy 40pt right

// The single sublayer that gets replicated
let dot = CALayer()
dot.frame = CGRect(x: 0, y: 140, width: 20, height: 20)
dot.backgroundColor = UIColor.systemBlue.cgColor
dot.cornerRadius = 10
replicator.addSublayer(dot)

view.layer.addSublayer(replicator)

    The replicator does not create new CALayer objects in memory. It reuses the same render data with different transforms applied by the GPU. This is why 100 copies cost almost the same as 1.

    See 06-sublayer-management.md for the standard sublayer management API that CAReplicatorLayer replaces in high-copy-count scenarios.

    instanceTransform

    The instanceTransform is applied cumulatively to each copy. Copy 0 (the original) gets no transform. Copy 1 gets one application. Copy 2 gets two applications. And so on.

    // Shift each copy 50pt right → creates a horizontal row
replicator.instanceTransform = CATransform3DMakeTranslation(50, 0, 0)

// Rotate each copy 30 degrees → creates a circular arrangement (with 12 copies)
replicator.instanceTransform = CATransform3DMakeRotation(.pi / 6, 0, 0, 1)

// Scale each copy smaller → creates a perspective trail
replicator.instanceTransform = CATransform3DConcat(
    CATransform3DMakeTranslation(30, 0, 0),
    CATransform3DMakeScale(0.9, 0.9, 1.0)
)

    Color Offsets

    Each copy can shift its color channels incrementally:

    replicator.instanceColor = UIColor.white.cgColor  // base color (usually white to not tint)

replicator.instanceRedOffset   = -0.05   // each copy loses red
replicator.instanceGreenOffset = 0.0
replicator.instanceBlueOffset  = 0.02    // each copy gains blue
replicator.instanceAlphaOffset = -0.1    // each copy is more transparent

    The offsets are additive per copy. Copy 3's red = base red + (3 * redOffset). Values are clamped to [0, 1].

    Dot Loading Indicator

    Three dots that bounce with staggered timing:

    func makeDotLoader(in parent: CALayer, center: CGPoint) {
    let replicator = CAReplicatorLayer()
    replicator.frame = CGRect(x: center.x - 40, y: center.y - 10, width: 80, height: 20)
    replicator.instanceCount = 3
    replicator.instanceTransform = CATransform3DMakeTranslation(30, 0, 0)
    replicator.instanceDelay = 0.15

    let dot = CALayer()
    dot.frame = CGRect(x: 0, y: 5, width: 10, height: 10)
    dot.backgroundColor = UIColor.systemBlue.cgColor
    dot.cornerRadius = 5

    // Scale animation: each dot pulses, staggered
    let pulse = CABasicAnimation(keyPath: "transform.scale")
    pulse.fromValue = 1.0
    pulse.toValue = 0.3
    pulse.duration = 0.6
    pulse.autoreverses = true
    pulse.repeatCount = .infinity
    dot.add(pulse, forKey: "pulse")

    replicator.addSublayer(dot)
    parent.addSublayer(replicator)
}

    The instanceDelay means dot 1 starts its animation 0.15s after dot 0, and dot 2 starts 0.3s after dot 0. This creates the cascading pulse effect.

    Activity Spinner

    Dots arranged in a circle with fading opacity — the classic loading indicator:

    func makeSpinner(in parent: CALayer, center: CGPoint, dotCount: Int = 12) {
    let replicator = CAReplicatorLayer()
    replicator.frame = CGRect(x: 0, y: 0, width: 60, height: 60)
    replicator.position = center
    replicator.instanceCount = dotCount
    replicator.instanceDelay = 1.0 / Double(dotCount)  // total cycle = 1 second

    let angle = (2.0 * CGFloat.pi) / CGFloat(dotCount)
    replicator.instanceTransform = CATransform3DMakeRotation(angle, 0, 0, 1)

    let dot = CALayer()
    dot.frame = CGRect(x: 25, y: 0, width: 8, height: 8)
    dot.backgroundColor = UIColor.white.cgColor
    dot.cornerRadius = 4

    // Opacity animation
    let fade = CABasicAnimation(keyPath: "opacity")
    fade.fromValue = 1.0
    fade.toValue = 0.2
    fade.duration = 1.0
    fade.repeatCount = .infinity
    dot.add(fade, forKey: "fade")

    replicator.addSublayer(dot)
    parent.addSublayer(replicator)
}

    Each dot fades at a slightly offset time, creating the spinning illusion. The dot is positioned off-center (at x: 25), and the rotation transform sweeps it around the circle.

    Reflection Effect

    Mirror a layer with a faded copy underneath:

    func addReflection(to layer: CALayer) {
    let replicator = CAReplicatorLayer()
    replicator.frame = layer.bounds
    replicator.instanceCount = 2

    // Flip vertically and push down by the layer height
    var reflectTransform = CATransform3DIdentity
    reflectTransform = CATransform3DTranslate(reflectTransform, 0, layer.bounds.height + 4, 0)
    reflectTransform = CATransform3DScale(reflectTransform, 1, -1, 1)  // vertical flip
    replicator.instanceTransform = reflectTransform

    // Fade the reflection
    replicator.instanceAlphaOffset = -0.6

    // Move the original content into the replicator
    let content = CALayer()
    content.frame = layer.bounds
    content.contents = layer.contents
    replicator.addSublayer(content)

    layer.addSublayer(replicator)
}

    Performance

    CAReplicatorLayer is one of the most efficient ways to draw repeated elements:

    • The GPU draws one sublayer's content and stamps it N times with different transforms
    • No additional layer tree overhead — instanceCount = 100 does not create 100 layer objects
    • instanceDelay is handled by the animation system, not by spawning separate timers
    • Memory usage is constant regardless of instance count (only the source sublayer is stored)

    Compare this to manually creating 100 CALayer objects with 100 CABasicAnimation instances — the replicator achieves the same visual result with dramatically less overhead.

    CAEmitterLayer

    Start here — the plain idea

    A CAEmitterLayer is a particle fountain. You don't animate each speck of confetti or each snowflake by hand — there could be thousands. Instead you describe the rules once ("spawn 30 a second, send them up at this speed, fade them over 2 seconds, let gravity pull them down") and the GPU runs the whole simulation for you, frame after frame, with zero per-frame CPU cost. Think of it as setting up a sprinkler and walking away, versus throwing each water droplet yourself.

    The mental model has exactly two parts. The emitter is the nozzle — it decides where particles are born and how the whole spray renders. The cell is the blueprint for a single particle — what it looks like and how it behaves (speed, lifetime, color, spin). One nozzle can hold several blueprints (confetti in six colors = six cells), and a cell can even spawn its own sub-particles, which is how a firework rocket bursts into sparks. Almost every property comes with a ...Range sibling that adds randomness, because nothing looks alive when every particle is identical.

    CAEmitterLayer is Core Animation's GPU-accelerated particle system. It lets you spawn thousands of particles with randomized motion, color, scale, and lifetime -- all composited on the GPU with zero per-frame CPU cost after setup. Every confetti burst, snow scene, or fire effect you see in iOS apps likely uses this class.

    See 02-calayer-fundamentals.md -- CAEmitterLayer is a CALayer subclass. Everything about the layer tree, geometry, and compositing applies here.

    The Two-Object Model

    A particle system has two parts:

    • CAEmitterLayer -- the emitter. Controls where particles spawn and how they render.
    • CAEmitterCell -- the template. Defines what each particle looks like and how it behaves.

    One emitter can have many cells (e.g., confetti with different shapes). One cell can also contain sub-emitter cells (particles that spawn particles).

    Setting Up a Basic Emitter

    import UIKit

class ParticleViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        view.backgroundColor = .black

        let emitter = CAEmitterLayer()
        emitter.emitterPosition = CGPoint(x: view.bounds.midX, y: view.bounds.midY)
        emitter.emitterSize = CGSize(width: 1, height: 1)
        emitter.emitterShape = .point

        let cell = CAEmitterCell()
        cell.contents = makeCircleImage(diameter: 10, color: .white).cgImage
        cell.birthRate = 20
        cell.lifetime = 3.0
        cell.velocity = 100
        cell.emissionRange = .pi * 2  // emit in all directions
        cell.scale = 0.5
        cell.alphaSpeed = -0.3  // fade out over lifetime

        emitter.emitterCells = [cell]
        view.layer.addSublayer(emitter)
    }

    func makeCircleImage(diameter: CGFloat, color: UIColor) -> UIImage {
        let renderer = UIGraphicsImageRenderer(size: CGSize(width: diameter, height: diameter))
        return renderer.image { ctx in
            color.setFill()
            ctx.cgContext.fillEllipse(in: CGRect(x: 0, y: 0, width: diameter, height: diameter))
        }
    }
}

    Emitter Layer Properties

    emitterPosition and emitterSize

    emitterPosition is a CGPoint defining where particles originate. emitterSize defines the area of emission. For a .point shape, size is irrelevant. For .line, .rectangle, or .circle, size controls the spawn region.

    // Emit from a line across the top of the screen
emitter.emitterPosition = CGPoint(x: view.bounds.midX, y: 0)
emitter.emitterSize = CGSize(width: view.bounds.width, height: 1)
emitter.emitterShape = .line

    emitterShape

    Controls the geometry of the emission region:

    Shape Description
    .point All particles from a single point
    .line Along a horizontal line
    .rectangle Within a rectangular area
    .circle Along or within a circle
    .sphere 3D sphere (particles emit in 3D)
    .cuboid 3D box

    emitterMode

    Determines where on the shape particles spawn:

    Mode Description
    .points Random points within the shape
    .outline Only on the edge/outline
    .surface On the surface (3D shapes)
    .volume Throughout the entire volume

    renderMode

    Controls compositing order and blending:

    Mode Effect
    .oldestFirst Older particles draw first (behind newer ones)
    .oldestLast Older particles draw on top
    .backToFront Depth-sorted (for 3D shapes)
    .additive Colors add together -- great for fire, glow
    .unordered No sorting (fastest)

    birthRate (on the layer)

    A global multiplier on all cells' birth rates. Set to 0 to stop emitting while letting existing particles finish their lifetime.

    // Stop emitting gracefully
emitter.birthRate = 0  // existing particles continue until they expire

    CAEmitterCell Properties

    Core Behavior

    let cell = CAEmitterCell()
cell.contents = circleImage.cgImage    // CGImage for each particle
cell.birthRate = 50                     // particles per second
cell.lifetime = 2.0                    // seconds each particle lives
cell.lifetimeRange = 0.5              // randomize: 1.5 to 2.5 seconds

    Motion

    cell.velocity = 150                    // initial speed (points/sec)
cell.velocityRange = 50               // randomize: 100 to 200
cell.emissionLongitude = -.pi / 2     // primary direction (up)
cell.emissionLatitude = 0             // tilt in/out of screen (3D)
cell.emissionRange = .pi / 4         // cone spread (45 degrees each side)
cell.xAcceleration = 0                // horizontal force (wind)
cell.yAcceleration = 200             // vertical force (gravity pulls down)

    emissionLongitude is the angle in the XY plane. 0 = right, .pi/2 = down, -.pi/2 = up, .pi = left. emissionRange adds randomness around that direction.

    Scale and Rotation

    cell.scale = 0.3                       // base scale
cell.scaleRange = 0.2                  // randomize: 0.1 to 0.5
cell.scaleSpeed = -0.05               // shrink over lifetime
cell.spin = 0                          // rotation speed (radians/sec)
cell.spinRange = .pi                  // random spin

    Color

    cell.color = UIColor.white.cgColor     // base tint (multiplied with contents)
cell.redRange = 0.3                    // randomize red: ±0.3
cell.greenRange = 0.2                  // randomize green: ±0.2
cell.blueRange = 0.1                   // randomize blue: ±0.1

    Stopping Gracefully

    Never remove the emitter layer abruptly -- existing particles vanish. Instead:

    // Stop spawning new particles; existing ones finish naturally

// After all particles have expired, remove the layer

    emitter.removeFromSuperlayer()
}

    Performance Notes

    CAEmitterLayer is entirely GPU-driven. After you configure the cells, the CPU does no per-frame work. This makes it extremely efficient for thousands of particles. Keep in mind:

    • Particle images should be small (8-32 points). Larger images increase GPU fill rate.
    • birthRate * lifetime = approximate active particle count. Keep this reasonable (under ~5000 for complex scenes).
    • .additive render mode is cheap on the GPU but can cause visual blowout.
    • Sub-emitters multiply particle counts exponentially. Use sparingly.

    CATiledLayer

    Start here — the plain idea

    CATiledLayer exists for one problem: showing something far too big to hold in memory all at once — a gigapixel photo, a city map, a 50-page PDF zoomed way in. If you loaded the whole thing as one bitmap, you'd need hundreds of megabytes of RAM. The fix is the same one map apps use: chop the content into a grid of small tiles and only draw the handful you can actually see right now.

    The mental shift is this — instead of "draw the whole image," CATiledLayer calls your draw(in:) once per visible tile, handing you a tiny rect and saying "just fill this square." You draw only that piece. Two twists trip people up. First, those draw calls happen on background threads (several at once), so anything shared and mutable you touch in there is a data race waiting to happen. Second, it has levels of detail — when the user zooms out, it draws low-res tiles instead of shrinking the giant ones, saving memory and time. Think of it as a just-in-time delivery system for pixels: nothing gets drawn until it's about to be seen.

    CATiledLayer is designed for one specific problem: displaying content that is far too large to fit in memory as a single bitmap. Maps, PDF documents, gigapixel images -- anything where the full resolution would consume hundreds of megabytes of RAM. CATiledLayer solves this by breaking the content into tiles and drawing each one on demand, on a background thread.

    See 04-layer-contents.md -- CATiledLayer replaces the standard approach of setting layer.contents to a single CGImage with a dynamic tiling system.

    How Tiling Works

    When a CATiledLayer needs to display, it:

    1. Divides its bounds into a grid of tiles (default 256x256 points).
    2. Determines which tiles are currently visible in the viewport.
    3. Calls draw(in: CGContext) on a background thread for each visible tile.
    4. The context's clip rect tells you exactly which tile region to draw.
    5. Newly drawn tiles fade in over fadeDuration seconds.
    6. Off-screen tiles are cached but can be purged under memory pressure.

    This means the layer never holds the entire content in memory. Only the visible tiles plus a small buffer exist at any time.

    Basic Setup

    import UIKit

class TiledView: UIView {
    override class var layerClass: AnyClass {
        return CATiledLayer.self
    }

    var tiledLayer: CATiledLayer {
        return layer as! CATiledLayer
    }

    override init(frame: CGRect) {
        super.init(frame: frame)
        tiledLayer.tileSize = CGSize(width: 256, height: 256)
        tiledLayer.levelsOfDetail = 4
        tiledLayer.levelsOfDetailBias = 2
        tiledLayer.contentsScale = UIScreen.main.scale
    }

    required init?(coder: NSCoder) {
        fatalError("init(coder:) has not been implemented")
    }

    override func draw(_ rect: CGRect) {
        guard let context = UIGraphicsGetCurrentContext() else { return }

        // rect is the tile being drawn -- NOT the full bounds
        // Draw only what falls within this tile

        let tileX = Int(rect.origin.x / 256)
        let tileY = Int(rect.origin.y / 256)

        // Alternate checkerboard colors per tile
        let isEven = (tileX + tileY) % 2 == 0
        context.setFillColor(isEven ? UIColor.systemGray5.cgColor : UIColor.systemGray3.cgColor)
        context.fill(rect)

        // Draw tile coordinates for debugging
        let label = "(\(tileX), \(tileY))" as NSString
        let attrs: [NSAttributedString.Key: Any] = [
            .font: UIFont.monospacedSystemFont(ofSize: 14, weight: .medium),
            .foregroundColor: UIColor.label
        ]
        label.draw(at: CGPoint(x: rect.origin.x + 10, y: rect.origin.y + 10),
                   withAttributes: attrs)
    }
}

    The critical insight: draw(_ rect:) receives the rect for one tile, not the entire layer. You only draw content for that tile region.

    Key Properties

    tileSize

    tiledLayer.tileSize = CGSize(width: 512, height: 512)

    The size of each tile in points. Larger tiles mean fewer draw calls but more memory per tile. Smaller tiles mean faster initial display but more overhead. The default of 256x256 is a good balance for most content.

    levelsOfDetail (LOD)

    tiledLayer.levelsOfDetail = 4

    The number of zoom-out levels beyond the base resolution. Each level halves the resolution. With levelsOfDetail = 4:

    Level Resolution
    0 (base) 1x (full detail)
    1 1/2x
    2 1/4x
    3 1/8x

    When the user zooms out, CATiledLayer draws lower-resolution tiles instead of scaling down the full-resolution ones. This saves memory and drawing time.

    levelsOfDetailBias

    tiledLayer.levelsOfDetailBias = 2

    Extra zoom-in levels beyond 1:1. Each bias level doubles the resolution. With levelsOfDetailBias = 2, the layer can render at 2x and 4x the base resolution when the user zooms in. This provides sharper content at high zoom without pre-rendering the entire image at 4x.

    fadeDuration

    tiledLayer.fadeDuration = 0.15  // default is 0.25

    How long new tiles take to fade in. Set to 0 for instant display. The default 0.25s provides a smooth appearance but can look sluggish if tiles draw quickly.

    PDF Rendering with CATiledLayer

    PDFs are resolution-independent and can be enormous. CATiledLayer draws each visible tile at the current zoom level by rendering the PDF into that tile's region.

    class PDFTiledView: UIView {
    override class var layerClass: AnyClass { CATiledLayer.self }

    var pdfPage: CGPDFPage? {
        didSet { setNeedsDisplay() }
    }

    override init(frame: CGRect) {
        super.init(frame: frame)
        let tiled = layer as! CATiledLayer
        tiled.levelsOfDetail = 4
        tiled.levelsOfDetailBias = 3
        tiled.tileSize = CGSize(width: 512, height: 512)
        backgroundColor = .white
    }

    required init?(coder: NSCoder) { fatalError() }

    override func draw(_ rect: CGRect) {
        guard let context = UIGraphicsGetCurrentContext(),
              let page = pdfPage else { return }

        // Fill background white (PDF pages may not cover full tile)
        context.setFillColor(UIColor.white.cgColor)
        context.fill(rect)

        // PDFs use a flipped coordinate system
        context.saveGState()

        let pageRect = page.getBoxRect(.mediaBox)
        context.translateBy(x: 0, y: bounds.height)
        context.scaleBy(x: 1, y: -1)

        let scale = bounds.width / pageRect.width
        context.scaleBy(x: scale, y: scale)

        context.drawPDFPage(page)
        context.restoreGState()
    }
}

    Thread Safety Warning

    draw(in:) is called on background threads. This is unusual for UIKit/Core Animation code. You must be careful:

    override func draw(_ rect: CGRect) {
    // SAFE: UIGraphicsGetCurrentContext() is set up per-thread by CATiledLayer
    guard let context = UIGraphicsGetCurrentContext() else { return }

    // SAFE: Immutable data, constants, local variables
    let color = UIColor.systemBlue.cgColor
    context.setFillColor(color)

    // DANGEROUS: Accessing shared mutable state without synchronization
    // self.someArray.append(rect)  // Data race!

    // DANGEROUS: UIKit calls (except graphics context)
    // let image = UIImage(named: "tile")  // May not be thread-safe in all cases

    // SAFE: Use a serial queue or lock for shared state
    // serialQueue.sync { self.loadedTiles.insert(tileKey) }
}

    Use pre-loaded data, immutable state, or explicit synchronization (serial queues, locks) for any shared resources accessed during tile drawing.

    Invalidating Tiles

    To force a tile to redraw (e.g., the underlying data changed):

    // Redraw a specific region
tiledLayer.setNeedsDisplay(CGRect(x: 512, y: 0, width: 256, height: 256))

// Redraw everything (clears the tile cache)
tiledLayer.setNeedsDisplay()

    Calling setNeedsDisplay() on the entire layer clears the tile cache and redraws all visible tiles. Use the rect-based version for targeted updates.

    Comparison: Approaches for Large Content

    Approach Best For Drawback
    CATiledLayer + UIScrollView PDFs, huge images, custom maps Background-thread drawing complexity
    MKMapView Geographic maps specifically Limited to map content
    UICollectionView Grid of discrete items Not for continuous content
    Custom Metal renderer Real-time zoomable content, terrain Much more code and complexity

    CATiledLayer is the right choice when you have a large, continuous, mostly-static content surface that needs pan and zoom. For real-time content, consider Metal.

    Performance Tips

    • Tile size trade-off: 256x256 is standard. Use 512x512 for content that is expensive to set up per-draw (e.g., PDF rendering) since fewer tiles means fewer draw calls.
    • Precompute tile images: If possible, pre-render tiles to disk (as PNG/JPEG files) and simply draw the pre-rendered image in draw(_:rect:). This is how map tile servers work.
    • fadeDuration = 0: If your tiles draw fast enough, eliminate the fade for snappier feel.
    • levelsOfDetail: Use the minimum levels needed. Each additional LOD level means more tiles to potentially render.
    • Avoid setNeedsDisplay() on the full layer unless absolutely necessary -- it discards the entire tile cache.

    The foundation

    • CATiledLayer = draws huge content as a grid of small tiles, on demand, so it never holds the whole thing in memory.
    • draw(_ rect:) = called once per tile; the rect is one square to fill, not the full bounds.
    • background threads = those draw calls run off the main thread (often several at once), so guard any shared mutable state.
    • tileSize = grid granularity, default 256×256; bigger means fewer draw calls but more memory per tile.
    • levelsOfDetail = extra zoom-out levels (low-res tiles when zoomed out).
    • levelsOfDetailBias = extra zoom-in levels (sharper tiles when zoomed in).
    • fadeDuration = how long new tiles fade in; set to 0 for instant.
    • setNeedsDisplay(rect) = redraw just one region instead of dumping the whole tile cache.

    CAFilter (Private API)

    Start here — the plain idea

    CAFilter is the hidden engine behind almost every visual effect you've admired in iOS — the frosted blur behind the lock screen clock, the saturated glassy tab bar, the dimmed notification background. It's a little GPU effect you attach to a layer: blur me, desaturate me, boost my contrast. Apple uses it everywhere but never exposed it publicly, so you reach it through the Objective-C runtime by name. That's also why it's off-limits for the App Store — treat this whole lesson as "looking under the hood," not shipping code.

    Two distinctions clear up the confusion. First, a filter on a layer changes that layer's own content. To blur whatever is behind a layer (real frosted glass), you need a CABackdropLayer — a special layer that captures the backdrop first, then filters it. That backdrop-capture is literally what UIVisualEffectView does internally. Second, filters chain: stack [blur, saturate, brightness] and the output of one feeds the next, so order matters — blurring then saturating looks different from the reverse. Think of it as a pedalboard for a layer's pixels.

    WARNING: Everything in this lesson uses private Apple APIs. These are NOT allowed in App Store submissions. Use them for learning, reverse engineering, personal tools, and portfolio projects only. Apple can change or remove these APIs at any time without notice.

    CAFilter is a private Core Animation class that applies GPU-backed visual effects to layers. It is the mechanism Apple uses internally for blur, saturation, brightness, and compositing effects throughout iOS -- from the lock screen blur to notification backgrounds to the tab bar translucency. Understanding CAFilter reveals how Apple builds its visual design system.

    See 13-blur-effects.md -- for the math behind Gaussian blur and other blur algorithms.

    See 16-blending-and-compositing.md -- for Porter-Duff compositing modes and blend mode math.

    Loading CAFilter

    CAFilter is not exposed in any public header. You access it via Objective-C runtime:

    import UIKit

// Get the CAFilter class
let CAFilter: AnyObject.Type = NSClassFromString("CAFilter")! as! AnyObject.Type

// Create a filter by name
func makeFilter(name: String) -> NSObject {
    let filter = CAFilter.perform(NSSelectorFromString("filterWithName:"), with: name)!
    return filter.takeUnretainedValue() as! NSObject
}

    Applying Filters to a Layer

    Filters are set via KVC (key-value coding) on the layer:

    let blurFilter = makeFilter(name: "gaussianBlur")
blurFilter.setValue(10.0, forKey: "inputRadius")

// Apply to a layer's content
layer.setValue([blurFilter], forKey: "filters")

    The "filters" key takes an array. Multiple filters are applied in order.

    // Remove all filters
layer.setValue(nil, forKey: "filters")

    Blur Filters

    gaussianBlur

    The standard Gaussian blur. This is what UIVisualEffectView uses internally.

    class BlurDemoViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()

        // Background content to blur
        let imageView = UIImageView(image: UIImage(named: "cityscape"))
        imageView.frame = view.bounds
        imageView.contentMode = .scaleAspectFill
        view.addSubview(imageView)

        // Overlay layer with blur filter
        let blurLayer = CALayer()
        blurLayer.frame = CGRect(x: 40, y: 200, width: 300, height: 200)
        blurLayer.backgroundColor = UIColor(white: 1, alpha: 0.1).cgColor
        blurLayer.cornerRadius = 20

        let blur = makeFilter(name: "gaussianBlur")
        blur.setValue(20.0, forKey: "inputRadius")
        blurLayer.setValue([blur], forKey: "filters")

        view.layer.addSublayer(blurLayer)
    }
}

    Important: A filter on a layer blurs that layer's own content (its contents image and sublayer tree). To blur content behind the layer (backdrop blur), you need CABackdropLayer (covered below).

    variableBlur

    Variable blur applies different blur amounts across the layer using a mask image. Apple uses this for scroll-edge fading blurs in navigation bars and scroll views.

    let variableBlur = makeFilter(name: "variableBlur")
variableBlur.setValue(15.0, forKey: "inputRadius")

// Create a gradient mask: clear at top (no blur), white at bottom (full blur)
let maskImage = makeGradientMask(size: CGSize(width: 300, height: 200))
variableBlur.setValue(maskImage, forKey: "inputMaskImage")

layer.setValue([variableBlur], forKey: "filters")

    The mask image controls blur intensity per-pixel: white = full blur, black = no blur, gray = partial blur.

    func makeGradientMask(size: CGSize) -> CGImage {
    let renderer = UIGraphicsImageRenderer(size: size)
    let image = renderer.image { ctx in
        let gradient = CGGradient(
            colorsSpace: CGColorSpaceCreateDeviceGray(),
            colors: [UIColor.black.cgColor, UIColor.white.cgColor] as CFArray,
            locations: [0.0, 1.0]
        )!
        ctx.cgContext.drawLinearGradient(
            gradient,
            start: CGPoint(x: 0, y: 0),
            end: CGPoint(x: 0, y: size.height),
            options: []
        )
    }
    return image.cgImage!
}

    Color Filters

    colorSaturate

    Adjusts color saturation. inputAmount = 0 is full grayscale, 1 is normal, > 1 boosts saturation.

    let saturate = makeFilter(name: "colorSaturate")
saturate.setValue(0.0, forKey: "inputAmount")  // grayscale
layer.setValue([saturate], forKey: "filters")

    colorBrightness

    Adjusts brightness. Positive values brighten, negative darken.

    let brightness = makeFilter(name: "colorBrightness")
brightness.setValue(0.2, forKey: "inputAmount")  // slightly brighter
layer.setValue([brightness], forKey: "filters")

    colorContrast

    Adjusts contrast. 1 is normal, < 1 reduces contrast, > 1 increases it.

    let contrast = makeFilter(name: "colorContrast")
contrast.setValue(1.5, forKey: "inputAmount")  // high contrast
layer.setValue([contrast], forKey: "filters")

    colorHueRotate

    Rotates all hues by an angle in radians.

    let hueRotate = makeFilter(name: "colorHueRotate")
hueRotate.setValue(Float.pi, forKey: "inputAngle")  // 180 degree hue shift
layer.setValue([hueRotate], forKey: "filters")

    colorInvert

    Inverts all colors. No parameters needed.

    let invert = makeFilter(name: "colorInvert")
layer.setValue([invert], forKey: "filters")

    Compositing Filters

    Compositing filters control how a layer blends with the content beneath it. These are set on the compositingFilter property (a single filter, not an array).

    // Additive blending (light adds together)
layer.compositingFilter = "plusL"

// Other compositing modes:
layer.compositingFilter = "multiplyBlendMode"
layer.compositingFilter = "screenBlendMode"
layer.compositingFilter = "overlayBlendMode"
layer.compositingFilter = "sourceOver"       // default (normal alpha compositing)
layer.compositingFilter = "plusD"            // Porter-Duff plus (clamped)
layer.compositingFilter = "plusL"            // linear dodge (unclamped add)

    class CompositingDemoViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()
        view.backgroundColor = .black

        // Red circle
        let red = CALayer()
        red.frame = CGRect(x: 80, y: 300, width: 200, height: 200)
        red.backgroundColor = UIColor.red.cgColor
        red.cornerRadius = 100
        view.layer.addSublayer(red)

        // Blue circle with additive blending
        let blue = CALayer()
        blue.frame = CGRect(x: 140, y: 340, width: 200, height: 200)
        blue.backgroundColor = UIColor.blue.cgColor
        blue.cornerRadius = 100
        blue.compositingFilter = "plusL"  // where they overlap = magenta
        view.layer.addSublayer(blue)

        // Green circle with additive blending
        let green = CALayer()
        green.frame = CGRect(x: 110, y: 380, width: 200, height: 200)
        green.backgroundColor = UIColor.green.cgColor
        green.cornerRadius = 100
        green.compositingFilter = "plusL"
        view.layer.addSublayer(green)
    }
}

    With plusL, overlapping red + blue = magenta, red + green = yellow, all three = white. This is additive light blending.

    CABackdropLayer

    CABackdropLayer is the private layer class that captures whatever is rendered behind it. This is what makes UIVisualEffectView work -- it grabs the backdrop, applies blur and saturation, then composites the result.

    let CABackdropLayer: AnyClass = NSClassFromString("CABackdropLayer")!

class BackdropBlurViewController: UIViewController {
    override func viewDidLoad() {
        super.viewDidLoad()

        // Content behind the blur
        let imageView = UIImageView(image: UIImage(named: "landscape"))
        imageView.frame = view.bounds
        imageView.contentMode = .scaleAspectFill
        view.addSubview(imageView)

        // Container view for the blur panel
        let blurPanel = UIView(frame: CGRect(x: 30, y: 200, width: 320, height: 220))
        blurPanel.layer.cornerRadius = 24
        blurPanel.clipsToBounds = true
        view.addSubview(blurPanel)

        // Create the backdrop layer
        let backdrop = CABackdropLayer.alloc() as! CALayer
        backdrop.perform(NSSelectorFromString("init"))
        backdrop.frame = blurPanel.bounds

        // Apply blur + saturation to the captured backdrop
        let blur = makeFilter(name: "gaussianBlur")
        blur.setValue(30.0, forKey: "inputRadius")

        let saturate = makeFilter(name: "colorSaturate")
        saturate.setValue(1.8, forKey: "inputAmount")

        backdrop.setValue([blur, saturate], forKey: "filters")
        blurPanel.layer.addSublayer(backdrop)

        // Add a tint layer on top
        let tint = CALayer()
        tint.frame = blurPanel.bounds
        tint.backgroundColor = UIColor(white: 1, alpha: 0.15).cgColor
        blurPanel.layer.addSublayer(tint)
    }
}

    This creates the same frosted-glass effect as UIVisualEffectView but with full control over blur radius, saturation, and tinting.

    Filter Chaining

    Multiple filters apply in sequence. The output of one filter feeds into the next:

    let blur = makeFilter(name: "gaussianBlur")
blur.setValue(8.0, forKey: "inputRadius")

let saturate = makeFilter(name: "colorSaturate")
saturate.setValue(1.5, forKey: "inputAmount")

let brightness = makeFilter(name: "colorBrightness")
brightness.setValue(0.05, forKey: "inputAmount")

// Blur first, then boost saturation, then brighten
layer.setValue([blur, saturate, brightness], forKey: "filters")

    Order matters. Blurring before saturating gives a different result than saturating before blurring.

    Animating Filter Values

    Filter properties can be animated through the layer's key paths:

    // Animate blur radius from 0 to 20
let blurAnim = CABasicAnimation(keyPath: "filters.gaussianBlur.inputRadius")
blurAnim.fromValue = 0
blurAnim.toValue = 20
blurAnim.duration = 1.0
layer.add(blurAnim, forKey: "blurAnimation")

    The key path format is filters.<filterName>.<propertyName>. The filter name is the string you passed to filterWithName:.

    Practical Example: Custom Blur Card

    Putting it all together -- a frosted glass card without UIVisualEffectView:

    class FrostedCardViewController: UIViewController {
    override func viewDidLoad() {

        let backdropClass = NSClassFromString("CABackdropLayer")! as! CALayer.Type
        let backdrop = backdropClass.init()
        backdrop.frame = card.bounds

        // 2. Apply blur + saturation
        let blur = makeFilter(name: "gaussianBlur")
        blur.setValue(25.0, forKey: "inputRadius")
        let sat = makeFilter(name: "colorSaturate")
        sat.setValue(1.8, forKey: "inputAmount")
        backdrop.setValue([blur, sat], forKey: "filters")
        card.layer.addSublayer(backdrop)

        // 3. White tint overlay
        let tint = CALayer()
        tint.frame = card.bounds
        tint.backgroundColor = UIColor(white: 1.0, alpha: 0.2).cgColor
        card.layer.addSublayer(tint)

        // 4. Border highlight
        let border = CAShapeLayer()
        border.path = UIBezierPath(roundedRect: card.bounds, cornerRadius: 20).cgPath
        border.fillColor = nil
        border.strokeColor = UIColor(white: 1, alpha: 0.3).cgColor
        border.lineWidth = 0.5
        card.layer.addSublayer(border)

        // 5. Content label
        let label = UILabel()
        label.text = "Frosted Glass"
        label.font = .systemFont(ofSize: 24, weight: .semibold)
        label.textColor = .white
        label.sizeToFit()
        label.center = CGPoint(x: card.bounds.midX, y: card.bounds.midY)
        card.addSubview(label)
    }
}

    Layer-Backed Views

    Start here — the plain idea

    Here's the secret most people learn late: a UIView doesn't actually draw anything. Every view has a CALayer riding inside it, and that is what shows up on screen. The view is the manager — it handles touches, layout, and lifecycle — while the layer is the worker that holds the pixels. Think of the view as the driver and the layer as the engine: you talk to the driver, but the engine does the moving.

    This explains a lot of "why does it work that way" moments. view.alpha is just a polite wrapper that sets layer.opacity; view.backgroundColor sets layer.backgroundColor. Some powers only exist on the engine — shadows, 3D transforms, anchor points, per-corner rounding — so for those you drop down to view.layer. And UIView.animate { } isn't magic either: inside that closure UIKit quietly opens a CATransaction and asks the layer for CABasicAnimation objects, which is exactly why your property changes only animate when they happen inside the block. Knowing the engine is there turns Core Animation from mysterious into obvious.

    Every UIView is backed by a CALayer. The view is the controller; the layer is the renderer. Understanding this relationship is essential because many visual capabilities -- shadows, 3D transforms, custom corner radii, anchor points -- only exist at the layer level. And knowing how UIView's animation system wraps Core Animation explains why certain things work the way they do.

    See 01-how-rendering-works.md -- the full rendering pipeline from view layout through GPU compositing.

    See 10-structs-vs-classes.md -- UIView is a class (reference type). Multiple variables can point to the same view and its layer.

    Every UIView Has a Layer

    let button = UIButton(type: .system)
print(type(of: button.layer))  // CALayer

let label = UILabel()
print(type(of: label.layer))   // CALayer

let imageView = UIImageView()
print(type(of: imageView.layer))  // CALayer

    The layer is created automatically when the view is initialized. You never create it yourself. The view owns the layer and sets itself as the layer's delegate.

    Property Proxying: View vs. Layer

    Many UIView properties are thin wrappers around layer properties:

    let myView = UIView()

// These are equivalent pairs:
myView.alpha = 0.5
myView.layer.opacity  // 0.5

myView.backgroundColor = .red
myView.layer.backgroundColor  // red CGColor

myView.isHidden = true
myView.layer.isHidden  // true

myView.clipsToBounds = true
myView.layer.masksToBounds  // true

    Setting view.alpha internally sets layer.opacity. Setting view.backgroundColor internally sets layer.backgroundColor. They are the same value.

    frame Differences

    view.frame and layer.frame are usually the same, but they diverge when a transform is applied:

    let box = UIView(frame: CGRect(x: 0, y: 0, width: 100, height: 100))

// Rotate 45 degrees
box.transform = CGAffineTransform(rotationAngle: .pi / 4)

// view.frame is the AXIS-ALIGNED bounding box of the rotated view
print(box.frame)  // approximately (-20.7, -20.7, 141.4, 141.4)

// layer.bounds is unchanged
print(box.layer.bounds)  // (0, 0, 100, 100)

// layer.position is the anchor point location
print(box.layer.position)  // (50, 50) — center of original frame

    After a transform, view.frame returns the bounding rectangle that encloses the rotated view. Do not set frame when a non-identity transform is active -- set bounds and center instead.

    Overriding layerClass

    By default, UIView creates a CALayer. You can override layerClass to get a specialized layer subclass:

    class ShapeView: UIView {
    override class var layerClass: AnyClass {
        return CAShapeLayer.self
    }

    var shapeLayer: CAShapeLayer {
        return layer as! CAShapeLayer
    }

    func configure() {
        shapeLayer.path = UIBezierPath(ovalIn: bounds).cgPath
        shapeLayer.fillColor = UIColor.systemBlue.cgColor
        shapeLayer.strokeColor = UIColor.white.cgColor
        shapeLayer.lineWidth = 3
    }
}

    This is more efficient than creating a UIView and adding a CAShapeLayer as a sublayer. The view's backing layer is the shape layer, so no extra layer exists.

    How UIView.animate Wraps Core Animation

    When you write:

    UIView.animate(withDuration: 0.5) {
    myView.alpha = 0.0
}

    Here is exactly what happens internally:

    1. UIKit opens a CATransaction with duration 0.5.
    2. It sets a flag that tells the view's layer delegate to return animations instead of NSNull().
    3. Inside the closure, myView.alpha = 0.0 sets layer.opacity = 0.0.
    4. The layer asks its delegate (the UIView) for an action for "opacity".
    5. Because the animation flag is set, UIView returns a CABasicAnimation with the correct fromValue (captured before the closure) and toValue.
    6. The animation is added to the layer.
    7. The transaction commits.

    This is why properties must change inside the closure. Outside the closure, the UIView delegate returns NSNull() (no animation).

    UIViewPropertyAnimator Wraps CASpringAnimation

    let animator = UIViewPropertyAnimator(
    duration: 0.5,
    dampingRatio: 0.7
) {
    myView.center.y += 200
}
animator.startAnimation()

    Internally, UIViewPropertyAnimator creates CASpringAnimation objects. The dampingRatio maps to the spring's damping and stiffness parameters. You get the convenience of UIKit with the physics of Core Animation.

    Layer-Only Features

    These capabilities exist only on CALayer and have no UIView equivalent:

    Shadows

    view.layer.shadowColor = UIColor.black.cgColor
view.layer.shadowOffset = CGSize(width: 0, height: 4)
view.layer.shadowRadius = 8
view.layer.shadowOpacity = 0.3

// IMPORTANT: always set shadowPath for performance
view.layer.shadowPath = UIBezierPath(
    roundedRect: view.bounds,
    cornerRadius: view.layer.cornerRadius
).cgPath

    Without shadowPath, Core Animation must ray-trace the shadow from the layer's composited alpha channel every frame. With shadowPath, it uses the precomputed path.

    3D Transforms

    UIView exposes only CGAffineTransform (2D). For 3D rotation and perspective, you must use the layer:

    var transform = CATransform3DIdentity
transform.m34 = -1.0 / 500  // perspective
transform = CATransform3DRotate(transform, .pi / 4, 1, 0, 0)  // tilt
view.layer.transform = transform

    sublayerTransform (Shared Perspective)

    var perspective = CATransform3DIdentity
perspective.m34 = -1.0 / 1000
containerView.layer.sublayerTransform = perspective
// Now all sublayers share the same vanishing point

    Anchor Point

    view.layer.anchorPoint = CGPoint(x: 0, y: 0)  // top-left pivot
// view.layer.position stays the same, so the view visually shifts

    Changing the anchor point changes where position refers to, which shifts the view's visual location. To change anchor point without moving the view:

    func setAnchorPoint(_ point: CGPoint, for view: UIView) {
    let oldFrame = view.frame
    view.layer.anchorPoint = point
    view.frame = oldFrame
}

    Individual Corner Radii (iOS 11+)

    view.layer.cornerRadius = 20
view.layer.maskedCorners = [.layerMinXMinYCorner, .layerMaxXMinYCorner]
// Only top-left and top-right corners are rounded

    Border

    view.layer.borderWidth = 1
view.layer.borderColor = UIColor.separator.cgColor

    The Drawing Chain

    When a view needs to display content:

    // 1. Mark view as needing display
view.setNeedsDisplay()

// 2. This calls layer.setNeedsDisplay() internally

// 3. During the display phase, the layer calls its delegate (the view)
//    which triggers:
override func draw(_ rect: CGRect) {
    // Your drawing code here
    // The context is backed by the layer's bitmap
}

// 4. The drawn bitmap becomes layer.contents
// 5. The render server composites it on the GPU

    UIView's draw(_ rect:) is actually invoked by the layer's display mechanism. The view is the layer's delegate, and the layer calls draw(_:in:) on its delegate, which UIView redirects to your draw(_ rect:) override.

    shouldRasterize

    view.layer.shouldRasterize = true
view.layer.rasterizationScale = UIScreen.main.scale

    When shouldRasterize is true, the layer and all its sublayers are flattened into a single bitmap and cached. This can dramatically improve performance for complex, static layer trees (e.g., a cell with shadows, rounded corners, and multiple sublayers).

    Pitfalls: - The rasterized image is cached at rasterizationScale. If you forget to set this, it rasterizes at 1x on a 3x display and looks blurry. - If the layer content changes frequently, rasterization hurts performance because the cache is regenerated every frame. - The cache consumes memory.

    // Good: complex but rarely changing content
cell.layer.shouldRasterize = true
cell.layer.rasterizationScale = UIScreen.main.scale

// Bad: content that animates continuously
animatingView.layer.shouldRasterize = true  // cache invalidated every frame

    The Rendering Pipeline for UIView

    The full pipeline, from your code to pixels:

    1. Layout phase (layoutSubviews) -- positions and sizes subviews.
    2. Display phase (draw(_ rect:)) -- draws content into the layer's backing store.
    3. Commit -- the layer tree is serialized and sent to the render server (a separate process).
    4. Render server -- composites all layers on the GPU.
    5. Display -- the composited frame buffer is sent to the display hardware.

    Steps 1-3 happen in your app process. Steps 4-5 happen in the render server process (backboardd). This split is why animations run smoothly even when your main thread is busy -- the render server has its own copy of the layer tree.

    Offscreen Rendering Triggers

    Certain layer configurations force the GPU to render into an offscreen buffer before compositing. This is expensive -- it doubles the rendering work for that layer.

    Trigger Why It Causes Offscreen Rendering
    shadowColor without shadowPath GPU must compute shadow from alpha channel
    cornerRadius + masksToBounds GPU must clip content to rounded rect
    layer.mask (mask layer) GPU must intersect content with mask
    layer.allowsGroupOpacity + transparent GPU must composite sublayers first, then apply opacity

    How to avoid: - Always set shadowPath when using shadows. - Use layer.cornerRadius without masksToBounds when possible (the background rounds, but content is not clipped). - Prefer shouldRasterize for static content with unavoidable offscreen triggers. - Use Instruments > Core Animation to spot offscreen rendering (yellow highlights).

    // SLOW: shadow without path + rounded corners with masking
view.layer.shadowColor = UIColor.black.cgColor
view.layer.shadowOpacity = 0.3
view.layer.shadowRadius = 8
view.layer.cornerRadius = 16
view.layer.masksToBounds = true  // triggers offscreen rendering

// FAST: shadow with path, no masking
view.layer.shadowPath = UIBezierPath(
    roundedRect: view.bounds, cornerRadius: 16
).cgPath
view.layer.cornerRadius = 16
// Don't set masksToBounds unless you need content clipping

    Building Complex Layer Trees in UIView

    You can add sublayers in your UIView subclass for complex visuals:

    class FancyCardView: UIView {

    private let borderLayer = CAShapeLayer()

        super.init(frame: frame)

            UIColor.systemPurple.cgColor,
            UIColor.systemPink.cgColor

        borderLayer.fillColor = nil
        borderLayer.strokeColor = UIColor(white: 1, alpha: 0.3).cgColor

        layer.masksToBounds = true
    }

            roundedRect: bounds.insetBy(dx: 0.5, dy: 0.5),
            cornerRadius: 20

    }
}

    Metal and CALayer

    Start here — the plain idea

    Most layers hold pixels that Core Animation drew for you. A CAMetalLayer is different: it hands you the canvas and steps back. It's the doorway through which your own GPU code — Metal shaders — gets its rendered frames onto the screen and composited alongside ordinary UIKit layers. Even MTKView, the "easy" Metal view, is just a CAMetalLayer with a tidy wrapper bolted on; learning the layer directly means you control the timing, pixel format, and color yourself.

    The core ritual is the drawable. A drawable is one frame's worth of canvas — a texture you draw into, then "present" so Core Animation shows it. The layer keeps a small pool of two or three of these and nextDrawable() blocks until one is free, which is actually your built-in frame pacing. Pair that with a CADisplayLink — a timer locked to the screen's refresh — and you get a clean loop: the screen says "I'm about to refresh," you render a fresh frame into a drawable, you present it, repeat. That's the whole heartbeat of every Metal app, games included.

    CAMetalLayer is the bridge between Metal's GPU rendering pipeline and Core Animation's compositing system. Every Metal-rendered pixel that appears on screen goes through a CAMetalLayer. Even MTKView is just a convenience wrapper around it. Understanding CAMetalLayer directly gives you full control over presentation timing, pixel format, color space, and display features like ProMotion and EDR.

    See 03-your-first-shader.md -- for Metal device creation, command queues, and pipeline setup.

    See 17-metal-in-uikit.md -- for a complete Metal + UIKit integration guide.

    Configuration Properties

    device

    metalLayer.device = MTLCreateSystemDefaultDevice()!

    The Metal device that will render into this layer. Must be set before requesting drawables.

    pixelFormat

    metalLayer.pixelFormat = .bgra8Unorm     // standard 8-bit per channel (default)
metalLayer.pixelFormat = .bgra8Unorm_srgb // sRGB color space
metalLayer.pixelFormat = .rgba16Float     // 16-bit float (HDR/EDR)

    .bgra8Unorm is the standard choice. Use .rgba16Float for EDR (Extended Dynamic Range) content where pixel values can exceed 1.0.

    framebufferOnly

    metalLayer.framebufferOnly = true  // default

    When true, the drawable's texture can only be used as a render target -- you cannot read from it or use it as a shader input. This enables GPU optimizations. Set to false only if you need to sample from the drawable texture (e.g., for post-processing passes that read back the framebuffer).

    contentsScale

    metalLayer.contentsScale = UIScreen.main.scale

    Maps points to pixels. On a 3x display, a 100x100 point layer renders 300x300 pixels. Always set this or your content will look blurry.

    drawableSize

    // Read the actual pixel dimensions
print(metalLayer.drawableSize)  // e.g., (1020, 1020) on a 3x display

// Or set it explicitly to decouple from frame * scale
metalLayer.drawableSize = CGSize(width: 512, height: 512)

    If not set manually, drawableSize is computed as frame.size * contentsScale. Setting it explicitly allows rendering at a lower resolution for performance.

    The Render Loop

    The fundamental cycle for Metal rendering:

    func render() {
    // 1. Get a drawable -- this is the texture we'll render into
    guard let drawable = metalLayer.nextDrawable() else { return }

    // 2. Create a render pass descriptor pointing to the drawable's texture
    let passDescriptor = MTLRenderPassDescriptor()
    passDescriptor.colorAttachments[0].texture = drawable.texture
    passDescriptor.colorAttachments[0].loadAction = .clear
    passDescriptor.colorAttachments[0].storeAction = .store
    passDescriptor.colorAttachments[0].clearColor = MTLClearColor(
        red: 0.1, green: 0.1, blue: 0.2, alpha: 1.0
    )

    // 3. Create a command buffer and encoder
    guard let commandBuffer = commandQueue.makeCommandBuffer(),
          let encoder = commandBuffer.makeRenderCommandEncoder(
              descriptor: passDescriptor
          ) else { return }

    // 4. Encode render commands
    encoder.setRenderPipelineState(pipelineState)
    encoder.setVertexBuffer(vertexBuffer, offset: 0, index: 0)
    encoder.drawPrimitives(type: .triangle, vertexStart: 0, vertexCount: 3)
    encoder.endEncoding()

    // 5. Present the drawable (schedules display after GPU finishes)
    commandBuffer.present(drawable)

    // 6. Commit the command buffer to the GPU
    commandBuffer.commit()
}

    Key points: - nextDrawable() blocks if all drawables are in use (the layer has a pool of 2-3). This is your frame pacing mechanism. - present(drawable) tells Core Animation to display this texture once the GPU is done. - commit() submits the work to the GPU. Rendering happens asynchronously.

    Off-Screen Rendering to Texture

    You can render to a Metal texture without displaying it, then assign it to a regular CALayer.contents:

    func renderToTexture() -> MTLTexture {
    let textureDescriptor = MTLTextureDescriptor.texture2DDescriptor(
        pixelFormat: .bgra8Unorm,
        width: 512,
        height: 512,
        mipmapped: false
    )
    textureDescriptor.usage = [.renderTarget, .shaderRead]

    let texture = device.makeTexture(descriptor: textureDescriptor)!

    let passDescriptor = MTLRenderPassDescriptor()
    passDescriptor.colorAttachments[0].texture = texture
    passDescriptor.colorAttachments[0].loadAction = .clear
    passDescriptor.colorAttachments[0].storeAction = .store
    passDescriptor.colorAttachments[0].clearColor = MTLClearColor(
        red: 0.2, green: 0.5, blue: 0.8, alpha: 1.0
    )

    let commandBuffer = commandQueue.makeCommandBuffer()!
    let encoder = commandBuffer.makeRenderCommandEncoder(descriptor: passDescriptor)!
    // Draw your content...
    encoder.endEncoding()
    commandBuffer.commit()
    commandBuffer.waitUntilCompleted()

    return texture
}

// Display the texture in a regular CALayer
func displayTexture(_ texture: MTLTexture) {
    let ciImage = CIImage(mtlTexture: texture, options: nil)!
    let cgImage = CIContext().createCGImage(ciImage, from: ciImage.extent)!
    someLayer.contents = cgImage
}

    The foundation

    • CAMetalLayer = the doorway that puts your own GPU-rendered frames on screen and composites them with UIKit.
    • drawable = one frame's canvas (a texture); you render into it, then present it.
    • nextDrawable() = grabs a free drawable from the small pool, blocking if none are ready — your frame pacing.
    • present + commit = present(drawable) schedules display once the GPU finishes; commit() submits the work.
    • CADisplayLink = a callback synced to the screen refresh; the heartbeat that drives the render loop.
    • essential config = device, pixelFormat, framebufferOnly, and contentsScale (set it or it's blurry).
    • drawableSize = actual pixel dimensions; lower it to render at reduced resolution for performance.
    • vs MTKView = MTKView is CAMetalLayer plus automatic setup; use the raw layer for full control or sublayer embedding.
    • EDR / wide color = wantsExtendedDynamicRangeContent + .rgba16Float for HDR; Display P3 for a wider gamut.

    Start here — the plain idea

    Recipe 1: App Store Card Expand

    Visual goal: A card in a grid expands to fill the screen with a spring animation, rounding down corners and fading in detail content -- like the App Store Today tab.

    import UIKit

class CardExpandViewController: UIViewController {
    private let card = UIView()
    private let cardImage = UIImageView()
    private var isExpanded = false
    private var collapsedFrame = CGRect.zero

    override func viewDidLoad() {
        super.viewDidLoad()
        view.backgroundColor = .systemBackground

        collapsedFrame = CGRect(x: 24, y: 120, width: view.bounds.width - 48, height: 280)

        card.frame = collapsedFrame
        card.backgroundColor = .systemIndigo

        card.layer.cornerCurve = .continuous  // Apple's squircle

        card.layer.shadowColor = UIColor.black.cgColor
        card.layer.shadowOpacity = 0.2
        card.layer.shadowRadius = 12
        card.layer.shadowOffset = CGSize(width: 0, height: 6)
        card.layer.shadowPath = UIBezierPath(
            roundedRect: card.bounds, cornerRadius: 20
        ).cgPath

        cardImage.frame = card.bounds
        cardImage.backgroundColor = .systemIndigo
        cardImage.contentMode = .scaleAspectFill
        card.addSubview(cardImage)

        view.addSubview(card)

        let tap = UITapGestureRecognizer(target: self, action: #selector(toggleExpand))
        card.addGestureRecognizer(tap)
    }

    @objc func toggleExpand() {
        isExpanded.toggle()
        let targetFrame = isExpanded ? view.bounds : collapsedFrame
        let targetRadius: CGFloat = isExpanded ? 0 : 20

        // Spring animation for frame
        let posAnim = CASpringAnimation(keyPath: "position")
        posAnim.fromValue = card.layer.presentation()?.position ?? card.layer.position
        posAnim.toValue = CGPoint(x: targetFrame.midX, y: targetFrame.midY)
        posAnim.damping = 18
        posAnim.stiffness = 220
        posAnim.mass = 1.0
        posAnim.duration = posAnim.settlingDuration

        let boundsAnim = CASpringAnimation(keyPath: "bounds.size")
        boundsAnim.fromValue = card.layer.presentation()?.bounds.size ?? card.layer.bounds.size
        boundsAnim.toValue = targetFrame.size
        boundsAnim.damping = 18
        boundsAnim.stiffness = 220
        boundsAnim.mass = 1.0
        boundsAnim.duration = boundsAnim.settlingDuration

        let cornerAnim = CASpringAnimation(keyPath: "cornerRadius")
        cornerAnim.fromValue = card.layer.presentation()?.cornerRadius ?? card.layer.cornerRadius
        cornerAnim.toValue = targetRadius
        cornerAnim.damping = 18
        cornerAnim.stiffness = 220
        cornerAnim.mass = 1.0
        cornerAnim.duration = cornerAnim.settlingDuration

        let shadowAnim = CABasicAnimation(keyPath: "shadowOpacity")
        shadowAnim.fromValue = card.layer.shadowOpacity
        shadowAnim.toValue = isExpanded ? 0.0 : 0.2
        shadowAnim.duration = 0.3

        // Apply final values
        CATransaction.begin()
        CATransaction.setDisableActions(true)
        card.layer.position = CGPoint(x: targetFrame.midX, y: targetFrame.midY)
        card.layer.bounds.size = targetFrame.size
        card.layer.cornerRadius = targetRadius
        card.layer.shadowOpacity = isExpanded ? 0.0 : 0.2
        card.layer.shadowPath = UIBezierPath(
            roundedRect: CGRect(origin: .zero, size: targetFrame.size),
            cornerRadius: targetRadius
        ).cgPath
        cardImage.frame = CGRect(origin: .zero, size: targetFrame.size)
        CATransaction.commit()

        card.layer.add(posAnim, forKey: "position")
        card.layer.add(boundsAnim, forKey: "bounds")
        card.layer.add(cornerAnim, forKey: "corner")
        card.layer.add(shadowAnim, forKey: "shadow")
    }
}

    Recipe 2: Pull-to-Refresh Spinner

    Visual goal: An arc stroke that grows as the user pulls down, then spins continuously on release.

    class PullToRefreshLayer: CALayer {
    let arcLayer = CAShapeLayer()
    private var isSpinning = false

    override init() {
        super.init()
        setup()
    }

    required init?(coder: NSCoder) { fatalError() }

    override init(layer: Any) {
        super.init(layer: layer)
    }

    private func setup() {
        let radius: CGFloat = 14
        let center = CGPoint(x: 22, y: 22)
        let path = UIBezierPath(
            arcCenter: center,
            radius: radius,
            startAngle: -.pi / 2,
            endAngle: 3 * .pi / 2,
            clockwise: true
        )

        arcLayer.path = path.cgPath
        arcLayer.fillColor = nil
        arcLayer.strokeColor = UIColor.label.cgColor
        arcLayer.lineWidth = 2.5
        arcLayer.lineCap = .round
        arcLayer.strokeEnd = 0
        arcLayer.frame = CGRect(x: 0, y: 0, width: 44, height: 44)
        addSublayer(arcLayer)

        bounds = CGRect(x: 0, y: 0, width: 44, height: 44)
    }

    /// Call during scroll: progress 0.0 to 1.0
    func updatePullProgress(_ progress: CGFloat) {
        guard !isSpinning else { return }
        CATransaction.begin()
        CATransaction.setDisableActions(true)
        arcLayer.strokeEnd = min(progress, 0.9)
        // Slight rotation as user pulls
        arcLayer.transform = CATransform3DMakeRotation(progress * .pi, 0, 0, 1)
        CATransaction.commit()
    }

    /// Call when user releases past threshold
    func startSpinning() {
        isSpinning = true

        arcLayer.strokeEnd = 0.75

        let rotation = CABasicAnimation(keyPath: "transform.rotation.z")
        rotation.fromValue = 0
        rotation.toValue = 2 * Float.pi
        rotation.duration = 0.8
        rotation.repeatCount = .infinity
        rotation.timingFunction = CAMediaTimingFunction(name: .linear)
        arcLayer.add(rotation, forKey: "spin")
    }

    /// Call when refresh completes
    func stopSpinning() {
        isSpinning = false
        arcLayer.removeAnimation(forKey: "spin")

        let fadeOut = CABasicAnimation(keyPath: "opacity")
        fadeOut.toValue = 0
        fadeOut.duration = 0.25
        arcLayer.add(fadeOut, forKey: "fadeOut")

        DispatchQueue.main.asyncAfter(deadline: .now() + 0.25) {
            self.arcLayer.strokeEnd = 0
            self.arcLayer.opacity = 1
            self.arcLayer.removeAllAnimations()
        }
    }
}

    Recipe 6: iMessage Bubble Pop

    Visual goal: A chat bubble scales up from small with a spring, fades in, and slides up slightly -- like iMessage incoming bubbles.

    func animateBubbleAppear(_ bubbleView: UIView) {
    let finalCenter = bubbleView.center
    let startCenter = CGPoint(x: finalCenter.x, y: finalCenter.y + 20)

    // Set initial state
    CATransaction.begin()
    CATransaction.setDisableActions(true)
    bubbleView.layer.transform = CATransform3DMakeScale(0.1, 0.1, 1)
    bubbleView.layer.opacity = 0
    bubbleView.layer.position = startCenter
    CATransaction.commit()

    // Scale spring
    let scaleAnim = CASpringAnimation(keyPath: "transform.scale")
    scaleAnim.fromValue = 0.1
    scaleAnim.toValue = 1.0
    scaleAnim.damping = 12
    scaleAnim.stiffness = 250
    scaleAnim.mass = 0.8
    scaleAnim.duration = scaleAnim.settlingDuration

    // Opacity fade
    let opacityAnim = CABasicAnimation(keyPath: "opacity")
    opacityAnim.fromValue = 0
    opacityAnim.toValue = 1
    opacityAnim.duration = 0.15

    // Position slide
    let posAnim = CASpringAnimation(keyPath: "position.y")
    posAnim.fromValue = startCenter.y
    posAnim.toValue = finalCenter.y
    posAnim.damping = 14
    posAnim.stiffness = 200
    posAnim.mass = 0.8
    posAnim.duration = posAnim.settlingDuration

    // Group them
    let group = CAAnimationGroup()
    group.animations = [scaleAnim, opacityAnim, posAnim]
    group.duration = max(scaleAnim.settlingDuration, posAnim.settlingDuration)

    // Set final values
    bubbleView.layer.transform = CATransform3DIdentity
    bubbleView.layer.opacity = 1
    bubbleView.layer.position = finalCenter

    bubbleView.layer.add(group, forKey: "bubblePop")
}

    }

        posAnim.beginTime = CACurrentMediaTime() + delay
        posAnim.duration = posAnim.settlingDuration
        posAnim.fillMode = .backwards  // hold initial state before beginTime

        // Opacity
        let opacityAnim = CABasicAnimation(keyPath: "opacity")
        opacityAnim.fromValue = 0
        opacityAnim.toValue = 1
        opacityAnim.duration = 0.3
        opacityAnim.beginTime = CACurrentMediaTime() + delay
        opacityAnim.fillMode = .backwards

        // Set final model values
        layer.transform = CATransform3DIdentity
        layer.opacity = 1

        layer.add(posAnim, forKey: "staggerPosition")
        layer.add(opacityAnim, forKey: "staggerOpacity")
    }
}

// In your table view controller:
class StaggeredListViewController: UITableViewController {
    private var hasAnimated = false
    private let items = Array(0..<30)

    override func viewDidAppear(_ animated: Bool) {
        super.viewDidAppear(animated)

        guard !hasAnimated else { return }
        hasAnimated = true

        let visibleCells = tableView.visibleCells
        for (index, cell) in visibleCells.enumerated() {
            let delay = Double(index) * 0.06  // 60ms stagger
            cell.animateAppear(delay: delay)
        }
    }

    override func tableView(_ tableView: UITableView, numberOfRowsInSection section: Int) -> Int {
        return items.count
    }

    override func tableView(_ tableView: UITableView, cellForRowAt indexPath: IndexPath) -> UITableViewCell {
        let cell = tableView.dequeueReusableCell(withIdentifier: "Cell") ??
                   UITableViewCell(style: .default, reuseIdentifier: "Cell")
        cell.textLabel?.text = "Item \(items[indexPath.row])"
        return cell
    }
}

    Key timing: The stagger delay (60ms between cells) is critical. Too short and cells appear to animate together. Too long and the animation feels sluggish. Apple typically uses 40-80ms stagger.

    Apple's approach: Apple often combines the stagger with a slight blur-to-sharp transition using CAFilter (private API). The spring parameters are tuned so that earlier cells have nearly settled by the time later cells begin, creating a smooth wave effect.

    Animation Parameter Reference

    Quick reference for spring parameters that feel "Apple-like":

    Effect Damping Stiffness Mass Notes
    Snappy tap response 15-20 300-400 0.5-0.8 Quick settle, minimal overshoot
    Bouncy pop-in 8-12 200-300 0.6-1.0 Visible overshoot, playful
    Card expand/collapse 16-20 200-250 1.0-1.2 Weighty, deliberate
    Badge bounce 6-10 250-350 0.4-0.6 Pronounced bounce
    Keyboard-style slide 18-22 280-350 0.9-1.0 Fast, professional
    Rubber band 4-8 100-200 0.5-0.8 Exaggerated bounce

    The general rule: higher stiffness = faster animation, lower damping = more overshoot, higher mass = more inertia.
