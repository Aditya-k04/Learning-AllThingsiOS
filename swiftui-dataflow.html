    01 — SwiftUI Is a Dependency Graph

    The single most important mental model: SwiftUI is not a view tree that you redraw. It's a graph of values and the views that depend on them. When a value changes, SwiftUI re-evaluates only the views that read it. Everything in this series is a consequence of that one idea.

    The graph (AttributeGraph)

    Underneath SwiftUI is a private framework called AttributeGraph. Think of it as a spreadsheet:

    • Each attribute is a cell: a piece of state, a derived value, or a view's body.
    • Edges are "depends on" relationships: body depends on the @State it reads.
    • When a cell changes, the graph marks everything downstream invalid (dirty).
    • On the next run loop, SwiftUI re-evaluates only the dirty attributes — it recomputes those bodys, diffs the result, and updates the rendered layers.
    @State count ──(read by)──▶ CounterView.body ──▶ rendered Text
       ▲
       └── Button action mutates count  ⇒  body marked dirty  ⇒  body re-runs

    You never tell SwiftUI "re-render." You mutate a value; the graph figures out who cared.

    "Reading a value" is what creates the edge

    The dependency edge is created the moment your body reads the value during evaluation. This is the crux:

    struct ProfileView: View {
    var model: UserModel          // @Observable

    var body: some View {
        Text(model.name)          // reading `name` ⇒ edge: body depends on model.name
        // model.email is NOT read here ⇒ NO edge ⇒ email changes don't touch this view
    }
}

    If you don't read it, you don't depend on it, and you won't be invalidated by it. This is why @Observable is efficient (lesson 05) and why "my view isn't updating" almost always means "the edge you assumed exists was never created" (lesson 12).

    Three things SwiftUI needs for each view

    From WWDC's Demystify SwiftUI, every view has:

    1. Identity — which view is this, across updates? (lesson 02)
    2. Lifetime — how long does its state live? (tied to identity)
    3. Dependencies — what values, when changed, invalidate its body?

    This series is really about #3 (dependencies) and how #1/#2 make state persist while the value-type view struct is thrown away and recreated constantly.

    The update cycle (what actually happens)

    1. Something mutates a source of truth (@State, an @Observable property, an environment value).
    2. The graph marks dependent body attributes dirty.
    3. SwiftUI schedules an update (coalesced — many mutations in one run loop tick = one update).
    4. Dirty bodys re-run, producing new value-type view trees.
    5. SwiftUI diffs new vs. old using identity, and mutates only the rendered objects that changed.

    Your body running is cheap and frequent. It is not the same as drawing. A body re-evaluation that produces an identical result causes no visible redraw. This is why "body runs a lot" is normal, and why lesson 11 is about reducing needless body runs, not eliminating them.

    Why views are structs (and disposable)

    A View is a value type — a lightweight description, recreated on every parent update. It holds no real storage. The durable stuff (your @State, your @Observable object) lives in the graph, outside the struct, keyed by identity. The struct is the recipe; the graph is the kitchen.

    This is the reverse of UIKit, where the UIView object is the durable thing you mutate.

    Exercises

    1. Add let _ = Self._printChanges() to two views — one that reads a property and one that doesn't. Mutate the property and watch which body re-runs in the console.

    3. Put a print("body") in a parent and a child. Trigger a parent state change and observe how invalidation flows downward.

    02 — View Identity and Lifetime

    State persists because of identity. If you don't understand identity, you'll hit the two classic bugs: state that resets unexpectedly, and state that refuses to reset when you want it to.

    Identity ≠ the view struct

    The view struct is recreated constantly (lesson 01). So SwiftUI can't use "is this the same struct instance?" to decide whether state carries over. Instead it uses identity: a stable notion of "this is the same UI element as last update."

    State (@State, @StateObject, the storage behind @Observable owned by @State) is keyed to identity:

    • Same identity across updates → same storage → state persists.
    • New/changed identity → fresh storage → state is reset (and old state is destroyed).

    Two kinds of identity

    1. Structural identity (the default)

    SwiftUI infers identity from a view's position in the view tree — the shape of your body. As long as the structure is stable, the view at a given position keeps its identity.

    if isExpanded {
    DetailView()      // identity A — lives only while isExpanded == true
} else {
    SummaryView()     // identity B
}

    The if/else creates two distinct identities. Flipping isExpanded destroys one and creates the other — their state does not transfer.

    ⚠️ AnyView erases structural identity. SwiftUI can't see the shape through it, so it often can't preserve identity across updates → state loss + worse diffing. Avoid AnyView in hot paths (lesson 11).

    2. Explicit identity

    You assign identity yourself:

    • ForEach(items, id: \.id) — each row's identity is its id.
    • .id(someValue) — forces an identity; changing the value destroys and recreates the view (and resets its state).
    ProfileView(user: user)
    .id(user.id)   // when user.id changes, ProfileView is rebuilt fresh — state resets

    This is the deliberate tool for "reset this subtree when X changes."

    Lifetime: how long state lives

    Lifetime = how long an identity exists in the graph. State is allocated when an identity first appears and destroyed when it disappears.

    ForEach(rows) { row in
    RowView(row: row)   // each identity's @State lives as long as that row is in `rows`
}

    Remove a row → its identity disappears → its @State is destroyed. Add it back later → new identity → fresh state. This is why @State is for ephemeral, view-local data you're happy to lose when the view goes away — and why durable data belongs in a model (lesson 06), not in @State on a transient row.

    The two classic bugs

    Bug A — "my state keeps resetting." Something upstream is changing this view's identity every update: an .id() bound to a value that changes too often, an if/else that swaps branches, an AnyView erasing structure, or a ForEach id that isn't stable (e.g. array index used as id while the array reorders).

    Bug B — "my state won't reset when the data changes." Same identity, new data — SwiftUI correctly kept the state. Fix by attaching .id(newValue) to force a fresh identity, or by deriving the view from the data instead of caching it in @State.

    Why this matters for the rest of the series

    • @State storage (lesson 03) is keyed to identity — that's why it persists across body re-runs.
    • The environment (lesson 07) propagates along the identity/structure of the tree — and presentation boundaries (sheets) start a new identity branch, which is the root cause of the injection-rejection in lesson 09.

    Exercises

    1. Put a @State var n = 0 Stepper inside both branches of an if/else. Increment, flip the branch, flip back — watch the state reset (two identities).
    2. Wrap a stateful view in .id(toggleValue). Toggle and confirm state resets on every change.
    3. In a ForEach, use the array index as id, then reorder the array. Observe state attaching to the wrong rows. Fix it with a stable id.

    Checkpoint

    • [ ] State is keyed to identity, not to the (disposable) view struct.
    • [ ] I can distinguish structural identity from explicit identity (.id(), ForEach id).
    • [ ] I know AnyView and unstable ForEach ids erase/scramble identity.
    • [ ] I can deliberately reset a subtree with .id(value) and explain why it works.
    • [ ] I understand lifetime = how long an identity exists, and that disappearing identity destroys its state.

    03 — @State Storage Internals

    @State looks like a variable on a struct, but a struct is a disposable value (lesson 01). So where does the value actually live, and why does it survive?

    @State is a handle, not the storage

    @State is a property wrapper whose stored part is tiny: a reference to a box of storage that SwiftUI owns, allocated in the graph and keyed to the view's identity (lesson 02).

    struct Counter: View {
    @State private var count = 0   // `count` is a handle to SwiftUI-managed storage
    var body: some View {
        Button("\(count)") { count += 1 }
    }
}
    • The Counter struct is recreated on every parent update.
    • But @State's storage box is not recreated — the new struct's handle is reconnected to the same box (because identity is the same).
    • That's why count keeps its value across hundreds of body re-runs.

    Conceptually:

    Counter struct (recreated)        SwiftUI graph (durable)
   @State count  ──handle──▶       [ box: 0 → 1 → 2 ... ]   keyed to Counter's identity

    Init-once semantics

    The initializer you write (= 0) is the default, used only the first time this identity appears. On later updates, SwiftUI ignores it and reconnects the existing box.

    @State private var count = expensiveDefault()   // expensiveDefault() may run more than you think

    ⚠️ The expression is evaluated whenever the struct is created (it's a normal Swift initializer), but its result is only adopted on first appearance. So: - Don't rely on @State's initializer to "reset" on later updates — it won't. - Avoid expensive work in a @State default; it can run on every struct recreation even though the value is discarded. Prefer .task/lazy init, or compute in the model.

    Passing a value in from a parent into @State? It seeds the state once. If the parent's value changes later, the @State does not update — by design. If you need it to track the parent, you want a plain let (read-only) or a Binding (lesson 04), not @State.

    Mutation → invalidation

    Writing through the @State handle does two things: 1. Updates the box. 2. Tells the graph "the dependents of this box are dirty" → those bodys re-run (lesson 01).

    This is why count += 1 updates the UI with no manual call. It's also why mutating a value no one read still works but causes no visible change (no edges → nothing to invalidate).

    Value types vs. reference types in @State

    • Value type (Int, String, your struct): @State owns the value. Mutating any part (count += 1, myStruct.field = x) is a write → invalidation.
    • Reference type (a plain class): @State owns the pointer. Mutating the object's properties does not go through the handle, so SwiftUI sees nothing and won't update. This is the entire reason the Observation framework (lesson 05) exists: to make class-property mutations observable. For an @Observable class you own, use @State — @State holds it stably, and observation handles the property-level updates.
    @State private var model = EditorModel()   // EditorModel is @Observable — correct
// reading model.title in body creates the edge; mutating model.title updates the UI

    @State vs the older @StateObject

    @StateObject (iOS 13–16) is the legacy equivalent for ObservableObject: it owns a reference type with init-once semantics, surviving body re-runs. In modern code, @State + @Observable replaces @StateObject + ObservableObject. Same ownership/lifetime idea, better observation (lesson 05). You'll still see @StateObject in older parts of a codebase.

    Exercises

    1. Put print("struct init") in a @State private var x = { print("default"); return 0 }() pattern (use an immediately-invoked closure). Trigger parent updates and watch "default" evaluate even though x doesn't reset.
    2. Try to "update" a child's @State by passing a new value from the parent each render. Confirm it ignores the new value after first appearance. Then switch to Binding and confirm it tracks.
    3. Put a plain class (not @Observable) in @State, mutate a property in a button, and watch the UI fail to update. Add @Observable and watch it work.

    Checkpoint

    • [ ] @State is a handle to SwiftUI-owned storage keyed to identity — not storage on the struct.
    • [ ] The initializer is a first-appearance default; later updates reconnect the existing box.
    • [ ] A value passed into @State seeds it once and won't track the parent afterward.
    • [ ] I know why mutating a plain-class property in @State doesn't update the UI, and that @Observable fixes it.

    04 — Binding Mechanics

    A Binding is the answer to: "how do I let a child write to a source of truth it doesn't own?" Understanding that it's just a get/set pair dissolves most confusion about $.

    What a Binding actually is

    Binding<Value> is a lightweight struct wrapping two closures — a getter and a setter — plus dependency bookkeeping:

    // Conceptually:
struct Binding<Value> {
    let get: () -> Value
    let set: (Value) -> Void
}

    It owns no storage. It's a two-way reference to a value that lives somewhere else (a @State box, an @Observable property, etc.). Reading the binding calls get; assigning calls set, which writes back to the real source → invalidation (lesson 01).

    The projected value: what $ means

    Every property wrapper can expose a projected value via $. For @State, the projected value is a Binding to its storage:

    struct Parent: View {
    @State private var name = ""        // `name` → the value; `$name` → Binding<String>
    var body: some View {
        TextField("Name", text: $name)  // hand the child a write-path to our state
        Child(name: $name)
    }
}

struct Child: View {
    @Binding var name: String           // a borrowed write-path; Child owns no storage
    var body: some View {
        TextField("Name", text: $name)  // $name here re-projects the binding
    }
}

    $name doesn't copy the string — it hands over the get/set pair pointing at Parent's @State box. When Child writes, it writes straight through to Parent's storage, and the graph invalidates everyone who read it.

    $ on @State → Binding. $ on a @Bindable @Observable → Binding to that object's property (lesson 06). $ on @Binding → the same binding, re-projected. Same idea everywhere: $ = "give me the write-path."

    Deriving bindings

    You can transform a binding without breaking the two-way link:

    // Binding to a nested property:
TextField("City", text: $user.address.city)     // dot-chaining preserves write-through

// Custom binding (full control of get/set):
let upper = Binding(
    get: { text },
    set: { text = $0.uppercased() }
)

// Binding to an optional with a default:
TextField("Nick", text: Binding($user.nickname, default: ""))   // if you have such a helper
// or unwrap:
if let nick = Binding($user.nickname) { TextField("Nick", text: nick) }

    Binding also supports map-like helpers in practice via custom inits. The rule: as long as set writes back to a real source of truth, the chain stays live.

    Bindings and dependencies (a subtle point)

    Holding a Binding does not by itself subscribe a view to changes. The dependency edge is created when you read the value (call get) during body (lesson 01). A view that only passes a binding through to a child, without reading it, won't re-render when the value changes — but the child that reads it will. This is usually what you want (it avoids re-rendering pass-through parents).

    Stale-binding & write-back traps

    1. Custom binding reading a stale snapshot. If your custom get closure captures a copy instead of reading the live source, it returns stale data: swift // BAD: captures the value at creation time let b = Binding(get: { snapshotCopy }, set: { snapshotCopy = $0 }) // GOOD: reads/writes the live source each call let b = Binding(get: { model.value }, set: { model.value = $0 })
    2. Binding to a computed property with no setter → you can't; set has nowhere to write.
    3. Binding to a let → impossible; there's no write-path. Use @State/@Bindable/@Binding on the owner.
    4. Animating through a binding: the animation attaches at the set site / the reading view, not the binding itself — keep that in mind when a write isn't animating.

    Mental model summary

    @State (owns box) ──$──▶ Binding(get/set) ──passed down──▶ @Binding (borrows) ──$──▶ Binding ...
        ▲                                                                                │
        └──────────────────────── set writes straight back to the box ──────────────────┘

    A Binding is a wire, not a tank. It carries reads and writes back to whoever owns the storage.

    Exercises

    1. Build Parent with @State name and a Child with @Binding name. Add Self._printChanges() to both. Type in the child; observe the parent's box updating and which bodies re-run.
    2. Write a custom Binding that uppercases on set. Then deliberately make it stale (capture a copy) and watch it break.
    3. Make a parent that passes a binding to a child but never reads it. Confirm the parent doesn't re-render when the value changes, but the child does.

    Checkpoint

    • [ ] A Binding is a get/set pair referencing storage elsewhere — it owns nothing.
    • [ ] $ produces the projected value; for @State/@Binding that's a Binding.
    • [ ] I can derive nested, custom, and optional bindings without breaking write-through.
    • [ ] Holding a binding ≠ subscribing; the edge comes from reading it.
    • [ ] I can spot stale-snapshot and no-write-path binding bugs.

    05 — Observation Framework Internals

    @Observable (iOS 17+, the Observation module) is the piece that makes class property mutations drive SwiftUI updates — and does it with surgical precision: only views that read a changed property re-render. This lesson opens the hood.

    The problem it solves

    A plain class in @State doesn't update the UI when its properties change (lesson 03) — SwiftUI only sees the pointer, not the mutation. The old fix, ObservableObject + @Published, broadcasts a single "something changed" signal that re-renders every observing view, even ones reading unrelated properties. @Observable replaces the broadcast with per-property tracking.

    Expand the macro (do this in Xcode)

    @Observable is a macro. Right-click an @Observable class → Expand Macro. You'll see roughly:

    @Observable
class EditorModel {
    var title = ""
    var body  = ""
}

    expands to something like:

    class EditorModel {
    @ObservationTracked var title = ""
    @ObservationTracked var body  = ""

    private let _$observationRegistrar = ObservationRegistrar()

    // For each property, generated accessors:
    var title: String {
        get {
            _$observationRegistrar.access(self, keyPath: \.title)   // ← registers a read
            return _title
        }
        set {
            _$observationRegistrar.withMutation(of: self, keyPath: \.title) { // ← announces a write
                _title = newValue
            }
        }
    }
    // ...same for `body`...
}

    Two generated calls are the whole trick: - access(self, keyPath:) in every getter — "someone is reading this exact property right now." - withMutation(of:keyPath:) in every setter — "this exact property is changing."

    The ObservationRegistrar is the bookkeeper connecting reads to interested observers per key path.

    How SwiftUI uses it: withObservationTracking

    The public primitive is:

    withObservationTracking {
    // 1. run this; every `access(...)` during it is recorded
    render(using: model.title)        // reads model.title ⇒ tracked
} onChange: {
    // 2. called once, the next time ANY tracked property is about to change
    scheduleUpdate()
}

    SwiftUI evaluates a view's body inside withObservationTracking. So: 1. While body runs, every @Observable property it reads calls access → SwiftUI records "this body depends on model.title." 2. When something later sets model.title, withMutation fires the onChange → SwiftUI marks that body dirty and re-evaluates it (lesson 01). 3. Re-evaluation re-arms tracking for the next change (onChange is one-shot).

    Result: reading model.title in body is exactly what subscribes the view to title — and only title. Read nothing, depend on nothing. This is the lesson-01 "edge = read" rule, mechanized.

    Why this is more precise than @Published

    ObservableObject + @Published @Observable
    Granularity Per-object (objectWillChange fires for any @Published write) Per-property (per key path actually read)
    A view reading only a when b changes Re-renders (wasteful) Does not re-render
    Boilerplate @Published on each property none
    View wrappers @StateObject / @ObservedObject / @EnvironmentObject @State / plain ref / @Environment / @Bindable

    Things the macro changes about your reasoning

    • Computed properties are tracked transitively. A computed var fullName { "\(first) \(last)" } isn't itself stored, but reading it reads first/last, so the view depends on those. Good.
    • Collections & nested @Observable. Mutating an element triggers tracking for the array property you read. Nested @Observable objects track their own properties — reading model.child.name subscribes to child.name.
    • Non-stored / @ObservationIgnored. Mark a property @ObservationIgnored to opt it out of tracking (e.g. caches, non-UI state).
    • Reads outside body aren't tracked. Reading a property in a button action or .task does not create a dependency — only reads during body evaluation count.

    Threading

    @Observable doesn't add thread safety. Mutations that drive UI must land on the main actor. Annotate UI-facing models with @MainActor (see Learning-Swift/21-concurrency-advanced). withMutation notifying observers off the main thread → updates scheduled off-main → undefined UI behavior.

    Exercises

    1. Expand the macro on a 3-property @Observable and read the generated access/withMutation calls. Write down which getter registers which key path.
    2. Two sibling views read different properties of one model. Add Self._printChanges() to both; mutate each property; prove per-property precision.
    3. Add a computed property combining two stored ones; confirm a view reading only the computed value updates when either stored value changes.
    4. Mark a property @ObservationIgnored, read it in a body, mutate it, and confirm the view does not update.

    Checkpoint

    • [ ] I can explain why @Observable re-renders less than @Published.
    • [ ] I know reads outside body aren't tracked, and that @MainActor is still my responsibility.

    06 — Observable Ownership: @State, plain refs, @Bindable

    Observation (lesson 05) handles updates. This lesson handles ownership: who creates the object, who keeps it alive, who borrows it, and how you get two-way bindings into it.

    The three roles

    Role Wrapper Meaning
    Owner @State This view creates the object and controls its lifetime (init-once, lesson 03).
    Borrower (read/observe) plain property (var model: M) This view uses an object owned elsewhere; reading its props still subscribes (lesson 05).
    Borrower (needs bindings) @Bindable This view needs $model.prop write-paths into an object owned elsewhere. 
    @Observable class EditorModel { var title = ""; var saved = false }

struct EditorScreen: View {
    @State private var model = EditorModel()        // OWNER — lifetime tied to this view's identity
    var body: some View { EditorForm(model: model) } // pass the plain reference down
}

struct EditorForm: View {
    @Bindable var model: EditorModel                 // BORROWER that needs bindings
    var body: some View {
        TextField("Title", text: $model.title)       // $ on @Bindable → Binding<String>
        Toggle("Saved", isOn: $model.saved)
    }
}

struct TitleLabel: View {
    var model: EditorModel                            // BORROWER that only reads
    var body: some View { Text(model.title) }         // reading subscribes to .title only
}

    Why @State for the owner (not @StateObject)

    For an @Observable class, @State provides exactly the ownership semantics you need: it holds the reference stably across body re-runs, keyed to identity (lesson 03), and creates it once. @StateObject is the legacy owner for ObservableObject; don't use it with @Observable. (@ObservedObject/@EnvironmentObject are likewise legacy — see the comparison table in lesson 05.)

    @Bindable in depth

    @Bindable does one job: expose $ bindings to an @Observable object's properties. It does not own or change the object's lifetime.

    • As a property: @Bindable var model: EditorModel — when a child needs bindings to a passed-in object.
    • Inline in body: when you only need a binding briefly: swift var body: some View { @Bindable var model = model // local rebind; legal inside body TextField("Title", text: $model.title) }
    • From the environment: read with @Environment(EditorModel.self), then @Bindable locally to get bindings (lesson 07).

    $model.title builds a Binding whose get/set read/write model.title through the observed accessors (lesson 04 + 05) — so writes both update the object and trigger precise invalidation.

    Object identity & lifetime (don't recreate by accident)

    The object lives as long as the owning @State identity lives (lesson 02). Watch for:

    • Creating the model in the wrong place. SomeView(model: EditorModel()) in a parent's body makes a new object every time the parent re-renders. Own it with @State instead, or inject it once.
    • Resetting via .id(). Changing the owner's .id() destroys and recreates its @State model — sometimes intended (reset the editor for a new document), often a surprise.
    • Sharing one object across screens. Don't give each screen its own @State copy. Own it once high up (or in the environment, lesson 07) and pass/inject the same reference.

    Choosing the wrapper (decision)

    Do I create/own this object's lifetime here?
├─ Yes → @State
└─ No  → Do I need $ bindings to its properties?
         ├─ Yes → @Bindable
         └─ No  → plain `var` (reading still observes)

Is it provided app-wide / across boundaries? → inject + read with @Environment(Type.self)  (lesson 07)

    Exercises

    1. Build owner (@State) → form (@Bindable) → label (plain var). Confirm: form edits propagate, label updates, and only views reading a changed property re-run.
    2. Deliberately create the model inside a parent's body (Child(model: EditorModel())) and watch edits vanish on every parent re-render. Fix with @State.
    3. Read an @Observable from the environment and use an inline @Bindable var model = model to bind a TextField.

    • [ ] I know @State (not @StateObject) owns an @Observable.

    • [ ] I can explain and avoid "new object every render" and unintended .id() resets.

    07 — The Environment as a Graph

    The environment is SwiftUI's dependency-injection mechanism: a typed bag of values that flows down the view tree, where any descendant can read the nearest ancestor's value without it being passed through every initializer in between.

    The mental model: a propagated, typed dictionary

    Think of EnvironmentValues as an immutable dictionary that travels down the tree:

    • Keys are types (EnvironmentKey conformers, or for objects, the object's type).
    • Each view node sees the dictionary handed down by its parent.
    • A .environment(...) modifier produces a modified copy for that view's subtree — the original is untouched, descendants see the new value.
    • @Environment(\.key) reads the value from the dictionary that reached this node.
    Root  { colorScheme: .light, theme: .blue }
 └─ A  .environment(\.theme, .purple)
     └─ B   sees { colorScheme: .light, theme: .purple }   ← nearest ancestor wins
         └─ C  @Environment(\.theme) ⇒ .purple

    It propagates with the tree's structure (lesson 02). That detail is exactly why presentation boundaries break it (lesson 09).

    Two flavors: values and objects

    1. Environment values (key + default) — works for any type

    struct ThemeKey: EnvironmentKey { static let defaultValue: Color = .blue }
extension EnvironmentValues {
    var theme: Color {
        get { self[ThemeKey.self] }
        set { self[ThemeKey.self] = newValue }
    }
}
// inject:  .environment(\.theme, .purple)
// read:    @Environment(\.theme) private var theme

    Crucial property: there is always a defaultValue. If no ancestor injected one, the read returns the default — no crash. (iOS 18+ lets you declare custom values with the @Entry macro instead of the boilerplate, but the default is still required.)

    2. Environment objects (@Observable, iOS 17+) — by type, no default

    @Observable class Session { var user: User? }
// inject:  .environment(session)            // note: the object, no key path
// read:    @Environment(Session.self) private var session

    Crucial property: there is no default. If no ancestor injected a Session, the read traps at runtime (lesson 09). This is the modern replacement for the legacy @EnvironmentObject (which has the same no-default, crash-if-missing behavior).

    Value (\.key) Object (Type.self)
    Requires defaultValue Yes No
    Missing injection returns default (safe) crash
    Use for config, theming, system values shared app/session models

    Reads resolve to the nearest ancestor

    When you read, SwiftUI walks up the tree to the closest .environment that set that key/type. Inner injections shadow outer ones for their subtree:

    ContentView()
    .environment(\.theme, .blue)        // app-wide default
// ...somewhere deep...
SpecialSection()
    .environment(\.theme, .red)         // this subtree only sees .red

    Outside SpecialSection, .theme is still .blue. This shadowing is a feature (scoped overrides), and a footgun if you inject the wrong scope.

    Reading the environment creates a dependency

    @Environment(\.theme) isn't a one-time copy — reading it in body registers a dependency (lesson 01). If an ancestor changes the injected value, views reading it re-evaluate. For environment objects, you additionally get Observation's per-property tracking (lesson 05) once you read the object's properties.

    Built-in environment values

    SwiftUI ships dozens: \.colorScheme, \.horizontalSizeClass, \.dynamicTypeSize, \.locale, \.dismiss, \.openURL, \.scenePhase, \.isEnabled, \.layoutDirection, … Same machinery — they're injected by the system high in the tree (and by modifiers like .disabled()), and you read them with @Environment(\.key). Studying these is the fastest way to internalize "injected high, read anywhere below."

    Exercises

    1. Make a custom value key with default .blue. Read it in a deep child without injecting — confirm you get .blue (no crash). Then inject .purple at a mid-level ancestor and confirm shadowing.
    2. Make an @Observable Session, inject at the app root, read it three levels down. Then remove the injection and observe the crash (preview of lesson 09).
    3. Inject \.theme, .red on a subtree and confirm siblings outside it still read the outer value.

    Checkpoint

    • [ ] I picture EnvironmentValues as an immutable, typed dictionary copied-and-modified down the tree.
    • [ ] I know values (key + default) are crash-safe; objects (Type.self, no default) crash if missing.
    • [ ] Reads resolve to the nearest ancestor; inner injections shadow outer ones.
    • [ ] Reading environment in body creates a dependency (and observes object properties).

    08 — How Injection Happens (Step by Step)

    Injection is three steps — write, propagate, read — and almost every "it's nil / it's the default / it didn't update" bug is a violation of one of injection's scope rules. This lesson makes the rules explicit.

    The write → propagate → read chain

    @main
struct MyApp: App {
    @State private var session = Session()          // 1. OWN the object (lesson 06)
    var body: some Scene {
        WindowGroup {
            RootView()
                .environment(session)                // 2. WRITE: add to env for RootView's subtree
        }
    }
}

struct DeepChild: View {
    @Environment(Session.self) private var session   // 3. READ: nearest ancestor's Session
    var body: some View { Text(session.user?.name ?? "—") }
}
    1. Write — .environment(value) (or .environment(\.key, value)) attaches the value to the subtree of the view it's applied to.
    2. Propagate — SwiftUI carries the (copy-modified) environment dictionary down the tree structure to every descendant (lesson 07).
    3. Read — @Environment(...) pulls the value during body evaluation, creating a dependency (lesson 01).

    Scope Rule 1 — injection flows to descendants only, never to self or up

    .environment(...) modifies the environment for the subtree of the view it's attached to — i.e. the modified view and its children as seen by SwiftUI's modifier model. The classic confusion:

    struct A: View {
    @Environment(\.theme) private var theme         // reads the env A RECEIVED from its parent
    var body: some View {
        Child()
            .environment(\.theme, .red)              // affects Child's subtree — NOT A's own `theme`
    }
}

    A's own theme property is whatever A's parent provided; the .environment(\.theme, .red) here changes what Child sees, not what A sees. You cannot read, in a view's own body, an environment value that the same body injects on a child. Injection goes down, reads come from above.

    If you need a value in A and below, inject it on A from A's parent.

    Scope Rule 2 — read the environment in body, never in init

    @Environment properties are populated by SwiftUI after the view is created, as part of placing it in the graph. During init, the wrapper has no value yet:

    struct Bad: View {
    @Environment(\.theme) private var theme
    init() {
        // theme is NOT available here — reading it gives the default/garbage
    }
    var body: some View { Text("ok").foregroundStyle(theme) } // ✅ available here
}

    Anything that depends on an environment value must happen in body (or a child that reads it), not in init. This is a frequent source of "it's always the default value."

    Scope Rule 3 — inject above the reader, in the same tree branch

    The injecting modifier must be on an ancestor of the reading view, within the same propagation branch. Three ways this fails:

    1. Injected below / as a sibling of the reader — reader is not in the injected subtree → default or crash.
    2. Reader is in a different branch — e.g. a .sheet/.toolbar/destination that SwiftUI roots elsewhere (lesson 09).
    3. Injected on the wrong view — e.g. on the child returned by body when you needed it on self (Rule 1).

    Scope Rule 4 — last writer down the path wins (shadowing)

    If multiple ancestors inject the same key/type, the nearest one to the reader wins (lesson 07). Deliberate for scoped overrides; accidental when a stray .environment mid-tree clobbers the app-wide value for a subtree.

    Propagation through modifiers & containers

    Environment flows through normal containers (VStack, List, ForEach, NavigationStack's inline content) because they're structural children. It also flows through most modifiers. It does not automatically flow across a boundary that SwiftUI hosts as a separate tree — modally presented content and some toolbar/scene contexts. That's lesson 09, and it's the #1 real-world injection failure.

    Putting the rules together (diagnostic)

    "My @Environment read is wrong/missing." Ask, in order: 1. Am I reading in body (not init)? (Rule 2) 2. Is the .environment(...) on a real ancestor of this view? (Rule 3) 3. Did I inject it on the child but try to read it on self? (Rule 1) 4. Is there a presentation boundary (sheet/cover/popover/destination/toolbar/hosting) between injector and reader? (lesson 09) 5. Did a nearer ancestor shadow it? (Rule 4)

    Exercises

    1. Reproduce Rule 1: read \.theme in A while injecting .red on A's child; print both — see A unaffected.
    2. Reproduce Rule 2: try to use @Environment(\.locale) in init (store it to a let); observe it's the default, then move the work to body.
    3. Reproduce Rule 3: inject a Session on a sibling of the reader and watch it fail; move the injection to a common ancestor and watch it work.

    Checkpoint

    • [ ] I can recite write → propagate → read.
    • [ ] Rule 1: injection affects descendants, not the injecting view's own read.
    • [ ] Rule 2: environment is available in body, not init.
    • [ ] Rule 3: inject on an ancestor in the same branch.
    • [ ] Rule 4: nearest injection shadows the rest.
    • [ ] I have the 5-question diagnostic memorized.

    09 — When Injection Is Rejected

    This is the lesson that pays for the series. "Injection rejected" shows up two ways: a hard crash ("No ObservableObject of type X found…") or a silent default (the value is just wrong). Both come from the same root cause — the reader is not in the injector's propagation subtree — usually at a boundary SwiftUI roots as a separate tree.

    The crash you'll actually see

    Fatal error: No ObservableObject of type Session found.
A View.environmentObject(_:) for Session may be missing as an ancestor of this view.

    (or, for @Environment(Session.self) with @Observable, a precondition trap with the same meaning.)

    It means: a view read an environment object (@Environment(Type.self) or legacy @EnvironmentObject) that no ancestor injected on this branch. Recall lesson 07: objects have no default, so "missing" = crash. (Environment values with a defaultValue never crash — they silently return the default, which is the other failure mode.)

    Root cause #1 — Presentation boundaries (the big one)

    Modally presented content is inserted parallel to your view, not as a child of it. SwiftUI roots the sheet/cover/popover in a separate branch, so your environment does not propagate into it. This is by design, not a bug.

    Affected: .sheet, .fullScreenCover, .popover, .alert/.confirmationDialog content, and (historically/inconsistently) some .toolbar and context-menu content.

    struct Home: View {
    @State private var session = Session()
    @State private var showSettings = false
    var body: some View {
        Button("Settings") { showSettings = true }
            .environment(session)                 // ✅ propagates to Home's subtree
            .sheet(isPresented: $showSettings) {
                SettingsView()                    // ❌ NOT a child of Home — no Session here → CRASH
            }
    }
}

    Fix — re-inject at the boundary:

    .sheet(isPresented: $showSettings) {
    SettingsView()
        .environment(session)                     // ✅ re-inject into the sheet's tree
}

    Rule of thumb: every presentation boundary is a fresh environment root. Re-inject what the presented content needs. Many teams centralize this (a helper that injects the standard set of app objects into every presented destination).

    Nuance by type: environment values (\.colorScheme, custom keys) often do get carried into sheets by the system in modern iOS, but environment objects are the ones that reliably need re-injection. Don't rely on the difference — re-inject objects at every boundary and you're always correct.

    Root cause #2 — NavigationStack destinations

    navigationDestination(for:) builds destinations lazily, and depending on where you attach things, the destination may not sit under your injector. Inject the destination's dependencies on the destination view itself (or attach .environment to the NavigationStack's content so it covers destinations), and verify — destinations are a common "default value, no crash" surprise for environment values, and a crash for objects.

    Root cause #3 — UIKit / UIHostingController boundaries

    When you bridge to UIKit (lesson links below), a new UIHostingController starts a fresh SwiftUI environment — your outer SwiftUI environment doesn't cross the UIKit gap. You must set the environment on the hosted root explicitly:

    let root = SettingsView().environment(session)    // inject before hosting
let host = UIHostingController(rootView: root)

    Same for any SwiftUI view you instantiate to embed in UIKit. (See Learning-SwiftUI/24-uikit-integration.)

    Root cause #4 — Timing: background launch / body before injection

    If a view's body runs before the environment is set — classically when the app is launched into the background and the scene renders before your injection point initializes — an object read can crash even though the code "looks" injected.

    Fixes: - Inject as high and as early as possible (on the WindowGroup's root content). - For resilience, prefer an environment value with a safe default over a no-default object, or make the @Environment read optional and handle nil: swift // value-based, optional, never crashes: @Environment(\.session) private var session: Session? // custom key, default nil - Don't do environment-dependent work in init (lesson 08, Rule 2).

    Root cause #5 — The scope-rule violations (no boundary involved)

    From lesson 08, even with no boundary you get rejected/wrong values when: - You inject below or beside the reader (reader not in the subtree). - You read in init (value not populated yet). - You inject on the child but read on self. - A nearer ancestor shadowed the value with a different one.

    These usually present as the silent default (for values) rather than a crash.

    Crash vs. silent — quick table

    You read… Missing/wrong injection result
    @Environment(Type.self) (object, @Observable) Crash (no default)
    legacy @EnvironmentObject var x: T Crash (no default)
    @Environment(\.customKey) (value, has default) Silent default
    @Environment(\.systemKey) (colorScheme, etc.) System-provided default

    The crash is friendlier — it tells you immediately. The silent default is the one that wastes an afternoon; lesson 12 covers catching it.

    Defensive patterns (what good codebases do)

    1. Inject app-wide objects once, at the top, on WindowGroup's content.
    2. Re-inject at every presentation/hosting boundary — centralize via a custom modifier: swift extension View { func injectAppEnvironment(_ s: Session, _ t: Theme) -> some View { self.environment(s).environment(t) } } // use on every sheet/cover/destination/hosted root
    3. Prefer value+default (or optional) over no-default objects for anything that might be read during early/background launch.
    4. Never read environment in init.
    5. Treat each boundary as a new environment root — make it a review checklist item.

    Exercises

    1. Reproduce the crash: inject a Session object, present a .sheet reading it without re-injecting → crash. Read the message. Then fix by re-injecting.
    2. Reproduce the silent default: custom value key with default .blue, read inside a sheet without re-injecting → confirm you get .blue (no crash) while the presenter shows .purple.
    3. Bridge a SwiftUI view through UIHostingController without setting the environment → crash; fix by injecting before hosting.
    4. Simulate the timing bug: do environment-dependent work in init and watch it use the default; move it to body.
    5. Write the injectAppEnvironment modifier and apply it to one sheet and one navigation destination.

    • [ ] I can read the "No ObservableObject of type X found" crash as "object not injected on this branch."
    • [ ] I know modals are parallel roots — environment doesn't propagate into sheets/covers/popovers; re-inject.
    • [ ] I know NavigationStack destinations and UIHostingController are boundaries too.
    • [ ] I understand the background-launch timing crash and its fixes (inject high/early, optional/default).
    • [ ] I can predict crash vs. silent default from object-vs-value.
    • [ ] I have the defensive patterns (re-inject at every boundary, value+default/optional, never in init).

    10 — The Full Chain

    Now assemble everything into one trace: an @Observable model owned at the root, injected into the environment, read deep in the tree, bound for editing, mutated, and the resulting precise invalidation. For each link, note exactly where it can break (with the lesson that explains it).

    The complete example

    @Observable
final class Session {                         // ← Observation (lesson 05)
    var displayName = ""
    var unreadCount = 0
}

@main
struct ChatApp: App {
    @State private var session = Session()    // LINK 1 — OWN (lesson 06): lifetime = app

    var body: some Scene {
        WindowGroup {
            RootView()
                .environment(session)          // LINK 2 — INJECT (lesson 07/08)
        }
    }
}

struct RootView: View {
    var body: some View { InboxView() }        // propagation flows down (lesson 07)
}

struct InboxView: View {
    @Environment(Session.self) private var session   // LINK 3 — READ (lesson 08)
    var body: some View {
        VStack {
            Text("Unread: \(session.unreadCount)")    // edge → session.unreadCount (lesson 01/05)
            NameEditor()
            Button("Mark read") { session.unreadCount = 0 }   // LINK 5 — MUTATE
                .sheet(isPresented: .constant(false)) {
                    NameEditor().environment(session)          // LINK 6 — RE-INJECT at boundary (lesson 09)
                }
        }
    }
}

struct NameEditor: View {
    @Environment(Session.self) private var session
    var body: some View {
        @Bindable var session = session                    // LINK 4 — BIND (lesson 06)
        TextField("Name", text: $session.displayName)      // two-way write-path (lesson 04)
    }
}

    Trace a mutation end to end

    User types in the TextField: 1. The Binding's set fires → writes session.displayName (lesson 04). 2. The setter calls withMutation(of:keyPath: \.displayName) (lesson 05). 3. The ObservationRegistrar notifies observers tracking displayName. Only views whose body read displayName are marked dirty (lesson 01). 4. NameEditor.body re-runs (it reads displayName). InboxView's Text("Unread:") does not — it only read unreadCount (precision, lesson 05). 5. SwiftUI diffs and updates just the changed Text in the field.

    User taps "Mark read": 1. session.unreadCount = 0 → withMutation(\.unreadCount). 2. The Text("Unread: …") (read unreadCount) re-runs; NameEditor (read only displayName) does not.

    This is the whole system working together: ownership keeps the object alive, environment delivers it without prop-drilling, @Bindable provides write-paths, Observation routes each mutation to exactly the right body.

    Where each link breaks

    Link Mechanism Breaks when… Lesson
    1 OWN @State holds object, keyed to identity Created in a body instead → new object each render; or .id() resets it 02,03,06
    2 INJECT .environment(obj) writes to subtree Injected below/beside reader, or on wrong view 08
    3 READ @Environment(Type.self) pulls nearest Read in init; nearest ancestor shadows; missing → crash 08,09
    4 BIND @Bindable → $ get/set Bound to a non-@Observable; stale custom binding 04,06
    5 MUTATE setter → withMutation Mutated off main actor; mutated a copy not the shared ref 05,06
    6 BOUNDARY re-inject into modal/host tree Forgot to re-inject into sheet/cover/destination/UIHostingController → crash/default 09
    — INVALIDATE read = edge Expected update but never read the value in body (no edge) 01,12

    The one-paragraph summary of the whole series

    SwiftUI is a dependency graph. Durable state lives in the graph keyed to identity, not in the disposable view struct. @State owns it; Binding is a get/set wire to it; @Observable makes class-property reads/writes trackable so only the bodys that read a property re-run; the environment carries shared objects down the tree by type. Injection is write→propagate→read, scoped to descendants and available only in body — and it gets rejected at boundaries (modals, navigation destinations, UIKit hosts) and on scope-rule violations, crashing for no-default objects and silently defaulting for values.

    1. Build the example. Add Self._printChanges() to InboxView and NameEditor. Edit the name and tap "Mark read"; confirm only the relevant body re-runs each time.
    2. Break link 1: create Session() inside InboxView's body and watch edits not persist.
    3. Break link 6: remove the .environment(session) from the sheet and trigger the crash.
    4. Break the invalidate edge: stop reading unreadCount in body (store it in a let in init) and watch the count stop updating.

    11 — Invalidation and Performance

    Now that you know how updates propagate, you can control how often. Performance work in SwiftUI is almost entirely about not creating dependencies you don't need and not destroying identity you wanted to keep.

    The cost model

    • body re-evaluation is cheap-ish but not free — it allocates value-type view trees and diffs them.
    • The expensive failures are: (a) a huge subtree re-evaluating on every keystroke, (b) work done inside body that should be cached, (c) lost identity forcing teardown/rebuild (lesson 02), (d) AnyView defeating the diffing.
    • Goal: each state change re-evaluates the smallest set of bodies, and each body does the least work.

    Lever 1 — Read narrowly (fewer edges)

    The edge is the read (lesson 01/05). So:

    • Read only what you display. Don't read model.everything in a parent if only a leaf needs one field — pass the object down and let the leaf read its field.
    • Split big views. A monolithic body that reads 10 properties re-runs on any of the 10. Extract subviews so each reads only its slice. This is the highest-leverage fix for "whole screen re-renders."
    // BAD: one body depends on title, body, isSaving, wordCount...
// GOOD:
HeaderView(model: model)     // reads title only
StatusView(model: model)     // reads isSaving only
EditorBody(model: model)     // reads body only
    • @ObservationIgnored for model fields that never drive UI (caches, tokens) so reads don't subscribe (lesson 05).

    Lever 2 — Preserve identity (avoid teardown)

    • Avoid AnyView in hot paths — it erases structural identity and tanks diffing (lesson 02). Prefer @ViewBuilder, Group, or generics.
    • Stable ForEach ids — never the array index when the array can reorder/insert; use a stable model id. Wrong ids cause mass rebuilds and state attaching to the wrong rows.
    • Don't churn .id(...) — changing it destroys and rebuilds the subtree every time.

    Lever 3 — Do less inside body

    body can run frequently, so keep it pure and cheap:

    • No expensive computation in body. Sorting/formatting on every run is a hidden cost. Cache in the model (a computed @Observable property is fine — it's only recomputed when read and its inputs change) or precompute.
    • No allocations of heavy objects (formatters, etc.) in body. Hoist DateFormatter/NumberFormatter to static or model level.
    • No side effects in body — never mutate state during body; it causes "Modifying state during view update" and extra invalidation loops.

    Lever 4 — Equatable views to short-circuit diffing

    For an expensive subtree that depends only on a few inputs, conform it to EquatableView (via .equatable() or Equatable) so SwiftUI can skip re-evaluating when inputs are unchanged:

    struct Row: View, Equatable {
    let item: Item
    static func == (l: Row, r: Row) -> Bool { l.item.id == r.item.id && l.item.version == r.item.version }
    var body: some View { /* expensive */ }
}
// use: Row(item: item).equatable()

    Use sparingly — a wrong == causes stale UI (the opposite bug). Measure first.

    Lever 5 — Scope animations & transactions

    withAnimation over a state mutation that touches a big @Observable graph can animate more than intended and amplify invalidation. Scope the state and the animation to the smallest unit; use .transaction to contain it. (Ties into the animation-decision framework — see references/files-app-document-scan-interaction for layer choices.)

    Common over-invalidation bugs (and fixes)

    Symptom Cause Fix
    Whole screen re-renders on one keystroke One big body reads many props Split into subviews that read their slice
    Every row rebuilds on list change Index used as ForEach id, or AnyView Stable ids; remove AnyView
    State resets randomly Identity churn (.id(), branch swap) Stabilize identity (lesson 02)
    Janky scroll Heavy work / allocations in cell body Cache/precompute; hoist formatters
    Update storm / "modifying state during update" Side effect in body Move to .task/action/onChange

    Measure, don't guess

    • let _ = Self._printChanges() — see why a body re-ran (which dependency or that the identity changed). Lesson 12.
    • Instruments → SwiftUI template — view-body durations, update counts, long updates.
    • Instruments → Hitches/Animation — frame drops correlated to update storms.
    • Profile on a real device — simulator timings mislead.

    Exercises

    1. Take a screen that re-renders fully on one field change; split it so only one subview re-runs. Prove it with _printChanges().
    2. Replace an AnyView-based list cell with a generic/@ViewBuilder one; compare update counts in Instruments.
    3. Add a wrong == to an Equatable view and watch stale UI; fix it.
    4. Move a DateFormatter out of a cell body and measure the scroll improvement.

    12 — Debugging Data Flow

    A field guide for the three questions you'll actually ask in the codebase: "Why did this re-render?", "Why is this NOT updating?", and "Why is the environment missing?" Plus exercises to map this onto your own app.

    Tool 1 — Self._printChanges()

    Drop this as the first line of a body:

    var body: some View {
    let _ = Self._printChanges()
    // ...
}

    It logs why the body re-ran: - @self — the view value itself changed (a stored let/property differs). - @identity — the view got a new identity (state will have reset! lesson 02). - a property name (e.g. _session) — that dependency changed. - @dependencies — an environment/observed dependency changed.

    This single tool answers "why did this re-render?" and, by its absence, "why isn't this updating?" (if it never prints, no edge exists). Private API — debug builds only; strip before shipping.

    Tool 2 — The "not updating" decision tree

    View isn't updating when data changes
├─ Does body even re-run? (add _printChanges)
│   ├─ No  → NO DEPENDENCY EDGE. You're not reading the value in body.
│   │        • Reading it in init / a stored let? (lesson 03/08) → read in body
│   │        • Reading a plain (non-@Observable) class property? (lesson 03/05) → make it @Observable
│   │        • Mutating a COPY, not the shared reference? (lesson 06)
│   │        • Property is @ObservationIgnored? (lesson 05)
│   └─ Yes → body runs but UI looks stale
│            • Wrong Equatable/== short-circuiting it? (lesson 11)
│            • Binding reading a stale snapshot? (lesson 04)
│            • Mutation off the main actor? (lesson 05/06)

    Tool 3 — The "environment missing/wrong" decision tree

    @Environment value is nil / default / crashing
├─ Crash "No ObservableObject of type X found"
│   → object not injected ON THIS BRANCH (lesson 09)
│     • Is there a sheet/cover/popover/destination/UIHostingController between injector and reader?
│       → RE-INJECT at that boundary (the #1 cause)
│     • Injected on a sibling/below the reader? → move to a common ancestor
│     • Body running before injection (background launch)? → inject higher/earlier, or use optional/default
├─ Silent default value (no crash)
│   → it's an environment VALUE (has defaultValue)
│     • Same boundary/scope checks as above (lesson 08 Rule 1–4)
│     • Reading in init? → move to body
│     • A nearer ancestor shadowed it? (lesson 07/08 Rule 4)
└─ Value updates late / not at all
    → are you reading it in body (edge) vs capturing once? (lesson 01)

    Tool 4 — Reproduce the rejections on purpose

    You debug faster when you've seen each failure. Build a scratch project and trigger: 1. Object-missing crash in a .sheet (then fix by re-inject). (lesson 09) 2. Silent default for a value key in a sheet. 3. init read returning the default. 4. Identity churn resetting @State (via .id() / branch swap). (lesson 02) 5. Off-main mutation failing to update.

    Keep this scratch project — it's your reference for "which symptom is which."

    Tool 5 — Instruments & view debugging

    • Instruments → SwiftUI template: body evaluation counts/durations, "long view body" call-outs. Best for "what's re-rendering too much."
    • Instruments → Hitches / Animation: ties update storms to dropped frames.
    • Debug → View Hierarchy (Xcode): inspect the rendered tree (note: rendered ≠ your body structure — modifiers/containers expand).
    • Real device, release-ish config, for timing.

    Tool 6 — Read the generated code

    • Expand Macro on @Observable (lesson 05) to see access/withMutation.
    • When confused about identity, simplify the body until _printChanges stops saying @identity — the last change you made was the culprit.

    Mapping exercises (do these against YOUR codebase)

    1. Find the root injections. Grep for .environment( and .environmentObject( at/near @main/WindowGroup. List every app-wide object injected. → That's your DI surface.
    2. Find every boundary. Grep for .sheet, .fullScreenCover, .popover, navigationDestination, UIHostingController. For each, verify the presented content re-injects what it reads. Flag any that rely on inheritance. (This is where lesson-09 bugs hide.)
    3. Find the over-readers. Open your 3 biggest bodys; count distinct model properties read. Pick one and split it so each subview reads its slice; confirm with _printChanges.
    4. Find legacy mixing. Grep for ObservableObject, @Published, @StateObject, @EnvironmentObject. Note where legacy and @Observable coexist — those seams are common bug sites.
    5. Audit ownership. Grep for object construction inside body (SomeModel( not behind @State). Each is a potential "new object every render" bug (lesson 06).
    6. Write the standard injector. Create injectAppEnvironment(...) (lesson 09) and apply it to every boundary found in #2.
