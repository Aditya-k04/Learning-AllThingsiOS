WidgetKit & Live Activities
Timelines, widget families, Live Activities and the Dynamic Island — high-craft, product-facing surfaces.

    WidgetKit & Live Activities — Overview

    Widgets, Live Activities, and the Dynamic Island are some of the highest-craft surfaces on iOS — tiny, glanceable, and unforgiving of sloppiness. They share one runtime model: your code runs briefly, produces SwiftUI views and a schedule, and the system renders them later, without your app running.

    The fundamental constraint

    A widget is not a live app. You don't get a render loop. You hand the system a timeline of pre-rendered states with dates, and it shows each one at the right time. No timers, no networking at display time — everything is computed in advance.

    Internalizing this "archived snapshots, not a running app" model explains every WidgetKit rule.

    What you'll learn

    1. 01-timeline-provider — the TimelineProvider, snapshots, entries, and refresh policies.
    2. 02-widget-views-families — building the SwiftUI view, supported families, and the containerBackground rules.
    3. 03-live-activities — ActivityKit: live, updating content on the Lock Screen.
    4. 04-dynamic-island — the compact, minimal, and expanded presentations.
    5. 05-interactivity-and-app-intents — buttons and toggles in widgets via App Intents, and deep links.

    Where it runs

    All of this lives in a widget extension — a separate target/process from your app, with its own memory budget. Sharing data with the main app means an App Group container (shared UserDefaults or a shared SwiftData/file store), covered alongside persistence.

    Why it's worth your time

    These surfaces are pure SwiftUI (SwiftUI) under tight constraints, they're extremely product-facing, and the Dynamic Island in particular is a showcase of the fluid, physical motion you care about. They reward craft more than almost anything else on the platform.

    The Timeline Provider

    The TimelineProvider is the engine of a widget. It answers one question: "what should this widget show, and when should it change?" You return a list of timed entries; the system caches them and renders each at its date — no further code of yours runs in between.

    Three methods

    struct Provider: TimelineProvider {
    // 1. A static placeholder for the gallery / redacted state. Instant, no data.
    func placeholder(in: Context) -> Entry {
        Entry(date: .now, value: "—")
    }
    // 2. A single snapshot for the widget gallery / transient previews.
    func getSnapshot(in: Context, completion: @escaping (Entry) -> Void) {
        completion(Entry(date: .now, value: latestValue()))
    }
    // 3. The real schedule: multiple future entries + a refresh policy.
    func getTimeline(in: Context, completion: @escaping (Timeline<Entry>) -> Void) {
        var entries: [Entry] = []
        let now = Date()
        for hour in 0..<5 {                       // pre-render the next 5 hours
            let date = Calendar.current.date(byAdding: .hour, value: hour, to: now)!
            entries.append(Entry(date: date, value: forecast(at: date)))
        }
        completion(Timeline(entries: entries, policy: .atEnd))
    }
}

    Each Entry is a TimelineEntry with a date (when it becomes visible) plus your data.

    Refresh policies

    The policy tells the system when to ask for a fresh timeline:

    • .atEnd — request a new timeline after the last entry's date.
    • .after(date) — refresh at a specific time.
    • .never — only when the app explicitly reloads it.

    You don't control timing precisely

    The system budgets widget refreshes (battery, frequency caps). A timeline asking for updates every minute will be throttled. Design around it: pre-compute several entries so the widget appears to update frequently from a single refresh, and push urgent changes from the app with:

    WidgetCenter.shared.reloadTimelines(ofKind: "ForecastWidget")

    Getting data in

    Because the provider runs in the extension, it reads from the shared App Group store the main app writes to — not from your app's memory. Fetch/network in the app, persist to the shared container, and let the provider read it synchronously. Heavy work in getTimeline risks the extension being killed for exceeding its memory budget.

    Configurable widgets

    User-customizable widgets (choose a city, pick an account) use AppIntentTimelineProvider with an AppIntent configuration — the same App Intents system from 05-interactivity-and-app-intents and App Intents & Siri.

    Widget Views & Families

    A widget's appearance is a SwiftUI view fed one timeline entry. The discipline is making one view adapt across several families (sizes), and following the platform rules that keep widgets crisp and legible.

    The widget definition

    @main
struct ForecastWidget: Widget {
    var body: some WidgetConfiguration {
        StaticConfiguration(kind: "ForecastWidget", provider: Provider()) { entry in
            ForecastView(entry: entry)
                .containerBackground(.fog.gradient, for: .widget)  // required iOS 17+
        }
        .configurationDisplayName("Forecast")
        .description("Your local forecast at a glance.")
        .supportedFamilies([.systemSmall, .systemMedium, .systemLarge,
                            .accessoryCircular, .accessoryRectangular, .accessoryInline])
    }
}

    Adapt to the family

    The same view renders at very different sizes. Read the environment and branch:

    struct ForecastView: View {
    @Environment(\.widgetFamily) private var family
    let entry: Entry
    var body: some View {
        switch family {
        case .systemSmall:        CompactForecast(entry)
        case .accessoryCircular:  Gauge(value: entry.temp) { Text("°") }
        default:                  DetailedForecast(entry)
        }
    }
}

    accessory* families are the Lock Screen / watch complications — monochrome, vibrancy-tinted; design them as glyphs and gauges, not full layouts.

    The rules that trip people up

    • containerBackground(for: .widget) is required (iOS 17+) — it's how the system insets and renders your background in StandBy, the gallery, and on the Home Screen. Without it the widget looks broken.
    • No scrolling, no video, no interactive UIKit. Widgets are static images plus (since iOS 17) Button/Toggle backed by App Intents — nothing else is interactive.
    • Use system fonts and .widgetAccentable() so tinted Home Screen modes recolor your widget correctly.
    • Keep the view pure — it's rendered from an archived entry; it can't fetch or run timers.

    Animation

    Widgets get limited, system-driven transitions between timeline entries (iOS 17+): SwiftUI will animate numeric text and view changes between entries automatically, and you can hint with .contentTransition. You don't drive these frame-by-frame — you describe two states and the system tweens, the same declarative model as SwiftUI animation.

    🆕 Liquid Glass rendering (iOS 26+)

    On iOS 26 and later, widgets adopt the Liquid Glass material and the system's accented/tinted rendering modes automatically. Design for it: read \.widgetRenderingMode and use .widgetAccentable() so your content recolors correctly in tinted and accented Home Screen modes, and keep contrast legible against the translucent material (the same "respect the substrate" point as Liquid Glass & Siri). The required containerBackground(for: .widget) is what lets the system apply the material correctly.

    Live Activities

    A Live Activity is a glanceable, updating view on the Lock Screen and in the Dynamic Island — a ride approaching, a game score, a timer, a delivery. It's WidgetKit's views plus a live update channel (ActivityKit), bounded by strict time and size limits.

    The shape

    You define the activity's data in two parts: attributes (fixed for the activity's life) and content state (the part that updates):

    import ActivityKit

struct DeliveryAttributes: ActivityAttributes {
    struct ContentState: Codable, Hashable {   // the changing part
        var status: String
        var etaMinutes: Int
    }
    let orderNumber: String                     // fixed for the whole activity
}

    Lifecycle: start, update, end

    // Start (from the foreground app)
let activity = try Activity.request(
    attributes: DeliveryAttributes(orderNumber: "A-1024"),
    content: .init(state: .init(status: "Preparing", etaMinutes: 30), staleDate: nil))

// Update — locally, or remotely via a push to activity.pushToken
await activity.update(.init(state: .init(status: "On the way", etaMinutes: 12), staleDate: nil))

// End
await activity.end(.init(state: .init(status: "Delivered", etaMinutes: 0), staleDate: nil),
                   dismissalPolicy: .after(.now + 60))

    Updating without your app running

    The power move is remote updates via push: each activity has a pushToken; your server sends a special liveactivity push with the new ContentState, and the system re-renders the views — your app never wakes. This is how a delivery or sports app keeps the Lock Screen current. Local updates work too but only while your app has execution time.

    The views

    The Live Activity declares a SwiftUI layout for the Lock Screen banner and for the Dynamic Island presentations (next lesson). They're the same view-building skills as 02-widget-views-families, under even tighter size limits.

    struct DeliveryLiveActivity: Widget {
    var body: some WidgetConfiguration {
        ActivityConfiguration(for: DeliveryAttributes.self) { context in
            // Lock Screen / banner UI
            LockScreenView(context: context)
        } dynamicIsland: { context in
            // compact / minimal / expanded — see next lesson
        }
    }
}

    Constraints to design around

    Live Activities have a maximum lifetime (hours, then the system ends them), a small memory budget, and update throttling. Keep ContentState tiny, set sensible staleDates so the UI can show "outdated" gracefully, and always provide an .end path.

    The Dynamic Island

    The Dynamic Island is the live, morphing pill around the front camera. It's the presentation layer of a Live Activity, and it's a masterclass in fluid motion — shapes that grow, merge, and split with physical springs. You author three discrete states; the system handles the morphing between them.

    The three presentations

    dynamicIsland: { context in
    DynamicIsland {
        // EXPANDED — when long-pressed or for important updates. Four regions:
        DynamicIslandExpandedRegion(.leading)  { Label(context) }
        DynamicIslandExpandedRegion(.trailing) { ETA(context) }
        DynamicIslandExpandedRegion(.center)   { Title(context) }
        DynamicIslandExpandedRegion(.bottom)   { ProgressBar(context) }
    } compactLeading: {
        Image(systemName: "bicycle")           // COMPACT — left of the camera
    } compactTrailing: {
        Text("\(context.state.etaMinutes)m")   // COMPACT — right of the camera
    } minimal: {
        Image(systemName: "bicycle")           // MINIMAL — when another activity shares the island
    }
}
    • Compact — the default: small leading + trailing content hugging the camera.
    • Minimal — a single tiny glyph, used when multiple activities compete for the island.
    • Expanded — a rich layout shown on long-press or for momentary alerts.

    You describe states; the system animates

    You never animate the pill morphing. You declare what each state looks like and the system tweens between them with its signature springs — the same "two states, let the system interpolate" model as SwiftUI animation (Animation Basics) and the WWDC spring talk (Animate with Springs). Your job is making each static state beautiful and legible.

    Design constraints

    • The hardware camera cutout is sacred — content flows around it; never assume a rectangle.
    • Compact content is tiny: a glyph and a few characters. Reserve detail for expanded.
    • Use .foregroundStyle, vibrancy, and SF Symbols; the island is rendered with system materials.
    • Keep the leading/trailing compact items visually balanced — they read as one unit split by the camera.

    Tap targets and deep links

    Tapping the island/Live Activity launches your app. Route it with .widgetURL(URL) on the Lock Screen view, or Link destinations in expanded regions, so different parts deep-link to different screens — the same deep-link handling as a widget tap (05-interactivity-and-app-intents).

    Why study it

    The Dynamic Island is the platform's clearest demonstration that premium motion = declarative states + great springs + respect for physical constraints — exactly the thesis running through your interaction-math and Core Animation work.

    Interactivity & App Intents

    Before iOS 17 a widget could only deep-link into the app. Now a widget or Live Activity can contain Buttons and Toggles that run code in place — check off a reminder, toggle a light — without launching the app. The mechanism is App Intents.

    Buttons backed by intents

    A widget button doesn't take a closure (there's no running app to run it). It takes an AppIntent — a serializable description of an action the system performs on your behalf:

    import AppIntents

struct ToggleTaskIntent: AppIntent {
    static var title: LocalizedStringResource = "Toggle Task"
    @Parameter var taskID: String
    init() {} 
    init(taskID: String) { self.taskID = taskID }

    func perform() async throws -> some IntentResult {
        TaskStore.shared.toggle(taskID)       // mutate shared (App Group) store
        return .result()
    }
}

// In the widget view (iOS 17+):
Button(intent: ToggleTaskIntent(taskID: task.id)) {
    Image(systemName: task.done ? "checkmark.circle.fill" : "circle")
}

    When tapped, the system runs perform() in your extension, you mutate the shared store, and WidgetKit reloads the timeline to reflect the change — all without opening the app.

    Why App Intents, not closures

    The widget UI is an archived snapshot; there's no live code to call. An AppIntent is a value the system can serialize, schedule, and execute in a fresh process. That same property is why App Intents also power Siri, Shortcuts, and Spotlight — one definition, many entry points (covered in App Intents & Siri).

    Deep links for everything else

    For actions that genuinely need the app (open a detail screen), use a URL:

    // widget view
.widgetURL(URL(string: "myapp://task/\(task.id)"))
// or a Link on a sub-region
Link(destination: URL(string: "myapp://task/\(task.id)")!) { RowView(task) }

    Handle it in the app with .onOpenURL { url in router.handle(url) }.

    Keep perform() fast and shared-state based

    perform() runs with a tight budget. Do the minimal mutation against the App Group store; don't network or do heavy work there. If a button needs network, mutate optimistic state, reload the timeline, and let the app reconcile later.

    🆕 Control widgets (iOS 18+)

    Beyond Home/Lock-Screen widgets, WidgetKit now builds Controls — the toggles and buttons that live in Control Center, the Lock Screen, and on the Action button. They're a ControlWidget, and like interactive widgets they're driven entirely by App Intents:

    struct FlashlightControl: ControlWidget {
    var body: some ControlWidgetConfiguration {
        StaticControlConfiguration(kind: "com.you.flashlight") {
            ControlWidgetToggle("Flashlight", isOn: FlashlightState.isOn,
                                action: ToggleFlashlightIntent()) { isOn in
                Image(systemName: isOn ? "flashlight.on.fill" : "flashlight.off.fill")
            }
        }
    }
}
    • ControlWidgetButton — a one-shot action (run an AppIntent).
    • ControlWidgetToggle — a binary state bound to an AppIntent.

    This is the same "one App Intent, many surfaces" idea: the intent you write for a widget button can also power a Control and a Siri phrase. See App Intents & Siri.
