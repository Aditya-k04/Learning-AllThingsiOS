Stack Decisions
A framework for choosing the right rendering tool for any interaction.

    Choosing the Right Rendering Stack — Learning Roadmap

    This curriculum teaches you to decide which technology stack to use for any interaction or animation — before you write a line of code. The goal: when you imagine an interaction, you already see the stack in your head ("this is SwiftUI for layout + a Metal .layerEffect for the reveal", or "this is explicit Core Animation because I need staggered per-layer timing").

    This is the meta-skill that sits above the other Learning folders. Once you have it, Core Animation, Shaders, and SwiftUI stop being separate worlds and become rungs on one ladder you climb up and down deliberately.

    Why this matters

    Most people pick a stack by habit ("I always use SwiftUI") or by cargo-culting a tutorial. Then they hit a wall — the animation stutters, the gesture can't interrupt, the effect can't be expressed — and they bolt on hacks.

    The fix is to understand that all rendering on Apple platforms is one layered system that bottoms out at the GPU compositor. Every animation API is a different altitude on that ladder. Master the map and three questions, and stack selection becomes mechanical.

    See core-animation-smooth-60-120fps for the render-pipeline deep dive this curriculum builds on.

    The one rule

    Start at the highest rung that can express the effect. Drop down only when you hit a specific, nameable limitation.

    Each rung you descend costs you code and gains you control. If you can't name the limitation forcing you lower, you're over-engineering. This single principle is the spine of the whole curriculum — see 07-escalation-ladder.

    Prerequisites

    • Helpful: Basic SwiftUI and UIKit familiarity. You should know what a view, a layer, and a gesture are.
    • Helpful: Skim how rendering works so the GPU-compositor floor of the ladder makes sense.
    • Not required: Metal/shader knowledge. You'll learn when to reach for it, not how to write it (that's Learning-Shaders).

    01 — The Rendering Stack Map

    Everything visual on iOS/macOS/tvOS/watchOS becomes CALayers composited by the GPU in a separate process (the render server / backboardd). Every animation API you know is a different altitude on one ladder that bottoms out there. Learn the ladder and stack choice stops being mysterious.

    Foundation: 01-how-rendering-works explains the app → render-server → GPU → display pipeline. This lesson assumes that floor and builds the ladder on top of it.

    The full ladder, high to low

    ┌──────────────────────────────────────────────────────────────┐
│ ALTITUDE        TECHNOLOGY                  YOU SPECIFY...      │
├──────────────────────────────────────────────────────────────┤
│ Highest    SwiftUI .animation/withAnimation  the END STATE     │
│            SwiftUI PhaseAnimator/Keyframe     a sequence        │
│            SwiftUI transitions                in/out behavior   │
│ ───────────────────────────────────────────────────────────── │
│            UIKit UIView.animate               end state + curve │
│            UIViewPropertyAnimator             + interruptibility│
│ ───────────────────────────────────────────────────────────── │
│            Core Animation (CAAnimation)       per-property keys │
│            CALayer subclasses (Shape/Emitter) + layer geometry  │
│ ───────────────────────────────────────────────────────────── │
│            CADisplayLink custom engine        EVERY FRAME       │
│            Spring physics (Wave/UIDynamics)   forces & velocity │
│ ───────────────────────────────────────────────────────────── │
│            SwiftUI Canvas / TimelineView      CPU draw commands │
│            Core Graphics (Quartz 2D)          CPU pixels        │
│ ───────────────────────────────────────────────────────────── │
│            Metal shader effects (SwiftUI)     per-pixel (easy)  │
│            MTKView / CAMetalLayer (raw Metal) EVERY PIXEL       │
│ ───────────────────────────────────────────────────────────── │
│ Lowest     GPU compositor (always running)    nothing — it just│
│                                               composites layers │
└──────────────────────────────────────────────────────────────┘

    The higher you are, the less you say and the more the system does. The lower you go, the more control and more code.

    Rung-by-rung: what each is for

    SwiftUI declarative (.animation, withAnimation, transitions)

    You change @State; the system interpolates from old to new. You never think about frames. Default rung for 90% of app UI. → 12-animation-basics

    SwiftUI PhaseAnimator / KeyframeAnimator

    Canned multi-step sequences without a manual state machine. A "like" button that scales up, overshoots, settles. Still declarative — you list phases, system drives them.

    UIKit UIView.animate

    The UIKit equivalent of declarative SwiftUI. Creates CABasicAnimations under the hood. Reach here only in UIKit codebases or when you need a UIKit-only API.

    UIViewPropertyAnimator

    Adds interruptibility and scrubbing. You can pause, reverse, and set fractionComplete from a gesture. The rung for grabbable, reversible transitions. → 05-how-the-user-touches-it

    Core Animation (explicit)

    CABasicAnimation, CAKeyframeAnimation, CAAnimationGroup on CALayers. You get per-property, per-layer timing control — beginTime offsets, staggered groups, exact timing functions per layer. Plus specialized layers: CAShapeLayer (vector paths), CAEmitterLayer (particles), CAGradientLayer (gradients), CAReplicatorLayer (cloned layers). → 09-cabasicanimation

    CADisplayLink custom engine

    A callback fired once per display refresh. You compute the value of every frame yourself. Needed when each frame depends on live input the system can't predict: physics, audio levels, sensor data, scroll offset, a shader uniform. Wave is built on this. → core-animation-smooth-60-120fps

    Spring physics (Wave, UIDynamics)

    A specialization of the frame-driven rung: a physics simulation drives the value. Best for gesture-driven, interruptible motion with natural velocity transfer. → damped-harmonic-oscillator

    SwiftUI Canvas + TimelineView / Core Graphics

    CPU immediate-mode 2D drawing. Flexible, but every pixel is drawn on the CPU on (or off) the main thread. Fine for modest element counts; falls over with thousands of elements or heavy per-pixel math.

    Metal shader effects (SwiftUI .colorEffect / .distortionEffect / .layerEffect)

    GPU per-pixel effects applied to existing SwiftUI views. iOS 17+. The easy entry to shaders — dissolve, shimmer, ripple, warp — without setting up a render pipeline. → 06-fragment-shaders

    Raw Metal (MTKView, CAMetalLayer)

    Full GPU control: vertex + fragment + compute, multi-pass, your own pipeline. Generative backgrounds, fluid sims, large particle systems. The bottom rung you'd ever hand-write. → 24-metal-and-calayer

    GPU compositor

    Not something you call — it's the floor. It takes the layer tree and composites it every refresh. Transforms, opacity, and masks on layers are essentially free here because the compositor does them in hardware without redrawing content. This is why staying high is cheap. → 06-cost-model-cpu-gpu

    The two clocks

    There are only two ways time advances in this whole stack:

    1. The system's interpolation clock — declarative rungs. You hand over A, B, and a curve; the system ticks it. You don't see frames.
    2. The manual clock — CADisplayLink / render loop. You get a tick per refresh and compute the frame yourself.

    Spring physics and Metar render loops both live on the manual clock. Core Animation lives on the system clock but lets you offset and group it precisely.

    Knowing which clock you need is half of Question 2.

    The crucial insight: rungs compose vertically

    "SwiftUI + Metal" is not a choice between two rungs. It's SwiftUI owning the high rungs (layout, state, navigation) with a Metal effect occupying a low rung inside it. The same composition shows up everywhere:

    • SwiftUI view → .layerEffect Metal shader (high + low)
    • UIViewRepresentable → wraps an MTKView (SwiftUI hosts raw Metal)
    • CADisplayLink → drives a SwiftUI @State (manual clock feeds declarative view)
    • SwiftUI host → UIKit UIViewPropertyAnimator for an interruptible transition

    So "what's my stack?" is really "what altitude does each concern live at?" Layout might be high, the effect low, the timing in the middle.

    Checkpoint

    You should now be able to: - Name the rungs of the ladder from SwiftUI down to the GPU compositor. - Explain why transforms/opacity are cheap (the compositor floor). - Explain the two clocks and which rungs use which. - Articulate why "SwiftUI + Metal" is vertical composition, not an either/or.

    02 — The Three Questions Framework

    Given any interaction, three questions deterministically narrow your stack. Each maps to one axis of the ladder. Ask them in order, before any code.

    Q1. WHAT am I changing?          → picks the WHAT axis   ([[03-what-are-you-changing]])
Q2. WHAT drives it over time?    → picks the TIMING axis ([[04-who-drives-time]])
Q3. HOW does the user touch it?  → picks the TOUCH axis  ([[05-how-the-user-touches-it]])

    The answers combine into a stack. Most of the time Q1 alone tells you whether you can stay high or must drop to a shader; Q2 and Q3 then refine which high-rung tool.

    Q1 — What am I actually changing?

    This is the most decisive question. It alone determines whether you can stay in the cheap, high rungs or are forced down to pixels.

    What you're changing Rung it forces Tool
    Position, scale, rotation, opacity Stay highest — free on GPU SwiftUI / UIKit
    Per-pixel color, warp, ripple, dissolve, glow Drop to shader Metal effects / MTKView
    Vector geometry — strokes, morphing paths, rings Middle (geometry layer) CAShapeLayer / SwiftUI Shape+.trim
    Hundreds+ independent particles Middle/low CAEmitterLayer / SpriteKit / Metal
    3D content Separate engine RealityKit / SceneKit
    Live 2D drawing, modest count CPU draw rung Canvas / Core Graphics

    The single most important fork: Am I moving/fading a view, or am I changing its pixels? Moving = free, stay high. Changing pixels = shader, drop low. Detail in 03-what-are-you-changing.

    Q2 — What drives the change over time?

    Picks your clock and timing rung.

    Driver Rung Tool
    "From A to B with this curve" Declarative SwiftUI .animation / UIView.animate
    A scripted multi-step sequence Declarative+ PhaseAnimator / KeyframeAnimator / CAAnimationGroup
    Springs / momentum / gesture velocity Physics Wave / CASpringAnimation / UIDynamics
    Every frame depends on live input Manual clock CADisplayLink
    Position tied to a 0→1 value I control Scrub UIViewPropertyAnimator.fractionComplete
    Precise per-layer timing offsets Explicit CA beginTime + grouped CAAnimations

    If you said "every frame depends on something only known at runtime" → you're on the manual clock (CADisplayLink). Everything else can ride the system clock. Detail in 04-who-drives-time.

    Q3 — How does the user touch it?

    Picks the interaction model — often the difference between two otherwise-equal high-rung tools.

    Interaction Rung Tool
    Tap → animate, never interrupted Highest SwiftUI / UIKit declarative
    User can grab mid-flight & redirect Interruptible UIViewPropertyAnimator / additive CA / spring-with-velocity
    Finger drags continuously, flicks with momentum Track + release direct value-set during gesture + spring on release
    Static, no touch Highest whatever Q1/Q2 said

    The trap: building a fire-and-forget animation, then discovering users can interrupt it and it looks broken. If there's any chance the user grabs it, design for interruption from the start. Detail in 05-how-the-user-touches-it.

    Worked example: decode the two canonical combos

    "SwiftUI + Metal for a shader reveal"

    • Q1: I'm changing pixels (a dissolve/mask reveal) → shader forced. But layout, the content being revealed, and state are ordinary.
    • Q2: "from hidden to shown with a curve" → declarative timing is fine; I just feed progress into the shader.
    • Q3: tap to trigger, not interrupted → highest.
    • Stack: SwiftUI owns layout/state/timing; a Metal .layerEffect/.colorEffect occupies one low rung for the reveal pixels. Vertical composition — SwiftUI + Metal.

    "UIKit + CA for fine layer timing"

    • Q1: moving/fading several layers → could stay high...
    • Q2: ...except I need staggered, exact per-layer timing (beginTime offsets, a CAAnimationGroup choreographed to the millisecond). SwiftUI can't express that cleanly → drop to explicit Core Animation.
    • Q3: fire-and-forget → no interruption machinery.
    • Stack: UIKit view controllers + explicit CAAnimations on individual CALayers. The timing precision (Q2), not the what (Q1), is what forced the drop.

    Notice: different questions forced the drop in each case. That's the whole point of asking all three.

    The framework as a one-liner

    Q1 decides if you can stay high or must go to pixels. Q2 decides your clock. Q3 decides interruptibility. The stack is the union of the three answers, applied per concern.

    Checkpoint

    • Given an interaction, you can state Q1/Q2/Q3 answers in under a minute.
    • You can explain why "shader reveal" and "fine layer timing" drop for different reasons.
    • You understand the answers combine per concern, not as a single global choice.

    03 — Q1: What Are You Changing? (The What Axis)

    The single most decisive question. It determines whether you stay in the cheap, high rungs or must descend to pixels. Get this wrong and you either over-engineer (writing a shader to move a box) or hit a wall (trying to ripple pixels with .offset).

    The five categories

    Every animatable thing falls into one of these. Identify yours first.

    1. Transform & opacity — the free tier

    Position, scale, rotation, anchor, alpha, cornerRadius (with a shaped layer), shadow (with shadowPath).

    These are composited by the GPU without redrawing content. The render server already has the layer's rasterized contents; it just multiplies a matrix and an alpha. Animating these is nearly free even for hundreds of layers.

    • Stay at the highest rung. SwiftUI .offset/.scaleEffect/.rotationEffect/.opacity, or UIKit transform/alpha, or CALayer.transform.
    • Rule: if your effect can be expressed as "the same pixels, moved/scaled/faded", you never need to go lower than SwiftUI.

    See 06-cost-model-cpu-gpu for why these are free and what silently isn't (e.g. animating cornerRadius on a non-shaped layer triggers redraws).

    2. Per-pixel procedural change — the shader tier

    Color shifts that vary per pixel, warping/distortion, ripples, dissolves, mask reveals driven by noise, glow/bloom, chromatic aberration, animated gradients with non-linear math, blur that you control.

    The defining trait: the output of each pixel is computed, not just repositioned. No transform can express it. This is the only category that genuinely forces Metal.

    • First reach: SwiftUI Metal effects — .colorEffect (recolor per pixel), .distortionEffect (move pixels), .layerEffect (sample the view's own pixels — needed for blur, dissolve, glow). iOS 17+. → 06-fragment-shaders
    • Drop to raw Metal (MTKView/CAMetalLayer) only if you need vertex control, compute passes, multi-pass, or to render generated content rather than effect an existing view.
    • Your "shader reveal" lives here.

    3. Vector geometry — the path tier

    Drawing/animating strokes, progress rings, morphing one shape into another, animated checkmarks, signature trails, graph lines.

    The trait: you're changing a path/shape, not pixels or position.

    • UIKit/CA: CAShapeLayer with animated strokeEnd (draw-on), strokeStart, lineWidth, or path (morph — paths must have matching control-point counts). → 16-cashapelayer
    • SwiftUI: a Shape + .trim(from:to:) animated, or Path interpolation.
    • Don't reach for a shader for vector work — CAShapeLayer rasterizes crisply and animates strokeEnd essentially free.

    4. Many independent elements — the particle tier

    Confetti, sparks, snow, a burst of hearts, fireworks — dozens to thousands of small sprites with individual lifetimes/velocities.

    The trait: too many elements to animate as individual views.

    • CAEmitterLayer — the built-in particle system. Cheap, declarative emission/lifetime/velocity. Best first reach. → 20-caemitterlayer
    • SpriteKit — when you also want collisions, physics, and a scene graph.
    • Metal instancing — when you need thousands+ with custom per-particle shading.
    • Anti-pattern: 200 SwiftUI views in a ForEach each animated — the view-diffing cost will destroy your frame rate.

    5. 3D / spatial — the scene tier

    Depth, lighting, 3D models, AR.

    • RealityKit (modern, AR-first) or SceneKit (older, still fine for non-AR 3D).
    • Raw Metal only if you're writing your own 3D engine.

    The decision in one diagram

    What am I changing?
│
├─ Same pixels, just moved/scaled/faded?  → TRANSFORM tier  → SwiftUI/UIKit (free)
│
├─ Each pixel's value is computed?         → SHADER tier     → .layerEffect → raw Metal
│
├─ A path / stroke / shape outline?        → PATH tier       → CAShapeLayer / Shape+trim
│
├─ Hundreds of small elements?             → PARTICLE tier   → CAEmitterLayer / SpriteKit
│
└─ Depth / 3D / AR?                         → SCENE tier      → RealityKit / SceneKit

    The classic mistakes this prevents

    • Writing a shader to move a view. Transform tier — .offset is free. A shader is 100× the code for the same result.
    • Trying to ripple/dissolve with .scaleEffect + .opacity. That's the shader tier; no transform composes into a ripple. You'll fake it badly forever.
    • Animating 300 confetti pieces as SwiftUI views. Particle tier — use CAEmitterLayer. View diffing can't keep up.
    • Reaching for Metal to draw a progress ring. Path tier — Shape + .trim or CAShapeLayer.strokeEnd. Crisp and free.
    • Morphing shapes by cross-fading two images. Path tier — animate the actual path.

    Checkpoint

    • You can drop any visual change into one of the five tiers instantly.
    • You know the only tier that forces Metal is per-pixel procedural change.
    • You know transforms/opacity are free and never justify dropping rungs.

    04 — Q2: What Drives the Change Over Time? (The Timing Axis)

    Once Q1 tells you what tier you're in, Q2 picks the clock — who advances the animation each frame. This is where most "it stutters / it can't be interrupted / it feels robotic" problems are actually decided.

    The two clocks (recap from 01-the-rendering-stack-map)

    1. System interpolation clock — you give A, B, and a curve; the system ticks every frame for you. You never see a frame.
    2. Manual clock (CADisplayLink) — you get a callback per refresh and compute the value yourself.

    Everything below is a flavor of one of these.

    The five timing drivers

    A. Declarative interpolation — system clock

    "Go from A to B over 0.3s with ease-in-out." The default. The system owns the frames.

    • SwiftUI withAnimation(.easeInOut) / .animation(...)
    • UIKit UIView.animate(withDuration:)
    • Core Animation CABasicAnimation
    • Use when: the motion is fully known up front and nothing external influences it mid-flight.
    • Tradeoff: you can't easily redirect it once started (without dropping to interruptible tools — 05-how-the-user-touches-it).

    B. Scripted sequence — system clock

    A multi-step canned animation: scale up → overshoot → settle; or a staged reveal of several elements.

    • SwiftUI PhaseAnimator (loop through discrete phases) / KeyframeAnimator (independent tracks per property).
    • UIKit/CA CAAnimationGroup + CAKeyframeAnimation, with beginTime offsets for staggering. → 12-caanimationgroup
    • Use when: the choreography is fixed but multi-step. This is the rung "fine layer timing" lives on when you need exact per-element offsets.

    C. Physics simulation — manual clock (usually), or CASpringAnimation on system clock

    A spring, momentum, or force field computes the value. Motion emerges from initial velocity + parameters, not a fixed duration.

    • Wave / custom CADisplayLink springs — full control, velocity transfer, interruptible. → damped-harmonic-oscillator
    • CASpringAnimation / SwiftUI .spring — declarative spring; easier but less interruptible.
    • UIDynamics — collisions, attachments, gravity for UIKit. → fluid-dynamics-for-ui
    • Use when: motion should feel physical, or carries gesture velocity, or must be interruptible mid-flight.

    D. Frame-perfect manual — manual clock

    Every frame's value depends on something only known at runtime this frame: audio level, sensor data, scroll offset, a continuously-updating shader uniform, a physics step you compute.

    • CADisplayLink is the only answer. → core-animation-smooth-60-120fps
    • Use when: "the system can't possibly interpolate this because I don't know the values in advance."
    • This is what drives a shader's time uniform, an audio visualizer, or a parallax tied to live scroll.

    E. Scrub / progress-driven — system clock, paused

    The animation's position is a 0→1 value you set — from a slider, a scroll offset, or a drag. Time doesn't flow; you are the playhead.

    • UIViewPropertyAnimator with fractionComplete — the cleanest tool. Pause an animation and scrub it.
    • SwiftUI: bind progress to state directly, or drive a shader progress uniform.
    • CA trick: layer.speed = 0 + set timeOffset to scrub a CAAnimation.
    • Use when: the user (or scroll) controls position, not play/pause.

    Decision diagram

    What advances each frame?
│
├─ A fixed curve, known up front          → DECLARATIVE   → withAnimation / UIView.animate / CABasic
│
├─ A fixed multi-step script              → SEQUENCE      → PhaseAnimator / CAAnimationGroup + beginTime
│
├─ Springs / momentum / gesture velocity  → PHYSICS       → Wave / CASpringAnimation / UIDynamics
│
├─ Live runtime input every frame         → MANUAL CLOCK  → CADisplayLink
│
└─ A 0→1 value I control (scroll/drag)    → SCRUB         → UIViewPropertyAnimator.fractionComplete

    The key tells

    • "Duration" makes sense → declarative or sequence (system clock).
    • "Velocity" or "it should feel bouncy/heavy" → physics.
    • "It depends on [live thing] every frame" → manual clock, CADisplayLink.
    • "The user controls how far along it is" → scrub.

    Why this axis causes the most pain

    People default to declarative for everything, then: - a gesture-driven thing feels robotic (should've been physics), - a scroll-linked effect lags (should've been scrub or manual clock), - an interruption snaps (declarative can't redirect — needs physics or scrub).

    Choosing the clock correctly up front is the difference between "feels Apple-grade" and "feels like a web animation."

    Checkpoint

    • You can name the clock for any interaction.
    • You know CADisplayLink is only for live-input-per-frame, not "I want 120fps" (the compositor already gives you that for free — see 06-cost-model-cpu-gpu).
    • You know scrub ≠ playback: the user is the playhead.

    05 — Q3: How Does the User Touch It? (The Interaction Axis)

    Two interactions can change the same thing (Q1) with the same driver (Q2) and still need different stacks — because of how the user touches them. This axis is the tiebreaker, and it's the one most often discovered too late.

    The three interaction models

    1. Fire-and-forget

    Tap a button → it animates → done. The user cannot grab it mid-flight. The animation runs to completion uninterrupted.

    • Any high rung works. SwiftUI withAnimation, UIView.animate, CABasicAnimation.
    • Simplest case — no special machinery.
    • Examples: a checkmark drawing in, a toast sliding up, a screen push.

    2. Interruptible / redirectable

    The user can grab the thing while it's animating and send it somewhere else — and it must respond smoothly, carrying its current velocity, not snapping or restarting.

    This is the rung that plain declarative animation cannot do well. A UIView.animate in flight, if you start a new one, jumps. You need one of:

    • UIViewPropertyAnimator — pause, reverse, retarget; designed for this.
    • Additive Core Animation (isAdditive = true) — multiple animations compose instead of replacing, so a new one blends from the current presentation state. → core-animation-smooth-60-120fps
    • Spring physics with velocity transfer (Wave) — a new spring starts from the current value and current velocity, so redirection is seamless. → damped-harmonic-oscillator
    • Examples: a sheet you can grab mid-dismiss, a card mid-fling, PiP video being re-flung before it settles.

    3. Continuously gesture-tracked + release

    The user's finger is the animation while touching (1:1 tracking, no easing), then on release the thing flies off / snaps with momentum.

    The defining pattern — two phases with a handoff: - During drag: set the value directly from the gesture translation. No animation, no spring response — response: 0 / instantaneous. Any easing here feels laggy. - On release: hand the gesture's velocity to a spring/decay so it continues naturally.

    • Tools: direct value-set during the gesture + a spring (Wave / CASpringAnimation / UIDynamics) seeded with the gesture velocity on release. → momentum-and-deceleration
    • Velocity units matter: gesture recognizers give pts/s; some spring APIs want normalized velocity. Convert. → momentum-and-deceleration
    • Examples: drag-to-dismiss, a draggable card, a custom bottom sheet, swipe-to-delete with throw.

    Decision diagram

    How does the user touch it?
│
├─ Can't touch mid-flight (tap → run)        → FIRE-AND-FORGET → any high rung
│
├─ Can grab & redirect mid-flight            → INTERRUPTIBLE   → UIViewPropertyAnimator
│                                                                 / additive CA / spring+velocity
│
└─ Finger tracks 1:1, then releases w/ throw → TRACK + RELEASE → direct set during drag
                                                                  + spring seeded w/ velocity

    The trap: discovering interruption too late

    The most common animation bug isn't a bug in the animation — it's that it was built fire-and-forget and then a user grabbed it. You see: - a snap/jump when a second gesture starts, - the view teleporting to a stale target, - velocity ignored so a fast flick feels mushy.

    Mitigation: during planning, ask "can the user touch this while it's moving?" If there's any chance, design at the interruptible or track+release rung from the start. Retrofitting interruption onto declarative code usually means rewriting it on the physics rung anyway.

    Why physics wins for interactive motion

    Spring physics is over-represented in interactive UI for a structural reason: a spring is defined by its current state (position + velocity) and a target, not by a fixed timeline. That means: - you can change the target any frame and it smoothly re-aims, - you can inject gesture velocity on release and it continues naturally, - interruption is free — there's no timeline to break.

    This is exactly why libraries like Wave exist and why Apple's own interactive transitions feel the way they do. If Q3 is "interruptible" or "track+release", physics is almost always the right driver (Q2), regardless of Q1.

    Cross-link: the PiP / sheet / card patterns in custom-split-view-gesture-driven and bottom-sheet-blur-backdrop are all this rung.

    Checkpoint

    • You can classify an interaction as fire-forget / interruptible / track+release.
    • You know declarative animation breaks under interruption and why physics doesn't.
    • You know track+release is two phases: direct-set during drag, velocity-seeded spring on release.
    • You ask "can the user touch it mid-flight?" before implementing.

    06 — The Cost Model: What's Free, What's Expensive, Where It Runs

    The three questions (02-three-questions-framework) tell you which stack. This lesson tells you what each rung costs — so your choice survives contact with 60/120fps. The golden rule of smoothness: keep work off the CPU's main thread and let the GPU compositor do what it does for free.

    Foundation: core-animation-smooth-60-120fps and 01-how-rendering-works cover the pipeline. This lesson is the budget view.

    The pipeline, as a budget

    Your app (CPU, main thread)          ← expensive work here = dropped frames
   │  commits layer tree (IPC)
   ▼
Render server / backboardd (separate process)
   │  composites layers
   ▼
GPU                                   ← transforms/opacity/masks: nearly free
   ▼
Display (60 or 120 Hz)                ← you have 16.6ms / 8.3ms per frame

    You have 16.6ms per frame at 60Hz, 8.3ms at 120Hz ProMotion. Anything on the main thread that overruns that budget drops a frame. The render server runs independently — which is why a CABasicAnimation keeps animating even if your main thread is briefly busy.

    The free tier (GPU compositing)

    These run on the GPU in the render server without redrawing layer contents. Animate them freely, even on many layers:

    • transform (translate, scale, rotate, 3D)
    • opacity / alpha
    • position / bounds (geometry)
    • shadowPath-based shadows (path supplied)
    • masking by another layer (cheap if simple)

    This is why the transform tier never justifies dropping rungs. Moving 200 cards is free; the compositor just re-multiplies matrices.

    The expensive tier (rasterization / offscreen passes)

    These force the GPU or CPU to redraw, often into an offscreen buffer, every frame they change:

    Operation Why it's costly Cheaper alternative
    cornerRadius on a layer with contents + masksToBounds offscreen pass per frame pre-rounded image, or .cornerCurve on a shaped layer, or avoid animating it
    shadowOpacity/shadowRadius without shadowPath recomputes shadow shape each frame set shadowPath
    Non-opaque blending / many overlapping transparent layers overdraw mark opaque where possible, flatten
    drawRect: / Core Graphics each frame CPU rasterization, main thread cache to an image; draw once
    Live blur (UIVisualEffectView animated) repeated offscreen sampling animate sparingly; or a cheaper shader
    Large CAEmitterLayer overdraw fill-rate bound fewer/smaller particles, additive blend
    SwiftUI heavy body recompute on every frame CPU diffing move work out of body, use drawingGroup() (Metal-back)

    "Offscreen pass" = the GPU can't composite directly; it renders to a temporary buffer first, then composites that. Each one is a context switch. A few are fine; per-frame in a scroll is death. Profile with Color Offscreen-Rendered in the simulator / Instruments.

    Where each rung does its work

    Rung CPU cost GPU cost Thread
    SwiftUI/UIKit transform animation tiny (commit once) free composite render server
    Core Animation explicit tiny (commit once) free–cheap render server
    CADisplayLink engine your code runs every frame depends what you set main thread (keep it lean!)
    Canvas / Core Graphics full CPU raster upload texture main (or bg) thread
    Metal shader effects tiny fragment cost × pixels GPU
    Raw Metal tiny–moderate whatever you write GPU

    Two danger rungs: - CADisplayLink — your callback is on the main thread every frame. If it does heavy work, you drop frames. Keep per-frame work to arithmetic; never allocate, never lay out, never decode images there. - Canvas / Core Graphics — CPU rasterization. Fine for static or low-frequency drawing; bad for per-frame animation of complex scenes. If it's hot, that's your signal to drop to a shader.

    Shader cost model

    Metal fragment shaders cost roughly (fragment complexity) × (number of pixels covered). Implications:

    • A full-screen shader on a 120Hz ProMotion display at 3x is a lot of pixels × 120/sec. Keep the fragment function cheap (avoid loops, expensive pow/sin where a cheaper approximation works — see 05-math-for-shaders).
    • Effecting a small view is cheap; effecting the whole screen needs care.
    • .layerEffect (samples the view's pixels) is pricier than .colorEffect (no sampling). Use the lightest that does the job.

    "I want 120fps" is not a reason to use CADisplayLink

    A common confusion: ProMotion 120fps is automatic for compositor-driven animations. A CABasicAnimation or SwiftUI transform animation already runs at the display's refresh rate. You only need CADisplayLink (and preferredFrameRateRange) when you're driving values yourself every frame (manual clock). Reaching for CADisplayLink to "get 120fps" on something the compositor could animate is a downgrade — more code, more main-thread risk, same frame rate.

    The budget checklist (apply to any stack choice)

    1. Can I express this as transform/opacity? → free, stop here.
    2. Am I triggering offscreen passes? → check the expensive-tier table; supply paths, pre-render, flatten.
    3. Is per-frame work on the main thread? → only with CADisplayLink, and keep it arithmetic-only.
    4. Is a shader covering huge pixel counts? → simplify the fragment function.
    5. Profile: Color Offscreen-Rendered, Color Blended Layers, Instruments Core Animation / Time Profiler.

    Checkpoint

    • You know transforms/opacity are free and which operations trigger costly offscreen passes.
    • You know CADisplayLink and Canvas put work on the (main) CPU and why that's the risk.
    • You know 120fps is free for compositor animations — CADisplayLink isn't how you "get" it.
    • You can sketch a shader's cost as complexity × pixels.

    07 — The Escalation Ladder (The Discipline)

    The three questions tell you where on the ladder to land. This lesson is the discipline that keeps you honest: start at the highest rung that can express the effect, and descend only when you hit a specific, nameable limitation.

    The rule, restated

    Every rung down costs code and gains control. You must be able to name the wall that forced you lower. "I felt like using Metal" is not a wall. "Transform can't express a per-pixel dissolve" is a wall.

    This prevents the two failure modes: - Over-engineering — writing a CADisplayLink engine or a shader for something withAnimation does in one line. - Under-reaching — fighting a high rung to fake an effect it structurally can't do (rippling with .scaleEffect), instead of dropping one rung to the tool that does it cleanly.

    The ladder with its "walls"

    Descend one rung only when you can state the wall.

    SwiftUI declarative
   │  WALL: needs interruption / scrubbing / UIKit-only API
   ▼
UIViewPropertyAnimator / UIKit
   │  WALL: needs exact per-layer timing offsets, or a specialized
   │        layer (Shape/Emitter/Gradient/Replicator)
   ▼
Explicit Core Animation
   │  WALL: each frame depends on live runtime input
   ▼
CADisplayLink custom engine
   │  WALL: I'm changing per-pixel values, not positions
   ▼
Metal shader effects (.colorEffect/.distortionEffect/.layerEffect)
   │  WALL: need vertex control / compute / multi-pass / generated content
   ▼
Raw Metal (MTKView / CAMetalLayer)

    (Parallel branch off "needs physics / velocity / interruption": spring engine (Wave / UIDynamics) — a specialization of the CADisplayLink rung. → 05-how-the-user-touches-it)

    Named walls — the only valid reasons to descend

    From → To Valid wall (name it before descending)
    SwiftUI → UIKit/PropertyAnimator "Must be interruptible / scrubbable", or a UIKit-only API
    SwiftUI → spring engine "Gesture velocity must transfer; must redirect mid-flight"
    UIKit → explicit Core Animation "Need beginTime stagger / per-layer timing functions / CAAnimationGroup" or "need CAShapeLayer/CAEmitterLayer/CAGradientLayer"
    Declarative → CADisplayLink "Each frame's value depends on live input (audio/sensor/scroll/uniform)"
    Anything → shader effect "I'm changing pixel values (color/warp/dissolve/blur), not positions"
    Shader effect → raw Metal "Need vertices / compute / multi-pass / to render generated content, not effect an existing view"

    If none of these sentences is true, stay where you are.

    Worked descents

    A draggable, flickable card

    • SwiftUI declarative? Wall: gesture velocity must transfer and the card must redirect mid-flight. → descend to spring engine. Stop. (Q1 is transform → no shader, no CA needed.)

    A shimmer sweep across a skeleton loader

    • SwiftUI declarative? Wall: it's a per-pixel moving gradient/specular highlight — pixels, not position. → descend past CA/CADisplayLink straight to shader effect (.colorEffect/.layerEffect) with a progress uniform driven declaratively. Stop. (No raw Metal — effect on an existing view suffices.)

    A staggered, choreographed multi-icon entrance

    • SwiftUI declarative? You can stagger with .delay, but Wall: you need millisecond-exact, per-layer timing functions and grouping. → descend to explicit Core Animation (CAAnimationGroup + beginTime). Stop. (Q1 transform → no shader.)

    An audio-reactive waveform

    • Declarative? Wall: every frame's height depends on live audio level. → descend to CADisplayLink sampling the audio tap. Possibly + shader if the bars are per-pixel glowy. (Q2 forced the manual clock.)

    A generative animated mesh-gradient background

    • Shader effect on a view? Wall: there's no underlying view to effect — it's generated full-screen content with multi-octave noise. → descend to raw Metal (MTKView/CAMetalLayer). (Note: SwiftUI's MeshGradient may suffice for simple cases — try the high rung first!)

    The "try high first" habit

    Before descending, spend 60 seconds asking: is there a newer high-rung API that already does this? Apple keeps pushing capabilities up the ladder: - MeshGradient (SwiftUI) instead of a custom gradient shader for simple cases. - PhaseAnimator/KeyframeAnimator instead of manual CA sequencing. - SF Symbol effects instead of hand-animating symbol layers. - .symbolEffect, .scrollTransition, .visualEffect — high-rung tools that used to require dropping down.

    The ladder's high rungs get taller every year. A wall that was real on iOS 15 may be gone on the current SDK — check via Learning-SwiftUI or the Axiom swiftui-26-ref skill before assuming you must descend.

    Ascending back up

    After descending for one concern, come back up for the others. You dropped to a shader for the reveal pixels — but the layout, state, and trigger timing stay on the SwiftUI rung. Don't let one low-rung concern drag your whole implementation down. This is the vertical composition principle in practice.

    Checkpoint

    • You can state the wall for any descent, and you refuse to descend without one.
    • You check for a newer high-rung API before dropping.
    • You ascend back up for concerns that didn't need the low rung.

    08 — Stack Combos Cookbook

    Reference sheet. Each combo lists when it wins, when not to use it, and how the pieces compose vertically (recall: rungs stack). Keep this open while planning.

    1. SwiftUI alone

    • Wins: standard app UI — state changes, layout, navigation, list reordering, sheets, simple transforms/fades. The default for ~90% of screens.
    • Not when: per-pixel effects, frame-exact multi-layer timing, gesture motion needing velocity transfer.
    • Composition: one rung. withAnimation + transitions.
    • Tells: "tap → thing moves/fades/appears."

    2. SwiftUI + PhaseAnimator / KeyframeAnimator

    • Wins: canned multi-step sequences — button bounce, attention pulse, staged entrance — without a manual state machine. KeyframeAnimator when properties need independent timelines.
    • Not when: the sequence is gesture-driven or interruptible.
    • Composition: SwiftUI declarative, just richer.
    • Tells: "a scripted little routine that plays on an event."

    3. SwiftUI + Canvas + TimelineView

    • Wins: custom 2D drawing that updates over time with modest element counts — a sparkline, a simple visualizer, hand-drawn progress. CPU-drawn but very flexible.
    • Not when: thousands of elements, heavy per-pixel math, or hot per-frame redraw (→ shader). See 06-cost-model-cpu-gpu.
    • Composition: TimelineView provides the clock; Canvas draws each tick (CPU).
    • Tells: "I need to draw custom shapes that change, not many of them."

    4. SwiftUI + Metal shader effects ← the "shader reveal" combo

    • Wins: per-pixel effects applied to existing SwiftUI views — dissolve/mask reveal, shimmer, ripple, heat distortion, glow, recolor. iOS 17+ via .colorEffect / .distortionEffect / .layerEffect.
    • Not when: you need vertex control, compute, multi-pass, or to render generated content (→ raw Metal). Or when it's just a transform (→ stay SwiftUI).
    • Composition: SwiftUI owns layout/state/timing; a Metal fragment function owns the effect pixels. Drive the shader with a progress/time uniform from SwiftUI state. → 06-fragment-shaders
    • Tells: "the pixels themselves change, but it's a normal view underneath."

    5. UIKit + UIViewPropertyAnimator

    • Wins: interruptible, scrubbable, reversible transitions — a sheet you can grab and reverse, a custom VC transition driven by fractionComplete, pause/resume.
    • Not when: pure fire-and-forget (overkill — use UIView.animate), or per-pixel (→ shader).
    • Composition: UIKit views; the animator is the timing authority you can scrub. → 05-how-the-user-touches-it
    • Tells: "the user can grab it mid-animation and reverse/redirect it."

    6. UIKit/AppKit + explicit Core Animation ← the "fine layer timing" combo

    • Wins: precise multi-layer choreography — beginTime stagger, per-layer timing functions, CAAnimationGroup; and the specialized layers: CAShapeLayer (paths/strokes), CAEmitterLayer (particles), CAGradientLayer (gradients), CAReplicatorLayer (clones).
    • Not when: a single view move (SwiftUI is far less code), or per-pixel procedural (→ shader).
    • Composition: view controllers host CALayers; you animate layers directly with millisecond control. → 09-cabasicanimation
    • Tells: "several layers must move in an exactly-timed sequence," or "I need a particle/shape/gradient layer."

    7. CADisplayLink custom engine

    • Wins: each frame's value comes from live runtime input — audio levels, sensors, scroll-linked parallax, a continuously updated shader uniform, hand-rolled physics.
    • Not when: the system can interpolate it (don't use this just to "get 120fps" — that's free). See 06-cost-model-cpu-gpu.
    • Composition: the display link is the manual clock; it sets values on layers/SwiftUI state/shader uniforms each frame. Keep the callback arithmetic-only.
    • Tells: "the value literally isn't known until that frame."

    8. Spring physics (Wave / UIDynamics) ← the interactive-motion combo

    • Wins: gesture-driven, interruptible motion with natural velocity transfer — PiP windows, draggable cards, custom sheets, swipe-to-dismiss with throw. Redirect mid-flight for free.
    • Not when: non-interactive canned motion (a fixed-duration declarative animation is simpler).
    • Composition: direct value-set during the gesture; on release, a spring seeded with gesture velocity drives the value (often setting a CALayer/SwiftUI position each frame). → damped-harmonic-oscillator, momentum-and-deceleration
    • Tells: "finger drags it, then it flies/settles," or "it must feel physical and grabbable."

    9. Raw Metal (MTKView / CAMetalLayer)

    • Wins: generated full-screen content, fluid sims, large custom particle systems, multi-pass effects, anything needing vertex/compute control.
    • Not when: SwiftUI shader effects already do it (combo #4), or it's simple gradient/transform.
    • Composition: wrap MTKView in UIViewRepresentable to host inside SwiftUI; SwiftUI still owns surrounding UI. → 24-metal-and-calayer
    • Tells: "there's no underlying view — I'm generating the pixels from scratch."

    10. SpriteKit

    • Wins: 2D sprite/particle scenes with built-in physics and a scene graph — game-like bursts, interactive 2D worlds.
    • Not when: plain UI (heavyweight), or you only need emission without physics (→ CAEmitterLayer).
    • Composition: SKView hosted in SwiftUI/UIKit; SpriteKit owns its scene.

    11. RealityKit / SceneKit

    • Wins: 3D content and AR. RealityKit for modern/AR; SceneKit for legacy non-AR 3D.
    • Not when: 2D UI.
    • Composition: ARView/SCNView (or RealityView in SwiftUI) hosts the 3D scene.

    Quick lookup by intent

    I want to... Combo
    Move/fade/scale views on state change #1 SwiftUI
    Play a canned bounce/pulse sequence #2 PhaseAnimator
    Draw a custom shape that changes over time #3 Canvas+TimelineView
    Dissolve / shimmer / ripple / glow a view #4 SwiftUI + Metal effects
    Let the user grab & reverse a transition #5 UIViewPropertyAnimator
    Choreograph many layers with exact timing #6 explicit Core Animation
    Draw a progress ring / animate a stroke #6 (CAShapeLayer) or #1 (Shape+.trim)
    Emit confetti / sparks / particles #6 (CAEmitterLayer) or #10 SpriteKit
    React to audio / sensors / scroll per frame #7 CADisplayLink
    Drag-and-flick a card / sheet with momentum #8 spring physics (Wave)
    Generate a full-screen animated background #9 raw Metal (or SwiftUI MeshGradient)
    Build a 2D mini-game effect #10 SpriteKit
    Show 3D / AR content #11 RealityKit

    Composition table — who owns what

    Concern Usually owned by
    Layout & state SwiftUI (highest rung)
    Navigation SwiftUI / UIKit
    Timing of a canned animation declarative
    Interruptible timing PropertyAnimator / spring
    Per-frame value from live input CADisplayLink
    Per-pixel effect Metal (effect or raw)
    Vector path CAShapeLayer / Shape
    Particles CAEmitterLayer / SpriteKit
    3D RealityKit / SceneKit

    The lesson of this table: a single interaction routinely spans several rows. Name the rung for each concern, not one rung for the whole feature.

    12 worked cases

    Each: one-sentence interaction → Q1/Q2/Q3 → stack → why.

    1. A button that bounces when tapped

    • Q1 transform · Q2 sequence · Q3 fire-forget
    • Stack: SwiftUI PhaseAnimator (or .symbolEffect.bounce for SF Symbols).
    • Why: canned multi-step, no pixels, no interruption. Stays top of ladder.

    2. Pull-to-refresh with a custom spinner

    • Q1 transform (+ maybe path for the spinner) · Q2 scrub during pull, then physics on release · Q3 track+release
    • Stack: SwiftUI scroll offset drives the spinner state directly (scrub); spring settle on release.
    • Why: the pull is the playhead; release hands off to physics. → momentum-and-deceleration

    3. A photo dissolve/shimmer reveal

    • Q1 pixels · Q2 declarative progress · Q3 fire-forget
    • Stack: SwiftUI + Metal .layerEffect with a progress uniform animated by withAnimation.
    • Why: Q1 forces a shader; everything else stays SwiftUI. The canonical SwiftUI + Metal combo. → 06-fragment-shaders

    4. Picture-in-Picture window you drag and fling

    • Q1 transform · Q2 physics · Q3 track+release + interruptible
    • Stack: spring engine (Wave) — direct set during drag, velocity-seeded spring on release, re-flingable mid-flight.
    • Why: interruption + velocity transfer ⇒ physics, regardless of the transform-only Q1. → custom-split-view-gesture-driven

    5. A staggered list entrance (rows cascade in)

    • Q1 transform · Q2 sequence with exact offsets · Q3 fire-forget
    • Stack: SwiftUI .transition + per-row .delay if loose timing is fine; drop to explicit Core Animation (beginTime stagger) only if you need millisecond-exact choreography.
    • Why: the wall for descending is timing precision (the "fine layer timing" reason). Try SwiftUI first.

    6. An audio-reactive equalizer

    • Q1 transform (bar heights) or pixels (if glowy) · Q2 manual clock · Q3 fire-forget
    • Stack: CADisplayLink sampling the audio tap, setting bar heights each frame; add a shader if bars glow.
    • Why: value unknown until each frame ⇒ manual clock. → 06-cost-model-cpu-gpu (keep the callback lean).

    7. A progress ring filling up

    • Q1 path · Q2 declarative · Q3 fire-forget
    • Stack: SwiftUI Circle().trim(from:0,to:progress) animated, or CAShapeLayer.strokeEnd.
    • Why: vector geometry ⇒ path tier; no shader, no physics. → 16-cashapelayer

    8. Confetti burst on success

    • Q1 particles · Q2 physics (built into emitter) · Q3 fire-forget
    • Stack: CAEmitterLayer (or SpriteKit if you want collisions).
    • Why: hundreds of elements ⇒ particle tier; never as individual views. → 20-caemitterlayer

    9. A bottom sheet you drag, with rubber-banding past the top

    • Q1 transform · Q2 scrub during drag + physics on release · Q3 track+release + interruptible
    • Stack: direct set during drag with rubber-banding beyond bounds; spring settle to a detent on release.
    • Why: track+release + bounded resistance. → bottom-sheet-blur-backdrop

    10. A custom interactive VC transition you can reverse

    • Q1 transform · Q2 scrub · Q3 interruptible
    • Stack: UIKit UIViewPropertyAnimator driven by fractionComplete from a pan gesture.
    • Why: scrubbable + reversible ⇒ PropertyAnimator is purpose-built. → 05-how-the-user-touches-it

    11. A living mesh-gradient background

    • Q1 pixels (generated) · Q2 manual/time · Q3 fire-forget
    • Stack: try SwiftUI MeshGradient first; if it needs animated multi-octave noise, raw Metal (MTKView in UIViewRepresentable).
    • Why: generated full-screen pixels ⇒ shader, and "no underlying view" ⇒ raw Metal over .layerEffect. Check the high rung (MeshGradient) before descending — see 07-escalation-ladder.

    12. A morphing tab-bar icon (one shape becomes another)

    • Q1 path · Q2 declarative or spring · Q3 fire-forget (or interactive)
    • Stack: CAShapeLayer animating path (matched control points), or SF Symbol .symbolEffect if a symbol covers it.
    • Why: shape morph ⇒ path tier; check SF Symbol effects (high rung) before hand-rolling. → dynamic-island-morphing

    The planning ritual (do this every time)

    1. One sentence: describe the interaction.
    2. Q1 / Q2 / Q3: answer all three out loud.
    3. Per concern: assign a rung to layout, timing, effect, interaction — not one rung to the whole thing.
    4. Escalation check: is each descent justified by a named wall? Is there a newer high-rung API?
    5. Cost check: any offscreen passes, main-thread per-frame work, or huge shader pixel counts? (06-cost-model-cpu-gpu)
    6. Now you have the stack in your head before writing code.

    You can now: - Run any interaction through the flowchart in under a minute. - Assign rungs per concern and compose them vertically. - Justify every descent with a named wall and check for high-rung alternatives. - Sanity-check the cost before committing.
