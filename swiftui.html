    01 — SwiftUI Philosophy: Declarative UI and the View Protocol

    SwiftUI is a declarative UI framework. Instead of writing step-by-step instructions to build your interface, you describe what the interface should look like for a given state. SwiftUI handles the how.

    Declarative vs Imperative

    Imperative (UIKit) — Describe HOW

    // UIKit: you manually create, configure, and position everything
let label = UILabel()
label.text = "Hello, World!"
label.font = .systemFont(ofSize: 24, weight: .bold)
label.textColor = .systemBlue
label.translatesAutoresizingMaskIntoConstraints = false
view.addSubview(label)
NSLayoutConstraint.activate([
    label.centerXAnchor.constraint(equalTo: view.centerXAnchor),
    label.centerYAnchor.constraint(equalTo: view.centerYAnchor)
])

    You create an object. You set properties. You add it to a parent. You configure constraints. If the data changes, you manually update the label again.

    Declarative (SwiftUI) — Describe WHAT

    // SwiftUI: you declare what the UI looks like
Text("Hello, World!")
    .font(.title)
    .foregroundStyle(.blue)

    That's it. No creating objects, no adding to parents, no constraints. You describe the result, and SwiftUI figures out how to produce it.

    The View Protocol

    Every SwiftUI view conforms to the View protocol:

    See 13-protocols for how protocols work in Swift.

    protocol View {
    associatedtype Body: View
    @ViewBuilder var body: Self.Body { get }
}

    The protocol requires exactly one thing: a computed property called body that returns some View. You never call body yourself — SwiftUI calls it when it needs to know what this view looks like.

    some View — Opaque Return Types

    The some keyword means "there is a specific concrete type here, but I'm not telling you what it is."

    See 14-generics for opaque return types and some.

    struct ContentView: View {
    var body: some View {  // compiler knows this is Text, but callers just see "some View"
        Text("Hello")
    }
}

    Why not write var body: Text? Because as your view grows, the concrete type becomes absurdly complex — something like ModifiedContent<VStack<TupleView<(Text, ModifiedContent<Text, _PaddingLayout>)>>, _BackgroundModifier<Color>>. The some View opaque type hides this complexity while preserving type safety.

    Views Are Structs, Not Classes

    See 10-structs-vs-classes for the difference between value and reference types.

    SwiftUI views are structs — value types. This is a deliberate choice:

    struct CounterView: View {
    @State private var count = 0

    var body: some View {
        Button("Count: \(count)") {
            count += 1
        }
    }
}
    • Structs are cheap — no heap allocation, no reference counting overhead
    • Structs are immutable by default — aligns with declarative philosophy
    • No inheritance — composition over inheritance (you combine small views, not subclass big ones)
    • No identity issues — two identical structs are equal; no worrying about object identity

    Views Are Recreated, Not Mutated

    This is the most important mental model shift. When state changes, SwiftUI doesn't mutate your existing view. It creates a new struct value, diffs it against the old one, and updates only what changed on screen.

    // When count changes from 0 to 1:
// 1. SwiftUI creates a NEW CounterView struct
// 2. Calls body on the new struct
// 3. Compares the new body output to the old
// 4. Updates only the text "Count: 0" → "Count: 1" on screen

    Your view struct is a lightweight description. The real on-screen rendering is managed by SwiftUI's internal framework.

    The SwiftUI Rendering Cycle

    State changes → body recomputed → diff against previous → update screen
      ↑                                                        |
      └────────────────────────────────────────────────────────┘
                         (user interaction)
    1. User taps a button (or a timer fires, or network data arrives)
    2. A @State or @Observable property changes
    3. SwiftUI marks the view as needing update
    4. SwiftUI calls body on the view to get the new description
    5. SwiftUI compares new vs old and applies minimal changes to the screen
    6. The screen updates — user sees the change

    Common beginner misconception: "body is called once when the view appears."

    NO. body is called on every state change. This is why body must be fast — it runs frequently. Never do expensive work (network calls, heavy computation) inside body.

    Your First SwiftUI View

    import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Text("Hello, SwiftUI!")
                .font(.largeTitle)
                .fontWeight(.bold)

            Text("This is your first view")
                .font(.subheadline)
                .foregroundStyle(.secondary)
        }
    }
}

#Preview {
    ContentView()
}

    Previews

    The #Preview macro creates a live preview in Xcode's canvas. You see your view rendered in real time without running the full app.

    #Preview {
    ContentView()
}

#Preview("Dark Mode") {
    ContentView()
        .preferredColorScheme(.dark)
}

    The App Entry Point

    import SwiftUI

@main
struct MyApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}
    • @main marks the app's entry point (like main() in C)
    • App protocol is the top-level protocol (similar to View but for the app lifecycle)
    • WindowGroup creates a window containing your root view
    • On iOS, you typically have one WindowGroup; on macOS, it enables multiple windows

    Why SwiftUI Uses Closures Everywhere

    See 07-closures for closure syntax and trailing closures.

    Notice how VStack { ... } and Button("Tap") { ... } use trailing closure syntax. This is because SwiftUI views take their child content as closures — specifically @ViewBuilder closures that can contain multiple views:

    // This trailing closure builds the VStack's children
VStack {
    Text("First")
    Text("Second")
    Text("Third")
}

// Without trailing closure syntax, it would look like this:
VStack(content: {
    Text("First")
    Text("Second")
    Text("Third")
})

    The closure isn't executed later — it's evaluated immediately to produce the child views. @ViewBuilder is a result builder that lets you list multiple views without wrapping them in an array.

    UIKit vs SwiftUI — Mental Model

    UIKit (Imperative) SwiftUI (Declarative)
    Create objects, configure, add to hierarchy Describe the view tree in body
    Mutate existing views on state change Return a new description; framework diffs
    View controller lifecycle (viewDidLoad, etc.) .onAppear, .task, state-driven
    Delegate pattern Closures and bindings
    Auto Layout constraints Stack-based layout (VStack, HStack, ZStack)
    UITableView with data source List with ForEach
    Storyboards / XIBs / code Code only (with live previews)

    Exercises

    1. Create a new SwiftUI project in Xcode. Modify ContentView to display your name in large bold text with a subtitle underneath.

    2. Add multiple previews: Create two #Preview blocks — one for light mode and one for dark mode. Observe how SwiftUI automatically adapts colors.

    3. Break the rules on purpose: Try putting a print("body called") statement inside body. Add a @State variable and a button that changes it. Watch the console — how many times does body get called?

    4. Think about it: Why does SwiftUI use structs instead of classes for views? What problems does this solve compared to UIKit's class-based UIView?

    5. Experiment with nesting: Create a VStack containing an HStack containing two Text views. Notice how composition replaces inheritance.

    02 — Views and Modifiers: Building Blocks of SwiftUI

    Every SwiftUI interface is built from views and modifiers. A view describes what appears on screen. A modifier transforms a view into a new view with different appearance or behavior.

    What Is a Modifier?

    A modifier is a method on View that returns a new view wrapping the original. It does not mutate the existing view — it creates a new one.

    Text("Hello")           // This is a Text view
    .padding()           // Returns ModifiedContent<Text, _PaddingLayout>
    .background(.blue)   // Returns ModifiedContent<ModifiedContent<Text, _PaddingLayout>, _BackgroundModifier>

    Each modifier wraps the previous result. The final type is a nested chain of ModifiedContent wrappers — but you never see this because some View hides it.

    Modifier Order Matters

    This is the single most important thing about modifiers. The order changes the result because each modifier wraps the view that came before it.

    // Version A: padding first, then background
Text("Hello")
    .padding()
    .background(.red)
// Result: red background extends to fill the padding area

// Version B: background first, then padding
Text("Hello")
    .background(.red)
    .padding()
// Result: red background only behind the text, padding is outside (clear)

    Think of it like wrapping a gift: padding first makes the box bigger, then background paints the bigger box. Background first paints the small box, then padding adds clear space around it.

    Visual Proof

    VStack(spacing: 20) {
    Text("Padding then Background")
        .padding()
        .background(.red)

    Text("Background then Padding")
        .background(.red)
        .padding()

    Text("Border shows the frame")
        .padding()
        .border(.blue, width: 2)
        .padding()
        .border(.red, width: 2)
}

    The border modifier is your best debugging tool — it shows exactly where each layer of the view ends.

    Common Modifiers

    Spacing and Sizing

    Text("Padded")
    .padding()                          // default padding on all sides
    .padding(.horizontal, 20)           // 20 points on leading and trailing
    .padding(.top, 8)                   // 8 points on top only

Text("Framed")
    .frame(width: 200, height: 100)     // exact size
    .frame(maxWidth: .infinity)         // take all available width
    .frame(minHeight: 44)              // at least 44 points tall
    .frame(maxWidth: .infinity, alignment: .leading) // left-aligned within full width

    Colors and Styles

    Text("Styled")
    .foregroundStyle(.blue)             // text/symbol color
    .background(.yellow)                // background color
    .tint(.green)                       // accent color for controls

Text("Gradient")
    .foregroundStyle(.linearGradient(
        colors: [.blue, .purple],
        startPoint: .leading,
        endPoint: .trailing
    ))

    Typography

    Text("Typography")
    .font(.title)                       // system title size
    .fontWeight(.bold)                  // weight
    .font(.system(size: 18, weight: .medium, design: .rounded))

    Shape and Clipping

    Image("photo")
    .clipShape(Circle())                // clip to circle
    .clipShape(RoundedRectangle(cornerRadius: 12))

Text("Rounded")
    .padding()
    .background(.blue, in: RoundedRectangle(cornerRadius: 8))

    Effects

    Text("Shadow")
    .shadow(color: .black.opacity(0.3), radius: 4, x: 0, y: 2)

Text("Blurred")
    .blur(radius: 3)

Text("Faded")
    .opacity(0.5)

    Overlay and Background with Alignment

    .overlay places content on top of a view. .background places content behind it. Both respect alignment.

    // Badge on top-right corner
Image(systemName: "bell")
    .font(.title)
    .overlay(alignment: .topTrailing) {
        Circle()
            .fill(.red)
            .frame(width: 12, height: 12)
    }

// Underline effect using background alignment
Text("Underlined")
    .background(alignment: .bottom) {
        Rectangle()
            .frame(height: 2)
            .foregroundStyle(.blue)
    }

    Conditional Modifiers

    Ternary in Modifier Values (Preferred)

    struct HighlightView: View {
    @State private var isActive = false

    var body: some View {
        Text("Tap me")
            .foregroundStyle(isActive ? .white : .primary)
            .padding()
            .background(isActive ? .blue : .gray.opacity(0.2))
            .clipShape(RoundedRectangle(cornerRadius: 8))
            .onTapGesture { isActive.toggle() }
    }
}

    Avoid if for Modifiers (When Possible)

    Using if/else in body creates different view types for each branch, which breaks animations and view identity:

    // BAD: creates two different views — SwiftUI can't animate between them smoothly
var body: some View {
    if isActive {
        Text("Active").foregroundStyle(.blue)
    } else {
        Text("Active").foregroundStyle(.gray)
    }
}

// GOOD: same view, different modifier values — smooth animation
var body: some View {
    Text("Active")
        .foregroundStyle(isActive ? .blue : .gray)
}

    Custom View Modifiers

    When you apply the same set of modifiers repeatedly, extract them into a ViewModifier:

    struct CardStyle: ViewModifier {
    func body(content: Content) -> some View {
        content
            .padding()
            .background(.white)
            .clipShape(RoundedRectangle(cornerRadius: 12))
            .shadow(color: .black.opacity(0.1), radius: 8, x: 0, y: 4)
    }
}

// Use with .modifier()
Text("Card Content")
    .modifier(CardStyle())

    Cleaner Syntax with Extensions

    extension View {
    func cardStyle() -> some View {
        modifier(CardStyle())
    }
}

// Now it reads naturally
Text("Card Content")
    .cardStyle()

    Parameterized Modifiers

    struct RoundedBackground: ViewModifier {
    var color: Color
    var cornerRadius: CGFloat = 12

    func body(content: Content) -> some View {
        content
            .padding()
            .background(color, in: RoundedRectangle(cornerRadius: cornerRadius))
    }
}

extension View {
    func roundedBackground(_ color: Color, cornerRadius: CGFloat = 12) -> some View {
        modifier(RoundedBackground(color: color, cornerRadius: cornerRadius))
    }
}

Text("Custom")
    .roundedBackground(.blue, cornerRadius: 20)

    Environment Modifiers Cascade

    Some modifiers set values that cascade to all children. These are called environment modifiers:

    VStack {
    Text("Title")      // inherits .title font
    Text("Subtitle")   // also inherits .title font
    Text("Override")
        .font(.caption) // overrides parent's font
}
.font(.title)           // applies to all children
.foregroundStyle(.blue) // applies to all children

    Not all modifiers cascade. .padding() and .background() only apply to the view they're attached to. .font(), .foregroundStyle(), and .tint() cascade to children.

    Debugging View Updates

    SwiftUI provides Self._printChanges() to understand why a view's body is recomputed:

    struct DebugView: View {
    @State private var count = 0

    var body: some View {
        let _ = Self._printChanges()  // prints to console on every body call

        Button("Count: \(count)") {
            count += 1
        }
    }
}
// Console output: "DebugView: @self, @identity, _count changed."

    Use this when you suspect unnecessary recomputations.

    Exercises

    1. Modifier order experiment: Create a Text view and apply .padding(), .background(.blue), and .border(.red, width: 2) in different orders. Create at least 3 permutations and observe the visual differences.

    2. Build a card component: Create a custom ViewModifier called CardModifier that adds padding, a white background, rounded corners, and a shadow. Use it on several different views.

    3. Conditional styling: Build a view with a @State boolean. When true, the text should be bold blue on a light blue background. When false, it should be regular gray text. Use ternary operators, not if/else.

    4. Environment modifiers: Create a VStack with 5 Text views. Set .font(.headline) and .foregroundStyle(.purple) on the VStack. Override the font on just one child. Verify the override works while others inherit.

    5. Debugging: Add Self._printChanges() to a view with multiple @State properties. Change them one at a time and observe which property triggered the recomputation.

    03 — Text and Images: Displaying Content in SwiftUI

    Text and images are the most fundamental building blocks of any interface. SwiftUI makes both simple to create and rich to customize.

    Text Basics

    Text("Hello, World!")

    That's a complete SwiftUI view. No configuration needed — it picks a reasonable font, size, and color automatically.

    Text Modifiers

    Text("Styled Text")
    .font(.title)
    .fontWeight(.bold)
    .italic()
    .underline()
    .strikethrough(color: .red)

    Each modifier returns a new Text view (or a modified view). They can be combined in any order — though .font() conventionally comes first for readability.

    System Font Styles

    SwiftUI provides semantic font styles that adapt to the user's Dynamic Type settings:

    VStack(alignment: .leading, spacing: 8) {
    Text("Large Title").font(.largeTitle)   // 34pt default
    Text("Title").font(.title)               // 28pt default
    Text("Title 2").font(.title2)            // 22pt default
    Text("Title 3").font(.title3)            // 20pt default
    Text("Headline").font(.headline)         // 17pt semibold
    Text("Body").font(.body)                 // 17pt — the default
    Text("Callout").font(.callout)           // 16pt
    Text("Subheadline").font(.subheadline)   // 15pt
    Text("Footnote").font(.footnote)         // 13pt
    Text("Caption").font(.caption)           // 12pt
    Text("Caption 2").font(.caption2)        // 11pt
}

    Always prefer semantic styles over fixed sizes. They scale with the user's accessibility settings.

    Custom Fonts

    Text("Custom Font")
    .font(.custom("Georgia", size: 20))

Text("System Design Variant")
    .font(.system(size: 18, weight: .medium, design: .rounded))
    // design options: .default, .rounded, .serif, .monospaced

    Text Interpolation and Formatting

    See 03-strings for Swift string interpolation fundamentals.

    let score = 42
let percentage = 0.857

Text("Score: \(score)")
Text("Percentage: \(percentage, format: .percent)")
Text("Price: \(29.99, format: .currency(code: "USD"))")
Text("Date: \(Date.now, format: .dateTime.day().month().year())")

    SwiftUI's Text supports format styles directly in interpolation — no need for NumberFormatter.

    Concatenating Text Views

    You can combine Text views with + to create mixed styling within a single line:

    Text("Hello, ")
    .fontWeight(.regular)
+ Text("World")
    .fontWeight(.bold)
    .foregroundStyle(.blue)
+ Text("!")
    .foregroundStyle(.red)

    This creates a single text run with mixed formatting — impossible to achieve with modifiers on a single Text.

    Multiline Text

    Text("This is a long paragraph that will wrap across multiple lines automatically when it exceeds the available width.")
    .lineLimit(3)               // max 3 lines, truncates with "..."
    .lineSpacing(4)             // space between lines
    .multilineTextAlignment(.center)  // center-align wrapped text
    .truncationMode(.middle)    // where to place "..." (.head, .middle, .tail)

Text("Expand to fit")
    .lineLimit(1...5)           // at least 1 line, up to 5

    Label: Icon + Text

    Label pairs text with an icon. SwiftUI can display both, or just one, depending on context:

    Label("Favorites", systemImage: "star.fill")
Label("Downloads", systemImage: "arrow.down.circle")
Label("Settings", image: "custom-gear")  // custom asset image

// In a List, Label shows both icon and text
// In a toolbar, it might show only the icon
// Label adapts to its context automatically

    SF Symbols

    Apple provides thousands of free vector icons called SF Symbols. They scale with text, support colors, and animate.

    // Basic SF Symbol
Image(systemName: "star.fill")
Image(systemName: "heart.circle.fill")
Image(systemName: "arrow.triangle.2.circlepath")

// Sizing — SF Symbols scale with .font()
Image(systemName: "star.fill")
    .font(.title)             // same size as title text
    .font(.system(size: 40))  // explicit size

// Colors
Image(systemName: "heart.fill")
    .foregroundStyle(.red)

// Symbol rendering modes
Image(systemName: "battery.75percent")
    .symbolRenderingMode(.palette)
    .foregroundStyle(.green, .gray)

Image(systemName: "cloud.sun.rain.fill")
    .symbolRenderingMode(.multicolor)  // Apple's built-in colors

// Symbol effects (iOS 17+)
Image(systemName: "wifi")
    .symbolEffect(.variableColor.iterative)

Image(systemName: "bell")
    .symbolEffect(.bounce, value: notificationCount)

    Use the SF Symbols app (free from Apple) to browse all available symbols.

    Image: Asset Catalog Images

    // Load from asset catalog
Image("sunset-photo")

// Common modifiers for photos
Image("sunset-photo")
    .resizable()                   // REQUIRED to resize — without this, image renders at native resolution
    .scaledToFit()                 // scale to fit within available space, maintaining aspect ratio
    .frame(width: 300, height: 200)

Image("profile-photo")
    .resizable()
    .scaledToFill()                // scale to fill, may crop
    .frame(width: 100, height: 100)
    .clipShape(Circle())           // clip to circle

    The .resizable() Rule

    Without .resizable(), the image renders at its native pixel size — often overflowing the screen. Always add .resizable() before any sizing modifiers.

    // WRONG — image at native resolution, may be huge
Image("photo")
    .frame(width: 200, height: 200)  // frame exists but image overflows

// RIGHT — image scales to fit the frame
Image("photo")
    .resizable()
    .scaledToFit()
    .frame(width: 200, height: 200)

    Aspect Ratio

    Image("landscape")
    .resizable()
    .aspectRatio(16/9, contentMode: .fit)  // force 16:9 aspect ratio

    AsyncImage: Loading Remote Images

    // Basic — shows placeholder while loading
AsyncImage(url: URL(string: "https://example.com/photo.jpg"))

// With placeholder and error handling
AsyncImage(url: URL(string: "https://example.com/photo.jpg")) { phase in
    switch phase {
    case .empty:
        ProgressView()          // loading indicator
    case .success(let image):
        image
            .resizable()
            .scaledToFit()
    case .failure:
        Image(systemName: "photo.slash")
            .foregroundStyle(.secondary)
    @unknown default:
        EmptyView()
    }

    04 — Layout Fundamentals: Stacks, Spacers, and Frames

    SwiftUI uses a stack-based layout system. Instead of Auto Layout constraints, you compose views using horizontal, vertical, and layered stacks. Understanding how SwiftUI negotiates size between parent and child is the key to mastering layout.

    VStack: Vertical Layout

    VStack arranges children from top to bottom:

    VStack {
    Text("First")
    Text("Second")
    Text("Third")
}

    Alignment and Spacing

    VStack(alignment: .leading, spacing: 12) {
    Text("Left-aligned")
    Text("Also left-aligned")
    Text("Short")
}
// alignment options: .leading, .center (default), .trailing

    HStack: Horizontal Layout

    HStack arranges children from leading to trailing (left to right in English):

    HStack {
    Image(systemName: "star.fill")
    Text("Favorites")
}

HStack(alignment: .firstTextBaseline, spacing: 8) {
    Text("Big").font(.largeTitle)
    Text("small").font(.caption)
}
// alignment options: .top, .center (default), .bottom, .firstTextBaseline, .lastTextBaseline

    ZStack: Layered Layout

    ZStack layers children back to front. The first child is at the back:

    ZStack {
    Color.blue                        // fills available space — background layer
    Text("On top of blue")            // front layer
        .foregroundStyle(.white)
        .font(.title)
}

ZStack(alignment: .bottomTrailing) {
    Image("photo")
        .resizable()
        .scaledToFit()
    Text("Photo credit")
        .font(.caption)
        .padding(4)
        .background(.black.opacity(0.6))
        .foregroundStyle(.white)
}

    Spacer

    Spacer() is a flexible view that expands to fill available space, pushing other views apart:

    HStack {
    Text("Leading")
    Spacer()           // pushes "Trailing" to the right edge
    Text("Trailing")
}

VStack {
    Text("Top")
    Spacer()           // pushes "Bottom" to the bottom
    Text("Bottom")
}

HStack {
    Spacer()
    Text("Centered-Right")
    Spacer()
    Spacer()  // two spacers on right, one on left — text sits at 1/3 point
}

    Spacer(minLength: 20) sets a minimum size — the spacer won't shrink below 20 points.

    Divider

    Divider() draws a thin horizontal line (in VStack) or vertical line (in HStack):

    VStack {
    Text("Section 1")
    Divider()
    Text("Section 2")
}

    Frame: Explicit Sizing

    .frame() proposes a specific size to its child:

    Text("Fixed Size")
    .frame(width: 200, height: 50)

// Take all available width, fixed height
Text("Full Width")
    .frame(maxWidth: .infinity)
    .frame(height: 44)
    .background(.blue)

// Minimum and maximum constraints
Text("Flexible")
    .frame(minWidth: 100, maxWidth: 300, minHeight: 44)

// Alignment within the frame
Text("Top-Leading")
    .frame(width: 200, height: 200, alignment: .topLeading)
    .border(.gray)

    maxWidth: .infinity Pattern

    This is the most common frame pattern — making a view take all available horizontal space:

    Button("Full Width Button") { }
    .frame(maxWidth: .infinity)
    .padding()
    .background(.blue)
    .foregroundStyle(.white)
    .clipShape(RoundedRectangle(cornerRadius: 12))

    Padding

    .padding() adds space between a view's content and its edge:

    // Default padding on all sides (~16 points, varies by platform)
Text("Padded").padding()

// Specific edges
Text("Horizontal").padding(.horizontal, 20)
Text("Top only").padding(.top, 8)
Text("Custom").padding(EdgeInsets(top: 10, leading: 20, bottom: 10, trailing: 20))

// Multiple padding modifiers stack
Text("Double padded")
    .padding(.horizontal, 8)
    .padding(.vertical, 4)

    How SwiftUI Layout Works

    SwiftUI uses a three-step negotiation process:

    1. Parent Proposes Size

    The parent view offers available space to the child: "You have 300 x 500 points."

    2. Child Reports Size

    The child decides its own size based on the proposal: "I need 120 x 20 points."

    3. Parent Positions Child

    The parent places the child within the available space according to alignment.

    // Example: how a Text in a .frame() works
Text("Hello")                      // Text wants 42 x 20 (just enough for the text)
    .frame(width: 200, height: 100) // Frame proposes 200 x 100 to Text
                                    // Text still reports 42 x 20
                                    // Frame renders as 200 x 100, places Text centered within
    .border(.red)                   // You see the 200x100 frame with text centered

    Key insight: children choose their own size. A parent can propose, but it cannot force. Text always sizes to fit its content. Color always accepts the full proposed size. Image (without .resizable()) always uses its native size.

    Layout Priority

    When space is limited, .layoutPriority() determines which views get space first:

    HStack {
    Text("This text might truncate if space is tight")
        .layoutPriority(1)  // gets space first

    Text("Less important")
    // layoutPriority defaults to 0 — gets remaining space
}

    Fixed Size

    .fixedSize() prevents a view from being compressed below its ideal size:

    Text("This long text will NOT be truncated — it may overflow instead")
    .fixedSize()

// Fix in one dimension only
Text("Fixed horizontally only")
    .fixedSize(horizontal: true, vertical: false)

    Alignment Guides

    For fine-grained control, .alignmentGuide() lets you customize exactly where a view aligns:

    HStack(alignment: .top) {
    Text("Normal")

    Text("Offset down")
        .alignmentGuide(.top) { dimension in
            dimension[.top] - 20  // shift this view 20 points down from top alignment
        }
}

    05 — State and Binding: Making Views Dynamic

    State is the engine of SwiftUI. When state changes, the UI updates automatically. Understanding where state lives and how it flows between views is the most important concept in SwiftUI.

    The Core Concept: Single Source of Truth

    Every piece of data in your app should have exactly one owner — one source of truth. Other views that need that data should reference it, not duplicate it.

    If two views both own copies of the same data, they will inevitably get out of sync. SwiftUI's property wrappers enforce this: @State for owning data, @Binding for referencing someone else's data.

    @State: Local View State

    @State creates a piece of state owned by the view. When the value changes, SwiftUI recomputes the view's body.

    See 24-property-wrappers for how property wrappers work under the hood.

    struct CounterView: View {
    @State private var count = 0

    var body: some View {
        VStack(spacing: 20) {
            Text("Count: \(count)")
                .font(.largeTitle)

            HStack(spacing: 16) {
                Button("Subtract") { count -= 1 }
                Button("Add") { count += 1 }
            }
        }
    }
}

    How @State Works on Structs

    Wait — structs are immutable. How can count += 1 work on a struct property?

    See 10-structs-vs-classes for value type immutability.

    The trick: @State doesn't actually store the value inside the struct. It stores the value in external storage managed by SwiftUI. The struct just holds a reference to that storage. When the struct is recreated (on every body call), it reconnects to the same external storage.

    // Conceptually, this is what @State does:
// - The actual Int is stored outside the struct, managed by SwiftUI
// - The struct has a reference to that external storage
// - When you write count += 1, you're modifying the external storage
// - SwiftUI sees the change and recreates the struct with a new body call
// - The new struct reconnects to the same external storage

    @State Should Be Private

    Mark @State properties as private. State is owned by the view — no other view should set it directly:

    @State private var isExpanded = false  // correct
@State var isExpanded = false          // works, but bad practice

    @Binding: A Reference to Someone Else's State

    @Binding creates a two-way connection to state owned by another view. The child can read AND write the parent's state.

    struct ParentView: View {
    @State private var isOn = false

    var body: some View {
        VStack {
            Text(isOn ? "ON" : "OFF")
                .font(.largeTitle)

            ToggleButton(isOn: $isOn)  // pass a binding using $
        }
    }
}

struct ToggleButton: View {
    @Binding var isOn: Bool  // does NOT own the data — just a reference

    var body: some View {
        Button(isOn ? "Turn Off" : "Turn On") {
            isOn.toggle()  // modifies the parent's @State
        }
    }
}

    The $ Prefix: Projected Value

    The $ prefix accesses the projected value of a property wrapper. For @State, the projected value is a Binding:

    See 24-property-wrappers for projected values.

    @State private var name = ""

// name     → the String value ("Hello")
// $name    → a Binding<String> that reads and writes the @State

    This is how you pass state to child views that need to modify it:

    TextField("Enter name", text: $name)  // TextField needs a Binding<String>
Toggle("Dark mode", isOn: $isDarkMode)  // Toggle needs a Binding<Bool>
Slider(value: $volume, in: 0...100)  // Slider needs a Binding<Double>

    Binding in Practice

    struct SettingsView: View {
    @State private var username = ""
    @State private var notificationsOn = true
    @State private var fontSize = 14.0

    var body: some View {
        VStack(alignment: .leading, spacing: 16) {
            TextField("Username", text: $username)
                .textFieldStyle(.roundedBorder)

            Toggle("Notifications", isOn: $notificationsOn)

            HStack {
                Text("Font size: \(Int(fontSize))")
                Slider(value: $fontSize, in: 10...30, step: 1)
            }

            // Divider between controls and summary
            Divider()

            // Summary shows current state — updates automatically
            Text("User: \(username.isEmpty ? "none" : username)")
            Text("Notifications: \(notificationsOn ? "on" : "off")")
            Text("Font size: \(Int(fontSize))pt")
        }
        .padding()
    }
}

    Constant Bindings for Previews

    When previewing a child view that takes a @Binding, use .constant() to provide a fixed value:

    struct ToggleButton: View {
    @Binding var isOn: Bool
    var body: some View {
        Button(isOn ? "ON" : "OFF") { isOn.toggle() }
    }
}

    ToggleButton(isOn: .constant(true))   // fixed binding for preview
}

    .constant() creates a binding that can be read but writes are silently ignored.

    State Change → Body Recompute → UI Update

    The entire SwiftUI rendering cycle depends on state:

    struct TrafficLight: View {
    @State private var color = "red"

    var body: some View {
        VStack(spacing: 20) {
            Circle()
                .fill(colorForName(color))
                .frame(width: 80, height: 80)

            Button("Next") {
                // State changes here...
                switch color {
                case "red": color = "green"
                case "green": color = "yellow"
                default: color = "red"
                }
                // ...trigger body to be called again
                // ...SwiftUI diffs the new body output
                // ...only the Circle's fill color updates on screen
            }
        }
    }

    func colorForName(_ name: String) -> Color {
        switch name {
        case "red": .red
        case "green": .green
        case "yellow": .yellow
        default: .gray
        }
    }
}

    Common Mistakes

    Mistake 1: Putting @State Too High

    // BAD: @State in the top-level view for data only one child needs
struct AppView: View {
    @State private var searchText = ""  // only SearchBar uses this

    var body: some View {
        VStack {
            SearchBar(text: $searchText)
            ProfileView()    // recomputes when searchText changes, even though it doesn't use it
            SettingsView()   // same — unnecessary recomputation
        }
    }
}

// GOOD: @State in the view that owns the data
struct SearchBar: View {
    @State private var searchText = ""  // owned here, only SearchBar recomputes
    var body: some View {
        TextField("Search", text: $searchText)
    }
}

    Mistake 2: Initializing @State from External Values

    // BAD: @State initialized from a parameter — only uses the initial value
struct EditView: View {
    @State private var text: String

    init(initialText: String) {
        _text = State(initialValue: initialText)  // only set once!
    }
    // If the parent passes a new initialText, this view won't update
}

// GOOD: use .onAppear or .task for one-time initialization
struct EditView: View {
    let initialText: String
    @State private var text = ""

    var body: some View {
        TextField("Edit", text: $text)
            .onAppear { text = initialText }
    }
}

    Mistake 3: Using @State for Data Shared Across Many Views

    @State is for local view data. For app-wide or shared data, you need @Observable and @Environment.

    See 15-observable-and-environment for sharing state across views. See 16-data-flow for state management patterns.

    Optional State and Bindings

    See 08-optionals for optionals in Swift.

    @State private var selectedItem: String? = nil

// Binding to optional — works with sheets and navigation
.sheet(item: $selectedItem) { item in
    DetailView(item: item)
}

    Lists and ForEach

    List is SwiftUI's primary container for displaying scrollable, vertically arranged rows of data. It is backed by UITableView on iOS and provides built-in support for selection, swipe actions, editing, and lazy loading. ForEach is the companion construct that generates views from collections.

    See 04-collection-types for the array and dictionary foundations that feed into lists.

    Static Lists

    The simplest list is hardcoded content. Each child becomes a row.

    struct StaticListView: View {
    var body: some View {
        List {
            Text("Row 1")
            Text("Row 2")
            Text("Row 3")
        }
    }
}

    You can mix different view types freely inside a List:

    List {
    Text("A label")
    Toggle("Wi-Fi", isOn: .constant(true))
    Button("Tap me") { print("tapped") }
}

    ForEach with Identifiable

    Dynamic lists use ForEach to generate rows from a collection. The elements must be uniquely identifiable so SwiftUI can track insertions, deletions, and moves efficiently.

    See 13-protocols for protocol conformance. Identifiable requires a single id property.

    struct Fruit: Identifiable {
    let id = UUID()
    let name: String
    let emoji: String
}

struct FruitListView: View {
    let fruits = [
        Fruit(name: "Apple", emoji: "🍎"),
        Fruit(name: "Banana", emoji: "🍌"),
        Fruit(name: "Cherry", emoji: "🍒"),
    ]

    var body: some View {
        List {
            ForEach(fruits) { fruit in
                HStack {
                    Text(fruit.emoji)
                    Text(fruit.name)
                }
            }
        }
    }
}

    When the data type conforms to Identifiable, you do not need to specify a key path -- ForEach uses \.id automatically.

    See 19-value-semantics -- because Fruit is a struct (value type), SwiftUI can diff efficiently.

    ForEach with id Key Path

    For simple types like String or Int that do not conform to Identifiable, provide a key path to a unique property.

    let colors = ["Red", "Green", "Blue"]

List {
    ForEach(colors, id: \.self) { color in
        Text(color)
    }
}

    \.self works when each element is unique and Hashable. Duplicate values will cause undefined behavior.

    Index-Based ForEach

    Use a range when you need index numbers rather than collection elements.

    ForEach(0..<5) { index in
    Text("Row \(index)")
}

    This is a constant range -- it must be known at compile time. For dynamic counts, use ForEach(Array(0..<dynamicCount), id: \.self) or prefer the collection-based form.

    Sections

    Group rows visually with Section. Each section can have a header and footer.

    List {

        Text("Banana")

    Add contextual actions that appear when the user swipes a row.

    onDelete and onMove

    For standard delete and reorder functionality, use the dedicated modifiers with EditButton().

    struct EditableListView: View {
    @State private var items = ["Alpha", "Beta", "Gamma", "Delta"]

    var body: some View {
        NavigationStack {
            List {
                ForEach(items, id: \.self) { item in
                    Text(item)
                }
                .onDelete { offsets in
                    items.remove(atOffsets: offsets)
                }

                }
            }
            .toolbar { EditButton() }
        }
    }

    onDelete and onMove are applied to ForEach, not to List.

    Change the visual presentation of the entire list.

    .listStyle(.grouped)       // flat section backgrounds
    .listStyle(.sidebar)       // collapsible sections, used in sidebars

    Pull to Refresh

    Add async refresh with .refreshable. This modifier requires an async closure.

    List {
    ForEach(items) { item in
        Text(item.name)
    }
}

    await loadItems()
}

    Selection

    Enable single or multi-select by binding a selection variable.

    @State private var selection: Set<String> = []
    let items = ["One", "Two", "Three", "Four"]

    Performance: List Is Lazy

    List only creates views for visible rows, similar to UITableView cell reuse. This makes it efficient for large data sets. A ScrollView with a VStack inside creates all views immediately, which is wasteful for long lists.

    // Efficient -- only visible rows are created
List(largeArray) { item in
    ExpensiveRowView(item: item)
}

// Wasteful for large data -- all 10,000 views created at once
ScrollView {
    VStack {
        ForEach(largeArray) { item in
            ExpensiveRowView(item: item)
        }
    }
}

    See 18-scrollview for when to use ScrollView with LazyVStack for custom scroll behavior.

    Navigation

    Navigation is how users move between screens in your app. SwiftUI's navigation system (iOS 16+) is built around NavigationStack and value-based NavigationLink, replacing the older NavigationView. The new system supports programmatic navigation, deep linking, and state restoration.

    See 09-enums for enum-based route definitions, which are the cleanest way to model navigation destinations.

    NavigationStack

    NavigationStack is the container that manages a stack of views. It provides the navigation bar and handles push/pop transitions.

    struct ContentView: View {
    var body: some View {
        NavigationStack {
            VStack {
                Text("Home Screen")
            }
            .navigationTitle("Home")
        }
    }
}

    Place .navigationTitle() on the content inside the stack, not on NavigationStack itself.

    NavigationLink (Simple)

    The basic form pushes a destination view when tapped.

    NavigationStack {
    List {
        NavigationLink("Settings") {
            SettingsView()
        }
        NavigationLink {
            ProfileView()
        } label: {
            Label("Profile", systemImage: "person")
        }
    }
    .navigationTitle("Menu")
}

    SwiftUI shows a disclosure indicator (chevron) automatically.

    Value-Based Navigation

    The preferred approach separates the link's value from its destination. This makes programmatic navigation possible.

    struct Item: Identifiable, Hashable {
    let id = UUID()
    let name: String
}

struct ItemListView: View {
    let items = [Item(name: "Alpha"), Item(name: "Beta"), Item(name: "Gamma")]

    var body: some View {
        NavigationStack {
            List(items) { item in
                NavigationLink(value: item) {
                    Text(item.name)
                }
            }
            .navigationDestination(for: Item.self) { item in
                Text("Detail for \(item.name)")
                    .navigationTitle(item.name)
            }
            .navigationTitle("Items")
        }
    }
}

    navigationDestination(for:) registers a mapping from a value type to a destination view. When a NavigationLink(value:) is tapped, SwiftUI finds the matching destination.

    Programmatic Navigation with NavigationPath

    NavigationPath is a type-erased stack that lets you push and pop values in code without user interaction.

    struct AppView: View {
    @State private var path = NavigationPath()

    var body: some View {
        NavigationStack(path: $path) {
            VStack(spacing: 20) {
                Button("Go to Detail") {
                    path.append(Item(name: "Pushed"))
                }
                Button("Go to Settings") {
                    path.append("settings")
                }
            }
            .navigationDestination(for: Item.self) { item in
                DetailView(item: item, path: $path)
            }
            .navigationDestination(for: String.self) { route in
                if route == "settings" {
                    SettingsView()
                }
            }
            .navigationTitle("Home")
        }
    }
}

    Key operations on NavigationPath:

    path.append(value)      // Push a new screen
path.removeLast()       // Pop one screen (go back)
path.removeLast(2)      // Pop two screens
path = NavigationPath() // Pop to root

    Enum-Based Routes

    For type-safe navigation, define an enum of all possible destinations.

    See 09-enums for associated values and Hashable conformance on enums.

    enum Route: Hashable {
    case profile(userId: String)
    case settings
    case detail(item: Item)
}

struct RouterView: View {
    @State private var path: [Route] = []

    var body: some View {
        NavigationStack(path: $path) {
            List {
                Button("My Profile") {
                    path.append(.profile(userId: "user123"))
                }
                Button("Settings") {
                    path.append(.settings)
                }
            }
            .navigationDestination(for: Route.self) { route in
                switch route {
                case .profile(let userId):
                    ProfileView(userId: userId)
                case .settings:
                    SettingsView()
                case .detail(let item):
                    DetailView(item: item)
                }
            }
            .navigationTitle("App")
        }
    }
}

    Using a typed array [Route] instead of NavigationPath gives you compile-time safety. Use NavigationPath when you need to push heterogeneous types.

    Navigation Title and Display Mode

    .navigationTitle("Title")
.navigationBarTitleDisplayMode(.large)   // big title, scrolls away
.navigationBarTitleDisplayMode(.inline)  // compact, centered
.navigationBarTitleDisplayMode(.automatic) // inherits from parent

    Deep Linking and State Restoration

    Encode and decode NavigationPath using Codable to support deep links and state restoration.

    struct PersistentNavView: View {
    @State private var path = NavigationPath()

    var body: some View {
        NavigationStack(path: $path) {
            HomeView()
                .navigationDestination(for: Item.self) { item in
                    DetailView(item: item)
                }
        }
        .onAppear { restorePath() }
        .onChange(of: path.count) { savePath() }
    }

    func savePath() {
        guard let data = try? JSONEncoder().encode(path.codable) else { return }
        UserDefaults.standard.set(data, forKey: "navPath")
    }

    func restorePath() {
        guard let data = UserDefaults.standard.data(forKey: "navPath"),
              let decoded = try? JSONDecoder().decode(
                  NavigationPath.CodableRepresentation.self, from: data
              ) else { return }
        path = NavigationPath(decoded)
    }
}

    All value types pushed onto the path must conform to both Hashable and Codable.

    Common Mistakes

    1. Nesting NavigationStacks -- This creates double navigation bars and broken behavior.

    // WRONG
NavigationStack {
    NavigationLink("Detail") {
        NavigationStack {  // Never nest these
            Text("Detail")
        }
    }
}

// CORRECT -- only one NavigationStack at the root
NavigationStack {
    NavigationLink("Detail") {
        Text("Detail")
            .navigationTitle("Detail")
    }
}

    2. NavigationLink outside a NavigationStack -- Links do nothing without a surrounding stack.

    3. Placing .navigationTitle on NavigationStack -- It goes on the content view inside the stack.

    4. Using NavigationView -- Deprecated in iOS 16. Always use NavigationStack or NavigationSplitView.

    Sheets and Alerts

    Sheets, alerts, and popovers are SwiftUI's modal presentation tools. Unlike navigation, which pushes screens onto a stack, modals appear on top of the current content and demand attention before the user returns.

    See 07-closures for trailing closure syntax -- every modal API uses closures to define content.

    Sheet (Boolean Presentation)

    The simplest sheet is driven by a Boolean binding.

    struct SheetDemoView: View {
    @State private var showSheet = false

    var body: some View {
        Button("Show Sheet") {
            showSheet = true
        }
        .sheet(isPresented: $showSheet) {
            SheetContentView()
        }
    }
}

struct SheetContentView: View {
    @Environment(\.dismiss) var dismiss

    var body: some View {
        VStack(spacing: 20) {
            Text("This is a sheet")
                .font(.title)
            Button("Dismiss") {
                dismiss()
            }
        }
    }
}

    Users can also dismiss by swiping down. The @Environment(\.dismiss) action works for all modal presentations.

    Sheet (Item-Based Presentation)

    Present a sheet when an optional value becomes non-nil. The value is passed into the sheet's content closure.

    See 08-optionals -- item-based presentation leverages optional binding.

    struct Note: Identifiable {
    let id = UUID()
    let title: String
    let body: String
}

struct NoteListView: View {
    @State private var selectedNote: Note?
    let notes = [
        Note(title: "First", body: "Hello"),
        Note(title: "Second", body: "World"),
    ]

    var body: some View {
        List(notes) { note in
            Button(note.title) {
                selectedNote = note
            }
        }
        .sheet(item: $selectedNote) { note in
            VStack {
                Text(note.title).font(.title)
                Text(note.body)
            }
            .padding()
        }
    }
}

    When the sheet is dismissed, selectedNote is automatically set back to nil.

    Full-Screen Cover

    .fullScreenCover presents a modal that covers the entire screen. There is no swipe-to-dismiss gesture by default.

    .fullScreenCover(isPresented: $showCover) {
    FullScreenView()
}

    The API mirrors .sheet exactly -- both boolean and item-based variants exist. Use full-screen covers for onboarding flows, login screens, or any context where accidental dismissal should be prevented.

    Presentation Detents (Half-Sheet)

    Control the height at which a sheet rests.

    .sheet(isPresented: $showSheet) {
    SheetContentView()
        .presentationDetents([.medium, .large])
        .presentationDragIndicator(.visible)
}

    Available detents: - .medium -- approximately half the screen - .large -- full height (default) - .fraction(0.3) -- 30% of screen height - .height(200) -- fixed height in points - Custom detents via the CustomPresentationDetent protocol

    .presentationDetents([.fraction(0.25), .medium, .large])

    Presentation Customization

    .sheet(isPresented: $showSheet) {
    ContentView()
        .presentationDetents([.medium, .large])
        .presentationDragIndicator(.visible)       // show the drag handle
        .presentationBackground(.ultraThinMaterial) // frosted glass background
        .presentationCornerRadius(24)               // custom corner radius
        .presentationContentInteraction(.scrolls)   // scroll before resize
        .interactiveDismissDisabled()               // prevent swipe to dismiss
}

    Dismissing Modals

    The standard approach is the dismiss environment action:

    @Environment(\.dismiss) var dismiss

Button("Done") { dismiss() }

    This works for sheets, full-screen covers, and navigation pops. Alternatively, set the boolean binding to false or the item binding to nil from the presenting view.

    Alerts

    Alerts are small modal dialogs with a title, optional message, and action buttons.

    struct AlertDemoView: View {
    @State private var showAlert = false

    var body: some View {
        Button("Delete All") {
            showAlert = true
        }
        .alert("Are you sure?", isPresented: $showAlert) {
            Button("Delete", role: .destructive) {
                deleteAll()
            }
            Button("Cancel", role: .cancel) { }
        } message: {
            Text("This action cannot be undone.")
        }
    }
}

    If you provide no buttons, SwiftUI adds a default "OK" button. The role: .cancel button is always placed appropriately by the system.

    Alert with Text Field

    Collect input directly in an alert.

@State private var newName = ""

    A confirmation dialog (action sheet on iPhone, popover on iPad) presents a list of actions.

@State private var sortOrder = "Name"

Button("Sort") { showDialog = true }

        Button("Name") { sortOrder = "Name" }

        Button("Cancel", role: .cancel) { }
    }

    Use confirmation dialogs when offering 3+ choices. For binary yes/no decisions, prefer alerts.

    Using item-based presentation (preferred when showing details for a tapped item):

            Button("Profile") { activeSheet = .profile }
            Button("Help") { activeSheet = .help }

        .sheet(item: $activeSheet) { sheet in

            case .settings: SettingsView()
            case .profile: ProfileView()
            case .help: HelpView()
            }
        }
    }
}

    Drawing and Shapes

    SwiftUI provides a rich set of built-in shapes and a Path API for drawing custom geometry. Shapes conform to the Shape protocol, which means they are views -- you can use them anywhere you would use Text or Image, and apply modifiers like .fill(), .stroke(), and .frame().

    See 13-protocols -- the Shape protocol is a great example of protocol-oriented design. You implement one method and get a full-featured view.

    Built-In Shapes

    VStack(spacing: 20) {
    Rectangle()
        .fill(.blue)
        .frame(width: 100, height: 60)

    RoundedRectangle(cornerRadius: 16)
        .fill(.green)
        .frame(width: 100, height: 60)

    Circle()
        .fill(.red)
        .frame(width: 80, height: 80)

    Ellipse()
        .fill(.orange)
        .frame(width: 120, height: 60)

    Capsule()
        .fill(.purple)
        .frame(width: 120, height: 50)
}

    RoundedRectangle also supports continuous corners via RoundedRectangle(cornerRadius: 16, style: .continuous), which matches Apple's icon corner style.

    Fill, Stroke, and StrokeBorder

    // Solid fill
Circle()
    .fill(.blue)

// Outline stroke -- the stroke line extends outside the shape's bounds
Circle()
    .stroke(.red, lineWidth: 4)

// Stroke border -- the stroke line stays inside the shape's bounds
Circle()
    .strokeBorder(.green, lineWidth: 4)

// Both fill and stroke (iOS 17+)
Circle()
    .fill(.blue)
    .stroke(.red, lineWidth: 4)

    Before iOS 17, you could not apply both .fill() and .stroke() on the same shape. The workaround was to layer two shapes in a ZStack.

    Stroke Styles

    Customize the stroke with dashes, line caps, and line joins.

    RoundedRectangle(cornerRadius: 12)
    .stroke(style: StrokeStyle(
        lineWidth: 3,
        lineCap: .round,
        lineJoin: .round,
        dash: [10, 5]  // 10pt dash, 5pt gap
    ))
    .frame(width: 200, height: 100)

    lineCap options: .butt (default), .round, .square. lineJoin options: .miter (default), .round, .bevel.

    The Shape Protocol

    Any type can become a shape by conforming to Shape and implementing path(in:).

    struct Triangle: Shape {
    func path(in rect: CGRect) -> Path {
        var path = Path()
        path.move(to: CGPoint(x: rect.midX, y: rect.minY))
        path.addLine(to: CGPoint(x: rect.maxX, y: rect.maxY))
        path.addLine(to: CGPoint(x: rect.minX, y: rect.maxY))
        path.closeSubpath()
        return path
    }
}

// Use it like any built-in shape
Triangle()
    .fill(.orange)
    .frame(width: 100, height: 100)

    The rect parameter tells you the available space. Always draw relative to rect, not absolute coordinates, so the shape scales correctly.

    Path API

    Path is the drawing primitive. It supports lines, arcs, curves, and more.

    struct StarShape: Shape {
    let points: Int
    let innerRatio: CGFloat

    func path(in rect: CGRect) -> Path {
        let center = CGPoint(x: rect.midX, y: rect.midY)
        let outerRadius = min(rect.width, rect.height) / 2
        let innerRadius = outerRadius * innerRatio
        let angleStep = .pi * 2 / CGFloat(points * 2)

        var path = Path()
        for i in 0..<(points * 2) {
            let radius = i.isMultiple(of: 2) ? outerRadius : innerRadius
            let angle = CGFloat(i) * angleStep - .pi / 2
            let point = CGPoint(
                x: center.x + cos(angle) * radius,
                y: center.y + sin(angle) * radius
            )
            if i == 0 {
                path.move(to: point)
            } else {
                path.addLine(to: point)
            }
        }
        path.closeSubpath()
        return path
    }
}

StarShape(points: 5, innerRatio: 0.4)
    .fill(.yellow)
    .frame(width: 150, height: 150)

    Key Path methods:

    path.move(to: point)                        // lift pen, move to point
path.addLine(to: point)                     // straight line
path.addArc(center:radius:startAngle:endAngle:clockwise:)  // arc
path.addQuadCurve(to:control:)              // quadratic Bezier
path.addCurve(to:control1:control2:)        // cubic Bezier
path.closeSubpath()                         // close current subpath

    Trim

    .trim(from:to:) draws only a portion of a shape's path. This is perfect for progress indicators and drawing animations.

    Circle()

    .stroke(.blue, lineWidth: 8)
    .rotationEffect(.degrees(-90))  // start from top
    .frame(width: 100, height: 100)

    InsettableShape

    To use .strokeBorder() on a custom shape, conform to InsettableShape by implementing inset(by:).

    struct RoundedTriangle: InsettableShape {
    var insetAmount: CGFloat = 0

    func path(in rect: CGRect) -> Path {
        let insetRect = rect.insetBy(dx: insetAmount, dy: insetAmount)
        var path = Path()
        path.move(to: CGPoint(x: insetRect.midX, y: insetRect.minY))
        path.addLine(to: CGPoint(x: insetRect.maxX, y: insetRect.maxY))
        path.addLine(to: CGPoint(x: insetRect.minX, y: insetRect.maxY))
        path.closeSubpath()
        return path
    }

    func inset(by amount: CGFloat) -> some InsettableShape {
        var shape = self
        shape.insetAmount += amount
        return shape
    }
}

    Colors and Materials

    Color in SwiftUI is more than just decoration -- it is a view, a shape style, and a bridge to the system's dynamic appearance. This lesson covers system colors, custom colors, gradients, materials, and how everything adapts to dark mode.

    System Colors

    SwiftUI provides named colors that adapt to the system appearance (light/dark mode, accessibility settings).

    Text("Primary").foregroundStyle(.primary)     // adapts to light/dark
Text("Secondary").foregroundStyle(.secondary) // dimmed
Text("Red").foregroundStyle(.red)
Text("Blue").foregroundStyle(.blue)
Text("Green").foregroundStyle(.green)

    Full list of named colors: .red, .orange, .yellow, .green, .mint, .teal, .cyan, .blue, .indigo, .purple, .pink, .brown, .gray, .black, .white, .clear.

    These are semantic colors -- .red is not always #FF0000. Apple adjusts the exact RGB values for readability in light and dark mode.

    Custom RGB Colors

    Color(red: 0.2, green: 0.5, blue: 0.8)           // RGB (0-1 range)
Color(red: 0.2, green: 0.5, blue: 0.8, opacity: 0.7) // with alpha
Color(hue: 0.6, saturation: 0.8, brightness: 0.9) // HSB
Color(white: 0.5)                                  // grayscale

    Asset Catalog Colors

    Define colors in your Xcode asset catalog (Assets.xcassets) for light/dark mode variants, then reference them by name.

    Color("BrandPrimary")   // looks up "BrandPrimary" in the asset catalog
Color("Background")     // separate light and dark appearances in the catalog

    This is the recommended approach for brand colors because Xcode lets you define different values for light mode, dark mode, and high contrast.

    foregroundStyle vs foregroundColor

    .foregroundStyle() is the modern API (iOS 15+). It accepts any ShapeStyle, not just Color.

    // Prefer this -- accepts colors, gradients, materials
Text("Styled")
    .foregroundStyle(.blue)

Text("Gradient text")
    .foregroundStyle(
        LinearGradient(colors: [.blue, .purple], startPoint: .leading, endPoint: .trailing)
    )

// Deprecated in favor of foregroundStyle
Text("Old way")
    .foregroundColor(.blue)

    .foregroundStyle supports up to three hierarchical levels:

    Label("Files", systemImage: "folder")
    .foregroundStyle(.blue, .blue.opacity(0.5))
// primary text uses .blue, secondary elements use .blue at 50% opacity

    ShapeStyle Protocol

    ShapeStyle is the protocol that unifies colors, gradients, and materials. Anything that conforms can be used with .foregroundStyle(), .fill(), .background(), and similar modifiers.

    Conforming types include: Color, LinearGradient, RadialGradient, AngularGradient, EllipticalGradient, Material, HierarchicalShapeStyle (.primary, .secondary, .tertiary, .quaternary).

    Linear Gradient

    A gradient that transitions between colors along a line.

    colors: [.blue, .purple],
    startPoint: .topLeading,
    endPoint: .bottomTrailing
)

        .init(color: .red, location: 0.0),

        .init(color: .green, location: 1.0),
    ],
    startPoint: .top,
    endPoint: .bottom
)

    Colors radiate outward from a center point.

    colors: [.yellow, .orange, .red],

    startRadius: 0,

    Animation Basics

    Animation in SwiftUI works by interpolating between old and new values when state changes. You do not write frame-by-frame logic. Instead, you change state, and SwiftUI smoothly transitions every affected property from its old value to its new value.

    See 10-structs-vs-classes -- views are structs. When state changes, SwiftUI creates a new struct, diffs it against the old one, and animates the differences.

    Implicit Animations

    Attach .animation() to a view. Whenever the specified value changes, all animatable properties on that view transition smoothly.

    struct ImplicitAnimationView: View {
    @State private var scale: CGFloat = 1.0

    var body: some View {
        Circle()
            .fill(.blue)
            .frame(width: 100, height: 100)
            .scaleEffect(scale)
            .animation(.easeInOut(duration: 0.5), value: scale)
            .onTapGesture {
                scale = scale == 1.0 ? 1.5 : 1.0
            }
    }
}

    The value: parameter tells SwiftUI which state change triggers the animation. Always provide it -- the old form without value: is deprecated.

    Explicit Animations

    Wrap the state change in withAnimation. This animates every view that depends on the changed state.

    struct ExplicitAnimationView: View {
    @State private var isExpanded = false

    var body: some View {
        VStack {
            RoundedRectangle(cornerRadius: 20)
                .fill(.green)
                .frame(width: isExpanded ? 300 : 100, height: isExpanded ? 300 : 100)

            Button("Toggle") {
                withAnimation(.spring()) {
                    isExpanded.toggle()
                }
            }
        }
    }
}

    Implicit vs explicit: Use implicit when a single view should always animate. Use explicit when a state change should animate multiple views simultaneously.

    Animation Curves

    Built-in timing curves control the speed profile of an animation.

    .animation(.linear(duration: 0.5), value: x)         // constant speed
.animation(.easeIn(duration: 0.5), value: x)          // starts slow, ends fast
.animation(.easeOut(duration: 0.5), value: x)         // starts fast, ends slow
.animation(.easeInOut(duration: 0.5), value: x)       // slow-fast-slow

    Spring Animations

    Springs feel natural because they overshoot and settle. They are the recommended default for interactive UI.

    // Basic spring
withAnimation(.spring()) {
    position = newPosition
}

// Customized spring
withAnimation(.spring(response: 0.4, dampingFraction: 0.6)) {
    position = newPosition
}

    Parameters: - response -- duration of one oscillation cycle (lower = faster) - dampingFraction -- how quickly oscillation stops (0 = infinite bounce, 1 = no bounce) - blendDuration -- how the spring blends with existing animations

    Spring Presets (iOS 17+)

    Convenient spring configurations:

    withAnimation(.bouncy) { ... }    // playful overshoot (dampingFraction ~0.6)
withAnimation(.smooth) { ... }    // no bounce, fluid motion (dampingFraction ~1.0)
withAnimation(.snappy) { ... }    // quick, minimal bounce (dampingFraction ~0.85)

// Customize presets
withAnimation(.bouncy(duration: 0.4)) { ... }
withAnimation(.smooth(duration: 0.3)) { ... }

    Animation Modifiers

    Chain modifiers to customize timing.

    // Delay before starting
withAnimation(.easeInOut(duration: 0.5).delay(0.3)) {
    showContent = true
}

// Repeat a fixed number of times
withAnimation(.linear(duration: 0.5).repeatCount(3, autoreverses: true)) {
    pulseScale = 1.2
}

// Repeat forever
withAnimation(.easeInOut(duration: 1.0).repeatForever(autoreverses: true)) {
    isGlowing = true
}

// Speed multiplier
withAnimation(.spring().speed(2.0)) {
    isExpanded.toggle()
}

    What Can Be Animated

    SwiftUI animates any property that conforms to the Animatable protocol. Built-in animatable properties include:

    .frame(width:height:)    // size changes
.offset(x:y:)            // position changes
.opacity()               // fade in/out
.scaleEffect()           // grow/shrink
.rotationEffect()        // rotation
.foregroundStyle()       // color changes
.background()            // background color
.clipShape()             // shape morphing
.trim(from:to:)          // drawing progress
.padding()               // spacing changes

    The Animatable Protocol

    For custom types, conform to Animatable to tell SwiftUI how to interpolate values.

    struct AnimatableCircle: Shape, Animatable {
    var progress: CGFloat

    var animatableData: CGFloat {
        get { progress }
        set { progress = newValue }
    }

    func path(in rect: CGRect) -> Path {
        let radius = min(rect.width, rect.height) / 2 * progress
        let center = CGPoint(x: rect.midX, y: rect.midY)
        return Path { p in
            p.addArc(center: center, radius: radius,
                     startAngle: .zero, endAngle: .degrees(360),
                     clockwise: false)
        }
    }
}

    animatableData tells SwiftUI which property to interpolate. For two values, use AnimatablePair:

    var animatableData: AnimatablePair<CGFloat, CGFloat> {
    get { AnimatablePair(width, height) }
    set {
        width = newValue.first
        height = newValue.second
    }
}

    Transitions

    Transitions define how a view enters and exits the view hierarchy. They only fire when a view is inserted or removed (controlled by if, switch, or conditional ForEach).

    struct TransitionView: View {
    @State private var showDetail = false

    var body: some View {
        VStack {
            Button("Toggle") {
                withAnimation(.spring()) {
                    showDetail.toggle()
                }
            }

            if showDetail {
                Text("Detail Content")
                    .padding()
                    .background(.blue)
                    .clipShape(RoundedRectangle(cornerRadius: 12))
                    .transition(.slide)
            }
        }
    }
}

    Built-in transitions:

    .transition(.opacity)              // fade
.transition(.slide)                // slide from leading edge
.transition(.scale)                // scale from center
.transition(.move(edge: .bottom))  // slide from specific edge
.transition(.push(from: .bottom))  // push with depth (iOS 16+)

    Combine and customize:

    // Different animations for insert and remove
.transition(.asymmetric(
    insertion: .scale.combined(with: .opacity),
    removal: .move(edge: .trailing).combined(with: .opacity)
))

    matchedGeometryEffect

    Create hero animations where a view appears to morph between two positions.

    struct HeroAnimationView: View {
    @Namespace private var animation
    @State private var isExpanded = false

    var body: some View {
        VStack {
            if isExpanded {
                RoundedRectangle(cornerRadius: 20)
                    .fill(.blue)
                    .matchedGeometryEffect(id: "card", in: animation)
                    .frame(width: 300, height: 400)
            } else {
                RoundedRectangle(cornerRadius: 10)
                    .fill(.blue)
                    .matchedGeometryEffect(id: "card", in: animation)
                    .frame(width: 100, height: 100)
            }
        }
        .onTapGesture {
            withAnimation(.spring()) {
                isExpanded.toggle()
            }
        }
    }
}

    Two views with the same id and namespace are treated as the same element -- SwiftUI interpolates position, size, and shape between them.

    Common Mistakes

    1. Animation not working: The .animation() modifier must come after the modifiers that change. And state must actually change for animation to trigger.

    2. Animating the wrong value: Make sure the value: parameter in .animation() matches what actually changes.

    3. Overusing implicit animations: When .animation() is placed too high in the view tree, it can animate unintended properties. Prefer explicit withAnimation for complex state changes.

    PhaseAnimator

    PhaseAnimator cycles through a sequence of phases, applying different view modifications at each step. The animation between phases is automatic.

    enum BouncePhase: CaseIterable {
    case initial, compress, stretch, overshoot, settle

    var scaleY: CGFloat {
        switch self {
        case .initial:   1.0
        case .compress:  0.7
        case .stretch:   1.3
        case .overshoot: 1.1
        case .settle:    1.0
        }
    }

    var offsetY: CGFloat {
        switch self {
        case .initial:   0
        case .compress:  20
        case .stretch:   -30
        case .overshoot: -10
        case .settle:    0
        }
    }
}

struct BounceView: View {
    var body: some View {
        PhaseAnimator(BouncePhase.allCases) { phase in
            Circle()
                .fill(.blue)
                .frame(width: 80, height: 80)
                .scaleEffect(y: phase.scaleY)
                .offset(y: phase.offsetY)
        } animation: { phase in
            switch phase {
            case .initial:   .spring(duration: 0.3)
            case .compress:  .easeIn(duration: 0.15)
            case .stretch:   .spring(response: 0.3, dampingFraction: 0.4)
            case .overshoot: .spring(response: 0.2, dampingFraction: 0.6)
            case .settle:    .smooth(duration: 0.3)
            }
        }
    }
}

    PhaseAnimator loops continuously by default. To trigger on demand, provide a trigger value:

    @State private var triggerCount = 0

PhaseAnimator(BouncePhase.allCases, trigger: triggerCount) { phase in
    // content
} animation: { _ in .spring }

Button("Bounce") { triggerCount += 1 }

    See 13-protocols -- CaseIterable conformance provides the allCases array used here.

    KeyframeAnimator

    KeyframeAnimator gives you timeline-level control. You define keyframe tracks for individual properties, each with precise timing.

    struct ShakeValues {
    var offset: CGFloat = 0
    var scale: CGFloat = 1
    var rotation: Double = 0
}

struct ShakeView: View {
    @State private var trigger = 0

    var body: some View {
        Text("Shake Me")
            .font(.largeTitle)
            .keyframeAnimator(
                initialValue: ShakeValues(),
                trigger: trigger
            ) { content, value in
                content
                    .offset(x: value.offset)
                    .scaleEffect(value.scale)
                    .rotationEffect(.degrees(value.rotation))
            } keyframes: { _ in
                KeyframeTrack(\.offset) {
                    SpringKeyframe(20, duration: 0.1)
                    SpringKeyframe(-20, duration: 0.1)
                    SpringKeyframe(10, duration: 0.1)
                    SpringKeyframe(-10, duration: 0.1)
                    SpringKeyframe(0, duration: 0.15)
                }
                KeyframeTrack(\.scale) {
                    CubicKeyframe(1.1, duration: 0.1)
                    CubicKeyframe(0.95, duration: 0.15)
                    CubicKeyframe(1.0, duration: 0.2)
                }
                KeyframeTrack(\.rotation) {
                    LinearKeyframe(3, duration: 0.1)
                    LinearKeyframe(-3, duration: 0.1)
                    LinearKeyframe(2, duration: 0.1)
                    LinearKeyframe(0, duration: 0.15)
                }
            }
            .onTapGesture { trigger += 1 }
    }
}

    Keyframe Types

    Type Behavior
    LinearKeyframe Constant speed between keyframes
    SpringKeyframe Spring physics interpolation
    CubicKeyframe Smooth cubic Bezier curve
    MoveKeyframe Jump immediately (no interpolation)

    CustomAnimation Protocol

    Build your own animation curves by conforming to CustomAnimation.

    struct ElasticAnimation: CustomAnimation {
    let duration: TimeInterval
    let bounces: Int

    func animate<V: VectorArithmetic>(
        value: V, time: Double, context: inout AnimationContext<V>
    ) -> V? {
        guard time < duration else { return nil } // nil means done

        let progress = time / duration
        let decay = pow(1 - progress, 2)
        let oscillation = sin(Double(bounces) * .pi * progress)
        let factor = 1 + decay * oscillation * 0.3

        return value.scaled(by: factor)
    }

    func shouldMerge<V: VectorArithmetic>(
        previous: Animation, value: V, time: Double,
        context: inout AnimationContext<V>
    ) -> Bool {
        false
    }
}

extension Animation {
    static func elastic(duration: TimeInterval = 0.8, bounces: Int = 4) -> Animation {
        Animation(ElasticAnimation(duration: duration, bounces: bounces))
    }
}

// Usage
withAnimation(.elastic()) {
    scale = 1.5
}

    Transaction Control

    Transactions let you override or suppress animations for specific state changes.

    // Suppress animation for a specific change
withTransaction(\.animation, nil) {
    position = newPosition  // no animation
}

// Override animation on a specific view
Text("Hello")
    .transaction { transaction in
        transaction.animation = nil  // never animate this view
    }

// Disable animation for child views
VStack { ... }
    .transaction { $0.animation = nil }

    Transactions propagate down the view hierarchy. A child can override its parent's transaction.

    Gestures

    SwiftUI provides a rich gesture system that lets users interact with your app through taps, drags, pinches, and rotations. Gestures in SwiftUI are structs that conform to the Gesture protocol, making them composable and type-safe.

    See 10-structs-vs-classes -- gestures are value types (structs) conforming to the Gesture protocol.

    See 12-animation-basics -- animating gesture-driven values for fluid interactions.

    Tap Gesture

    The simplest gesture. Use .onTapGesture for single taps, or pass a count for multi-tap recognition.

    struct TapExample: View {
    @State private var tapped = false
    @State private var doubleTapped = false

    var body: some View {
        VStack(spacing: 30) {
            Circle()
                .fill(tapped ? .green : .blue)
                .frame(width: 100, height: 100)
                .onTapGesture {
                    tapped.toggle()
                }

            Circle()
                .fill(doubleTapped ? .red : .gray)
                .frame(width: 100, height: 100)
                .onTapGesture(count: 2) {
                    doubleTapped.toggle()
                }
        }
    }
}

    Long Press Gesture

    Recognizes a press-and-hold. The minimumDuration parameter controls how long the user must press before the gesture fires.

    struct LongPressExample: View {
    @State private var isPressed = false

    var body: some View {
        Circle()
            .fill(isPressed ? .red : .blue)
            .frame(width: 100, height: 100)
            .onLongPressGesture(minimumDuration: 0.8) {
                isPressed.toggle()
            }
    }
}

    Drag Gesture

    DragGesture tracks finger movement and provides translation (how far the finger moved) and predictedEndTranslation (where inertia would carry it). This is the foundation for swipe-to-dismiss, reordering, and custom sliders.

    struct DraggableCard: View {
    @State private var offset = CGSize.zero
    @State private var lastOffset = CGSize.zero

    var body: some View {
        RoundedRectangle(cornerRadius: 16)
            .fill(.blue.gradient)
            .frame(width: 200, height: 120)
            .offset(offset)
            .gesture(
                DragGesture()
                    .onChanged { value in
                        offset = CGSize(
                            width: lastOffset.width + value.translation.width,
                            height: lastOffset.height + value.translation.height
                        )
                    }
                    .onEnded { value in
                        lastOffset = offset
                    }
            )
    }
}

    The value provided to .onChanged and .onEnded includes startLocation, location, translation, predictedEndLocation, and predictedEndTranslation.

    Magnify Gesture (Pinch to Zoom)

    iOS 17 renamed MagnificationGesture to MagnifyGesture. It reports a scale factor relative to the starting size (1.0 = no change).

    struct PinchToZoom: View {
    @State private var scale: CGFloat = 1.0
    @State private var lastScale: CGFloat = 1.0

    var body: some View {
        Image(systemName: "photo")
            .resizable()
            .scaledToFit()
            .frame(width: 200, height: 200)
            .scaleEffect(scale)
            .gesture(
                MagnifyGesture()
                    .onChanged { value in
                        scale = lastScale * value.magnification
                    }
                    .onEnded { value in
                        lastScale = scale
                    }
            )
    }
}

    Rotate Gesture

    iOS 17 renamed RotationGesture to RotateGesture. It reports the rotation angle.

    struct RotateExample: View {
    @State private var angle: Angle = .zero
    @State private var lastAngle: Angle = .zero

    var body: some View {
        Rectangle()
            .fill(.orange.gradient)
            .frame(width: 150, height: 150)
            .rotationEffect(angle)
            .gesture(
                RotateGesture()
                    .onChanged { value in
                        angle = lastAngle + value.rotation
                    }
                    .onEnded { value in
                        lastAngle = angle
                    }
            )
    }
}

    @GestureState vs @State

    @GestureState automatically resets to its initial value when the gesture ends. This is perfect for transient effects like highlighting during a press or showing a drag preview. Use @State when you need the value to persist after the gesture.

    struct GestureStateExample: View {
    // Resets to false when long press ends
    @GestureState private var isPressing = false

    var body: some View {
        Circle()
            .fill(isPressing ? .yellow : .gray)
            .frame(width: 100, height: 100)
            .gesture(
                LongPressGesture(minimumDuration: 1.0)
                    .updating($isPressing) { value, state, transaction in
                        state = value  // state is an inout binding to isPressing
                    }
                    .onEnded { _ in
                        print("Long press completed")
                    }
            )
    }
}

    The .updating closure receives three parameters: the current value of the gesture, an inout state (your @GestureState variable), and a transaction that controls animation.

    Composing Gestures

    SwiftUI provides three composition operators for combining gestures.

    Simultaneously -- both gestures are recognized at the same time:

    let pinchAndRotate = MagnifyGesture()
    .simultaneously(with: RotateGesture())

    Sequenced -- the second gesture only activates after the first succeeds:

    let longPressThenDrag = LongPressGesture(minimumDuration: 0.5)
    .sequenced(before: DragGesture())

    Exclusively -- only one gesture can be recognized (first match wins):

    let tapOrLongPress = TapGesture()
    .exclusively(before: LongPressGesture())

    Gesture Priority

    When a parent and child both have gestures, SwiftUI must decide which takes precedence.

    VStack {
    Circle()
        .frame(width: 100, height: 100)
        .onTapGesture { print("Circle tapped") }
}
// .gesture()               -- child gesture wins (default)
// .highPriorityGesture()   -- parent gesture wins
// .simultaneousGesture()   -- both gestures fire
.highPriorityGesture(
    TapGesture().onEnded { print("VStack tapped") }
)

    Real-World Example: Swipe to Dismiss

    A card that can be swiped down to dismiss, with spring animation and opacity fade.

    struct SwipeToDismissCard: View {
    @GestureState private var dragOffset: CGFloat = 0
    @State private var dismissed = false

    var body: some View {
        if !dismissed {
            RoundedRectangle(cornerRadius: 20)
                .fill(.blue.gradient)
                .frame(width: 300, height: 200)
                .overlay(Text("Swipe down to dismiss").foregroundStyle(.white))
                .offset(y: dragOffset)
                .opacity(1 - Double(abs(dragOffset)) / 300)
                .gesture(
                    DragGesture()
                        .updating($dragOffset) { value, state, _ in
                            state = value.translation.height
                        }
                        .onEnded { value in
                            if value.translation.height > 150 {
                                withAnimation(.spring()) {
                                    dismissed = true
                                }
                            }
                        }
                )
                .animation(.spring(), value: dragOffset)
        }
    }
}

    Observable and Environment

    SwiftUI's data observation system has evolved significantly. The modern approach, introduced in iOS 17, uses the @Observable macro to replace the older ObservableObject protocol. Understanding both systems is essential since most codebases still contain legacy patterns.

    See 24-property-wrappers -- @Observable is a macro that generates property wrapper-like observation code at compile time.

    See 18-memory-management -- @Observable types are classes; watch for retain cycles when closures capture them.

    The Evolution of Observation

    Era Model View Binding Injection
    iOS 13-16 ObservableObject + @Published @StateObject, @ObservedObject @EnvironmentObject
    iOS 17+ @Observable @State, direct reference @Environment

    The modern system is simpler: fewer property wrappers, less boilerplate, and more efficient re-rendering (only properties actually read by a view trigger updates).

    @Observable Macro (iOS 17+)

    Mark a class with @Observable and its properties are automatically tracked. No @Published needed.

    import Observation

@Observable
class UserProfile {
    var name = "Anonymous"
    var age = 0
    var isLoggedIn = false
}

    Use it in a view by simply referencing the object. SwiftUI automatically detects which properties you read and only re-renders when those specific properties change.

    struct ProfileView: View {
    var profile: UserProfile  // no property wrapper needed to read

    var body: some View {
        VStack {
            Text(profile.name)  // view re-renders only when name changes
            Text("Age: \(profile.age)")
        }
    }
}

    @State with @Observable

    When a view owns an @Observable object (creates and manages its lifecycle), use @State.

    struct ContentView: View {
    @State private var profile = UserProfile()

    var body: some View {
        VStack {
            Text(profile.name)
            Button("Change Name") {
                profile.name = "Alice"
            }
            // Pass to child views
            ProfileEditor(profile: profile)
        }
    }
}

    @Bindable

    When you need two-way bindings to an @Observable object's properties (for TextField, Toggle, etc.), wrap the reference with @Bindable.

    struct ProfileEditor: View {
    @Bindable var profile: UserProfile

    var body: some View {
        Form {
            TextField("Name", text: $profile.name)
            Stepper("Age: \(profile.age)", value: $profile.age)
            Toggle("Logged In", isOn: $profile.isLoggedIn)
        }
    }
}

    You can also create a local @Bindable inside body if you only need bindings temporarily:

    var body: some View {
    @Bindable var profile = profile
    TextField("Name", text: $profile.name)
}

    @Environment(\.dismiss) private var dismiss

    Legacy: ObservableObject Pattern (iOS 13-16)

    Understanding the legacy pattern is important for maintaining older codebases.

    // Model
class LegacyViewModel: ObservableObject {
    @Published var count = 0        // must mark each property with @Published
    var notObserved = "ignored"     // changes here don't trigger updates
}

// View that owns the object (survives re-render)
struct OwnerView: View {
    @StateObject private var vm = LegacyViewModel()

    var body: some View {
        ChildView(vm: vm)
    }
}

// View that borrows the object (does NOT own lifecycle)
struct ChildView: View {
    @ObservedObject var vm: LegacyViewModel

    var body: some View {
        Text("\(vm.count)")
    }
}

// Injection through environment
struct LegacyApp: App {
    @StateObject private var vm = LegacyViewModel()

    var body: some Scene {
        WindowGroup {
            ContentView()
                .environmentObject(vm)
        }
    }
}

struct SomeChild: View {
    @EnvironmentObject var vm: LegacyViewModel
    var body: some View { Text("\(vm.count)") }
}

    The key problem with the legacy system: every @Published property change re-renders every view observing the object, even if that view only reads unrelated properties. @Observable fixes this with property-level tracking.

    Data Flow

    SwiftUI's declarative paradigm relies on a strict data flow model: state drives the UI, and user actions modify state, which triggers re-rendering. Understanding where to place state, how to share it, and how to structure communication between views is the key to writing maintainable SwiftUI apps.

    See 05-state-and-binding -- fundamentals of @State and @Binding.

    See 15-observable-and-environment -- @Observable, @Environment, and observation mechanics.

    See 19-value-semantics -- SwiftUI's state management relies heavily on value semantics to detect changes.

    The Golden Rule: Single Source of Truth

    Every piece of state should have exactly one owner. Other views either receive a read-only copy or a binding to that state. When state is duplicated, views go out of sync.

    Owner View (@State / @Observable)
  |
  +-- Child A (read via init parameter)
  |
  +-- Child B (write via @Binding)
  |
  +-- Deep Child (read via @Environment)

    Data Flowing Down

    There are three ways to pass data from parent to child.

    Init parameters -- simple, explicit, best for most cases:

    struct ParentView: View {
    @State private var username = "Alice"

    var body: some View {
        // Read-only: pass the value
        GreetingView(name: username)

        // Read-write: pass a binding
        NameEditor(name: $username)
    }
}

struct GreetingView: View {
    let name: String
    var body: some View { Text("Hello, \(name)") }
}

struct NameEditor: View {
    @Binding var name: String
    var body: some View { TextField("Name", text: $name) }
}

    Environment -- for values needed by many descendants without threading through every intermediate view:

    struct AppRoot: View {
    @State private var settings = AppSettings()

    var body: some View {
        NavigationStack {
            HomeView()
        }
        .environment(settings)
    }
}

// Any descendant can access it directly
struct DeeplyNestedView: View {
    @Environment(AppSettings.self) private var settings
    var body: some View { Text("Font: \(settings.fontSize)") }
}

    Actions Flowing Up

    Children communicate upward through closures and bindings, never by reaching into parent state directly.

    Closures -- child fires an action, parent decides what to do:

    struct ItemRow: View {
    let item: Item
    let onDelete: () -> Void

    var body: some View {
        HStack {
            Text(item.name)
            Spacer()
            Button("Delete", role: .destructive, action: onDelete)
        }
    }
}

struct ItemList: View {
    @State private var items = Item.samples

    var body: some View {
        ForEach(items) { item in
            ItemRow(item: item) {
                items.removeAll { $0.id == item.id }
            }
        }
    }
}

    Bindings -- child modifies parent state directly through a two-way connection:

    struct FilterBar: View {
    @Binding var searchText: String
    @Binding var showFavorites: Bool

    var body: some View {
        HStack {
            TextField("Search", text: $searchText)
            Toggle("Favorites", isOn: $showFavorites)
        }
    }
}

    When to Use @State

    Use @State for simple, local, view-owned values. If the view is destroyed, the state should logically disappear too.

    Good uses: toggle booleans, sheet presentation flags, text field input, animation state, temporary form values.

    struct LoginForm: View {
    @State private var email = ""
    @State private var password = ""
    @State private var showingError = false

    var body: some View {
        Form {
            TextField("Email", text: $email)
            SecureField("Password", text: $password)
            Button("Log In") { attemptLogin() }
        }
        .alert("Invalid credentials", isPresented: $showingError) { }
    }

    private func attemptLogin() { /* ... */ }
}

    The ViewModel Pattern with @Observable

    For complex business logic, extract state into an @Observable class. Mark it @MainActor to ensure all property changes happen on the main thread.

    @Observable
@MainActor
class BookListViewModel {
    var books: [Book] = []
    var isLoading = false
    var errorMessage: String?

    private let repository: BookRepository

    init(repository: BookRepository) {
        self.repository = repository
    }

    func loadBooks() async {
        isLoading = true
        defer { isLoading = false }

        do {
            books = try await repository.fetchAll()
        } catch {
            errorMessage = error.localizedDescription
        }
    }

    func delete(_ book: Book) async {
        books.removeAll { $0.id == book.id }
        try? await repository.delete(book)
    }
}

struct BookListView: View {
    @State private var viewModel: BookListViewModel

    init(repository: BookRepository) {
        _viewModel = State(initialValue: BookListViewModel(repository: repository))
    }

    var body: some View {
        List(viewModel.books) { book in
            Text(book.title)
                .swipeActions {
                    Button("Delete", role: .destructive) {
                        Task { await viewModel.delete(book) }
                    }
                }
        }
        .overlay { if viewModel.isLoading { ProgressView() } }
        .task { await viewModel.loadBooks() }
    }
}

    Performance: Efficient Re-Rendering

    With @Observable, SwiftUI tracks which properties each view body actually reads. Only views that read a changed property are re-rendered.

    @Observable
class Model {
    var name = ""     // only NameView re-renders when this changes
    var score = 0     // only ScoreView re-renders when this changes
}

struct NameView: View {
    var model: Model
    var body: some View { Text(model.name) }
}

struct ScoreView: View {
    var model: Model
    var body: some View { Text("Score: \(model.score)") }
}

    This is a major improvement over ObservableObject, where any @Published change re-rendered every observing view.

    Grids and Adaptive Layout

    SwiftUI offers multiple grid systems: LazyVGrid and LazyHGrid for scrollable grids, Grid for precise alignment-based layouts, and layout protocols like ViewThatFits and AnyLayout for adaptive design across device sizes.

    LazyVGrid: Vertical Scrolling Grid

    LazyVGrid arranges items in a vertically scrolling grid. It only creates views that are currently visible, making it efficient for large data sets.

    struct PhotoGrid: View {
    let photos = (1...50).map { "photo_\($0)" }

    let columns = [
        GridItem(.flexible()),
        GridItem(.flexible()),
        GridItem(.flexible())
    ]

    var body: some View {
        ScrollView {
            LazyVGrid(columns: columns, spacing: 8) {
                ForEach(photos, id: \.self) { photo in
                    RoundedRectangle(cornerRadius: 8)
                        .fill(.blue.gradient)
                        .aspectRatio(1, contentMode: .fit)
                        .overlay(Text(photo).font(.caption))
                }
            }
            .padding()
        }
    }
}

    LazyHGrid: Horizontal Scrolling Grid

    Same concept but scrolls horizontally. Define rows instead of columns.

    struct HorizontalGrid: View {
    let rows = [
        GridItem(.fixed(80)),
        GridItem(.fixed(80))

    var body: some View {
        ScrollView(.horizontal) {
            LazyHGrid(rows: rows, spacing: 10) {

    GridItem Types

    GridItem controls how columns (or rows) are sized. There are three types.

    Fixed -- exactly the specified width:

    GridItem(.fixed(100))  // always 100 points wide

    Flexible -- fills available space, respecting min/max constraints:

    GridItem(.flexible(minimum: 80, maximum: 200))  // stretches to fill

    Adaptive -- fits as many items as possible within the available space:

    // Each item is at least 80pt wide; SwiftUI decides how many fit per row
GridItem(.adaptive(minimum: 80))

    Adaptive is especially powerful for responsive layouts. One GridItem(.adaptive(minimum: 100)) replaces the entire column array and automatically adjusts the column count based on screen width.

    struct AdaptiveGrid: View {
    let columns = [GridItem(.adaptive(minimum: 100))]

    var body: some View {
        ScrollView {
            LazyVGrid(columns: columns, spacing: 12) {
                ForEach(0..<20, id: \.self) { _ in
                    RoundedRectangle(cornerRadius: 8)
                        .fill(.orange.gradient)
                        .frame(height: 100)
                }
            }
            .padding()
        }
    }
}

    Custom Layout Protocol

    For complete control, conform to the Layout protocol. You implement two methods: sizeThatFits (propose a size) and placeSubviews (position children).

    struct RadialLayout: Layout {
    func sizeThatFits(proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) -> CGSize {
        proposal.replacingUnspecifiedDimensions()
    }

    func placeSubviews(in bounds: CGRect, proposal: ProposedViewSize, subviews: Subviews, cache: inout ()) {
        let radius = min(bounds.width, bounds.height) / 2.5
        let angle = Angle.degrees(360.0 / Double(subviews.count)).radians

        for (index, subview) in subviews.enumerated() {
            let x = bounds.midX + radius * cos(angle * Double(index) - .pi / 2)
            let y = bounds.midY + radius * sin(angle * Double(index) - .pi / 2)
            subview.place(at: CGPoint(x: x, y: y), anchor: .center, proposal: .unspecified)
        }
    }
}

// Usage
struct RadialDemo: View {
    var body: some View {
        RadialLayout {
            ForEach(0..<8, id: \.self) { i in
                Circle()
                    .fill(Color(hue: Double(i) / 8, saturation: 0.8, brightness: 0.9))
                    .frame(width: 40, height: 40)
            }
        }
        .frame(width: 300, height: 300)

    Lazy Scrolling Pattern

    Content inside ScrollView is NOT lazy. For large data sets, wrap content in LazyVStack or LazyHStack so views are only created when they scroll into the visible area. This is critical for performance.

    struct LazyScrollExample: View {
    var body: some View {
        ScrollView {
            LazyVStack(spacing: 8) {
                ForEach(0..<10_000, id: \.self) { index in
                    Text("Row \(index)")
                        .frame(maxWidth: .infinity, alignment: .leading)
                        .padding()
                        .background(.fill, in: RoundedRectangle(cornerRadius: 8))
                }
            }
            .padding()
        }
    }
}

    Without LazyVStack, SwiftUI would create all 10,000 views at once, causing a massive memory spike and launch delay.

    ScrollViewReader: Programmatic Scrolling

    Wrap your ScrollView in a ScrollViewReader to get a ScrollViewProxy that can scroll to any view by its .id().

    struct ChatView: View {
    @State private var messages = Message.samples
    @State private var newMessage = ""

    var body: some View {
        VStack {
            ScrollViewReader { proxy in
                ScrollView {
                    LazyVStack(alignment: .leading, spacing: 8) {
                        ForEach(messages) { message in
                            MessageBubble(message: message)
                                .id(message.id)
                        }
                    }
                    .padding()
                }
                .onChange(of: messages.count) {
                    // Scroll to the latest message
                    if let lastID = messages.last?.id {
                        withAnimation {
                            proxy.scrollTo(lastID, anchor: .bottom)
                        }
                    }
                }
            }

            HStack {
                TextField("Message", text: $newMessage)
                Button("Send") {
                    messages.append(Message(text: newMessage))
                    newMessage = ""
                }
            }
            .padding()
        }
    }
}

    The anchor parameter controls which part of the target view aligns with the visible area: .top, .center, .bottom, .leading, .trailing.

    Search and Filtering

    Search is one of the most common interactions in any app. SwiftUI provides the .searchable modifier that adds a platform-native search bar to navigation containers. Combined with Swift's collection filtering, you can build powerful, responsive search experiences with very little code.

    See 23-functional-programming for filter, map, and other higher-order functions used to filter data. See 03-strings for string comparison techniques like localizedCaseInsensitiveContains. See 07-lists-and-foreach for the List and ForEach foundations that display filtered results.

    Adding a Search Bar

    The .searchable modifier binds a search bar to a @State string. It must be placed inside a NavigationStack (or NavigationSplitView) to appear in the navigation bar.

    struct ContactListView: View {
    let contacts = ["Alice", "Bob", "Charlie", "Diana", "Eve"]
    @State private var searchText = ""

    var body: some View {
        NavigationStack {
            List(filteredContacts, id: \.self) { contact in
                Text(contact)
            }
            .navigationTitle("Contacts")
            .searchable(text: $searchText)
        }
    }

    var filteredContacts: [String] {
        if searchText.isEmpty {
            return contacts
        }
        return contacts.filter {
            $0.localizedCaseInsensitiveContains(searchText)
        }
    }
}

    The filteredContacts computed property re-evaluates every time searchText changes, and SwiftUI updates the list automatically.

    Search Bar Placement

    You can control where the search bar appears using the placement parameter.

    .searchable(text: $searchText, placement: .navigationBarDrawer(displayMode: .always))

    Available placements include: - .automatic — the system decides (default) - .navigationBarDrawer — below the navigation title (iOS) - .navigationBarDrawer(displayMode: .always) — always visible, not hidden by scrolling - .sidebar — in the sidebar (macOS / iPadOS split views) - .toolbar — in the toolbar area

    Search Suggestions

    Suggestions appear below the search bar as the user types. Use the .searchSuggestions modifier and mark each suggestion with .searchCompletion so tapping it fills the search field.

    struct SuggestingSearchView: View {
    @State private var searchText = ""
    let allFruits = ["Apple", "Apricot", "Banana", "Blueberry", "Cherry", "Date"]

    var suggestions: [String] {
        guard !searchText.isEmpty else { return [] }
        return allFruits.filter {
            $0.localizedCaseInsensitiveContains(searchText)
        }
    }

    var body: some View {
        NavigationStack {
            List(allFruits, id: \.self) { Text($0) }
                .navigationTitle("Fruits")
                .searchable(text: $searchText)
                .searchSuggestions {
                    ForEach(suggestions, id: \.self) { suggestion in
                        Text(suggestion)
                            .searchCompletion(suggestion)
                    }
                }
        }
    }
}

    Suggestions can include images and labels — they are full SwiftUI views. The .searchCompletion value is what gets placed into searchText when the user taps the suggestion.

    Search Scopes

    Scopes let users narrow the search domain. They appear as a segmented control beneath the search bar.

    enum SearchScope: String, CaseIterable {
    case all = "All"
    case favorites = "Favorites"
    case recent = "Recent"
}

struct ScopedSearchView: View {
    @State private var searchText = ""
    @State private var scope: SearchScope = .all

    var body: some View {
        NavigationStack {
            List { /* filtered content */ }
                .searchable(text: $searchText)
                .searchScopes($scope) {
                    ForEach(SearchScope.allCases, id: \.self) { scope in
                        Text(scope.rawValue).tag(scope)
                    }
                }
        }
    }
}

    The scope binding updates as the user selects a segment. Use it alongside searchText in your filtering logic.

    Search Tokens

    Tokens represent structured, discrete filters (like tags or categories). They appear as capsules in the search field.

    struct RecipeSearchView: View {
    @State private var searchText = ""
    @State private var tokens: [Cuisine] = []

    struct Cuisine: Identifiable, Hashable {
        let id = UUID()
        let name: String
    }

    let suggestedTokens = [Cuisine(name: "Italian"), Cuisine(name: "Japanese")]

    var body: some View {
        NavigationStack {
            List { /* filtered results */ }
                .searchable(text: $searchText, tokens: $tokens) { token in
                    Text(token.name)
                }
                .searchSuggestions {
                    ForEach(suggestedTokens) { token in
                        Text(token.name).searchCompletion(token)
                    }
                }
        }
    }
}

    When the user taps a token suggestion, SwiftUI appends it to the tokens array rather than filling searchText.

    Submitting Search

    React to the user pressing the search button on the keyboard with .onSubmit(of: .search).

    .searchable(text: $searchText)
.onSubmit(of: .search) {
    performSearch()
}

func performSearch() {
    // e.g., call an API, log analytics, etc.
    print("User searched for: \(searchText)")
}

    This is useful when you want to trigger a network request only on explicit submit rather than on every keystroke.

    Debouncing Search

    For expensive operations (network calls, large dataset filtering), debounce by using .task(id:) with a short delay.

    struct DebouncedSearchView: View {
    @State private var searchText = ""
    @State private var results: [String] = []

    var body: some View {
        NavigationStack {
            List(results, id: \.self) { Text($0) }
                .navigationTitle("Search")
                .searchable(text: $searchText)
                .task(id: searchText) {
                    // Wait 300ms before searching — if searchText changes,
                    // this task is cancelled and a new one starts.
                    try? await Task.sleep(for: .milliseconds(300))

                    // Check cancellation after the sleep
                    guard !Task.isCancelled else { return }
                    results = await fetchResults(for: searchText)
                }
        }
    }

    func fetchResults(for query: String) async -> [String] {
        // Simulate network call
        return ["Result for \(query)"]
    }
}

    Because .task(id:) cancels the previous task when the id changes, rapid typing only triggers one actual search after the user pauses.

    Filtering with Sections

    When your list uses sections, filter both the items and the sections themselves.

    struct SectionedContact {
    let letter: String
    let names: [String]
}

struct SectionedSearchView: View {
    let sections: [SectionedContact] = [
        SectionedContact(letter: "A", names: ["Alice", "Amanda"]),
        SectionedContact(letter: "B", names: ["Bob", "Brenda"]),
        SectionedContact(letter: "C", names: ["Charlie", "Carol"]),
    ]

    @State private var searchText = ""

    var filteredSections: [SectionedContact] {
        if searchText.isEmpty { return sections }
        return sections.compactMap { section in
            let filtered = section.names.filter {
                $0.localizedCaseInsensitiveContains(searchText)
            }
            return filtered.isEmpty ? nil : SectionedContact(letter: section.letter, names: filtered)
        }
    }

    var body: some View {
        NavigationStack {
            List {
                ForEach(filteredSections, id: \.letter) { section in
                    Section(section.letter) {
                        ForEach(section.names, id: \.self) { Text($0) }
                    }
                }
            }
            .navigationTitle("Contacts")
            .searchable(text: $searchText)
        }
    }
}

    The key technique is compactMap — it drops sections whose filtered names array is empty, so you don't end up with empty section headers.

    Async Patterns in SwiftUI

    SwiftUI provides first-class support for asynchronous work through modifiers like .task, .refreshable, and AsyncImage. These integrate seamlessly with Swift's structured concurrency, handling cancellation and lifecycle automatically so you can focus on the data flow rather than manual task management.

    See 20-concurrency-basics for async/await, Task, and structured concurrency fundamentals. See 21-concurrency-advanced for actors, @MainActor, and advanced concurrency patterns. See 16-error-handling for do/catch and error propagation.

    The .task Modifier

    .task launches an async operation when the view appears. SwiftUI automatically cancels the task when the view disappears — no manual cleanup required.

    struct UserProfileView: View {
    let userId: Int
    @State private var user: User?

    var body: some View {
        Group {
            if let user {
                Text(user.name)
            } else {
                ProgressView()
            }
        }
        .task {
            user = await fetchUser(id: userId)
        }
    }
}

    The task runs on the main actor by default (since the view's body is @MainActor), but the await call suspends without blocking the UI. The automatic cancellation means navigating away from this view stops any in-flight network request.

    .task(id:) — Re-run on Value Change

    When you pass an id, the task cancels and restarts every time that value changes. This is ideal for search, filtering, or any input-driven async work.

    struct SearchView: View {
    @State private var query = ""
    @State private var results: [String] = []

    var body: some View {
        List(results, id: \.self) { Text($0) }
            .searchable(text: $query)
            .task(id: query) {
                // Debounce: wait before fetching
                try? await Task.sleep(for: .milliseconds(300))
                guard !Task.isCancelled else { return }
                results = await search(query)
            }
    }
}

    Every keystroke cancels the previous task. Only the final pause actually triggers the search.

    Loading State Pattern

    A robust async view handles three states: loading, loaded, and error. Model this explicitly with an enum.

    enum LoadingState<T> {
    case idle
    case loading
    case loaded(T)
    case error(Error)
}

struct ArticleListView: View {
    @State private var state: LoadingState<[Article]> = .idle

    var body: some View {
        Group {
            switch state {
            case .idle:
                Color.clear
            case .loading:
                ProgressView("Loading articles...")
            case .loaded(let articles):
                List(articles) { article in
                    Text(article.title)
                }
            case .error(let error):
                ContentUnavailableView {
                    Label("Error", systemImage: "exclamationmark.triangle")
                } description: {
                    Text(error.localizedDescription)
                } actions: {
                    Button("Retry") { Task { await loadArticles() } }
                }
            }
        }
        .task { await loadArticles() }
    }

    func loadArticles() async {
        state = .loading
        do {
            let articles = try await ArticleService.fetchAll()
            state = .loaded(articles)
        } catch {
            state = .error(error)
        }
    }
}

    Always show a loading indicator and handle errors — never leave the user staring at a blank screen.

    Pull-to-Refresh

    .refreshable adds pull-to-refresh to scrollable views. The closure is async; SwiftUI shows the refresh indicator until it completes.

    struct RefreshableListView: View {
    @State private var items: [String] = ["Item 1", "Item 2"]

    var body: some View {
        List(items, id: \.self) { Text($0) }
            .refreshable {
                items = await fetchLatestItems()
            }
    }
}

    The refresh indicator dismisses automatically when the async closure returns. Do not manually toggle a boolean — SwiftUI handles the UI.

    AsyncImage

    AsyncImage loads and displays a remote image with built-in phase handling.

    AsyncImage(url: URL(string: "https://example.com/photo.jpg")) { phase in
    switch phase {
    case .empty:
        ProgressView()
    case .success(let image):
        image
            .resizable()
            .aspectRatio(contentMode: .fill)
    case .failure:
        Image(systemName: "photo")
            .foregroundStyle(.secondary)
    @unknown default:
        EmptyView()
    }
}
.frame(width: 100, height: 100)
.clipShape(RoundedRectangle(cornerRadius: 8))

    AsyncImage does not cache images across view updates. For production apps with many images, consider a dedicated image caching library or a custom solution.

    Lifecycle Modifiers

    SwiftUI provides several modifiers for reacting to view lifecycle and value changes.

    struct LifecycleView: View {
    @State private var count = 0

    var body: some View {
        Text("Count: \(count)")
            .onAppear {
                print("View appeared")
            }
            .onDisappear {
                print("View disappeared — clean up resources")
            }
            .onChange(of: count) { oldValue, newValue in
                print("Count changed from \(oldValue) to \(newValue)")
            }
    }
}
    • .onAppear runs synchronously every time the view appears (including re-appearing after navigation back). Use .task for async work instead.
    • .onDisappear is the place to stop timers or clean up non-async resources.
    • .onChange(of:) fires when a value changes. The iOS 17+ version provides both old and new values.

    Pagination with onAppear

    Load more data when the user scrolls to the bottom by attaching .onAppear to the last visible item.

    struct PaginatedListView: View {
    @State private var items: [Item] = []
    @State private var page = 0
    @State private var isLoading = false

    var body: some View {
        List(items) { item in
            Text(item.title)
                .onAppear {
                    if item.id == items.last?.id {
                        loadMore()
                    }
                }
        }
        .task { loadMore() }
    }

    func loadMore() {
        guard !isLoading else { return }
        isLoading = true
        Task {
            let newItems = await fetchItems(page: page)
            items.append(contentsOf: newItems)
            page += 1
            isLoading = false
        }
    }
}

    @Observable View Models with .task

    Combine .task with an @Observable view model for clean separation of concerns.

    @MainActor
@Observable
class PostsViewModel {
    var posts: [Post] = []
    var state: LoadingState<[Post]> = .idle

    func load() async {
        state = .loading
        do {
            posts = try await PostService.fetchAll()
            state = .loaded(posts)
        } catch {
            state = .error(error)
        }
    }

    func refresh() async {
        // Don't reset to .loading — keep current content visible
        do {
            posts = try await PostService.fetchAll()
            state = .loaded(posts)
        } catch {
            state = .error(error)
        }
    }
}

struct PostsView: View {
    @State private var viewModel = PostsViewModel()

    var body: some View {
        List(viewModel.posts) { post in
            Text(post.title)
        }
        .refreshable { await viewModel.refresh() }
        .task { await viewModel.load() }
    }
}

    The @MainActor annotation ensures all property updates happen on the main thread. The view model is @Observable, so SwiftUI tracks changes to its properties automatically.

    Cancellation

    .task cancels automatically when the view disappears. For manual Task instances, check cancellation explicitly.

    func processItems(_ items: [Item]) async throws {
    for item in items {
        // Check before expensive work
        try Task.checkCancellation()
        await process(item)
    }
}

// Or check without throwing:
func processItemsSafe(_ items: [Item]) async {
    for item in items {
        if Task.isCancelled { return }
        await process(item)
    }
}

    Common mistake: launching a detached Task { } inside a view without storing and cancelling it. This leaks work that continues after the view disappears. Prefer .task whenever possible.

    Common Mistakes

    1. No loading state — the view is blank while data loads. Always show ProgressView.
    2. No error handling — network failures silently produce an empty screen. Always catch errors.
    3. Blocking the main thread — performing synchronous heavy work in body or onAppear freezes the UI. Use .task for all async operations.
    4. Ignoring cancellation — long-running loops that never check Task.isCancelled waste resources.
    5. Forgetting @MainActor — updating @State or @Observable properties from a background context causes warnings or undefined behavior.

    Geometry and Preferences

    SwiftUI's layout system is declarative — views propose sizes down and report sizes up. But sometimes you need to read a view's actual size or position at runtime. GeometryReader gives you access to this information. Preference keys let child views send data up to ancestors. Together, they enable scroll effects, adaptive layouts, sticky headers, and other advanced patterns.

    See 13-protocols for protocol conformance, which is how you define PreferenceKey. See 04-layout-fundamentals for how SwiftUI's layout proposal and response system works.

    GeometryReader

    GeometryReader is a container view that gives its content a GeometryProxy with the container's size and position information.

    struct GeometryBasicsView: View {
    var body: some View {
        GeometryReader { geometry in
            Text("Width: \(geometry.size.width, specifier: "%.0f")")
                .position(x: geometry.size.width / 2, y: geometry.size.height / 2)
        }
    }
}

    Critical behavior: GeometryReader accepts all proposed space and fills its container completely. This is different from most SwiftUI views, which only take the space they need. If you put a GeometryReader inside a VStack, it will expand to fill all available space.

    GeometryProxy Properties

    The GeometryProxy provides three key pieces of information.

    GeometryReader { proxy in
    VStack {
        // 1. Size of the container
        Text("Size: \(proxy.size.width) x \(proxy.size.height)")

        // 2. Position in the global screen coordinate space
        Text("Global Y: \(proxy.frame(in: .global).minY)")

        // 3. Position relative to the parent
        Text("Local origin: \(proxy.frame(in: .local).origin)")
    }
}
    • proxy.size — the width and height available to the GeometryReader
    • proxy.frame(in: .global) — the frame in screen coordinates (useful for scroll effects)
    • proxy.frame(in: .local) — the frame relative to the GeometryReader itself (origin is usually zero)

    Named Coordinate Spaces

    You can define custom coordinate spaces and measure positions relative to them.

    struct CoordinateSpaceExample: View {
    var body: some View {
        ScrollView {
            VStack(spacing: 20) {
                ForEach(0..<20) { i in
                    GeometryReader { proxy in
                        Text("Item \(i) — Y in scroll: \(proxy.frame(in: .named("scroll")).minY, specifier: "%.0f")")
                            .frame(maxWidth: .infinity)
                    }
                    .frame(height: 50)
                }
            }
        }
        .coordinateSpace(name: "scroll")
    }
}

    The .coordinateSpace(name:) modifier defines a reference frame. Any descendant can measure its position relative to it using proxy.frame(in: .named("scroll")).

    GeometryReader Gotchas

    Because GeometryReader takes all proposed space, naively placing it in a layout can break your design.

    Problem:

    VStack {
    Text("Title")
    GeometryReader { proxy in  // This expands to fill remaining VStack space
        Text("Size: \(proxy.size.width)")
    }
    Text("Footer")  // Pushed to the bottom or off-screen
}

    Solution: Use GeometryReader inside an .overlay or .background so it measures without affecting layout.

    Text("Hello")
    .overlay {
        GeometryReader { proxy in
            Color.clear
                .onAppear {
                    print("Text size: \(proxy.size)")
                }
        }
    }

    This is the recommended pattern for measuring a view's size: put the GeometryReader in an overlay with Color.clear and read the size in .onAppear or via a preference key.

    Preference Keys

    Preferences let child views communicate data up to ancestor views — the reverse of the normal data flow. You define a PreferenceKey with a default value and a reduce function that combines values from multiple children.

    struct ViewHeightKey: PreferenceKey {
    static var defaultValue: CGFloat = 0
    static func reduce(value: inout CGFloat, nextValue: () -> CGFloat) {
        value = max(value, nextValue())
    }
}

    Setting a preference (child):

    Text("Measure me")
    .background {
        GeometryReader { proxy in
            Color.clear
                .preference(key: ViewHeightKey.self, value: proxy.size.height)
        }
    }

    Reading a preference (ancestor):

    struct ParentView: View {
    @State private var childHeight: CGFloat = 0

    var body: some View {
        VStack {
            ChildView()
            Text("Child height: \(childHeight)")
        }
        .onPreferenceChange(ViewHeightKey.self) { height in
            childHeight = height
        }
    }
}

    The reduce function determines how values from multiple children are combined. Common strategies: - max(value, nextValue()) — take the largest - value + nextValue() — sum all values - value = nextValue() — last writer wins

    Anchor Preferences

    AnchorPreference captures geometry information (bounds, positions) that can only be resolved relative to a specific coordinate space using a GeometryProxy.

    struct BoundsKey: PreferenceKey {
    static var defaultValue: Anchor<CGRect>? = nil
    static func reduce(value: inout Anchor<CGRect>?, nextValue: () -> Anchor<CGRect>?) {
        value = value ?? nextValue()
    }
}

struct HighlightableView: View {
    @State private var selectedBounds: Anchor<CGRect>?

    var body: some View {
        VStack {
            Text("Option A")
                .anchorPreference(key: BoundsKey.self, value: .bounds) { $0 }
        }
        .onPreferenceChange(BoundsKey.self) { selectedBounds = $0 }
        .overlayPreferenceValue(BoundsKey.self) { anchor in
            if let anchor {
                GeometryReader { proxy in
                    let rect = proxy[anchor]
                    RoundedRectangle(cornerRadius: 8)
                        .stroke(.blue, lineWidth: 2)
                        .frame(width: rect.width, height: rect.height)
                        .offset(x: rect.minX, y: rect.minY)
                }
            }
        }
    }
}

    Anchor preferences are powerful for drawing highlights, underlines, and indicators that track the position of a child view.

    .visualEffect (iOS 17+)

    .visualEffect provides geometry-aware transformations without the layout disruption of GeometryReader. The view's layout remains unchanged — only the visual rendering is affected.

    ScrollView {
    LazyVStack {
        ForEach(0..<30) { i in
            RoundedRectangle(cornerRadius: 12)
                .fill(.blue.gradient)
                .frame(height: 100)
                .visualEffect { content, proxy in
                    let frame = proxy.frame(in: .scrollView)
                    let distance = min(0, frame.minY)
                    content
                        .scaleEffect(1 + distance / 700)
                        .opacity(1 + distance / 300)
                }
        }
    }
}

    This is the modern replacement for many GeometryReader-in-scroll patterns. It does not affect layout, does not take extra space, and composes cleanly.

    See 18-metal-in-swiftui for combining .visualEffect with Metal shader functions.

    Common Pattern: Measuring View Size

    A reusable modifier that reads a view's size and writes it to a binding.

    struct SizePreferenceKey: PreferenceKey {
    static var defaultValue: CGSize = .zero
    static func reduce(value: inout CGSize, nextValue: () -> CGSize) {
        value = nextValue()
    }
}

extension View {
    func readSize(into binding: Binding<CGSize>) -> some View {
        self.background {
            GeometryReader { proxy in
                Color.clear
                    .preference(key: SizePreferenceKey.self, value: proxy.size)
            }
        }
        .onPreferenceChange(SizePreferenceKey.self) { binding.wrappedValue = $0 }
    }
}

// Usage
struct MeasuredView: View {
    @State private var textSize: CGSize = .zero

    var body: some View {
        Text("Hello, World!")
            .readSize(into: $textSize)
        Text("Text is \(textSize.width, specifier: "%.0f") x \(textSize.height, specifier: "%.0f")")
    }
}

    Common Pattern: Parallax Scroll Header

    struct ParallaxHeaderView: View {
    var body: some View {
        ScrollView {
            VStack(spacing: 0) {
                GeometryReader { proxy in
                    let minY = proxy.frame(in: .scrollView).minY
                    Image("header")
                        .resizable()
                        .scaledToFill()
                        .frame(width: proxy.size.width,
                               height: max(250, 250 + minY))
                        .offset(y: minY > 0 ? -minY : 0)

                }
                .frame(height: 250)

                // Content below the header
                ForEach(0..<30) { i in
                    Text("Row \(i)")
                        .frame(maxWidth: .infinity, alignment: .leading)
                        .padding()
                }
            }
        }
    }
}

    When the user pulls down, the header stretches. When scrolling up, it pins and the content scrolls over it.

    View Composition

    SwiftUI's power comes from composing small, focused views into complex interfaces. Every view you write is a struct with a body — and that body can contain other custom views. Mastering composition means your code stays readable, reusable, and performant even as your app grows.

    See 14-generics for generic type parameters used in reusable view containers. See 07-closures for the closure syntax that powers @ViewBuilder parameters. See 25-swift-internals for result builders, the compiler feature behind @ViewBuilder.

    Extracting Subviews

    The simplest form of composition is pulling part of a view into its own struct.

    // Before: one large view
struct ProfileView: View {
    var body: some View {
        VStack {
            Image(systemName: "person.circle.fill")
                .font(.system(size: 80))
                .foregroundStyle(.blue)
            Text("Alice")
                .font(.title)
            Text("iOS Developer")
                .font(.subheadline)
                .foregroundStyle(.secondary)
        }
    }
}

// After: extracted subview
struct AvatarView: View {
    let name: String

    var body: some View {
        Image(systemName: "person.circle.fill")
            .font(.system(size: 80))
            .foregroundStyle(.blue)
    }
}

struct ProfileView: View {
    var body: some View {
        VStack {
            AvatarView(name: "Alice")
            Text("Alice").font(.title)
            Text("iOS Developer")
                .font(.subheadline)
                .foregroundStyle(.secondary)
        }
    }
}

    Extract when a view is reused, when it has its own state, or when the parent body gets too long to read at a glance.

    Passing Data to Subviews

    There are three main ways to get data into a subview.

    1. Init parameters — simplest, for read-only data:

    struct GreetingView: View {
    let name: String
    var body: some View { Text("Hello, \(name)") }
}

    2. @Binding — for two-way data flow:

    struct ToggleRow: View {
    let title: String
    @Binding var isOn: Bool
    var body: some View {
        Toggle(title, isOn: $isOn)
    }
}

    3. @Environment — for data that many views need:

    struct ThemedText: View {
    @Environment(\.colorScheme) var colorScheme
    var body: some View {
        Text("Themed")
            .foregroundStyle(colorScheme == .dark ? .white : .black)
    }
}

    Use init parameters by default. Use @Binding when the child needs to modify the parent's state. Use @Environment for cross-cutting concerns like themes, locale, or app-wide settings.

    @ViewBuilder

    @ViewBuilder is a result builder that lets functions and computed properties return multiple views as a single composed view. SwiftUI's body property uses it implicitly.

    struct ContentView: View {
    @ViewBuilder
    var header: some View {
        Text("Welcome")
            .font(.largeTitle)
        Text("Here is your dashboard")
            .font(.subheadline)
            .foregroundStyle(.secondary)
    }

    var body: some View {
        VStack {
            header
            Divider()
            Text("Content goes here")
        }
    }
}

    You can also use @ViewBuilder on functions:

    @ViewBuilder
func badge(for count: Int) -> some View {
    if count > 0 {
        Text("\(count)")
            .font(.caption)
            .padding(4)
            .background(.red)
            .clipShape(Circle())
    }
    // If count is 0, returns EmptyView implicitly
}

    The @ViewBuilder attribute enables if/else, switch, and for statements inside the function body.

    Generic View Containers

    Combine generics with @ViewBuilder to create reusable containers that accept any content.

    struct Card<Content: View>: View {
    let title: String
    @ViewBuilder var content: () -> Content

    var body: some View {
        VStack(alignment: .leading, spacing: 12) {
            Text(title)
                .font(.headline)
            content()
        }
        .padding()
        .background(.background, in: RoundedRectangle(cornerRadius: 12))
        .shadow(radius: 2)
    }
}

// Usage
Card(title: "Weather") {
    Label("72°F", systemImage: "sun.max.fill")
    Label("Clear skies", systemImage: "cloud")
}

    The caller provides any views inside the trailing closure, and Card wraps them in a styled container. This is the same pattern Apple uses for VStack, List, and NavigationStack.

    Multi-Slot Containers

    For containers with distinct regions (header, content, footer), use multiple @ViewBuilder parameters.

    struct PageLayout<Header: View, Content: View, Footer: View>: View {
    @ViewBuilder var header: () -> Header
    @ViewBuilder var content: () -> Content
    @ViewBuilder var footer: () -> Footer

    var body: some View {

                .frame(maxWidth: .infinity)
                .background(.ultraThinMaterial)

                content()
                    .padding()
            }

            footer()
                .padding()
                .frame(maxWidth: .infinity)
                .background(.ultraThinMaterial)
        }
    }
}

    AnyView — Type Erasure

    AnyView wraps any view into a uniform type. This is occasionally necessary but should be avoided when possible because it prevents SwiftUI from optimizing diffs.

    // Avoid this pattern when possible
func makeView(for type: String) -> AnyView {
    switch type {
    case "text": return AnyView(Text("Hello"))
    case "image": return AnyView(Image(systemName: "star"))
    default: return AnyView(EmptyView())
    }
}

// Prefer @ViewBuilder instead
@ViewBuilder
func makeView(for type: String) -> some View {
    switch type {
    case "text": Text("Hello")
    case "image": Image(systemName: "star")
    default: EmptyView()
    }
}

    With @ViewBuilder, SwiftUI knows the exact types at compile time and can diff them efficiently. AnyView erases that information.

    Conditional Views

    if/else in body creates a _ConditionalContent — SwiftUI knows both branches and can animate between them.

    struct ToggleableView: View {
    @State private var showDetail = false

    var body: some View {
        VStack {
            if showDetail {
                DetailView()
                    .transition(.slide)
            } else {
                SummaryView()
                    .transition(.slide)
            }
            Button("Toggle") {
                withAnimation { showDetail.toggle() }
            }
        }
    }
}

    When a branch changes, SwiftUI removes the old view's state and creates fresh state for the new branch. This is by design — each branch is a distinct view identity.

    EquatableView

    By default, SwiftUI re-evaluates a view's body whenever any parent state changes — even if the view's inputs haven't changed. EquatableView lets you override this.

    struct ExpensiveView: View, Equatable {
    let data: [Int]

    static func == (lhs: Self, rhs: Self) -> Bool {
        lhs.data == rhs.data
    }

    var body: some View {
        // Complex rendering that you don't want re-evaluated unnecessarily
        Text("Items: \(data.count)")
    }
}

// Wrap in EquatableView
EquatableView(content: ExpensiveView(data: items))

    Only use this when profiling reveals unnecessary body evaluations. Premature optimization here adds complexity without benefit.

    UIKit Integration

    SwiftUI doesn't cover every UIKit feature. When you need a UIKit view (like MKMapView, WKWebView, or a custom UITextView) inside SwiftUI, you wrap it with UIViewRepresentable. When you need a UIKit view controller (like UIImagePickerController), you use UIViewControllerRepresentable. And when you need SwiftUI inside a UIKit app, you use UIHostingController.

    See 13-protocols for protocol conformance — both representable protocols require specific method implementations. See 18-memory-management for coordinator lifecycle and avoiding retain cycles.

    UIViewRepresentable

    This protocol bridges a UIKit UIView into SwiftUI. You implement two required methods.

    struct MyWrappedView: UIViewRepresentable {
    func makeUIView(context: Context) -> UILabel {
        let label = UILabel()
        label.text = "Hello from UIKit"
        label.textAlignment = .center
        return label
    }

    func updateUIView(_ uiView: UILabel, context: Context) {
        // Called when SwiftUI state changes
        // Update the UIKit view to reflect new state
    }
}
    • makeUIView(context:) — called once to create the UIKit view. Do initial setup here.
    • updateUIView(_:context:) — called every time the SwiftUI view's state changes. Sync the UIKit view with current SwiftUI state here.

    The Coordinator Pattern

    UIKit relies heavily on delegates and data sources. The Coordinator is a reference-type object (class) that acts as the delegate bridge between UIKit callbacks and SwiftUI state.

    struct WrappedTextView: UIViewRepresentable {
    @Binding var text: String

    func makeCoordinator() -> Coordinator {
        Coordinator(self)
    }

    func makeUIView(context: Context) -> UITextView {
        let textView = UITextView()
        textView.delegate = context.coordinator
        textView.font = .preferredFont(forTextStyle: .body)
        return textView
    }

    func updateUIView(_ uiView: UITextView, context: Context) {
        if uiView.text != text {
            uiView.text = text
        }
    }

    class Coordinator: NSObject, UITextViewDelegate {
        var parent: WrappedTextView

        init(_ parent: WrappedTextView) {
            self.parent = parent
        }

        func textViewDidChange(_ textView: UITextView) {
            parent.text = textView.text
        }
    }
}

    The data flow is: 1. SwiftUI @Binding changes → updateUIView is called → UIKit view updates 2. User types in UIKit → delegate method fires on Coordinator → Coordinator writes to @Binding → SwiftUI updates

    The if uiView.text != text guard in updateUIView prevents an infinite loop where setting the text triggers the delegate, which sets the binding, which triggers updateUIView again.

    Example: Wrapping MKMapView

    import MapKit

struct MapView: UIViewRepresentable {
    @Binding var region: MKCoordinateRegion

    func makeCoordinator() -> Coordinator {
        Coordinator(self)
    }

    func makeUIView(context: Context) -> MKMapView {
        let mapView = MKMapView()
        mapView.delegate = context.coordinator
        return mapView
    }

    func updateUIView(_ uiView: MKMapView, context: Context) {
        uiView.setRegion(region, animated: true)
    }

    class Coordinator: NSObject, MKMapViewDelegate {
        var parent: MapView

        init(_ parent: MapView) {
            self.parent = parent
        }

            parent.region = mapView.region
        }
    }
}

    UIViewControllerRepresentable

    For wrapping entire view controllers. The pattern is identical to UIViewRepresentable but uses UIViewController instead of UIView.

    Sizing Considerations

    When wrapping UIKit views, sizing can be tricky.

    struct SizedWrappedView: UIViewRepresentable {
    func makeUIView(context: Context) -> UILabel {
        let label = UILabel()
        label.text = "Hello"
        label.setContentHuggingPriority(.required, for: .horizontal)
        label.setContentCompressionResistancePriority(.required, for: .horizontal)
        return label
    }

    func updateUIView(_ uiView: UILabel, context: Context) {}
}

    If the UIKit view has a proper intrinsicContentSize, SwiftUI will respect it. If not, wrap the representable in a .frame() modifier to give it explicit dimensions.

    For UIHostingController, call hostingController.sizeThatFits(in:) to get the ideal size for the SwiftUI content.

    See 17-metal-in-uikit for wrapping Metal/MTKView as a UIViewRepresentable.

    Common Issues

    1. Coordinator lifecycle — The coordinator is created once by makeCoordinator() and persists for the view's lifetime. Do not create a new coordinator in updateUIView.
    2. Update cycles — Always guard against redundant updates in updateUIView to prevent infinite loops (e.g., if uiView.text != text).
    3. Memory — The coordinator holds a reference to the parent struct. Since the parent is a value type, this is safe. But if your coordinator subscribes to notifications or holds closures, remember to clean up in deinit or use dismantleUIView.
    4. Threading — UIKit views must be updated on the main thread. Coordinator delegate methods that return data asynchronously should dispatch to DispatchQueue.main before updating bindings.

    MVVM in SwiftUI

    The most common SwiftUI architecture is MVVM: Model (data), View (UI), ViewModel (logic and state). With the @Observable macro (iOS 17+), view models are simple classes.

    // Model
struct Task: Identifiable {
    let id = UUID()
    var title: String
    var isComplete: Bool
}

// ViewModel
@MainActor
@Observable
class TaskListViewModel {
    var tasks: [Task] = []
    var newTaskTitle = ""

    func addTask() {
        guard !newTaskTitle.isEmpty else { return }
        tasks.append(Task(title: newTaskTitle, isComplete: false))
        newTaskTitle = ""
    }

    func toggleTask(_ task: Task) {
        guard let index = tasks.firstIndex(where: { $0.id == task.id }) else { return }
        tasks[index].isComplete.toggle()
    }

    func deleteTasks(at offsets: IndexSet) {
        tasks.remove(atOffsets: offsets)
    }
}

// View
struct TaskListView: View {
    @State private var viewModel = TaskListViewModel()

    var body: some View {
        NavigationStack {
            List {
                ForEach(viewModel.tasks) { task in
                    HStack {
                        Image(systemName: task.isComplete ? "checkmark.circle.fill" : "circle")
                            .onTapGesture { viewModel.toggleTask(task) }
                        Text(task.title)
                            .strikethrough(task.isComplete)
                    }
                }
                .onDelete { viewModel.deleteTasks(at: $0) }
            }
            .navigationTitle("Tasks")
            .toolbar {
                ToolbarItem(placement: .bottomBar) {
                    HStack {
                        TextField("New task", text: $viewModel.newTaskTitle)
                        Button("Add", action: viewModel.addTask)
                    }
                }
            }
        }
    }
}

    @MainActor ensures all property updates happen on the main thread. @Observable automatically triggers SwiftUI updates when properties change.

    Repository Pattern

    Abstract data sources behind a protocol so your view model doesn't care whether data comes from a network, database, or mock.

    protocol PostRepository {
    func fetchAll() async throws -> [Post]
    func save(_ post: Post) async throws
}

class LivePostRepository: PostRepository {
    func fetchAll() async throws -> [Post] {
        let url = URL(string: "https://api.example.com/posts")!
        let (data, _) = try await URLSession.shared.data(from: url)
        return try JSONDecoder().decode([Post].self, from: data)
    }

    func save(_ post: Post) async throws {
        // network save
    }
}

class MockPostRepository: PostRepository {
    func fetchAll() async throws -> [Post] {
        return [Post(id: 1, title: "Mock Post")]
    }

    func save(_ post: Post) async throws { }
}

    In tests and previews, inject MockPostRepository. In production, inject LivePostRepository.

    Router Pattern

    Centralize navigation with an enum of routes and a router object.

    enum Route: Hashable {
    case detail(Item)
    case settings
    case profile(userId: Int)
}

@MainActor
@Observable
class Router {
    var path = NavigationPath()

    func navigate(to route: Route) {
        path.append(route)
    }

    func popToRoot() {
        path = NavigationPath()
    }
}

struct RootView: View {
    @State private var router = Router()

    var body: some View {
        NavigationStack(path: $router.path) {
            HomeView()
                .navigationDestination(for: Route.self) { route in
                    switch route {
                    case .detail(let item): DetailView(item: item)
                    case .settings: SettingsView()
                    case .profile(let id): ProfileView(userId: id)
                    }
                }
        }
        .environment(router)
    }
}

    Any child view can navigate by accessing the router from the environment.

    App Lifecycle

    The @main attribute marks your app's entry point. App contains Scene instances that define windows.

    @main
struct MyApp: App {
    @Environment(\.scenePhase) var scenePhase

    var body: some Scene {
        WindowGroup {
            ContentView()
        }
        .onChange(of: scenePhase) { oldPhase, newPhase in
            switch newPhase {
            case .active:
                print("App is active")
            case .inactive:
                print("App is inactive — save state")
            case .background:
                print("App is in background — persist data")
            @unknown default:
                break
            }
        }
    }
}

    Scene phases: - .active — app is in the foreground and interactive - .inactive — app is visible but not interactive (e.g., during app switcher) - .background — app is not visible. Save state and release resources here.

    Deep Linking

    Handle incoming URLs with .onOpenURL.

    struct ContentView: View {
    @Environment(Router.self) var router

    var body: some View {
        NavigationStack(path: $router.path) {
            HomeView()
        }
        .onOpenURL { url in
            // Parse url and navigate
            if let route = Route(from: url) {
                router.navigate(to: route)
            }
        }
    }
}

    The Composable Architecture (TCA) — Brief Overview

    TCA is a third-party architecture by Point-Free that enforces unidirectional data flow.

    User Action → Reducer → State → View

    Core concepts: - State — a struct holding all view state - Action — an enum of everything the user (or system) can do - Reducer — a pure function: (State, Action) → (State, Effect) - Store — holds state, dispatches actions, runs effects

    TCA provides strong testability, composability, and dependency management. The tradeoff is more boilerplate and a steeper learning curve. Consider it for large team projects where consistency matters.

    Common Architectural Mistakes

    1. Massive views — a 500-line body is a code smell. Extract subviews.
    2. Business logic in body — formatting, filtering, and validation belong in the view model.
    3. Over-abstraction — don't create abstractions until you have three similar things. Start concrete.
    4. Singleton abuse — injecting dependencies through environment is cleaner and more testable than global singletons.
    5. Ignoring accessibility — it's much harder to retrofit. Build it in from the start.
    6. Skipping error handling — every async operation can fail. Handle every error state.

    5. Stacked Bottom Sheet

    Link: src/animations/stacked-bottom-sheet

    What it does: Multiple bottom sheets stack on top of each other. Each new sheet pushes the previous one back (scales down, translates up). Dismiss via drag.

    Key technique: Provider pattern manages sheet stack. Each sheet tracks its depth in stack. scale = 1 - (depth * 0.05), translateY = depth * -20, cornerRadius increases with depth. Pan gesture with velocity threshold for dismiss.

    SwiftUI translation:

    // Custom sheet manager (ObservableObject) tracks stack
// Each sheet level:
.scaleEffect(1.0 - Double(depth) * 0.05)
.offset(y: -CGFloat(depth) * 20)
.clipShape(RoundedRectangle(cornerRadius: 12 + CGFloat(depth) * 4))
// Drag to dismiss with .gesture(DragGesture())
// Or use .sheet() with custom detents for simpler version

    UIKit translation: Custom UIPresentationController subclass. Override presentationTransitionWillBegin to scale presenting VC.

    Use for: Multi-level sheets, wizard flows, stacked modals.

    6. Clerk-Style Stacked Toasts

    Link: src/animations/clerk-toast

    What it does: Toasts stack at top of screen. New toasts push old ones down and scale them. Swipe to dismiss individual toasts.

    Key technique: Toast provider manages array. Each toast's style driven by its index: scale = 1 - index * 0.05, translateY = index * 8. Spring animation on insert/remove. Swipe gesture on individual toast for dismiss.

    SwiftUI translation:

    // Overlay with ForEach over toast stack
// Each toast:
.scaleEffect(1.0 - Double(index) * 0.05)
.offset(y: CGFloat(index) * 8)
.transition(.move(edge: .top).combined(with: .opacity))
.gesture(DragGesture().onEnded { if $0.translation.y < -50 { dismiss() } })

    Use for: Notification toasts, snackbars, alert stacks.

    7. Liquid Glass Playground

    Link: src/animations/liquid-glass-playground

    What it does: Draggable glass overlay with progressive haptics. Uses expo-glass-effect on iOS for native blur. Damped drag friction makes it feel physical.

    Key technique: dampedTranslation = translationY * DRAG_FRICTION. Progressive haptics: haptic intensity increases with drag distance. Spring return: withSpring(0, { duration: 600, dampingRatio: 1 }) (critically damped = no bounce).

    SwiftUI translation:

// Pre-iOS 26: .background(.ultraThinMaterial)
// Drag with friction:
let damped = translation * 0.6
// Spring return:
.animation(.spring(duration: 0.6, bounce: 0)) // critically damped
// Progressive haptics:
UIImpactFeedbackGenerator(style: intensityForDistance(damped))

    Use for: Glass overlays, draggable panels, pull-to-refresh.

    Spring Config Translation Guide

    Common springs from the demos → SwiftUI equivalents:

    Reanimated SwiftUI Feel
    { damping: 200, stiffness: 200 } .spring(duration: 0.6, bounce: 0) Critically damped, no bounce
    { damping: 100, stiffness: 200 } .spring(response: 0.35, dampingFraction: 0.7) Snappy with slight bounce
    { damping: 50, stiffness: 300, mass: 0.4 } .spring(response: 0.25, dampingFraction: 0.6) Quick and bouncy
    { damping: 80, stiffness: 120, mass: 0.8 } .spring(response: 0.5, dampingFraction: 0.75) Medium, natural
    withSpring(0, { duration: 600, dampingRatio: 1 }) .spring(duration: 0.6, bounce: 0) Smooth return, no overshoot
    withTiming({ duration: 400 }) .easeInOut(duration: 0.4) Linear timing curve

    UIKit equivalent: UIView.animate(withDuration:delay:usingSpringWithDamping:initialSpringVelocity:) or UISpringTimingParameters(mass:stiffness:damping:initialVelocity:)
