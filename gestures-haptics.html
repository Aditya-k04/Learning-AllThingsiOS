Gestures & Haptics
Custom and simultaneous recognizers, velocity & rubber-banding, and Core Haptics — the 'feel' of fluid interfaces.

    Gestures & Haptics — Overview

    Fluid interfaces feel alive because they respond continuously to the user and confirm actions physically. That's two systems working together: gesture recognition (turning raw touches into intent) and haptics (turning events into felt feedback). This course covers both — the input side of the "feel" you keep chasing in your interaction work.

    The big idea

    A great gesture-driven interaction is a pure function of the touch, not an animation that plays. The interface tracks the finger 1:1, and physics (springs, deceleration) takes over only on release.

    This is the thesis of Apple's Designing Fluid Interfaces, and it's why understanding gestures matters as much as understanding animation.

    What you'll learn

    1. 01-gesture-recognizers — the recognizer state machine, the built-in recognizers, and discrete vs continuous.
    2. 02-simultaneity-and-dependencies — making recognizers cooperate, fail, or require each other (the hard part).
    3. 03-velocity-and-rubber-banding — velocity, projection, and the rubber-band formula behind every Apple scroll edge.
    4. 04-core-haptics — UIFeedbackGenerator and the CHHapticEngine for custom textures.
    5. 05-designing-the-feel — combining gesture, spring, and haptic into one coherent interaction.

    The math of velocity, projection, and rubber-banding is Interaction Math applied. The spring that catches a released gesture is the same critically-damped spring you already studied. And the place gestures actually live — UIScrollView, hit-testing — is the responder chain.

    Gesture Recognizers

    A UIGestureRecognizer watches the raw touch stream on a view and collapses it into a single, meaningful gesture — a tap, a pan, a pinch. It saves you from implementing touchesBegan/Moved/Ended by hand and, crucially, coordinates with other recognizers.

    The state machine

    Every recognizer is a small state machine. The two shapes:

    • Discrete (tap, long-press): .possible → .recognized (fires once) or .failed.
    • Continuous (pan, pinch, rotation): .possible → .began → .changed (repeatedly) → .ended (or .cancelled).
    let pan = UIPanGestureRecognizer(target: self, action: #selector(handlePan))
view.addGestureRecognizer(pan)

@objc func handlePan(_ g: UIPanGestureRecognizer) {
    switch g.state {
    case .began:   startOffset = card.center
    case .changed:
        let t = g.translation(in: view)
        card.center = CGPoint(x: startOffset.x + t.x, y: startOffset.y + t.y)
    case .ended, .cancelled:
        let v = g.velocity(in: view)        // points/sec — feed a spring
        settle(with: v)
    default: break
    }
}

    You read translation(in:), velocity(in:), scale, or rotation depending on the recognizer — these are the continuous values you map to your effect.

    The built-ins

    UITapGestureRecognizer (set numberOfTapsRequired), UILongPressGestureRecognizer (continuous after the press threshold), UIPanGestureRecognizer, UIPinchGestureRecognizer, UIRotationGestureRecognizer, UISwipeGestureRecognizer (discrete, directional), and UIScreenEdgePanGestureRecognizer (interactive back-swipe).

    Translation reset trick

    translation accumulates from the gesture start. A common pattern is to read it, apply it, then zero it so you work in deltas:

    let t = g.translation(in: view)
card.center.x += t.x
g.setTranslation(.zero, in: view)   // now translation is the delta since last call

    Discrete vs continuous matters for design

    Discrete gestures commit an action (tap to select). Continuous gestures track a value (drag to move) — and tracking is where fluidity lives, because the user stays in control the whole time. Prefer continuous, interruptible gestures over discrete taps that trigger canned animations wherever the interaction allows.

    SwiftUI parallel

    SwiftUI's DragGesture, MagnificationGesture, etc. wrap these same recognizers. When SwiftUI's gesture API isn't enough (simultaneous recognition with a scroll view, precise failure requirements), drop to UIKit via UIViewRepresentable.

    Simultaneity & Dependencies

    A single view often has several recognizers, and they compete for the same touches. By default, only one recognizer wins a touch sequence. Coordinating them — letting some run together, making one wait for another to fail — is where most real gesture work happens, and it's all done through UIGestureRecognizerDelegate.

    The default: mutual exclusion

    If a view has both a tap and a pan, UIKit lets exactly one recognize. A pinch and a rotation on the same view won't both fire unless you explicitly allow it. The fix is one delegate method:

    func gestureRecognizer(_ g: UIGestureRecognizer,
    shouldRecognizeSimultaneouslyWith other: UIGestureRecognizer) -> Bool {
    return true   // let pinch + rotation + pan all drive the same view
}

    This is exactly how a photo viewer lets you pinch, rotate, and pan a picture at once.

    Dependencies: require failure / require success

    Two ordering tools:

    // Don't recognize a single-tap until the double-tap has FAILED
singleTap.require(toFail: doubleTap)

// Don't start my pan until the screen-edge pan has failed (let system back-swipe win)
myPan.require(toFail: edgePan)

    require(toFail:) is the classic single-vs-double-tap solution and the way you yield to system gestures at screen edges.

    Cooperating with UIScrollView

    The trickiest real case: a custom pan inside a scroll view. The scroll view's pan and yours fight. Options:

    • Implement shouldRecognizeSimultaneouslyWith to let both run, then decide in your handler.
    • Use gestureRecognizerShouldBegin(_:) to veto your recognizer based on direction (e.g. only take over for horizontal drags, let vertical scroll pass through).
    func gestureRecognizerShouldBegin(_ g: UIGestureRecognizer) -> Bool {
    guard let pan = g as? UIPanGestureRecognizer else { return true }
    let v = pan.velocity(in: view)
    return abs(v.x) > abs(v.y)   // only claim mostly-horizontal drags
}

    shouldReceive: filtering touches

    gestureRecognizer(_:shouldReceive:) lets you ignore touches that land on certain subviews (e.g. don't start a background pan when the touch began on a button). It's the precise alternative to toggling isUserInteractionEnabled everywhere.

    The hit-testing connection

    Before a recognizer ever sees a touch, hit-testing decided which view the touch belongs to (the responder chain). If a gesture "never fires," check hit-testing first — an overlapping view or disabled interaction is the usual culprit, not the recognizer config.

    Velocity & Rubber-Banding

    The difference between a gesture that feels mechanical and one that feels alive comes down to two things: respecting the velocity of the finger when it lifts, and resisting — with rubber-banding — when the content reaches a boundary. Both are simple formulas, and both are everywhere in iOS.

    Velocity: the handoff to physics

    When a drag ends, the finger has a velocity (points/second). A canned animation ignores it; a fluid one continues the motion at that speed and lets a spring or deceleration take over. This is the gesture→physics handoff.

    case .ended:
    let v = pan.velocity(in: view)         // e.g. (1200, -300) pts/sec
    // hand v to a spring as initial velocity, or project where it would land:
    let projected = project(initial: v.x, decelerationRate: 0.998)
}

    Projection: where would it land?

    UIScrollView's deceleration uses an exponential decay. The closed-form "final resting point" is the standard projection function — use it to decide which page/item the flick should snap to:

    func project(initial v: CGFloat, decelerationRate d: CGFloat) -> CGFloat {
    return (v / 1000) * d / (1 - d)        // displacement after a flick
}

    This is the math from Designing Fluid Interfaces — project the flick, find the nearest snap target, then spring there.

    Rubber-banding

    When content is dragged past its bounds (top of a scroll view, edge of a card stack), it shouldn't move 1:1 — it should resist, moving less and less the further you pull. Apple's formula:

    // x  = how far past the edge the finger is
// d  = dimension (e.g. view height), c ≈ 0.55 constant
func rubberBand(_ x: CGFloat, dim d: CGFloat, c: CGFloat = 0.55) -> CGFloat {
    return (1 - (1 / (x * c / d + 1))) * d
}

    The result asymptotically approaches a limit no matter how hard you pull — exactly the bouncy resistance of every iOS scroll edge. On release, a spring returns it to the boundary.

    The complete release behavior

    on .ended:
    if past a boundary  → spring back to the boundary (using release velocity)
    else if flicked     → project landing point, snap to nearest item, spring there
    else                → spring to nearest rest position

    That three-case release is the heart of nearly every premium iOS interaction.

    Core Haptics

    Haptics are the felt half of feedback — the tap you feel when a picker lands, the buzz when an action fails. Used well they make an interface feel physical and confident; overused they're annoying. iOS gives you two tiers: simple semantic generators and the full Core Haptics engine.

    Tier 1: UIFeedbackGenerator (use this 90% of the time)

    Three semantic generators cover most needs. They map to system-tuned patterns and respect the user's settings:

    // Impact — something hit/landed. Style sets the weight.
let impact = UIImpactFeedbackGenerator(style: .medium)  // .light/.medium/.heavy/.soft/.rigid
impact.prepare()                 // warms the Taptic Engine (lowers latency)
impact.impactOccurred()

// Selection — a value changed (picker tick, segmented control)
let selection = UISelectionFeedbackGenerator()
selection.selectionChanged()

// Notification — success / warning / error
UINotificationFeedbackGenerator().notificationOccurred(.success)

    The key detail: call prepare() just before you expect to fire. It wakes the Taptic Engine so the haptic lands in sync with the visual; without it there's a perceptible lag.

    Match haptics to motion

    A haptic that fires a frame off from its visual feels wrong. Trigger the feedback at the exact moment the visual event happens — the snap, the toggle, the boundary hit from 03-velocity-and-rubber-banding. A pull-to-refresh, for example, fires one .impactOccurred() precisely when the threshold is crossed, not when the release animation finishes.

    Tier 2: CHHapticEngine (custom textures)

    For bespoke feedback — a continuous rumble that tracks a drag, a layered "texture" — use Core Haptics directly. You build a pattern of transient events (sharp taps) and continuous events (sustained vibration), each with intensity and sharpness:

    import CoreHaptics
let engine = try CHHapticEngine()
try engine.start()

let tap = CHHapticEvent(eventType: .hapticTransient, parameters: [
    CHHapticEventParameter(parameterID: .hapticIntensity, value: 1.0),
    CHHapticEventParameter(parameterID: .hapticSharpness, value: 0.7)
], relativeTime: 0)

let pattern = try CHHapticPattern(events: [tap], parameters: [])
try engine.makePlayer(with: pattern).start(atTime: 0)
    • intensity — how strong (amplitude).
    • sharpness — timbre, from soft/round (0) to crisp/clicky (1).

    You can drive a continuous event's parameters in real time (a CHHapticParameterCurve) so the haptic intensity follows a gesture's velocity — the deepest level of "feel."

    Availability & etiquette

    Core Haptics needs a Taptic Engine (iPhone 8+); check CHHapticEngine.capabilitiesForHardware().supportsHaptics. Respect the system: haptics are silenced in Low Power Mode and by the user's settings — never rely on them as the only feedback channel.

    Designing the Feel

    This is the synthesis lesson: take a gesture, a spring, and a haptic and combine them into one interaction that feels like a physical object. Everything in this course and your interaction-math notes comes together here.

    The recipe

    Track the finger 1:1 while it's down. Resist at boundaries with rubber-banding. On release, continue with the finger's velocity into a spring. Confirm discrete moments with a haptic.

    That's it — four ingredients, applied consistently, are what separates "Apple-grade" from "fine."

    A worked example: a dismissable sheet

    @objc func handlePan(_ g: UIPanGestureRecognizer) {
    let t = g.translation(in: view).y
    switch g.state {
    case .changed:
        // track 1:1 downward; rubber-band upward (can't drag above its rest)
        sheet.transform = CGAffineTransform(translationX: 0,
            y: t > 0 ? t : rubberBand(-t, dim: view.bounds.height) * -1)
    case .ended:
        let v = g.velocity(in: view).y
        if t > 120 || v > 800 {                 // past threshold OR flicked down
            haptic.impactOccurred()             // confirm the commit
            dismissWithVelocity(v)              // spring off-screen at that speed
        } else {
            springBack(initialVelocity: v)      // not enough → spring home
        }
    default: break
    }
}

    Notice: the decision uses both displacement and velocity (a fast flick dismisses even a short drag), the haptic fires at the moment of commitment, and both outcomes are springs seeded with the release velocity.

    Principles that scale

    • Interruptible always. The user can grab the sheet mid-dismiss and drag it back. Springs are retargetable; canned animations aren't — this is why you prefer springs (Springs).
    • Velocity is sacred. Never drop the finger's velocity at release; it's the single biggest contributor to "fluid."
    • Haptics mark events, not motion. One crisp tap at the threshold beats a continuous buzz.
    • Tune by feel, then derive. Find the spring response/damping that feels right, then note the numbers — the workflow from the study method.

    What to avoid

    Fixed-duration animations for gesture-driven UI (they fight interruption), haptics on high-frequency events (exhausting), and 1:1 tracking past boundaries (feels broken — always rubber-band). These are the anti-patterns from Designing Fluid Interfaces.
