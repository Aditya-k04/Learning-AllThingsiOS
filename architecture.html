App Architecture
MVC → MVVM, unidirectional data flow, dependency injection, TCA, and modularization.

    App Architecture — Overview

    Architecture is how you arrange code so an app stays understandable and changeable as it grows. There's no single "right" pattern — there are trade-offs. This course covers the spectrum, from the MVC you inherit to the strict unidirectional flows of modern SwiftUI, so you can choose deliberately instead of by default.

    The questions architecture answers

    Where does state live? How does the view get it and signal intent? Where do side effects (network, disk) happen? How do you test logic without the UI? How do pieces stay decoupled so one change doesn't ripple everywhere?

    Every pattern is a different set of answers to those five questions.

    1. 01-mvc-to-mvvm — why Massive-View-Controller happens and how MVVM splits it.
    2. 02-unidirectional-data-flow — state down, events up; the SwiftUI-native mental model.
    3. 03-dependency-injection — decoupling from concrete services so code is testable.
    4. 04-tca — The Composable Architecture: reducers, effects, and when it's worth it.
    5. 05-modularization — splitting into Swift packages for build speed and boundaries.

    A guiding principle

    Keep logic out of views and side effects at the edges. A view should render state and emit intent; it shouldn't know how to talk to the network or the database. That single rule — pushing I/O to the boundary, keeping a testable core — is what most architectures are really enforcing.

    How it connects

    The data-flow patterns here are the application-level version of SwiftUI's dependency graph. Dependency injection is what makes the networking (API client) and persistence layers swappable for the tests in unit testing.

    MVC → MVVM

    Apple's original pattern is MVC — Model, View, Controller. It's simple and it's everywhere in UIKit, but it has a famous failure mode. MVVM is the most common response, and understanding why it exists matters more than the acronym.

    Classic MVC and its failure mode

    In theory: the Model holds data, the View displays it, the Controller mediates. In practice on iOS, UIViewController owns the views and becomes the dumping ground for networking, formatting, navigation, and business logic — the dreaded Massive View Controller. A 2,000-line controller that can't be tested because everything is tangled with UIViews.

    The root problem: the controller mixes presentation logic (how data becomes display values) with view lifecycle and side effects, so none of it can be isolated.

    MVVM: extract a testable ViewModel

    MVVM splits the controller in two:

    • ViewModel — owns presentation state and logic. Knows nothing about UIKit/SwiftUI. Pure, testable.
    • View (the controller/SwiftUI view) — observes the ViewModel and renders it; forwards user intent back.
    @MainActor
final class ProfileViewModel: ObservableObject {
    @Published private(set) var state: State = .loading
    enum State { case loading, loaded(ProfileVM), failed(String) }

    private let api: APIClient            // injected dependency (testable)
    init(api: APIClient) { self.api = api }

    func load(id: Int) async {
        do { state = .loaded(ProfileVM(try await api.user(id))) }
        catch { state = .failed("Couldn’t load profile.") }
    }
}

    The view becomes thin:

    struct ProfileView: View {
    @StateObject var vm: ProfileViewModel
    var body: some View {
        switch vm.state {
        case .loading: ProgressView()
        case .loaded(let p): ProfileContent(p)
        case .failed(let m): ErrorView(m)
        }
    }
    .task { await vm.load(id: id) }
}

    Why this is testable

    Because ProfileViewModel has no UIKit/SwiftUI imports and takes its APIClient by injection, a test creates it with a fake client and asserts the state transitions — no UI, no network, milliseconds to run. That's the whole payoff, and it depends on dependency injection.

    MVVM isn't free

    Pitfalls: ViewModels can themselves bloat (split them), bindings can create retain cycles (use [weak self] in closures, or value-type state), and two-way bindings can get hard to trace. MVVM tames MVC but doesn't enforce a direction of data flow — which is what the next pattern adds.

    Unidirectional Data Flow

    Unidirectional data flow (UDF) adds a rule MVVM lacks: data moves in one direction. State flows down into the view; the view sends actions up; actions produce a new state. No two-way bindings tangling who-changed-what. It's the model SwiftUI itself is built around, and it makes complex screens predictable.

    The loop

            ┌─────────── State ───────────┐
        │                             ▼
   (reducer)                       View renders
        ▲                             │
        └────────── Action ◄──────────┘
                 (user intent)

    State down, events up. The view is a pure function of state; the only way to change state is to send an action through a single place that produces the next state.

    A minimal UDF store

    @MainActor
final class Store<State, Action>: ObservableObject {
    @Published private(set) var state: State
    private let reduce: (inout State, Action) -> Void
    init(_ initial: State, reduce: @escaping (inout State, Action) -> Void) {
        self.state = initial; self.reduce = reduce
    }
    func send(_ action: Action) { reduce(&state, action) }   // the ONLY mutation path
}

    struct CounterState { var count = 0 }
enum CounterAction { case increment, decrement, reset }

let store = Store(CounterState()) { state, action in
    switch action {
    case .increment: state.count += 1
    case .decrement: state.count -= 1
    case .reset:     state.count = 0
    }
}
// View:  Text("\(store.state.count)")  ;  Button("+") { store.send(.increment) }

    Every change is an action through send → easy to log, replay, and test. You assert "given this state, sending this action yields that state" with no UI.

    Why it scales

    • Predictable: there's exactly one way state changes, so bugs are reproducible.
    • Debuggable: log every action to get a complete history of what happened.
    • Testable: the reducer is a pure function — the easiest thing in the world to test.

    This is the same precision as SwiftUI's dependency graph: describe state, and the system re-renders only what depends on what changed.

    Side effects

    Pure reducers can't network. UDF handles effects by having the reducer return effects (async work) that run outside and feed their results back as new actions — the model formalized by TCA (04-tca). Keeping effects at this boundary is the same "I/O at the edges" principle from 00-overview.

    Dependency Injection

    Dependency injection (DI) means a component receives the things it needs from outside, instead of creating them itself. It's the single most important technique for testable, decoupled code — and it's mostly just "pass it in," no framework required.

    The problem with hard-coded dependencies

    final class FeedViewModel {
    func load() async {
        let data = try? await URLSession.shared.data(from: url)   // hard-wired!
        // ...
    }
}

    This ViewModel is its network call. You can't test it without hitting the real server, can't preview it offline, can't swap implementations. It's welded to URLSession.

    Inject an abstraction

    Depend on a protocol, and pass an implementation in:

    protocol FeedLoading { func loadFeed() async throws -> [Post] }

final class FeedViewModel {
    private let loader: FeedLoading
    init(loader: FeedLoading) { self.loader = loader }     // injected
    func load() async { posts = (try? await loader.loadFeed()) ?? [] }
}

struct LiveFeedLoader: FeedLoading { /* real URLSession */ }
struct StubFeedLoader: FeedLoading {                        // for tests/previews
    func loadFeed() async throws -> [Post] { Post.samples }
}

    Now production injects LiveFeedLoader, tests inject StubFeedLoader, and SwiftUI previews inject sample data — same ViewModel, no changes. This is the seam every other course relies on: the API client protocol, the testing fakes in unit testing.

    Forms of injection

    • Initializer injection (preferred): pass dependencies in init. Explicit, compile-checked, no hidden state.
    • Property injection: set after creation. Use when init can't have the value yet.
    • Environment injection (SwiftUI): put a dependency in the Environment so a subtree can read it — Apple's built-in DI:
    struct APIKey: EnvironmentKey { static let defaultValue: APIClient = LiveAPIClient() }
extension EnvironmentValues { var api: APIClient {
    get { self[APIKey.self] } set { self[APIKey.self] = newValue } } }

ContentView().environment(\.api, MockAPIClient())   // swap for previews/tests

    This is the same environment mechanism from environment as a graph.

    Composition root

    Decide where concrete types are chosen: one place near the app's entry point (the "composition root") wires real implementations together, and nothing deep in the app says URLSession.shared or LiveFeedLoader(). Keep the new-ing of concrete dependencies at the top; everything below receives abstractions.

    You rarely need a DI framework

    For most apps, initializer injection + the SwiftUI environment is enough. Heavy DI containers add indirection that obscures the dependency graph. Reach for one only when manual wiring genuinely hurts.

    The Composable Architecture (TCA)

    TCA is a popular open-source library (by Point-Free) that takes unidirectional data flow and makes it rigorous, composable, and testable to an exhaustive degree. It's opinionated and has a learning curve — worth knowing whether or not you adopt it, because it crystallizes the ideas.

    The core types

    State    – a value type describing everything a feature needs
Action   – an enum of every event that can happen (user + system)
Reducer  – (inout State, Action) -> Effect : evolves state, returns side effects
Store    – runtime that holds State and processes Actions
Effect   – async work whose results feed back in as Actions

    @Reducer
struct Counter {
    @ObservableState struct State { var count = 0; var fact: String? }
    enum Action { case incr, factButtonTapped, factResponse(String) }

    var body: some Reducer<State, Action> {
        Reduce { state, action in
            switch action {
            case .incr:
                state.count += 1; return .none
            case .factButtonTapped:
                return .run { [count = state.count] send in           // side EFFECT
                    let fact = try await numberFact(count)
                    await send(.factResponse(fact))                   // feeds back as an action
                }
            case .factResponse(let fact):
                state.fact = fact; return .none
            }
        }
    }
}

    What it buys you

    • Composition: features are reducers you can nest and combine; a parent's state contains children's states, and actions route down. Big apps stay made of small, isolated features.
    • Effects are explicit & controlled: every side effect goes through Effect, and dependencies are injected through a typed dependency system — so tests run with controlled clocks and stubbed services.
    • Exhaustive testing: the TestStore asserts that every state mutation after every action is exactly what you declared — if anything else changes, the test fails. It's the strictest testability on the platform.

    The cost

    TCA adds real overhead: a lot of boilerplate, a strong learning curve, and a whole-app commitment (it works best when everything is TCA). For small apps or teams new to it, plain MVVM + the lightweight UDF store from 02-unidirectional-data-flow gets most of the benefit with far less ceremony.

    When to reach for it

    Large, long-lived apps with complex state and many engineers, where the discipline and exhaustive tests pay back the boilerplate. For a solo side project, it's usually more than you need — but studying it sharpens how you think about state and effects everywhere.

    Modularization

    As an app grows, a single target becomes slow to build and easy to tangle. Modularization — splitting the app into Swift packages with explicit dependencies — restores fast builds, enforced boundaries, and independently testable, previewable features.

    Why split up

    • Build speed: changing one feature recompiles only that module, not the whole app. This is the biggest day-to-day win on a large codebase.
    • Enforced boundaries: module A can only use module B if it explicitly depends on it. Accidental coupling becomes a compile error instead of a habit.
    • Isolation: a feature module can be built, tested, and SwiftUI-previewed on its own, without launching the full app.

    A typical structure

    App                     – thin shell: composition root, app lifecycle, wiring
├── Features/
│   ├── Feed            – one feature = one module (UI + logic)
│   ├── Profile
│   └── Settings
├── Core/
│   ├── Networking      – the API client (Topics-Networking)
│   ├── Persistence     – the store (Topics-Persistence)
│   └── DesignSystem    – shared components, colors, typography
└── Shared/
    └── Models          – domain types used across features

    Features depend on Core and Shared, never on each other directly — cross-feature communication goes through abstractions or the app shell.

    Swift Package Manager

    Local SPM packages are the standard tool. A Package.swift declares the module's products, dependencies, and targets:

    // swift-tools-version: 5.10
import PackageDescription
let package = Package(
    name: "Feed",
    products: [.library(name: "Feed", targets: ["Feed"])],
    dependencies: [.package(path: "../Networking"), .package(path: "../DesignSystem")],
    targets: [
        .target(name: "Feed", dependencies: ["Networking", "DesignSystem"]),
        .testTarget(name: "FeedTests", dependencies: ["Feed"])
    ])

    Add these as local packages in your Xcode project; SPM also manages third-party dependencies (TCA, etc.) in the same file.

    Keep boundaries clean with protocols

    Modules should depend on abstractions, not each other's concretes — the DI principle at module scale. The Feed module depends on a FeedLoading protocol (perhaps in a small Interfaces module), and the app shell injects the concrete Networking implementation. This keeps the dependency graph a tree, not a web.

    Don't over-modularize early

    Forty modules for a five-screen app is its own kind of pain — every change touches many Package.swift files. Start with a single target; split when build times or coupling actually hurt. Modularize in response to real pressure, not speculatively.
